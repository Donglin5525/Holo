# Holo 生活轨迹 AI 洞察升级实施方案

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将 Holo 从"个人数据查询与周期总结助手"升级为"基于真实记录的生活轨迹观察助手"，让财务、习惯、待办与想法数据能形成有证据、有温度、有深度、不过度推断的 AI 洞察。

**Architecture:** 本方案不新建并行 AI 系统，继续基于现有 HoloBackend Prompt 托管、iOS `HoloBackendAIProvider`、`MemoryInsightContextBuilder`、`MemoryInsightService` 和 `CrossModuleCorrelator` 演进。Prompt 负责定义洞察方法、表达边界和输出格式；代码负责提供 Prompt 生效链路、Prompt 版本绑定、生活轨迹上下文、个人基线和跨模块模式。

**Tech Stack:** Holo iOS Swift / SwiftUI / Core Data，HoloBackend Node.js / Hono / JSON Prompt Registry / 管理后台，现有 AI Chat Completions 网关。

## 一、背景与目标

### 1.1 当前能力边界

Holo 当前已经具备以下基础能力：

| 模块 | 已有能力 |
|------|----------|
| 财务 | 收入/支出记录、分类、预算、日消费、分类聚合、上期对比、异常描述 |
| 习惯 | 活跃习惯、打卡记录、完成率、连续打卡、断连检测 |
| 待办 | 创建/完成/更新/删除、逾期、优先级、完成统计、完成趋势 |
| 想法 | 文本记录、近期片段、心情/标签分布、文本内容注入 |
| AI Chat | 意图识别、执行动作、分析查询、流式回答 |
| 记忆洞察 | 日报/周报/月报/年度回顾、结构化 JSON 输出、洞察持久化 |
| 后端 | AI 网关、Prompt 托管、管理后台、日志、模型路由 |

本方案严格基于这些能力设计，不假设 Holo 拥有睡眠、定位、屏幕使用、社交关系、真实工作压力、健康指标等未接入数据。除非用户主动在想法、任务或记录中写出，否则 AI 不得生成这些维度的洞察。

### 1.2 升级目标

升级后的 Holo 要能从用户已有记录中观察：

1. 财务上钱流向哪里，是否偏离个人常态。
2. 习惯上哪些承诺被兑现，哪些习惯断连，是否出现恢复。
3. 待办上任务是否闭环，是否堆积，重要任务是否推进。
4. 想法上用户反复写到什么主题，但不做心理诊断。
5. 跨模块上多个现象是否在同一时间段并发出现，但不做武断因果判断。

最终用户感受到的不是"AI 在写漂亮总结"，而是：

> Holo 不替用户编故事，但能把用户自己留下的数据串成一条温柔、准确、可回看的生活轨迹。

### 1.3 核心原则

| 原则 | 说明 |
|------|------|
| 证据优先 | 每条洞察必须来自输入上下文中的事实 |
| 克制表达 | 不诊断、不审判、不人格评价 |
| 分层洞察 | 区分事实、变化、模式、关联、假设、建议 |
| 弱因果约束 | 跨模块只能说并发、相关、值得留意，不能说导致/证明 |
| 小切口建议 | 建议必须具体、轻量、可执行 |
| 后端优先 | Prompt 正式发布以 HoloBackend 托管版本为准 |

## 二、总体架构

### 2.1 当前链路

```text
后端 Prompt 管理后台
  -> /v1/prompts/:type
  -> HoloBackendPromptService.loadPrompt()
  -> HoloBackendAIProvider
  -> /v1/ai/chat/completions
  -> 上游模型
```

周期洞察链路：

```text
MemoryInsightContextBuilder
  -> MemoryInsightContext JSON
  -> AIProvider.generateMemoryInsight()
  -> memory_insight_generation Prompt
  -> MemoryInsightResponseParser
  -> MemoryInsightRepository 持久化
  -> 记忆长廊展示
```

> ⚔️ **[HIGH] 审查意见 #4 — generateMemoryInsight 不强制 JSON 输出**
>
> `HoloBackendAIProvider.generateMemoryInsight()` 构建 `/v1/ai/chat/completions` 请求时**没有设置 `responseFormat: .jsonObject`**。LLM 完全可能返回非 JSON 内容（尤其是新 Prompt 更长更复杂时），导致 `MemoryInsightResponseParser` 解析失败。
>
> 方案 3.3 节要求"输出严格 JSON"，但代码层面没有保障手段。**建议在 Phase 1 就修复此问题**，而不是等到 Phase 6 测试集才发现。
>
> 涉及文件：`HoloBackendAIProvider.swift` `generateMemoryInsight()` 方法。

### 2.2 升级后链路

```text
HoloBackend Prompt Registry
  -> Prompt 版本/历史/测试/回滚
  -> iOS Prompt 缓存按版本刷新
  -> AI 调用携带 promptType + promptVersion

MemoryInsightContextBuilder
  -> 原有四模块上下文
  -> LifeEvent 行为事件流
  -> DailyLifeSnapshot 每日快照
  -> PersonalBaseline 个人基线
  -> CrossModulePattern 跨模块模式
  -> MemoryInsightContext JSON

memory_insight_generation v5
  -> 事实/变化/模式/关联/假设/建议
  -> 严格 JSON
  -> evidence 绑定
```

> ⚔️ **[HIGH] 审查意见 #8 — 新模型名 CrossModulePattern 与现有 CrossModuleCorrelation 冲突**
>
> 方案定义了 `CrossModulePattern`（含 `patternType`、`summary`、`evidenceDates`、`modulePair`、`confidence`），但现有 `CrossModuleCorrelator.swift` 中已有 `CrossModuleCorrelation` 结构体（含 `modulePair`、`observation`、`signalStrength`、`summary`）。
>
> 这两个类型的职责高度重叠。方案没有说明：是替换 `CrossModuleCorrelation`？还是并存？如果并存，AI Prompt 应优先使用哪个？如果没有迁移策略，现有 6 条规则产出 `CrossModuleCorrelation`，新 5 条规则产出 `CrossModulePattern`，下游序列化/注入会混乱。

### 2.3 Prompt 与代码职责边界

| 类型 | 负责内容 | 不负责内容 |
|------|----------|------------|
| Prompt 改动 | AI 身份、洞察框架、语气、输出格式、禁止项、证据要求 | Prompt 是否实时生效、数据是否足够、缓存是否刷新 |

## 三、Prompt 改动范围

所有 Prompt 正式发布均通过 HoloBackend 管理后台保存到后端托管版本。真机本地 `PromptManager` 保留为 fallback，不作为正式发布源。

### 3.1 `system_prompt`

**定位：** Holo 的基础身份与全局行为边界。

**改动目标：**

- 从"个人数据管理助手"升级为"生活轨迹观察助手"。
- 明确不评价用户，只观察真实记录。
- 明确区分事实、观察、假设、建议。
- 明确跨模块只能表达并发现象。
- 明确不编造、不诊断、不审判。

**建议 Prompt：**

```text
你是 Holo，用户的个人生活轨迹观察助手。你的职责不是评价用户，而是基于真实记录，帮助用户看见财务、习惯、待办、想法之间的节奏、变化、偏离和恢复。

今天是 {{todayDate}}。

核心原则：
- 只基于上下文中的真实数据回答，禁止编造数字、日期、分类、任务或习惯。
- 区分事实、观察、假设和建议。没有证据时只能说"不足以判断"。
- 不做心理诊断，不评价人格，不使用羞辱、审判或命令式表达。
- 跨模块关系只能表达为"并发现象"或"值得留意的关联"，不能说"导致、证明、说明一定是因为"。
- 对金额、次数、完成率、日期保持精确。
- 如果用户要执行记账、创建任务、打卡等操作，而当前链路无法确认执行成功，不要假装已完成。
- 回复使用中文，克制、温和、具体。少说空泛鼓励，多给一个可执行的小切口。
```

### 3.2 `analysis_prompt`

**定位：** 用户主动发起数据分析查询时使用，例如"我这周怎么样""帮我分析一下最近状态"。

**改动目标：**

- 从普通分析报告升级为五层分析。
- 支持跨财务、习惯、待办、想法的综合观察。
- 明确数据不足时不能硬凑。

**建议 Prompt：**

```text
你是 Holo 的个人数据分析助手。你会收到结构化 JSON 上下文和用户问题，请基于数据生成分析。

分析顺序：
1. 事实：先列出上下文中明确存在的数据。
2. 变化：对比本期与上期、工作日与周末、不同分类或不同习惯的差异。
3. 模式：识别重复出现的时间、分类、任务节奏或习惯断连。
4. 关联：如果多个模块同时变化，只能描述并发关系，不做因果断言。
5. 建议：给出一个具体、轻量、可执行的下一步。

规则：
- 禁止编造数据。
- 禁止重新估算上下文没有给出的数字。
- 明确区分"数据支持的观察"和"个人建议"。
- 如果样本不足，请直接说明样本不足。
- 语气要有温度但不软化事实；锐利来自证据，不来自批评。
- 不输出 JSON，只输出 Markdown 文本。
- 控制在 300-500 字。
```

### 3.3 `memory_insight_generation`

**定位：** 记忆长廊日报/周报/月报的核心生成 Prompt。

**改动目标：**

- 从周期总结升级为生活轨迹回放。
- 每张卡片必须有 evidence。
- 不改变现有 JSON Schema，避免前端解析崩溃。
- 在现有 `title/body/evidence/suggestedQuestion/anomalySeverity` 内体现洞察层次。

**建议 Prompt：**

```text
你是 Holo 的生活轨迹洞察生成器。你的任务是基于用户一个周期内的结构化数据，生成可复看的记忆回放。

你要观察的不是单个模块，而是用户这一周期的生活轨迹：
- 财务：钱流向哪里，是否偏离个人常态，是否出现异常集中。
- 习惯：哪些承诺被兑现，哪些习惯断连，是否出现恢复。
- 待办：任务是否闭环，是否堆积，重要任务有没有推进。
- 想法：用户反复写到什么主题，情绪基调如何，但不得心理诊断。
- 跨模块：多个模块是否在同一时间段出现并发变化。

洞察层级：
- fact：明确事实。
- change：本期相对上期的变化。
- pattern：重复出现的模式。
- correlation：跨模块并发现象。
- hypothesis：低置信度假设，必须用"可能/值得留意"。
- suggestion：轻量建议。

硬性规则：
1. 只使用输入数据中的事实。
2. 不编造金额、日期、次数、完成率。
3. 不做人格判断或心理诊断。
4. 跨模块不得使用"导致、因为、证明、说明"。
5. 每张卡片必须有 evidence；没有 evidence 就不要生成。
6. 数据稀疏时少输出，不要硬凑。
7. 优先识别"偏离常态、连续变化、恢复迹象、任务堆积、预算异常、习惯断连"。

语言风格：
- 克制、温暖、具体。
- 不说教，不审判。
- 标题要有画面感，但正文必须落回证据。
- 建议只给一个小动作，不要泛泛规划人生。

输出严格 JSON，不要在 JSON 外添加任何文字。
```

**Schema 兼容要求：**

继续沿用现有输出：

```json
{
  "title": "string",
  "summary": "string",
  "cards": [
    {
      "id": "string",
      "type": "habit | finance | task | thought | milestone | cross_domain | overview | anomaly",
      "title": "string",
      "body": "string",
      "evidence": [],
      "suggestedQuestion": "string 或 null",
      "anomalySeverity": "warning | critical | info 或 null"
    }
  ],
  "suggestedQuestions": []
}
```

### 3.4 `annual_review`

**定位：** 年度回顾。

**改动目标：**

- 从月度摘要拼接升级为年度生活轨迹观察。
- 识别反复模式、转折点和恢复能力。
- 不逐月流水账。

**建议 Prompt：**

```text
你是 Holo 的年度生活轨迹回顾助手。你会收到用户一年内的月度洞察摘要和年度汇总数据。

你的目标不是逐月流水账，而是找出这一年里反复出现的模式、转折点和恢复能力。

请重点观察：
- 财务：全年支出结构、异常月份、预算压力与改善。
- 习惯：哪些习惯真正稳定，哪些反复中断，何时恢复。
- 待办：长期任务推进节奏，高优先级事项是否被完成。
- 想法：反复出现的主题和表达变化。
- 跨模块：多个模块在同一月份的并发现象，但不得做因果断言。

规则：
- 只基于输入数据。
- 不编造缺失月份。
- 不做心理诊断。
- 不评价人格。
- 识别转折点时必须引用月份或证据。
- 同时写出"值得记住的进展"和"下一年最小改进方向"。

输出严格 JSON，不要在 JSON 外添加任何文字。
```

### 3.5 `intent_recognition`

**定位：** 意图识别。该 Prompt 与执行链路强相关，风险高，本期只做小补丁。

**新增规则：**

```text
额外规则：
- "复盘、总结、看看这周/本月状态、我最近怎么样"优先识别为 generate_memory_insight 或 query_analysis。
- 如果用户想要跨财务、习惯、待办一起分析，analysisDomain 使用 crossModule。
- 如果一句话同时包含执行动作和分析查询，返回 clarification，不要混合执行。
```

### 3.6 `clarification`

**定位：** 意图不明确时追问。

**建议 Prompt：**

```text
当用户意图不明确时，你只追问一个最关键的问题。

规则：
- 不连续追问多个问题。
- 优先补足执行所需的最小字段。
- 语气自然，不要像表单。
- 如果涉及记账，优先确认金额或分类。
- 如果涉及任务，优先确认任务标题或时间。
- 如果涉及习惯，优先确认习惯名称。
- 如果涉及分析，优先确认时间范围或分析维度。
```

### 3.7 `response_template`

**定位：** 操作执行后的确认回复。

**建议 Prompt：**

```text
根据操作结果生成确认回复。

规则：
- 如果系统已经返回执行成功，才可以说"已记录/已创建/已完成"。
- 如果执行失败，说明失败原因，并给出一个下一步。
- 不夸张，不撒花式鼓励。
- 多动作结果要逐条说明。
- 金额、任务名、习惯名必须和执行结果一致。
```

**定位：** 轻量结构化字段提取。

**建议 Prompt：**

```text
从用户输入中提取结构化字段。只提取，不解释，不推理过度。

可提取字段：
- amount
- note
- type: expense | income
- title
- date
- mood
- weight
- habitName
- categoryCandidate
- priority
- dueDate
- tags
- description

规则：
- 不确定的字段不要填。
- 金额只保留数字。
- 日期尽量转为 yyyy-MM-dd。
- 只输出 JSON。
```

> 建议：
> - 方案 A：使用更准确的估算公式，如对中文文本用 `characterCount * 1.5`，英文用 `wordCount * 1.3`。
> - 方案 B：实现渐进式截断（当前只截一次就停），而非固定优先级的一次性截断。
> - 方案 C：将 token 预算逻辑抽为独立方法，方便后续替换为真正的 BPE tokenizer。

> 现有规则清单：
>
> | # | 方法名 | 检测内容 |
> |---|--------|----------|
> | 1 | `detectHabitFinanceCorrelation` | 习惯完成率降 + 餐饮占比升 |
> | 2 | `detectTaskFinanceCorrelation` | 待办逾期率高 + 支出升 |
> | 3 | `detectThoughtHabitCorrelation` | 负面情绪比高 + 习惯完成率降 |
> | 4 | `detectTaskHabitCorrelation` | 待办与习惯完成率同方向变化 |
> | 5 | `detectEmotionSpendingCorrelation` | 负面情绪 + 支出升 |
> | 6 | `detectWeekdayWeekendPattern` | 工作日/周末支出差异 |
>
> 方案提出的 5 条新规则与现有规则的重叠分析：
>
> | 方案新规则 | 重叠的现有规则 | 重叠程度 |
> |-----------|--------------|---------|
> | 1. 任务堆积与消费上升并发 | `detectTaskFinanceCorrelation` | **高度重叠**：触发条件几乎相同（逾期率>20% + 支出升>15%） |
> | 2. 习惯断连与餐饮集中 | `detectHabitFinanceCorrelation` | **高度重叠**：都检测习惯+餐饮关系 |
> | 3. 重要任务完成与习惯恢复 | 无直接对应 | 新增能力 |
> | 4. 周末补偿型消费 | `detectWeekdayWeekendPattern` | **中度重叠**：方案增加了工作日任务/习惯完成率低的条件 |
> | 5. 恢复迹象 | 无直接对应 | 新增能力 |
>
> 如果 5 条新规则与 6 条旧规则并行运行，对同一并发现象可能输出 2 条冗余洞察（如"任务堆积+消费上升"同时被旧规则 #2 和新规则 #1 检测到）。
>
> 建议：
> - 明确标注哪些新规则是**替换**旧规则（如 #1 替换 `detectTaskFinanceCorrelation`）
> - 哪些是**全新**（如 #3 恢复迹象、#5 恢复迹象）
> - 添加去重逻辑：如果同一天/同一模块对已被某条规则覆盖，后续规则应跳过

**新增规则：**

1. 任务堆积与消费上升并发

触发条件：

- 当前周期逾期率 > 20%
- 当前周期支出较上期上升 > 15%

> ⚔️ 与现有 `detectTaskFinanceCorrelation` 触发条件完全相同（逾期率 > 20%，支出升 > 15%）。是替换还是并行？

输出限制：

```text
任务逾期率偏高，同时支出较上期上升。
```

禁止：

```text
因为任务压力导致消费上升。
```

2. 习惯断连与晚间/餐饮消费集中并发

触发条件：

- 某正向习惯连续 3 天未完成
- 同期餐饮/外卖/夜宵支出高于个人基线或本周期日均

> ⚔️ 与现有 `detectHabitFinanceCorrelation` 检测内容重叠（都检测习惯+餐饮），但触发条件不同：旧规则看全期完成率降幅和餐饮占比升幅，新规则看连续 3 天断连和绝对金额对比。可以作为增强版替换旧规则，但需要明确标注。

3. 重要任务完成与习惯恢复

触发条件：

- 高优先级任务完成
- 后续 1-2 天习惯完成率回升

输出方向：

```text
高优任务推进后，习惯记录也出现恢复。
```

4. 周末补偿型消费候选

触发条件：

- 周末日均支出显著高于工作日
- 工作日任务完成率或习惯完成率偏低

> ⚔️ 现有 `detectWeekdayWeekendPattern` 只检测工作日/周末支出差异，不检查工作日任务/习惯完成率。新规则增加了跨模块条件，是**有价值的增强**。建议作为替换或升级。

必须表达为候选：

```text
这是一个值得留意的周末支出集中现象。
```

5. 恢复迹象

触发条件：

- 前半周期习惯完成率低或任务逾期增加
- 后半周期任务完成/习惯完成回升

这是最重要的温度来源，必须优先于单纯批评异常。

**验收标准：**

- 每条规则有最低样本量门槛。
- 每条输出带 `confidence`。
- 每条输出带证据日期。
- 不输出因果词。

## 六、用户可见体验

### 6.1 周报卡片示例

基于真实数据时，Holo 可以输出类似：

```text
这周不是失控，而是节奏被打断了两次。
周二和周三的任务逾期数量上升，同时这两天餐饮支出也更集中。数据只能说明它们在同一时间段出现，不能判断原因，但这两天值得单独回看。
```

前提数据：

- 周二/周三逾期任务数存在。
- 周二/周三餐饮支出存在。
- 周期内其他日期可对比。

### 6.2 习惯恢复示例

```text
你没有一直掉线，后半周接住了自己。
这个习惯本周前半段有中断，但周五和周六重新完成了打卡。相比单纯追求满分，这个恢复动作更值得保留。
```

前提数据：

- 某习惯前半周未完成。
- 周五/周六完成。
- 该习惯为活跃习惯。

### 6.3 待办推进示例

```text
表面上还有堆积，但关键的事并没有停。
本周仍有逾期任务，不过你完成了几个高优先级事项。与其只看"还有多少没做完"，更值得看见的是：重要任务已经有推进。
```

前提数据：

- 活跃/逾期任务存在。
- 高优先级完成任务存在。

### 6.4 财务小切口示例

```text
这周的变化主要不是全面变贵，而是集中在餐饮。
本周总支出高于上周，其中餐饮是最主要的增量来源。下周如果想轻一点，不需要全面收紧，先看餐饮里的高频项目就够了。
```

前提数据：

- 本周/上周支出对比存在。
- 分类聚合存在。
- 餐饮增量明显。

### 6.5 用户主动问"我最近怎么样"

```text
从 Holo 记录来看，这周有三个信号：

第一，财务上支出比上周更集中，主要集中在餐饮。

```

前提数据：

- Chat 分析上下文能拿到周期数据。
- 或用户触发的是周期洞察，不是纯今日上下文。

## 七、风险与约束

### 7.1 Prompt 变好但数据不足

风险：

- Prompt 想表达"生活轨迹"，但 context 只有聚合数据。

应对：

- Phase 1 只承诺表达升级。
- 深度洞察必须等 Phase 4 的事件流和基线。

### 7.2 跨模块误判因果

风险：

- AI 把"同时发生"说成"因为 A 导致 B"。

应对：

- Prompt 禁止因果词。
- 规则引擎输出 `correlation` 而不是 `cause`。
- 测试集检查禁止词。

### 7.3 Prompt 改动不生效

风险：

- iOS 内存缓存导致真机仍用旧 Prompt。

应对：

- Phase 2 修缓存刷新。
- 短期测试时要求重启 App。

### 7.4 旧洞察缓存挡住新版效果

风险：

- 数据 hash 没变时，旧洞察直接命中。

应对：

- Phase 3 将 promptVersion 纳入缓存条件。

> ⚔️ **[CRITICAL] 审查意见 #2+#3 补充 — 本节风险分析不完整**
>
> 除了 promptVersion 和 snapshotHash 的问题外，还有两个未识别的风险：
>
> **风险 7.4a：fetchInsight 不检查 sourceSnapshotHash**
> - 即使数据已变化（hash 不同），`fetchInsight` 仍会返回旧的 ready 记录（因为它只查 periodType + periodStart + status）。
> - `sourceSnapshotHash` 仅在 `markStaleIfNeeded` 中使用，需要外部主动调用才会触发 stale 标记。
> - 如果 `markStaleIfNeeded` 没有被正确调用，数据变化后旧洞察仍然会被命中。
>
> **风险 7.4b：Jaccard 相似度去重绕过所有缓存条件**
> - 即使 Phase 3 修了 fetchInsight，`checkSimilarityWithRecent` 仍可能返回旧洞察。
> - 此方法不检查 promptVersion、不检查 sourceSnapshotHash，只检查 token 集合相似度。
> - 需要 Phase 3 同时修改此方法（见审查意见 #3）。
>
> 建议将这两个风险加入本节。

### 7.5 过度感性导致不可信

风险：

- AI 输出漂亮但空泛的"懂你"。

应对：

- 每张卡必须 evidence。
- 数据稀疏时少输出。
- 建议必须是小动作。

> ⚔️ **[LOW] 审查意见 #16 — generateMemoryInsight 的 30 秒超时未讨论**
>
> `MemoryInsightService` 使用 `withThrowingTaskGroup` 竞速实现 30 秒硬超时。Phase 4 增加上下文后，AI 生成时间可能变长。如果 30 秒不够，洞察生成会静默失败（`saveFailed`）且用户无感知。
>
> 建议在"风险与约束"中增加：
> - 风险 7.6：Phase 4 后上下文增大导致生成超时
> - 应对：评估是否需要将超时从 30 秒调整为 45-60 秒，或增加超时重试

### 一、审批结论

本方案的产品方向成立：将 Holo 从“数据总结助手”升级为“生活轨迹观察助手”是合理的，且与现有财务、习惯、待办、想法、记忆洞察、HoloBackend Prompt 管理能力一致。

2. GLM/对抗审查批注。
3. 仍未消化进正文的修正建议。

因此，**当前版本不适合直接交给开发执行**。如果直接实施，会有较高概率重复建设已有后端能力，或在洞察缓存、Prompt 版本、跨模块模型上留下新的状态分裂。

建议下一步不是继续在本文档追加补丁，而是生成一份干净的 **V2 修订版**，将已确认成立的审查意见吸收进正文，并删除已过时或与代码现状冲突的设计。

### 二、必须采纳的修订项

#### 1. 重写 4.1 后端 Prompt 发布机制

原方案中关于 `managedPrompts.json` 增加元数据、新增 `managedPromptHistory.json`、重新实现历史和回滚的设计必须删除。

当前代码已具备：

- SQLite `prompt_versions` 表。
- Prompt 版本递增。
- 历史版本查询。
- 行级 diff。
- 非破坏式回滚。
- 管理后台历史页和 diff 查看器。

V2 中后端新增范围应收缩为：

1. 新增 `/v1/prompts/meta` 批量 metadata API。
2. 如确实需要变更说明，则新增 SQLite migration：`prompt_versions.change_note`。
3. 管理后台编辑页增加 `changeNote` 输入框。
4. 管理后台增加 Prompt 测试区。

不得再新增第二套 JSON 历史文件，避免状态源分裂。

#### 2. Phase 1 必须加入 JSON 输出强制约束

`memory_insight_generation` 要求严格 JSON，但当前 `HoloBackendAIProvider.generateMemoryInsight()` 没有传 `responseFormat: .jsonObject`。

V2 的 Phase 1 必须新增代码任务：

```swift
let request = buildRequest(
    purpose: .insight,
    messages: messages,
    responseFormat: .jsonObject
)
```

否则 Prompt 越复杂，`MemoryInsightResponseParser` 解析失败概率越高。

#### 3. 洞察缓存修复必须同时覆盖三条路径

原方案只说“缓存命中条件加入 promptVersion”不够。

V2 必须同时处理：

1. `fetchAvailable` / `fetchInsight` 当前只按 `periodType + periodStart + status` 查询的问题。
2. `MemoryInsightService.generateInsight()` 中 `sourceSnapshotHash` 只在取出后比较的问题。
3. `checkSimilarityWithRecent()` Jaccard 相似度去重不检查 `promptVersion` 的问题。

建议 V2 明确规则：

- 非 `forceRefresh` 命中缓存时，必须同时满足：
  - periodType 相同
  - periodStart 相同
  - status 为 ready
  - sourceSnapshotHash 相同
  - promptVersion 相同或符合兼容策略
- Jaccard 去重只允许比较相同 `promptVersion` 的洞察；Prompt 版本变化时跳过去重。

#### 4. 旧洞察 `promptVersion = 0` 需要迁移策略

现有 Core Data 中 `promptVersion` 默认值为 `0`。V2 必须定义旧记录策略。

建议采用保守策略：

- `promptVersion = 0` 视为“未知版本旧洞察”。
- 旧洞察仍可展示。
- 当用户主动重新生成或 Prompt 版本升级后，不再用 `promptVersion = 0` 记录阻挡新生成。
- 不做自动批量删除，避免破坏用户已生成的记忆内容。

#### 5. 不新建 `CrossModulePattern` 并行模型

原方案提出的 `CrossModulePattern` 与现有 `CrossModuleCorrelation` 职责重叠。

V2 应改为扩展现有 `CrossModuleCorrelation`：

```swift
struct CrossModuleCorrelation {
    let modulePair: [AnalysisDomain]
    let observation: String
    let signalStrength: Double
    let summary: String
    let patternType: String?
    let evidenceDates: [String]
}
```

这样可以复用现有 `CrossModuleCorrelator.detect()` 和现有上下文结构，避免新旧模型并存造成 Prompt 注入混乱。

#### 6. 跨模块规则要标注“替换/增强/新增”

V2 必须逐条说明新规则与现有 6 条规则的关系：

| 规则 | 处理方式 |
|------|----------|
| 任务堆积 + 消费上升 | 替换或增强 `detectTaskFinanceCorrelation`，不得并行重复输出 |
| 习惯断连 + 餐饮集中 | 增强 `detectHabitFinanceCorrelation`，需去重 |
| 重要任务完成 + 习惯恢复 | 新增 |
| 周末支出集中 + 工作日低完成 | 增强 `detectWeekdayWeekendPattern` |
| 恢复迹象 | 新增，且应优先展示 |

### 三、建议采纳的修订项

#### 1. `LifeEvent.value` 不使用 `Double`

财务金额当前大量使用 `Decimal`。V2 中 `LifeEvent.value` 不应使用 `Double` 表示金额。

建议二选一：

1. `valueText: String?`，用于传给 AI，例如 `"¥35.20"`。
2. 按事件类型拆分结构化字段，例如 `amount: Decimal?`、`count: Int?`、`rate: Double?`。

如果只是给 AI 看，`valueText` 更简单、风险更低。

#### 2. 删除或定义 `intensity`

`LifeEvent.intensity` 目前没有语义定义。V2 需要二选一：

1. 删除该字段，避免模型误用。
2. 明确定义为 `0...1` 的显著度分数，并给出不同事件类型的计算公式。

建议 V2 第一版删除 `intensity`，等规则引擎成熟后再加。

#### 3. 定义 PersonalBaseline 门槛

V2 应明确：

- 基线期不与观察期重叠。
- 默认取观察期之前 4 周。
- 至少需要 2 个有效周才生成基线。
- 空周跳过，不做插值。
- 数据不足时不输出“偏离常态”类洞察。

#### 4. Token budget 需要具体策略

V2 不应只写“更新 token budget 裁剪策略”，而要写清楚：

- 周报、月报、日报、年度回顾分别的预算。
- 新增 `LifeEvent`、`DailyLifeSnapshot`、`PersonalBaseline` 的裁剪顺序。
- 年度 `buildAnnualContext()` 是否也调用预算裁剪。
- 是否修正当前 `encoded.count / 2` 的估算方式。

建议采用渐进式裁剪：

1. 裁剪想法原文。
2. 裁剪普通 LifeEvent。
3. 保留 anomaly / highPriority task / habit recovery / budget overrun。
4. 重新估算，直到低于预算。

#### 5. Prompt v1 与上下文增强后需要二次校准

Phase 1 先改 Prompt，Phase 4 再改上下文结构，这个顺序可以接受，但 V2 必须新增：

- Phase 4 完成后，重新校准 `memory_insight_generation`。
- 用新上下文 fixtures 重跑 Prompt 测试。
- 不允许直接沿用 Phase 1 Prompt 作为最终版本。

### 四、可延后处理项

以下问题存在，但不阻塞 V2 生成：

1. 禁止词检测可以先做子串匹配，后续再引入更复杂的语义检测。
2. 30 秒超时可以先作为风险记录，等 Phase 4 上下文增大后再实测是否调到 45-60 秒。

4. `changeNote` 如果当前开发节奏紧，可以延后，不影响核心 Prompt 版本能力。

### 五、需要 GLM 重点复核的问题

请 GLM 重点复核以下判断是否准确：

1. 当前 HoloBackend 是否已经完全以 SQLite `prompt_versions` 作为 Prompt 主存储，`managedPrompts.json` 是否仅作为兼容降级。
2. `fetchAvailable` 当前是否确实只按 `periodType + periodStart + status` 查询，不包含 `sourceSnapshotHash` 和 `promptVersion`。
3. `checkSimilarityWithRecent()` 是否会在 Prompt 版本变化后仍复用旧洞察。

6. `LifeEvent.value` 是否应避免 `Double`，改用 `Decimal` 或 `String`。
