# Holo 记忆管理类型化重构方案

Date: 2026-06-06
Status: Revised after GLM adversarial review R2
Scope: 长期记忆候选生成、晋升规则、记忆管理页展示、HoloAI 召回摘要

## 0. 审查后修订摘要

GLM 第一轮对抗审查指出一个前置事实：当前长期记忆虽然已经可以生成、确认和保存，但普通 HoloAI 对话链路尚未实际注入长期记忆摘要。`HoloMemorySummaryProvider` 已实现，`AIUserContextMessageBuilder` 也有注入入口，但 `UserContextBuilder.buildContext()` 返回的 `memorySummary` 仍是 `nil`。因此，本方案的实施顺序必须调整：先证明 HoloAI 能使用已有长期记忆，再推进语义类型化。

R1 后采纳以下修订：

1. 新增 `Phase 0: 召回链路接通与验证`，作为所有类型化工作的前置条件。
2. 明确 `statMilestone` 与情景记忆不是同一层：`statMilestone` 是展示型长期成就素材，情景记忆仍负责近期、可过期、可观察的上下文。
3. 明确 `MemoryCandidateSemanticMapper` 不新增 LLM 调用；采用“洞察生成 prompt 一次输出候选语义字段 + 本地 Mapper 校验/转换”的方案。
4. 将 prompt 修改纳入 iOS 与 HoloBackend 双端同步，并明确后端部署要求。
5. 将 `expiresAt` 从“字段设计”升级为“必须有查询过滤和清理消费端”的实施项。
6. 将 `useScopes` 和 `prohibitedInferences` 从死字段升级为召回筛选与 prompt 注入规则。
7. 增加 feature flag、旧数据 fallback、自动化测试和并发安全要求。

R2 后继续采纳以下修订：

1. 新增字段兼容策略改为 Optional 或自定义 decoder，避免旧 JSON 因缺少 `useScopes` / `prohibitedInferences` 解码失败。
2. 删除旧数据自动语义分类，不再把旧 `.recurringPattern` 自动映射为 `stablePattern`。旧数据统一保留 `semanticType = nil`，并在 UI 标记为旧格式。
3. 降低 prompt 输出复杂度：LLM 首版只输出 `semanticType / displaySummary / aiUseSummary` 三个核心候选字段，其余字段由 Mapper 本地补默认值。
4. 明确 `confidence` 和 `sensitivity` 的合并规则：confidence 以 LLM 为主、缺失时 fallback；sensitivity 只允许本地安全规则升级，不允许降级。
5. 明确 `AIUserContextPurpose.chat` 到 `useScopes` 的映射；`intentRecognition` 不注入记忆。
6. Phase 0 增加开发者测试记忆写入路径，不依赖完整“洞察生成 -> 候选提取 -> 用户确认”链路。
7. `crossDomain` 默认不晋升，除非能拆成单一语义。

## 1. 背景

当前“记忆管理”页的问题不是单纯文案不够好，而是记忆候选仍在沿用洞察卡片的表达方式：标题像系统聚类标签，摘要像对用户的轻建议，类型也更接近技术来源，而不是“这条记忆以后如何帮助 HoloAI 理解用户”。

截图里的例子能代表这个问题：

- “体重习惯连续中断”用户很难理解：是体重目标终止、训练习惯中断，还是只是最近没有记录。
- “任务全部闭环”是内部效率话术，不是用户自然语言。
- “任务清零，支出偏高”把两个不同语义混在一起，既不利于用户审核，也不利于 HoloAI 未来正确召回。

因此，本方案不把目标定义为“把标题写得更口语化”，而是把长期记忆拆成可被 AI 使用的语义类型，并让页面摘要、AI 摘要、洞察建议分层。

## 2. 设计目标

1. 让用户在记忆管理页能快速判断：Holo 到底记住了什么，是否准确，是否应该保留。
2. 让 HoloAI 在后续对话、Daily Sense、周期洞察、目标分析中知道：这条记忆适合何时召回，不能如何误用。
3. 减少冷数据、普通统计、系统标签污染长期记忆。
4. 接通并验证现有长期记忆召回链路，不重建一套并行系统。
5. 保持情景记忆和长期记忆边界清晰，避免 UI 与召回语义混杂。

非目标：

- 不把记忆管理页做成洞察详情页。
- 不在记忆卡片里输出行动建议。
- 不把所有洞察卡片都晋升为长期记忆。
- 不在首版直接做复杂的手动分类编辑器。

## 3. 核心原则

### 3.1 记忆类型服务 AI 使用，而不是服务 UI 好看

类型不是为了给卡片换图标，而是决定：

- 这条记忆是否需要用户确认。
- 是否能静默写入。
- 过期时间应该多长。
- 哪些 HoloAI 场景可以召回。
- 召回时需要避免哪些推断。

### 3.2 展示摘要、AI 摘要、用户建议必须分层

一条长期记忆至少要拆成两层摘要：

| 字段 | 读者 | 作用 | 示例 |
| --- | --- | --- | --- |
| `displaySummary` | 用户 | 说明 Holo 记住的事实，便于审核 | 近 14 天内，体重、平板支撑、俯卧撑等身体相关记录多次中断。 |
| `aiUseSummary` | HoloAI | 说明未来如何使用，以及不能如何误判 | 用户近期身体状态追踪不稳定。健康或习惯洞察可参考此上下文，但需要确认用户当前是否仍关注这些目标。 |

行动建议只允许出现在对话、洞察、提醒中，不进入记忆管理页摘要。

### 3.3 一条记忆只表达一个语义

“任务清零，支出偏高”应拆成：

- 任务收尾节奏稳定。
- 本周支出被大额固定开销拉高。

长期记忆需要可审核、可召回、可删除。混合语义会让用户不知道删除的是哪部分，也会让 HoloAI 召回时误带无关上下文。

## 4. 记忆类型

首版采用 5 类语义类型。

### 4.1 阶段变化 `phaseShift`

定义：用户跨过了一个阶段，或者某个长期状态出现了可被证据支撑的台阶变化。

典型来源：

- 财务：存款、负债、预算控制进入新阶段。
- 健康：体重进入新稳定区间，运动能力进入新台阶。
- 任务：从经常拖延变成稳定收尾。
- 想法：从迷茫转向明确方向。

页面示例：

- 标题：体重进入新的稳定区间
- 页面摘要：近 30 天体重稳定低于上一阶段区间，相关记录有连续证据。
- AI 摘要：用户身体状态出现阶段性变化。健康洞察可把它作为长期进展背景，但仍需结合最近记录判断是否稳定。

晋升建议：

- 默认进入待确认。
- 涉及健康、财务、职业等高影响领域时必须确认。
- 证据不足时继续观察，不进入候选。

### 4.2 稳定习惯 `stablePattern`

定义：用户长期重复出现、对个性化理解有价值的行为倾向。

典型来源：

- 每周固定整理任务。
- 晚上更容易记录想法或身体数据。
- 周末支出更集中。
- 某类提醒在特定时段更容易被完成。

页面示例：

- 标题：任务整理形成固定节奏
- 页面摘要：最近数周内，任务创建和完成集中出现在固定时间段。
- AI 摘要：用户可能更适合在该时间段处理任务规划。后续提醒和目标拆解可参考这个节奏，不要把它表述为强制偏好。

晋升建议：

- 多周期、多证据、低敏感时可静默写入。
- 只有单周期证据时继续观察。
- 涉及健康、财务压力、情绪状态时要求确认。

### 4.3 偏离提醒 `driftSignal`

定义：用户近期偏离了自己曾经在意、持续追踪或明确设定的目标/节奏。

典型来源：

- 身体记录最近不连续。
- 戒烟目标出现反复。
- 任务完成节奏变慢。
- 睡眠、运动、预算控制明显偏离。

页面示例：

- 标题：身体记录最近不连续
- 页面摘要：近 14 天内，体重、平板支撑、俯卧撑等身体相关记录多次中断。
- AI 摘要：用户近期身体状态追踪不稳定。健康或习惯洞察可参考此上下文，但需要确认用户当前是否仍关注这些目标，避免直接归因为懒惰、放弃或失败。

晋升建议：

- 默认进入待确认，不静默写入。
- 默认设置 `expiresAt`，建议 14 到 30 天。
- 如果后续恢复，应该更新或归档，而不是长期保留为固定画像。

### 4.4 人生节点 `lifeEvent`

定义：来自想法、对话、任务、财务或档案更新中的重要生活事件。它不一定是结构化数据，但对长期理解非常重要。

典型来源：

- 升职加薪。
- 准备换工作。
- 搬家。
- 开始一段新计划。
- 明确一个长期目标。

页面示例：

- 标题：你记录了一次职业进展
- 页面摘要：想法记录中提到近期获得新的职业进展，并与收入变化有关。
- AI 摘要：用户近期有职业正向变化。后续职业、财务、目标规划相关对话可参考，但不要主动暴露敏感细节，除非用户话题相关。

晋升建议：

- 默认要求确认。
- 可长期保留。
- 敏感人生事件必须确认，不允许静默写入。

### 4.5 轻量统计收藏 `statMilestone`

定义：有纪念意义但不应强影响 HoloAI 判断的累计节点。

典型来源：

- 完成 50 个任务。
- 消费记录达到 100 笔。
- 连续记录 30 天。
- 某类事项累计达到一个数量。

页面示例：

- 标题：已完成 50 个任务
- 页面摘要：任务历史中累计完成数量达到新的记录节点。
- AI 摘要：这是轻量成就节点，仅在用户回顾历史或成就时召回，不用于推断用户当前效率或性格。

晋升建议：

- 不进入主要长期记忆列表，进入轻量记录。
- 默认不参与 HoloAI 核心上下文召回。
- 可用于年度回顾、记忆长廊、成就展示。

### 4.6 `statMilestone` 与情景记忆的边界

现有代码已经有两套记忆：

| 体系 | 模型 | 当前 UI | 主要用途 |
| --- | --- | --- | --- |
| 情景记忆 | `HoloEpisodicMemory` | 最近记住 | 近期观察、可过期上下文、记忆观察器自我去重 |
| 长期记忆 | `HoloLongTermMemory` | 待确认 / 已记住 | 经用户确认或低风险静默接受的长期理解 |

`statMilestone` 不替代情景记忆，也不应该让 UI 变成四个主 Section。首版建议：

- 情景记忆继续显示为“最近记住”，保留过期、隐藏、删除能力。
- `statMilestone` 仍存储在长期记忆模型中，但 `useScopes` 默认为 `[.displayOnly, .retrospective]`。
- UI 上可在“长期记忆”里用筛选或子分组显示“轻量记录”，不要新增与“最近记住”并列的第四个主 Section。
- HoloAI 普通对话默认不召回 `statMilestone`；只有年度回顾、记忆长廊、成就展示等回顾场景才使用。

## 5. 数据模型调整

现有模型：

```swift
enum HoloLongTermMemoryType: String, Codable, Equatable {
    case explicitUserPreference
    case stableFeedbackPreference
    case recurringPattern
    case longTermGoal
    case profileBackedFact
}

struct HoloLongTermMemory {
    var type: HoloLongTermMemoryType
    var title: String
    var summary: String
    ...
}
```

建议首版不要直接删除旧 `type`，而是新增语义字段，降低迁移风险：

```swift
enum HoloMemorySemanticType: String, Codable, Equatable {
    case phaseShift
    case stablePattern
    case driftSignal
    case lifeEvent
    case statMilestone
}

enum HoloMemoryUseScope: String, Codable, Equatable {
    case coreContext
    case recentInsight
    case goalPlanning
    case retrospective
    case displayOnly
}

struct HoloLongTermMemory {
    var type: HoloLongTermMemoryType
    var semanticType: HoloMemorySemanticType?
    var title: String
    var summary: String              // 兼容旧数据，后续等同 displaySummary
    var displaySummary: String?
    var aiUseSummary: String?
    var useScopes: [HoloMemoryUseScope]?          // 旧 JSON 兼容：缺失时使用 fallback
    var prohibitedInferences: [String]?           // 旧 JSON 兼容：缺失时视为空
    ...
}
```

兼容策略：

- 旧数据没有 `semanticType` 时不做自动语义分类，保留 `semanticType = nil`。
- 旧 `summary` 继续展示，新增数据优先用 `displaySummary`。
- `HoloMemorySummaryProvider` 后续优先给 prompt 注入 `aiUseSummary`，而不是页面摘要。
- `useScopes` / `prohibitedInferences` 必须是 Optional，或通过自定义 `init(from:)` 在缺失时给默认值；禁止直接新增非 Optional 字段导致旧 JSON 解码失败。
- 首版不强制批量重写旧 `summary`。UI 应对旧格式记忆显示“旧格式记忆，建议重新确认/删除”的轻提示，避免把原本系统话术继续伪装成新方案结果。
- 旧格式记忆默认不参与新的语义筛选；如果需要召回，只按旧 `title：summary` 低优先级注入，且受 feature flag 控制。

## 6. Prompt 输出规则

长期记忆候选生成不应直接复用洞察卡片标题。需要新增一层“记忆候选语义化”规则。

当前代码里，`MemoryInsightService` 会在洞察生成完成后，把满足 `patternType != nil && evidence 非空` 的洞察卡片转成通知 payload，再由 `HoloLongTermMemoryCandidateObserver` 创建长期记忆候选。这个链路解释了为什么“任务全部闭环”“体重习惯连续中断”会进入记忆管理页：长期记忆候选直接继承了洞察卡片的展示标题和正文。

因此首版必须增加一个桥接层，但它不能新增额外 LLM 调用。采用方案 A：

1. `memoryInsightGeneration` 在生成洞察卡片时，只为明确可成为长期记忆的卡片输出可选 `memoryCandidate` 字段。
2. `memoryCandidate` 首版只包含三个 LLM 负责字段：`semanticType`、`displaySummary`、`aiUseSummary`。
3. `MemoryCandidateSemanticMapper` 只做本地结构转换、字段校验、默认值填充和安全兜底。
4. Mapper 不负责“理解文本并重新分类”，也不对每张卡片再次调用 LLM。

目标链路：

```text
MemoryInsightService.generateInsight()
  -> parse JSON into MemoryInsightCard[]
  -> post .memoryInsightDidGenerate notification
     payload: card + optional memoryCandidate
  -> HoloLongTermMemoryCandidateObserver.extractCandidates(...)
  -> MemoryCandidateSemanticMapper.validateAndMap(memoryCandidate, card, evidence)
  -> HoloLongTermMemory candidate
  -> HoloMemoryPromotionPolicy.evaluate(...)
  -> HoloLongTermMemoryStore.upsertCandidate(...)
```

桥接层负责把 LLM 已输出的候选语义转换成长期记忆模型，而不是直接复用 `card.title` 和 `card.body`，也不是额外发起二次 LLM 判断。

推荐输出结构：

```json
{
  "semanticType": "phaseShift | stablePattern | driftSignal | lifeEvent | statMilestone",
  "displaySummary": "给用户审核的事实摘要，不含建议",
  "aiUseSummary": "给 HoloAI 使用的上下文摘要，说明适用场景和误用边界"
}
```

生成约束：

1. `title` 不允许出现“闭环、终端、清零、偏高、模式、趋势、画像、异常”等系统词，除非用户原文就是这样说的。
2. `displaySummary` 只描述事实和含义，不输出行动建议、鼓励语、教练式表达。
3. `aiUseSummary` 必须说明适合在哪些场景召回，以及要避免哪些误判。
4. 普通统计只能进入 `statMilestone`，不得伪装成阶段变化。
5. 混合语义必须拆分；一个候选只能有一个主语义。
6. 敏感、高影响推断必须要求确认。
7. 首版只允许 `habit` / `finance` / `task` / `milestone` 卡片输出 `memoryCandidate`。
8. `overview` / `anomaly` / `thought` 默认不输出 `memoryCandidate`。
9. `cross_domain` 默认不晋升为长期记忆候选；除非能拆成单一语义类型，否则只保留为洞察卡。

Mapper 默认值规则：

| 字段 | 默认策略 |
| --- | --- |
| `title` | 默认复用卡片标题，但必须通过禁用词校验；不合格则用 `displaySummary` 压缩生成本地标题或丢弃候选 |
| `useScopes` | 根据 `semanticType` 本地填充 |
| `prohibitedInferences` | 根据 `semanticType` 和 sensitivity 本地填充基础边界 |
| `confidence` | LLM 若输出合法值则采用；缺失时按 evidence 数量 fallback |
| `sensitivity` | LLM 若输出合法值则采用；本地关键词/来源规则只能升级，不能降级 |
| `expiresAt` | `driftSignal` 默认 14-30 天；其他类型默认 nil |

后端与 iOS 同步要求：

- 修改 iOS fallback prompt：`PromptManager.swift` 内嵌的 `memoryInsightGeneration`。
- 修改后端生效 prompt：`HoloBackend/src/prompts/defaultPrompts.json`。
- 升级 `PromptManager.promptVersions[.memoryInsightGeneration]`。
- 如果新增 prompt 类型，必须同步后端 prompt 配置和路由配置；首版不建议新增独立 prompt 类型。
- 涉及 `HoloBackend/` 后，必须执行后端 Docker 重建和部署，不能只提交本地代码。

## 7. 晋升规则

| 语义类型 | 默认决策 | 是否可静默写入 | 过期策略 | 主要召回场景 |
| --- | --- | --- | --- | --- |
| `phaseShift` | 待确认 | 否 | 长期保留 | 近期洞察、目标规划、长期回顾 |
| `stablePattern` | 观察或静默 | 是，需多周期证据且低敏感 | 可长期保留 | 今日状态、提醒、个性化建议 |
| `driftSignal` | 待确认 | 否 | 14-30 天 | 健康/习惯/目标洞察 |
| `lifeEvent` | 待确认 | 否 | 长期保留 | 对话、规划、年度回顾 |
| `statMilestone` | 轻量收藏 | 否 | 长期或归档 | 记忆长廊、成就回顾 |

关键调整：

- `driftSignal` 不应成为永久画像。它是近期上下文，需要过期或被恢复信号覆盖。
- `statMilestone` 默认不进 HoloAI 核心上下文，避免“记录 100 笔消费”影响对用户性格的理解。
- `stablePattern` 是唯一适合低风险静默写入的类型，但必须有跨周期证据。
- 晋升策略以 `semanticType` 为主；旧 `type` 只能作为兼容 fallback。
- `HoloLongTermMemoryCandidateObserver` 不能继续把所有洞察候选硬编码为 `.recurringPattern`。在模型迁移期，旧 `type` 至少要由 `semanticType` 映射得到。

旧 `type` 兼容映射建议：

| `semanticType` | 兼容旧 `type` |
| --- | --- |
| `phaseShift` | `.profileBackedFact` |
| `stablePattern` | `.recurringPattern` |
| `driftSignal` | `.recurringPattern` |
| `lifeEvent` | `.profileBackedFact` |
| `statMilestone` | `.recurringPattern` |

注意：上表只用于新数据写入时给旧 `type` 字段赋兼容值，不用于旧数据自动反推 `semanticType`。

过期消费端要求：

- `HoloLongTermMemoryStore` 增加 `cleanupExpired(now:)`。
- `queryPromptSummary()`、`queryConfirmed()`、`queryCandidates()` 必须过滤 `expiresAt < now`。
- `HoloMemoryCenterView.loadMemories()` 前触发长期记忆过期清理。
- App 生命周期进入前台时可触发轻量清理。
- 过期的 `driftSignal` 默认归档或从召回中排除，不直接硬删除，避免用户审计时找不到来源。

## 8. UI 信息架构

当前 UI 已经存在“最近记住（情景记忆）/ 待确认 / 已记住”三段。首版不建议增加第四个主 Section，而是调整命名和分组语义：

### 8.1 最近记住

保留现有情景记忆 Section。

定位：

- 近期观察。
- 可过期上下文。
- 可隐藏、可删除、可拒绝。
- 不等同于长期记忆。

首版不把 `statMilestone` 放进这里，避免与情景记忆混淆。

### 8.2 待确认

Holo 认为可能值得记住，但需要用户审核。

优先显示：

- 阶段变化
- 偏离提醒
- 人生节点

卡片操作：

- 记住
- 不准确
- 删除

### 8.3 长期记忆

已经确认、会影响 HoloAI 理解用户的内容。

默认按语义类型分组或筛选：

- 阶段变化
- 稳定习惯
- 人生节点

### 8.4 轻量记录

不作为主 Section，作为“长期记忆”里的筛选或子分组。不强影响 AI 判断，但适合保留为回顾或成就。

包括：

- 累计记录节点
- 连续打卡节点
- 历史数量里程碑

## 9. 示例改写

### 9.1 身体记录中断

当前：

- 标题：体重习惯连续中断
- 摘要：最近 14 天中体重习惯全部未完成，俯卧撑中断 11 天……

建议：

- 类型：`driftSignal`
- 标题：身体记录最近不连续
- 页面摘要：近 14 天内，体重、平板支撑、俯卧撑等身体相关记录多次中断。
- AI 摘要：用户近期身体状态追踪不稳定。健康或习惯洞察可参考此上下文，但需要确认用户当前是否仍关注这些目标，避免直接归因为懒惰、放弃或失败。
- 过期：30 天

### 9.2 任务全部闭环

当前：

- 标题：任务全部闭环
- 摘要：本月创建 4 个任务，全部完成，逾期 0……

建议：

- 类型：`stablePattern`
- 标题：任务收尾节奏稳定
- 页面摘要：本月创建的任务均已完成，未出现逾期。
- AI 摘要：用户近期任务执行和收尾表现稳定。后续任务规划可参考这个节奏，但不要据此推断用户一直高效。
- 召回：目标规划、今日状态、任务建议

### 9.3 支出偏高

当前：

- 标题：任务清零，支出偏高
- 摘要：本周完成 5 个任务，逾期 0 个……总支出比上周高 75%……

建议拆分。

候选 A：

- 类型：`stablePattern`
- 标题：本周任务收尾顺利
- 页面摘要：本周完成 5 个任务，未出现逾期。
- AI 摘要：用户本周任务执行状态较稳定。可用于近期回顾，不应长期推断为固定习惯，除非后续多周期重复。

候选 B：

- 类型：`driftSignal` 或不进入长期记忆
- 标题：本周支出被大额开销拉高
- 页面摘要：本周总支出高于上周，主要由大额或固定开销贡献。
- AI 摘要：近期财务洞察可参考这条背景，但需要区分固定必要支出和可调整支出，不要直接判断用户消费失控。

如果支出偏高只是房租或一次性固定支出，建议只保留为洞察卡，不晋升长期记忆。

## 10. 实施拆分

### Phase 0: 召回链路接通与验证

目标：先证明当前已确认长期记忆能进入普通 HoloAI 对话上下文。

- `UserContextBuilder.buildContext()` 在 `HoloAIFeatureFlags.memorySummaryInjectionEnabled` 开启时调用 `HoloMemorySummaryProvider.selectRelevantSummary(purpose:)`。
- `AIUserContextMessageBuilder.build(.chat)` 保持只在聊天场景注入，意图识别场景不注入长期记忆。
- 先使用现有 `title：summary` 格式，不引入新字段。
- 验证设置页“对话中利用已记住的信息辅助回答”开关与真实行为一致。
- 开发者验证允许直接写入测试用 confirmed memory，不依赖完整“打开长期记忆开关 -> 生成洞察 -> 候选提取 -> 用户确认”链路。
- 用户场景验证再覆盖完整链路：开启长期记忆开关、生成洞察、提取候选、确认后注入聊天。
- 用本地测试或日志确认 `context.memorySummary.lines` 非空时 prompt 中出现“--- 记忆摘要 ---”。
- 如果需要生产验证，通过后端请求日志确认 LLM 请求体包含记忆摘要。

验收标准：

- 有已确认长期记忆时，聊天 prompt 能看到最多 5 条记忆摘要。
- 关闭开关后，聊天 prompt 不注入记忆摘要。
- 没有长期记忆时，不出现空的记忆 Section。
- 开发者测试数据和用户真实确认数据两种路径都能完成注入验证。

### Phase 1: Prompt 与候选结构

- 更新 `memoryInsightGeneration` 或新增长期记忆候选提取 prompt。
- 输出 `semanticType / displaySummary / aiUseSummary`。
- `HoloLongTermMemoryCandidateObserver` 不再把所有候选写成 `.recurringPattern`。
- 新增 `MemoryCandidateSemanticMapper`，只做本地校验和结构转换，不新增 LLM 调用。
- 如果仍由 `MemoryInsightService` 发送通知，通知 payload 应包含语义化后的字段，而不是只传 `title/summary/patternType/evidence`。
- 同步修改 iOS fallback prompt 和 HoloBackend 生效 prompt。
- 升级 prompt version；涉及 HoloBackend 变更时完成 Docker 重建和部署。

### Phase 2: 模型与 Store 兼容

- 扩展 `HoloLongTermMemoryModels.swift`。
- 保持旧 JSON 可解码。
- `HoloLongTermMemoryStore` 保存新字段。
- 新字段使用 Optional 或自定义 decoder 默认值，禁止破坏旧 JSON。
- 旧数据不自动映射新 `semanticType`，保留 nil。
- UI 标记旧格式记忆，避免把旧摘要当成新方案产物。

### Phase 3: 晋升策略

- 按 `semanticType` 重写 `HoloMemoryPromotionPolicy.evaluate(candidate:)`。
- 为 `driftSignal` 增加默认过期。
- 为 `statMilestone` 增加轻量收藏路径。
- 增加长期记忆过期清理和查询过滤。
- 旧 `type` 只作为兼容，不再主导晋升。

### Phase 4: HoloAI 召回

- `HoloMemorySummaryProvider` 优先注入 `aiUseSummary`。
- 按 `useScopes` 与当前 `purpose` 选择记忆。
- 避免把 `displaySummary` 当成 AI 上下文。
- `AIUserContextPurpose.chat` 默认允许 `[.coreContext, .recentInsight]`；`AIUserContextPurpose.intentRecognition` 不注入长期记忆。
- 如需 `goalPlanning` / `retrospective` 等更细场景，必须让上游将 `HoloAICapabilityID` 或等价 purpose 传入 `UserContextBuilder.buildContext()`，不能假设聊天 purpose 已足够表达业务场景。
- 注入格式包含误用边界：

```text
--- 记忆摘要 ---
- [recentInsight, goalPlanning] 身体记录最近不连续：用户近期身体状态追踪不稳定。适合健康或习惯洞察参考。
  避免推断：不要判断用户已放弃目标；不要归因为懒惰或失败。
```

筛选算法：

1. 只取 `.confirmed` 和 `.silentlyAccepted`。
2. 排除 `expiresAt < now`。
3. 根据当前 `purpose` 映射到允许的 `useScopes`。
4. 排除 `.displayOnly`，除非 purpose 是回顾或记忆长廊。
5. 优先级：高置信、近期更新、非敏感、证据多。
6. 若匹配不足 5 条，可 fallback 到 `.coreContext`；仍不足则注入已有条目，不补无关记忆。
7. 注入后的记忆摘要总长度首版限制在 500-700 中文字或等价 token 内，超出时按优先级截断。

### Phase 5: UI 调整

- `HoloMemoryCenterView` 保留“最近记住”，将长期记忆部分改为“待确认 / 长期记忆”，轻量记录作为长期记忆的子分组或筛选。
- 卡片摘要只展示 `displaySummary`。
- 详情页展示类型、证据、确认状态；AI 摘要默认可隐藏或仅调试可见。

### Phase 6: 回滚与灰度

- 增加 feature flag：`semanticMemoryTypesEnabled`、`semanticMemoryPromptEnabled`、`semanticMemoryRecallEnabled`。
- Phase 0 可单独上线；Phase 1-5 分阶段打开。
- 关闭 `semanticMemoryPromptEnabled` 时，恢复旧洞察输出和旧候选提取。
- 关闭 `semanticMemoryRecallEnabled` 时，不注入新 `aiUseSummary/useScopes/prohibitedInferences`，但保留已保存字段。
- 不做破坏性批量迁移；旧数据保留可读。

## 11. 验证用例

### 11.1 Prompt 样例

输入：14 天内体重、平板支撑、俯卧撑记录中断。

预期：

- 输出 `semanticType=driftSignal`。
- 标题不出现“终端/闭环/清零/模式”。
- `displaySummary` 不包含建议。
- `aiUseSummary` 包含误判边界。
- 设置过期时间。

### 11.2 晋升策略

- `driftSignal + normal` 仍要求确认。
- `stablePattern + 多周期证据 + normal` 可静默写入。
- `lifeEvent + highImpact` 必须确认。
- `statMilestone` 不进入核心长期记忆召回。
- `expiresAt < now` 的长期记忆不会进入候选、已确认列表和 prompt 摘要。

### 11.3 UI

- 用户能在一屏内理解每条记忆“记住了什么”。
- 混合语义不会出现在同一张卡。
- 页面不展示行动建议。
- 情景记忆仍显示在“最近记住”，`statMilestone` 不与其混淆。

### 11.4 自动化测试

需要新增或扩展以下测试：

- `UserContextBuilder` / `AIUserContextMessageBuilder`：开关注入、关闭不注入、空记忆不注入。
- `MemoryCandidateSemanticMapperTests`：语义字段完整时成功映射，缺字段时降级或丢弃。
- `HoloMemoryPromotionPolicyTests`：按 `semanticType` 决策，不再被旧 `type` 主导。
- `HoloLongTermMemoryStoreTests`：过期过滤、过期归档/清理、旧 JSON 解码。
- `HoloMemorySummaryProviderTests`：按 `useScopes` 和 purpose 选择，`displayOnly` 默认不进聊天。

## 12. 风险与边界

1. 如果只改 UI 文案，不改候选结构，系统仍会持续生成低价值记忆。
2. 如果只加类型，不拆 `displaySummary` 和 `aiUseSummary`，HoloAI 仍会把面向用户的文案当上下文使用。
3. 如果 `driftSignal` 不过期，用户短期偏离会固化成长期画像。
4. 如果 `statMilestone` 参与核心召回，HoloAI 会被低价值统计节点污染。
5. 如果人生节点允许静默写入，会带来信任和隐私风险。
6. 如果 Phase 0 不先接通召回链路，后续所有 AI 摘要和召回规则都无法验证。
7. 如果 prompt 只改 iOS fallback 而不改 HoloBackend 生效 prompt，生产行为不会变化。
8. 如果长期记忆 Store 继续无串行写保护，后台候选写入和 UI 确认/删除同时发生时存在覆盖风险。

并发安全要求：

- `HoloLongTermMemoryStore` 应参考 `HoloEpisodicMemoryStore` 增加串行/并发 barrier 队列，避免读改写竞争。
- 所有确认、拒绝、删除、候选 upsert 走同一个写入队列。
- 测试覆盖并发 upsert 与 confirm/delete 的基本场景。

## 13. 结论

推荐采用“Phase 0 先接通召回 + 语义类型 + 双摘要 + 晋升矩阵”的重构方式。

记忆管理页的核心任务不是教育用户，也不是生成建议，而是让用户审核 Holo 对自己的长期理解。真正面向用户的建议应放在 HoloAI 对话、Daily Sense 或周期洞察里；长期记忆本身应该更像一组可控、可解释、可撤销的 AI 上下文。

进入实施前的最低条件：

- Phase 0 通过验证。
- `statMilestone` 与情景记忆边界被确认。
- prompt 双端同步和后端发版步骤被纳入实施计划。
- `MemoryCandidateSemanticMapper` 明确不新增 LLM 调用。
