# Holo 目标系统 v1 设计

## 概述

Holo 目标系统 v1 为用户提供一个从模糊长期愿望到可执行行动系统的轻量闭环。

用户可以在 HoloAI 中表达类似「我想学习 SwiftUI」「我想减脂」「我想准备考试」这样的长期目标。HoloAI 通过最多 3 轮追问澄清目标，再生成一个可编辑的目标计划草案。用户确认后，系统保存一个 `Goal`，并按用户选择创建关联的任务和习惯。

目标本身放在「个人 > 我的目标」中长期管理。任务模块继续负责一次性行动，习惯模块继续负责持续行为，Goal 只承担聚合、关联、进展解释和 AI 上下文感知。

## 设计目标

- 让 HoloAI 可以把用户的长期意图转化为结构化行动计划。
- 在「个人」模块下新增轻量的目标管理能力。
- 让任务和习惯可以归属于某个目标，并支持目标级进展统计。
- 让 HoloAI 后续对话和洞察能在用户授权后感知目标上下文。
- 保持 v1 范围克制，避免把 Goal 做成复杂项目管理系统。

## 非目标

- v1 不支持用户在「个人 > 我的目标」中手动新建目标。
- v1 不支持一个任务或习惯关联多个目标。
- v1 不提供复杂 OKR、里程碑、多层项目管理、甘特图或团队协作能力。
- v1 不让 AI 自动把目标标记为完成。
- v1 不把完整目标列表自动写入 `HoloProfile.md`。

## 核心定位

Goal 是 Holo 对「用户为什么要做这些任务和习惯」的理解层。

```text
Goal：回答「我为什么要持续做这些事」
Task：回答「我接下来要完成什么一次性行动」
Habit：回答「我需要长期坚持什么行为」
HoloAI：负责把模糊目标变成 Goal + Task + Habit 的草案
个人模块：负责长期查看和管理 Goal
```

示例：

```text
Goal：学习 SwiftUI 到能独立开发 App
  Task：看完 SwiftUI 官方教程
  Task：做一个 Todo demo
  Task：重构 Holo 中的一个真实页面
  Habit：每周学习 4 次，每次 45 分钟
  Habit：每周复盘一次学习笔记
```

```text
Goal：减脂 5kg
  Task：制定一周饮食计划
  Task：购买体脂秤
  Habit：每周健身 3 次
  Habit：每天记录体重
  Habit：每周最多一次放纵餐
```

## 用户故事

### 用户故事 1：通过 HoloAI 规划学习目标

用户在 HoloAI 中点击「规划目标」，输入「我想学习 SwiftUI」。

HoloAI 追问：
- 想学到什么程度？
- 学习这个技能是为了什么？
- 每周大概能投入多少时间？

用户回答后，HoloAI 生成目标草案：
- 一个学习 SwiftUI 的 Goal
- 若干阶段性任务
- 一个每周学习习惯
- 一个每周复盘习惯

用户在确认看板中轻量编辑标题、截止日期和习惯频率，勾选要创建的项目，授权保存目标，并授权 HoloAI 后续参考该目标。

### 用户故事 2：管理已有目标

用户进入「个人 > 我的目标」，看到所有目标列表。

每个目标展示：
- 标题
- 当前状态
- 粗粒度进展
- 关联任务完成情况
- 关联习惯近期坚持情况

用户点进目标详情，可以查看关联任务和习惯，也可以暂停、恢复、完成或删除目标。

### 用户故事 3：删除目标但保留行动

用户删除某个目标时，系统提示：

```text
删除目标后，基于该目标创建的任务和习惯不会被删除，只会解除与该目标的关联。
```

用户确认后，系统删除 Goal 或解除 Goal 记录，同时保留所有任务、习惯、打卡记录和任务完成历史。

### 用户故事 4：AI 在授权后感知目标

用户授权某个目标进入 AI 上下文后，后续向 HoloAI 提问「我最近是不是有点拖延」。

HoloAI 可以结合：
- 当前 active Goal
- 目标下任务完成情况
- 目标下习惯近期完成情况
- 用户的近期对话和记录

给出相关建议，例如：

```text
你最近的 SwiftUI 学习目标里，习惯还保持得不错，但「完成第一个 demo」这个任务已经停了 5 天。
要不要我帮你把它拆成一个 30 分钟能开始的小任务？
```

## 产品范围

### 目标创建入口

v1 有两个入口：

- HoloAI 快捷入口 Chip：「规划目标」
- 「个人 > 我的目标」空状态按钮：「让 HoloAI 帮你规划第一个目标」

「个人 > 我的目标」不提供手动新建按钮。空状态入口只负责跳转到 HoloAI 的目标规划流程。

### 创建方式

目标只能由 HoloAI 创建。

流程：

```text
用户进入目标规划
  -> HoloAI 识别目标类型
  -> 最多 3 轮追问
  -> HoloAI 推荐生成密度，用户可切换精简 / 完整模式
  -> HoloAI 生成 GoalDraft
  -> 用户在确认看板中编辑和勾选
  -> 用户授权保存
  -> 系统写入 Goal、任务、习惯及关联关系
```

### 生成模式

v1 支持两种生成策略，但底层统一输出 `GoalDraft`。

| 模式 | 用途 | 输出倾向 |
|------|------|----------|
| 精简模式 | 用户想快速开始 | 少量高置信任务和习惯 |
| 完整模式 | 用户想要阶段化计划 | 更多阶段任务、持续习惯和复盘节点 |

生成模式不改变数据结构，只影响 AI 生成草案时的密度和拆解粒度。

v1 不要求用户在看见任何信息前做强决策。默认策略是：

- HoloAI 根据目标类型、追问信息量、截止时间和用户投入时间自动推荐生成密度。
- 确认看板顶部提供「精简 / 完整」切换，用户可以在生成前或重新生成时调整。
- 如果用户没有主动选择，系统使用推荐模式。

这样保留用户选择权，同时避免把「精简还是完整」变成目标规划流程里的额外负担。

### 多轮追问

最多追问 3 轮。

追问优先级：

1. 目标结果：用户希望达到什么程度。
2. 动机和用途：为什么要做这个目标。
3. 时间和资源：截止时间、每周投入、已有基础或限制。

如果 3 轮后信息仍不完整，HoloAI 不继续追问，而是生成带不确定项提示的草案，让用户在确认看板中调整。

### HoloAI 路由入口

目标规划不作为普通 quick action prompt 进入 `sendMessage()` 的默认链路。

现有 `QuickActionBar` 的常规行为是：

```text
QuickActionBar
  -> ChatViewModel.sendQuickAction(action)
  -> inputText = action.prompt
  -> sendMessage()
  -> ConversationCoordinator.process()
```

`规划目标` 需要走独立分流：

```text
QuickAction.planGoal
  -> ChatViewModel.startGoalPlanning(seedText: nil)
  -> activeGoalPlanningSession = GoalPlanningSession(...)
  -> GoalPlanningCoordinator.start()
  -> assistant 追问消息写入聊天流
```

从「个人 > 我的目标」空状态进入时：

```text
PersonalGoalEmptyState
  -> deep link / tab switch 到 HoloAI
  -> ChatViewModel.startGoalPlanning(seedText: nil)
```

如果用户没有点快捷入口，而是在普通 HoloAI 输入中直接说「我想学习 SwiftUI」：

- v1 可以先让普通 `ConversationCoordinator` 识别为 `query` 或 `unknown`，并在回复中引导用户点击「规划目标」。
- 不在 v1 里强行让普通聊天自动切换到目标规划，避免误把闲聊想法变成创建流程。
- 后续版本可以在 `AIParseBatch` 中增加 `plan_goal_suggestion`，作为轻提示而非自动执行。

### GoalPlanningSession 状态机

`ChatViewModel` 需要持有当前目标规划会话：

```swift
@Published private(set) var activeGoalPlanningSession: GoalPlanningSession?
```

当 `activeGoalPlanningSession != nil` 时，`sendMessage()` 的分流规则是：

```text
用户发送消息
  -> 如果 activeGoalPlanningSession.status == collecting
       交给 GoalPlanningCoordinator.handleUserReply(...)
       不调用 ConversationCoordinator
  -> 如果 activeGoalPlanningSession.status == draftReady
       普通文本输入不继续写入规划；用户需要在确认看板操作
  -> 如果 session cancelled / confirmed
       清空 activeGoalPlanningSession
       恢复普通 ConversationCoordinator 流程
```

会话状态：

| 状态 | 说明 | 下一步 |
|------|------|--------|
| `collecting` | 正在追问和收集回答 | 继续追问，或达到 3 轮后生成草案 |
| `draftReady` | 已生成 GoalDraft | 展示确认看板 |
| `confirming` | 用户正在确认看板保存 | 禁用重复提交 |
| `confirmed` | 已成功写入 Goal/任务/习惯 | 清空 session，展示成功结果 |
| `cancelled` | 用户取消规划 | 清空 session，恢复普通聊天 |

v1 会话只保存在内存中。App 被杀、聊天页面释放、用户手动取消时，未确认草案可以丢弃。丢弃前如果已有草案，UI 需要提示「未保存的目标计划会丢失」。

聊天流中的表现：

- 追问以普通 assistant 消息展示。
- 用户回答以普通 user 消息展示。
- 草案生成后不直接在气泡中编辑，而是弹出确认看板。
- 取消规划时写入一条轻量 assistant 消息：「已取消这次目标规划。」

## 数据模型

### Goal

`Goal` 是结构化持久化实体，建议使用 Core Data 存储。

核心字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `UUID` | 目标唯一 ID |
| `title` | `String` | 目标标题 |
| `summary` | `String?` | 简短说明 |
| `domain` | `String` | 目标领域，如 learning、health、career、finance、life、project |
| `desiredOutcome` | `String?` | 用户希望达到的结果 |
| `motivation` | `String?` | 用户做这个目标的原因 |
| `status` | `String` | active、paused、completed |
| `deadline` | `Date?` | 可选截止日期 |
| `createdAt` | `Date` | 创建时间 |
| `updatedAt` | `Date` | 更新时间 |
| `completedAt` | `Date?` | 完成时间 |
| `source` | `String` | v1 固定为 holoAI |
| `allowAIContext` | `Bool` | 是否允许 HoloAI 后续参考该目标 |
| `lastInsightSummary` | `String?` | 最近一次 AI 目标解释摘要，可选 |

### GoalStatus

```swift
enum GoalStatus: String, Codable {
    case active
    case paused
    case completed
}
```

删除行为不需要在 v1 UI 中展示 `deleted` 状态。实现时可以根据 Core Data 现有风格选择硬删除或软删除，但用户侧表现是目标被移除。

### GoalDomain

v1 可以先用字符串或轻量枚举：

```swift
enum GoalDomain: String, Codable {
    case learning
    case health
    case career
    case finance
    case life
    case project
    case other
}
```

领域主要用于：
- AI 追问策略
- 图标和颜色
- 目标列表筛选或分组
- 后续洞察时的语境判断

### 任务和习惯关联

v1 采用单归属关系。

```text
一个 Goal 可以关联多个 TodoTask
一个 Goal 可以关联多个 Habit
一个 TodoTask 最多属于一个 Goal
一个 Habit 最多属于一个 Goal
```

建议使用 Core Data to-one 关系，而不是在 `TodoTask` / `Habit` 上保存裸 `goalId`。

| 实体 | 新关系 | 说明 |
|------|--------|------|
| `Goal` | `tasks: Set<TodoTask>` | 目标下关联任务，to-many |
| `TodoTask` | `goal: Goal?` | 所属目标，to-one，可选 |
| `Goal` | `habits: Set<Habit>` | 目标下关联习惯，to-many |
| `Habit` | `goal: Goal?` | 所属目标，to-one，可选 |

v1 不做多对多关系，避免进度重复计算、解除关联复杂化和 UI 解释成本。

删除规则：

- `Goal.tasks` 使用 `nullifyDeleteRule`，删除 Goal 时不删除 TodoTask，只解除关系。
- `TodoTask.goal` 使用 `nullifyDeleteRule`。
- `Goal.habits` 使用 `nullifyDeleteRule`，删除 Goal 时不删除 Habit，只解除关系。
- `Habit.goal` 使用 `nullifyDeleteRule`。

这个设计与现有 `TodoTask.list` 关系模式一致，也避免删除目标时手动批量清空 UUID 带来的对象刷新和孤儿引用问题。

### GoalDraft

`GoalDraft` 是 AI 生成后的临时草案，不直接等同于持久化实体。

```swift
struct GoalDraft: Codable, Equatable {
    let id: String
    var title: String
    var summary: String?
    var domain: GoalDomain
    var desiredOutcome: String?
    var motivation: String?
    var deadlineText: String?
    var tasks: [GoalTaskDraft]
    var habits: [GoalHabitDraft]
    var missingInfoWarnings: [String]
}
```

### GoalTaskDraft

```swift
struct GoalTaskDraft: Codable, Equatable, Identifiable {
    let id: String
    var isSelected: Bool
    var title: String
    var dueDateText: String?
    var priority: Int?
    var note: String?
}
```

### GoalHabitDraft

```swift
struct GoalHabitDraft: Codable, Equatable, Identifiable {
    let id: String
    var isSelected: Bool
    var name: String
    var frequency: String
    var targetCount: Int?
    var type: String
    var unit: String?
    var targetValue: Double?
}
```

## AI 架构

### 目标规划信号与意图边界

v1 不强制把目标规划做成普通 `IntentRouter` 动作。可以定义目标规划相关信号，但它们优先服务于独立 flow，而不是进入全局执行路由。

| 信号 | 用途 |
|------|------|
| `plan_goal` | 用户表达想建立长期目标或从快捷入口进入规划 |
| `continue_goal_planning` | 目标规划多轮追问中的继续回答 |
| `confirm_goal_plan` | 用户确认保存目标草案 |

v1 先把目标规划视为独立 flow，不走普通 `IntentRouter.route()` 立即执行。

推荐做法：

```text
普通快捷动作：
ChatViewModel -> ConversationCoordinator -> IntentRouter

目标规划：
ChatViewModel -> GoalPlanningCoordinator -> GoalPlanningProvider -> GoalDraftBoard
```

原因：
- 目标规划需要多轮状态。
- 目标规划不会在第一轮直接写库。
- 目标规划输出的是草案和确认看板，不是普通聊天卡片。

意图边界：

- `plan_goal` 只在目标规划 flow 内使用，或作为普通聊天中的「建议进入目标规划」信号。
- `continue_goal_planning` 不进入全局 `AIIntent` 路由，优先由 `activeGoalPlanningSession` 状态决定。
- `confirm_goal_plan` 由确认看板按钮触发，不依赖 LLM 判断。
- 普通用户输入「我想学习 SwiftUI」在 v1 不自动创建 `plan_goal`，避免误触发；HoloAI 可以回复并提示「可以点规划目标，我来帮你拆成任务和习惯」。

Prompt 变更范围：

- 目标规划 prompt 独立放在 `GoalPlanningPromptBuilder` 或专用 prompt 类型中。
- 现有意图识别 prompt 只需要增加轻量边界说明：目标规划由快捷入口启动，普通聊天不直接执行目标创建。
- 后端 managed prompt 如果承载 iOS 默认意图识别模板，需要同步补充该边界，避免本地和后端 prompt 语义分叉。

### GoalPlanningSession

目标规划需要一个轻量会话状态：

| 字段 | 说明 |
|------|------|
| `id` | 会话 ID |
| `initialUserText` | 用户最初表达 |
| `turnCount` | 已追问轮数 |
| `maxTurns` | v1 固定 3 |
| `answers` | 已收集回答 |
| `mode` | concise 或 complete，默认由 AI 推荐 |
| `status` | collecting、draftReady、confirming、confirmed、cancelled |
| `draft` | 当前 GoalDraft |

该会话可以先保存在内存中。若用户退出聊天或 App 被杀，v1 可以丢弃未确认草案，不做复杂恢复。

### 追问策略

不同领域使用不同追问模板。

学习类：
- 想学到什么程度？
- 学习它是为了做什么？
- 当前基础如何，每周能投入多少时间？

健康类：
- 目标是减脂、增肌、改善作息，还是饮食控制？
- 有什么身体限制或饮食禁忌？
- 每周能接受几次运动或记录？

职业类：
- 目标对应什么结果，如求职、晋升、作品集？
- 有没有截止时间？
- 当前已有基础是什么？

项目类：
- 最终交付物是什么？
- 有哪些必须完成的阶段？
- 期望什么时候完成？

### 草案输出规则

HoloAI 输出草案时必须遵守：

- 不直接创建任务或习惯。
- 不生成过多低置信建议。
- 每个任务必须是可执行动作。
- 每个习惯必须有明确频率或目标次数。
- 对不确定信息写入 `missingInfoWarnings`。
- 草案中的任务和习惯默认选中，但用户可以取消。

## 确认看板

确认看板是目标创建前的关键安全层。

### UI 形态

v1 使用 `fullScreenCover + NavigationStack` 展示确认看板。

选择 full screen 的原因：

- 看板需要同时编辑 Goal、任务草案和习惯草案，信息密度高。
- sheet 在移动端容易变成半屏挤压，编辑体验不稳定。
- 聊天流中的特殊卡片不适合承载多段编辑和授权开关。

交互结构：

```text
ChatView
  -> activeGoalPlanningSession.status == draftReady
  -> fullScreenCover(GoalDraftReviewView)
      -> 顶部导航：取消 / 保存
      -> Goal 基本信息
      -> 生成密度：推荐模式，可切换精简 / 完整并重新生成
      -> 任务草案区
      -> 习惯草案区
      -> 授权区
```

返回和取消：

- 点击「取消」时，如果用户已编辑草案，弹确认：「放弃这次目标计划？」
- 取消后 `activeGoalPlanningSession.status = cancelled`，关闭看板，聊天流写入取消提示。
- 保存成功后 `status = confirmed`，关闭看板，聊天流展示成功结果，并提供跳转目标详情入口。
- fullScreenCover 需要接入项目既有返回手势规范；如果当前工程要求 `.swipeBackToDismiss`，实现时必须补上。

### 看板能力

用户可以：
- 编辑 Goal 标题。
- 编辑 Goal 说明。
- 编辑 Goal 领域。
- 编辑可选截止日期。
- 勾选或取消任务草案。
- 编辑任务标题、截止日期、优先级和简短说明。
- 勾选或取消习惯草案。
- 编辑习惯名称、频率、目标次数和类型。
- 选择是否保存目标。
- 选择是否允许 HoloAI 后续参考该目标。

### 看板不支持

v1 不在确认看板中支持：
- 任务附件。
- 复杂提醒设置。
- 多级清单编辑。
- 标签管理。
- 习惯高级统计配置。
- 目标下的多层阶段树。

这些能力可在任务或习惯创建后进入对应模块继续编辑。

### 保存行为

用户点击确认后：

1. 创建 `Goal`。
2. 为选中的 `GoalTaskDraft` 创建 `TodoTask`，并设置 `task.goal = goal`。
3. 为选中的 `GoalHabitDraft` 创建 `Habit`，并设置 `habit.goal = goal`。
4. 保存 `allowAIContext` 授权状态。
5. 在 HoloAI 中展示创建成功结果。
6. 提供跳转到「个人 > 我的目标」或目标详情的入口。

如果任务或习惯部分创建失败，应保留已创建项，并在结果中明确显示失败项。v1 不要求事务级回滚。

## 进度模型

目标进度很难被真实量化，因此 v1 不强调精确百分比。

v1 不计算固定综合分数，也不使用 `任务 50% + 习惯 50%` 这类权重公式。UI 主要展示原始指标和可解释的粗粒度状态。

### 数据来源

进度由两部分构成：

- 任务推进情况：目标下任务完成数 / 总任务数。
- 习惯坚持情况：目标下习惯最近 7、14 或 30 天完成率。

默认使用近 14 天习惯完成率，兼顾敏感度和稳定性。

### 展示规则

v1 展示原始指标：

```text
任务：已完成 4/9
习惯：近 14 天完成率 68%
```

如果目标只有任务，只展示任务摘要。

如果目标只有习惯，只展示习惯摘要。

如果目标既没有任务也没有习惯，展示「尚未关联行动」。

新增任务或习惯不应被解释为目标倒退；它只改变原始分母，不触发负面文案。

### 粗粒度状态

| 状态 | 启发式条件 | 说明 |
|------|------------|------|
| 起步中 | 刚创建，或任务/习惯记录很少 | 目标尚未形成稳定推进证据 |
| 稳定推进 | 最近有任务完成，且主要习惯近期表现稳定 | 目标正在健康推进 |
| 有些停滞 | 近期无任务完成，或主要习惯明显低于目标频率 | 需要提醒或拆小行动 |
| 接近完成 | 大部分任务完成，且主要习惯近期稳定 | 可以提示用户确认是否完成 |
| 已暂停 | `status == paused` | 不参与 active 目标建议 |
| 已完成 | `status == completed` | 保留历史展示，不再作为 active 上下文 |

这些状态由规则引擎判断，不暴露精确分数。具体阈值可以在实现时保守设置，并允许后续根据真实使用数据调整。

### AI 的角色

AI 可以解释状态和给出建议，但不直接决定进度数字，也不自动完成目标。

AI 适合输出：

```text
任务推进良好，但习惯坚持一般。
```

不适合输出：

```text
你已经完成了 53.7%。
```

## 完成、暂停和删除

### 完成

v1 支持用户手动标记完成。

HoloAI 或系统可以在未来提示「建议完成」，但必须由用户确认后才把 `Goal.status` 改为 `completed`。

### 暂停

暂停后的目标：

- 保留所有关联。
- 不出现在 active 目标上下文中。
- 可以在个人目标页恢复。

是否仍进入 AI 上下文：
- 默认不进入。
- 即使 `allowAIContext == true`，`paused` 目标也不主动注入普通聊天上下文。

### 删除

删除目标时：

- 删除 Goal 或将 Goal 从用户可见列表移除。
- 通过 Core Data `nullifyDeleteRule` 解除 `TodoTask.goal` 和 `Habit.goal` 关系。
- 不删除任何任务。
- 不删除任何习惯。
- 不删除任务完成历史。
- 不删除习惯打卡历史。

## AI 上下文和授权

### 授权原则

保存目标和让 HoloAI 长期参考目标是两件事。

用户确认看板中需要两个明确动作：

- 保存为目标。
- 允许 HoloAI 在后续对话和洞察中参考此目标。

目标详情页也必须提供 `allowAIContext` 开关，用户可以随时撤销。

### 不写入 HoloProfile.md

v1 不把完整目标自动写入 `HoloProfile.md`。

`HoloProfile.md` 继续作为用户长期背景和偏好，例如职业、作息、沟通风格、健康提醒偏好。

目标数据保存在结构化 `Goal` 实体中。

### UserContextBuilder 扩展

`UserContextBuilder` 应动态读取：

- `status == active`
- `allowAIContext == true`

的目标，并拼接为 AI 上下文。

建议上下文格式：

```text
## 当前目标

- 学习 SwiftUI 到能独立开发 App
  - 状态：稳定推进
  - 任务：已完成 4/9
  - 习惯：近 14 天完成率 68%
  - 说明：用户希望用 SwiftUI 独立开发 Holo 功能

- 减脂 5kg
  - 状态：有些停滞
  - 任务：已完成 2/4
  - 习惯：近 14 天完成率 42%
```

为避免 prompt 过长，v1 不默认把所有授权目标注入每次 LLM 调用。注入策略：

- 普通聊天默认最多注入 1 个最近活跃或最相关的 active Goal。
- 当用户问题明显涉及目标、学习、计划、习惯坚持、拖延、复盘时，最多注入 3 个 active 且授权的目标。
- 分析查询和记忆洞察可以按场景注入更多摘要，但需要控制在短摘要级别。
- 每个 Goal 上下文控制为标题、状态、任务摘要、习惯摘要、用户动机的压缩版，避免完整展开任务/习惯列表。

排序优先级：

1. 最近更新。
2. 有停滞风险。
3. 创建时间较近。

## 与现有 Holo 架构的关系

### HoloAI

现有 HoloAI 已有：

- `AIParseBatch`
- `ConversationCoordinator`
- `IntentRouter`
- 多动作执行结果
- 聊天卡片渲染

目标规划不建议直接塞进普通多动作执行链路。目标规划需要多轮状态和确认看板，应通过独立的 `GoalPlanningCoordinator` 管理。

分流规则：

- `QuickAction.planGoal` 和个人目标空状态跳转直接调用 `ChatViewModel.startGoalPlanning()`。
- `activeGoalPlanningSession != nil` 时，用户后续输入优先交给 `GoalPlanningCoordinator`。
- 普通 `sendMessage()` 仅在没有 active session 时调用 `ConversationCoordinator`。
- `GoalPlanningCoordinator` 不直接调用 `IntentRouter` 写业务实体；只有确认看板保存时才通过 Repository 创建 Goal、任务和习惯。

### 任务模块

任务模块继续负责：

- 任务创建
- 截止日期
- 优先级
- 完成状态
- 任务详情编辑

Goal 只通过 Core Data to-one 关系 `TodoTask.goal` 关联任务，不改变任务模块的核心行为。

### 习惯模块

习惯模块继续负责：

- 打卡型习惯
- 数值型习惯
- 频率
- 目标次数
- 打卡记录
- 统计

Goal 只通过 Core Data to-one 关系 `Habit.goal` 关联习惯，不改变习惯模块的核心行为。

### 个人模块

个人模块新增「我的目标」入口。

这让个人模块从静态档案扩展为：

- 个人资料
- 目标管理
- 长期授权边界

### HoloProfile

HoloProfile 不承担目标系统的结构化存储。

它仍然用于长期用户背景。Goal 上下文由 `UserContextBuilder` 动态注入，便于授权撤销和状态过滤。

## 错误处理

### AI 追问失败

如果目标规划中的 AI 调用失败：

- 展示错误提示。
- 保留用户已输入内容。
- 允许重试。
- 不写入任何目标、任务或习惯。

### 草案解析失败

如果 AI 返回内容无法解析为 `GoalDraft`：

- 展示「计划生成失败，可以重试」。
- 记录 LLM 调用日志。
- 不展示不完整看板。

### 部分保存失败

如果确认保存时部分任务或习惯创建失败：

- 已成功创建的实体保留。
- 失败项在结果中展示。
- Goal 仍然保存。
- 用户可以稍后手动补充任务或习惯。

v1 不要求事务级全量回滚。

## 隐私和安全

- 用户必须明确授权目标进入 AI 上下文。
- 关闭授权后，后续普通聊天和洞察不主动使用该目标。
- 暂停和完成目标默认不进入 active 上下文。
- 删除目标不会删除任务和习惯，避免误删生活数据。
- AI 生成草案必须经过用户确认，不允许直接创建目标、任务或习惯。

### 仍然存在的问题

#### 1. 跨 Tab 跳转机制未设计（P1）

方案要求「个人 > 我的目标」空状态跳转到 HoloAI 并触发 `ChatViewModel.startGoalPlanning()`。

但 Holo 是 TabView 架构。PersonalView 在「个人」Tab，ChatView 在「聊天」Tab。PersonalView **无法直接访问** ChatViewModel — 它们属于不同的 Tab 内容。

要实现这个跳转，需要以下方案之一：

- **方案 A**：通过 `@Environment` 注入一个全局 action，TabView 层持有，ChatView/PersonalView 都能访问。
- **方案 B**：NotificationCenter 广播，ChatView 监听并触发 `startGoalPlanning()`。
- **方案 C**：`@AppStorage` / UserDefaults 写入标记，ChatView 在 `onAppear` 中检查并触发。

方案未说明选择哪种，也没评估各自的时序风险（比如 Tab 还没加载时发送广播会丢失）。

#### 2. 追问消息的持久化策略有歧义（P1）

方案说追问以 assistant 消息展示、用户回答以 user 消息展示。但没说明这些消息是否持久化到 `ChatMessage` Core Data 实体。

- **如果持久化**：用户取消规划后，聊天流里会留下一段"你在学什么程度？""学 SwiftUI"的孤立对话，没有上下文，看起来很奇怪。
- **如果不持久化**：App 被杀后聊天流里这段对话消失，但其他消息保留，用户可能困惑"刚才的对话去哪了"。
- **如果持久化但标记为 goalPlanning 类型**：需要在 ChatMessage 实体上加 `messageType` 字段，允许目标规划消息被过滤或特殊展示。

这个选择影响 ChatMessage 数据模型和聊天流 UI，不是实现细节。

#### 3. GoalPlanningCoordinator 内部仍是黑箱（P1）

路由和状态机定义清楚了，但 GoalPlanningCoordinator 自身的实现仍是一句话："多轮编排"。

关键问题未回答：

- **每轮追问是独立 LLM 调用还是流式？** 如果是独立调用，需要把历史 Q&A 作为 context 传入下一轮。3 轮追问 = 3 次 LLM 调用，每次携带累积的 context。
- **追问策略如何注入 prompt？** 是在 system prompt 中写"如果 domain == learning，问这三个问题"，还是 GoalPlanningPromptBuilder 为每轮动态生成 prompt？
- **何时决定停止追问？** 只靠 `turnCount >= 3`？还是 LLM 可以提前返回 `draftReady`？
- **GoalDraft 生成是第 4 次 LLM 调用？** 还是在第 3 轮追问的 LLM 调用中同时生成草案？

这些决定了 prompt 设计和 LLM 调用次数（直接影响成本和延迟），不是可以留到实施时再决定的。

#### 4. fullScreenCover 返回手势与状态机的交互（P1）

方案说确认看板用 fullScreenCover + 需要接入 `.swipeBackToDismiss`。

但用户在 `draftReady` 状态下滑动返回时，session 还在内存中。这意味着：

- 滑动返回 → 看板关闭 → 但 `activeGoalPlanningSession` 仍然是 `draftReady`
- 用户在聊天框输入文字 → 被分流逻辑拦截（"session 存在，不走 ConversationCoordinator"）
- 用户看不到任何 UI 反馈，输入被吞

需要在 `.swipeBackToDismiss` 的回调中同步处理 session 状态（提示"放弃草案？"或保持看板打开）。

#### 5. GoalHabitDraft.frequency String 到 HabitFrequency 的映射无保障（P2）

LLM 输出 `GoalHabitDraft.frequency` 是自由 String（可能是 "每周 3 次"、"daily"、"2 times a week"），但创建真实 Habit 时需要映射到 `HabitFrequency` 枚举的 rawValue（`daily` / `weekly` / `monthly`）。

如果 LLM 输出的 frequency 不匹配任何枚举值，创建会失败或产生错误数据。需要：

- 在 GoalPlanningPromptBuilder 的 prompt 中明确约束输出格式
- 或在确认看板中用 Picker 替代自由文本输入，强制用户选择频率
- 或在 GoalDraft 解析时做容错映射

#### 6. Core Data 关系迁移的注意事项（P2）

修订后改用 Core Data 关系，方向正确。但需要注意：

- 新增 Goal 实体：轻量迁移可以处理。
- 给现有 TodoTask 和 Habit 新增 `goal: Goal?` 关系：**轻量迁移也能处理**（可选关系，默认 nil）。
- 但删除规则（nullifyDeleteRule）必须在 `CoreDataStack+GoalEntities.swift` 中正确设置。现有代码中 `denyDeleteRule` 放 to-many 侧，`nullifyDeleteRule` 放 to-one 侧 — 方案的 Goal-TodoTask 关系需要在 Goal 侧（to-many）用 nullify 而非 deny，这一点虽然文档写了，但实施时容易出错，因为与现有 TodoList-TodoTask 的 deny/nullify 模式不同。

#### 7. Phase 1 交付后无法手动创建 Goal 的验证问题（P2）

Phase 1 完成后，「个人 > 我的目标」永远显示空状态（因为 Goal 只能由 HoloAI 创建，而 HoloAI 入口在 Phase 2 才有）。开发和测试时需要一个临时入口来创建测试数据，或者 Phase 1 必须和 Phase 2 一起交付。

方案可以补充：Phase 1 开发期间，GoalRepository 提供一个 debug 方法用于创建测试 Goal。

### 第二轮修复建议

| 优先级 | 问题 | 建议动作 |
|--------|------|---------|
| **P1** | 跨 Tab 跳转机制 | 明确 PersonalView → ChatView.startGoalPlanning() 的通信方案（推荐 NotificationCenter 或 TabView 层共享 action） |
| **P1** | 追问消息持久化策略 | 明确是否写入 ChatMessage，如果写入需要加 messageType 字段或取消时软删除 |
| **P1** | GoalPlanningCoordinator 内部实现 | 补充 LLM 调用次数、context 传递方式、何时停止追问的规则 |
| **P1** | fullScreenCover 返回手势与状态机 | 明确滑动返回时的 session 处理策略 |
| **P2** | frequency String 映射 | 在 prompt 中约束格式或在确认看板用 Picker |
| **P2** | 删除规则配置注意 | 实施时在 code review 中重点检查 nullifyDeleteRule 放置侧 |
| **P2** | Phase 1 验证 | 补充 debug 创建入口或注明 Phase 1+2 必须一起交付 |

### 总体结论（第二轮）

第一轮的 3 个 P0 架构问题已全部解决。方案现在的状态是**可以进入实施**，但上面 4 个 P1 问题建议在写代码前补齐，特别是 #3（GoalPlanningCoordinator 内部实现）——它是整个 AI 规划流程的核心，如果到实施时才设计 prompt 和调用链路，大概率会返工。

本轮修订针对下方对抗性审查补齐工程落地细节：

- 明确 `QuickAction.planGoal` 和个人目标空状态不走普通 `sendMessage()` 执行动作链路，而是进入 `ChatViewModel.startGoalPlanning()`。
- 明确 `ChatViewModel` 持有 `activeGoalPlanningSession`，并在 session 存在时把用户后续输入交给 `GoalPlanningCoordinator`。
- 将 `goalId: UUID?` 裸字段改为 Core Data to-one 关系：`TodoTask.goal`、`Habit.goal`，Goal 侧反向关系使用 `nullifyDeleteRule`。
- 明确确认看板采用 `fullScreenCover + NavigationStack`，保存和取消流程进入状态机。
- 合并原 Phase 2 和 Phase 3，保证「AI 追问 -> GoalDraft -> 确认看板 -> 写入」是一个可验证闭环。
- 移除固定 `50/50` 综合进度公式，改为展示原始指标和启发式粗粒度状态。
- 补充 AI 上下文注入的 token 控制策略：普通聊天不无条件注入全部授权目标。

## Claude 对抗性审查（2026-05-17）

### 总体评价

方案的产品定位清晰，边界意识好（"不做项目管理"反复强调），分阶段策略合理。但架构层面存在几个**硬伤**，不解决会直接卡住实施。

---

### 一、架构级硬伤（P0）

#### 1. GoalPlanningCoordinator 的路由入口是个黑箱

方案写的是：

```
目标规划：ChatViewModel -> GoalPlanningCoordinator -> GoalPlanningProvider -> GoalDraftBoard
```

但回避了最关键的问题：**ChatViewModel 怎么知道该走 GoalPlanningCoordinator 而不是 ConversationCoordinator？**

现有架构中，QuickAction 的实现就是把一段文本塞进 `inputText`，然后走 `sendMessage()` → `ConversationCoordinator.process()`。整个管道是无状态的，每条消息独立解析。方案需要回答：

- 方案 A：在 `sendMessage()` 入口处判断文本/意图，分流到不同 Coordinator？那判断逻辑写在哪？
- 方案 B：在 ConversationCoordinator 内部拦截 `plan_goal` 意图再转交？那 ConversationCoordinator 就不再是通用编排器了。
- 方案 C：QuickAction 不走 sendMessage()，直接调用 GoalPlanningCoordinator？那 ChatView 里会出现两条完全不同的消息处理路径。

这不仅是"实现细节"，它决定了 ChatViewModel 的状态机是否需要重构。**方案对此一字未提。**

#### 2. 多轮追问机制与现有无状态管道冲突

方案提出最多 3 轮追问，需要 `GoalPlanningSession` 维护状态（`turnCount`、`answers`、`status`）。但当前 ConversationCoordinator 是**完全无状态**的——每次 `process()` 调用都是独立的 LLM 请求 + 路由执行。

多轮追问意味着：

- ChatViewModel 需要持有一个 `GoalPlanningSession?`
- 后续消息需要识别"这是在回答目标追问"而非"这是一条新消息"
- 会话状态需要和聊天历史 UI 协调（追问和回答要在聊天流中自然出现）

方案说"可以先保存在内存中"、"App 被杀可以丢弃"，这是对复杂度的回避而非解决。**多轮状态管理是整个方案最难的工程问题，但方案把它当作了实现细节。**

#### 3. `goalId: UUID?` 裸字段 vs Core Data 关系 — 选了更难的路径

方案选择在 TodoTask 和 Habit 上加 `goalId: UUID?` 裸字段，而非使用 Core Data 的 `NSRelationshipDescription`。理由是"避免进度重复计算、解除关联复杂化"。

但这个选择的代价没被充分评估：

| 问题 | 裸 UUID | Core Data 关系 |
|------|---------|---------------|
| 删除 Goal 时清空 goalId | **必须手动遍历所有 Task/Habit** 写查询 + 批量更新 | 设 `nullifyDeleteRule` 自动处理 |
| 查询"某个 Goal 下的所有 Task" | `NSPredicate(format: "goalId == %@", goalId)` 全表扫描 | `goal.tasks` 直接访问，Core Data 延迟加载 |
| Goal 详情页显示关联任务 | 需要 2 次独立 fetch | 一次关系访问 |
| 反向关系（Task 知道自己的 Goal） | 需要额外查询 Goal 表 | Core Data 自动维护反向指针 |
| 数据一致性 | Goal 已删但 goalId 残留 → 孤儿引用 | Core Data 级别保证 |

现有代码中，TodoTask → TodoList 就是用 Core Data 关系（`list` 属性，to-one），而不是裸 UUID。**方案选择了一条与项目现有模式不一致、且需要更多手工维护的路径，却没有给出充分的理由。**

如果担心多对多复杂度，可以用 to-one 关系来规避，而不是退回裸字段。

---

### 二、产品/设计层问题（P1）

#### 4. "精简/完整模式" 是一个不必要的决策点

用户还没看到草案就要选"精简"还是"完整"。**用户在做选择时缺乏判断依据。** 建议直接去掉这个选择，让 AI 根据追问中获得的信息量自动决定输出密度。v1 本身就强调"范围克制"，加一个模式选择与这个原则矛盾。

#### 5. 确认看板的 UI 形态完全缺失

方案详细描述了确认看板的功能（编辑标题、勾选任务、编辑习惯等），但**完全没有说明它在 UI 上是什么形态**：

- 是 fullScreenCover？（现有 CLAUDE.md 说要加 `.swipeBackToDismiss`）
- 是 sheet？
- 是 push navigation？
- 还是聊天流中的一张特殊卡片？

不同形态直接影响：编辑体验、手势行为、返回逻辑、状态恢复。这不是"UI 细节"，是核心交互流程的一部分。

#### 6. 进度模型的 50/50 权重不靠谱

```
综合分数 = 任务完成率 * 0.5 + 习惯完成率 * 0.5
```

问题：

- 用户给"减脂 5kg"创建了 1 个任务、5 个习惯。50/50 意味着 1 个任务完成就占 50% 进度，5 个习惯合力才占另外 50%。
- 习惯频率不同（每日 vs 每周），但计算时用的是统一的"近 14 天完成率"。一个每周习惯在 14 天内只有 2 个数据点，统计意义很弱。
- 用户新增任务时，完成率立刻下降。这不符合直觉。

建议：v1 只做"展示原始数据"（任务 X/Y 已完成，习惯近 14 天 Z%），不计算综合分数。粗粒度状态可以基于启发式规则（"有推进 + 习惯稳定 = 稳定推进"），而非数值阈值。

---

### 三、遗漏和风险（P2）

#### 7. AI Prompt 注入目标上下文的 token 成本未评估

UserContextBuilder 已经组装了 7 个子上下文。方案要求再注入最多 3 个 active Goal（每个含标题、状态、任务摘要、习惯摘要、说明）。估算每个 Goal 约 100-150 tokens，3 个就是 300-450 tokens。

**每条聊天消息**都会携带这些 tokens。按当前架构（每条消息都调 LLM），这意味着持续的成本增加。方案没有评估这个影响，也没有考虑是否只在目标相关对话中注入。

#### 8. Core Data 轻量迁移的边界条件

方案说给 TodoTask 和 Habit 加可选字段，走轻量迁移。这本身没问题。但：

- 如果后续 v2 要把裸 `goalId` 改为 Core Data 关系，**轻量迁移无法处理这种结构性变更**，需要手动写 mapping model。
- 第一次就做对关系设计，可以避免后续痛苦的数据迁移。

#### 9. 删除 Goal 的实现被低估

方案要求"删除 Goal 时清空所有关联 Task/Habit 的 goalId"。如果用裸 UUID：

```swift
let batchUpdate = NSBatchUpdateRequest(entityName: "TodoTask")
batchUpdate.predicate = NSPredicate(format: "goalId == %@", goalId)
batchUpdate.propertiesToUpdate = ["goalId": NSNull()]
```

`NSBatchUpdateRequest` 不触发 `NSManagedObjectContext` 的自动通知，需要手动 `refreshAllObjects()` — 但开发规范明确写了 **"禁止 `refreshAllObjects()`，改为重新 fetch"**。这意味着删除 Goal 后，已经在内存中的 Task/Habit 对象的 goalId 不会自动更新，需要在相关视图做手动刷新。方案没有考虑这个实现路径。

#### 10. Phase 2 和 Phase 3 不应该分开

Phase 2（HoloAI 目标规划流程）输出 GoalDraft，Phase 3（确认看板）消费 GoalDraft。没有 Phase 3，Phase 2 的输出无处可去。这两个阶段应该合并为一个可交付的整体。分开写容易导致中间态无法验证。

#### 11. 缺少对现有 HoloAI 意图的冲突分析

方案新增 `plan_goal`、`continue_goal_planning`、`confirm_goal_plan` 三个意图。但现有 `AIIntent` 枚举有 16 个 case，`AIParseBatch` 的 prompt 已经引导 LLM 识别这些意图。**新增 3 个意图意味着要修改后端 Prompt（defaultPrompts.json / managedPrompts.json），以及 iOS 端的意图解析 prompt。** 方案对此只字未提。

特别地，用户在普通聊天中说"我想学习 SwiftUI"，现有意图系统可能把它解析为 `unknown` 或 `query`。加了 `plan_goal` 后，LLM 需要区分"闲聊一个想法"和"真的要规划目标"。这个边界比看起来模糊得多。

---

### 四、值得肯定的设计决策

| 决策 | 为什么对 |
|------|---------|
| 只能由 HoloAI 创建，不支持手动 | 避免 v1 做两套创建 UI，降低复杂度 |
| 单归属（Task/Habit → Goal 是 to-one） | 符合 v1 克制原则，多归属是 v2 的事 |
| 删除 Goal 不删 Task/Habit | 保护用户生活数据，这是正确的安全策略 |
| 粗粒度状态而非精确百分比 | 避免虚假精确，降低实现复杂度 |
| GoalPlanningCoordinator 与 ConversationCoordinator 分离 | 正确的架构直觉，多轮状态不应该污染通用管道 |
| allowAIContext 独立授权 | 隐私意识好，用户可控 |

### 六、总结

方案的产品侧思考扎实，但工程落地有 3 个 P0 级架构问题没回答——多轮状态管理、路由分流机制、数据关系设计。建议先把这三个问题补清楚再进入实施。
