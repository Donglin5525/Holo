# HoloAI 能力启动台与记忆层方案

> 日期：2026-05-24
> 范围：HoloAI 对话框输入区上方能力 Tab、短期记忆、长期记忆、记忆确认与来源透明。不引入 Agent 模式。
> V2 修订：吸收对抗性审查意见，明确短期记忆门面设计、Prompt 注入路径、Token 预算、长期记忆优先级和 Phase 顺序。

## Goal

把 HoloAI 从“用户自己想一句话问 AI”的聊天入口，升级为“引导用户理解自己、形成行动、持续被记住”的生活观察入口。

用户最终感受到的不是 HoloAI 多了几个快捷按钮，而是：

> Holo 知道我现在处在什么状态，能看见我最近的变化，也能在我确认后记住长期偏好和反复模式，帮助我更好地使用自己的记录。

本方案只做两件事：

1. **能力启动台**：替换当前低价值快捷按钮，展示 HoloAI 最值得使用的能力入口。
2. **Memory Layer**：工程化短期记忆和长期记忆，明确数据来源、晋升规则、确认策略和撤销能力。

本方案不引入 Agent。所有入口仍走确定性工作流，由本地规则和已有 AI 链路编排。

## Product Positioning

Holo 的核心价值不是“记账、创建任务、记录心情、习惯打卡”本身。这些是数据输入能力，是 Holo 理解用户的原材料。

HoloAI 的核心价值应表达为：

| 层级 | 价值 | 用户感知 |
|------|------|----------|
| 记录 | 快速把生活事件结构化 | “我不用打开多个页面录入” |
| 状态 | 理解今天和最近发生了什么 | “它知道我现在该关注什么” |
| 模式 | 发现反复出现的长期规律 | “它不是只看今天，它记得我的节奏” |
| 行动 | 把洞察变成可执行调整 | “它能帮我把下一步安排好” |

因此，输入框上方 Tab 不应再长期展示基础录入入口。它应该展示 HoloAI 对用户更有价值的能力：

- 新手引导
- 今日状态
- 最近分析
- 长期模式
- 规划目标

## Non-Goals

本轮明确不做：

| 不做 | 原因 |
|------|------|
| 不做“更多”入口 | 原来的记账、任务、心情、习惯打卡不应占据核心位置，否则启动台会退回功能抽屉 |
| 不做 Agent 自主工具调用 | 当前目标可以用确定性工作流实现，Agent 会显著增加不可控复杂度 |
| 不自动创建任务/习惯/预算 | 行动落地必须由用户确认 |
| 不把所有原始数据写入长期记忆 | 长期记忆只存稳定偏好、反复模式、重要事实和用户确认内容 |
| 不做向量数据库 | V1 用结构化 JSON 存记忆，并通过证据引用关联现有 Core Data 源实体，避免过早引入基础设施 |
| 不做医疗/心理诊断 | 健康和情绪只作为生活信号，不做诊断或人格判断 |
| 不让 AI 自由改写用户画像 | 长期记忆必须经过规则判断和用户可控机制 |

## Current Context

当前 HoloAI 已经具备本方案的基础能力：

| 能力 | 现状 |
|------|------|
| 对话编排 | `ConversationCoordinator` 负责意图解析、查询/操作分流、顺序执行 |
| 当前上下文 | `UserContextBuilder` 聚合今日消费、习惯、任务、想法、账户、个人档案、近期趋势、目标上下文 |
| 分析上下文 | `AnalysisContextBuilder` 支持 finance / habit / task / thought / health / goal / crossModule |
| 记忆洞察 | `MemoryInsightService` 生成周期洞察并保存结构化卡片 |
| 今日状态 | `DailySenseStateBuilder` 已有 stable / atRisk / recovering 规则引擎 |
| 洞察偏好 | `InsightPreferenceProfileService` + `InsightFeedbackAggregator` 已能聚合洞察反馈偏好 |
| 目标规划 | `GoalPlanningCoordinator` 已支持追问、草案生成、确认保存 |
| 基础快捷栏 | `QuickActionBar` + `QuickAction` 当前展示记账、任务、心情、打卡等低价值入口 |

主要缺口：

| 缺口 | 影响 |
|------|------|
| 能力入口仍是基础 CRUD | 用户不知道 HoloAI 的高阶能力是什么 |
| 短期/长期记忆没有统一抽象 | 近期状态、目标、偏好、洞察散落在不同 service |
| 长期记忆缺少晋升规则 | 容易把单次事件、AI 猜测、敏感推断错误固化 |
| 记忆缺少确认与撤销机制 | 用户会不信任“AI 记住了我” |
| 今日状态缺少低数据兜底 | 没有任务/习惯/记录时容易生成假洞察 |

## Target UX

### 1. 启动台入口

HoloAI 输入框上方展示能力入口，而不是基础操作按钮：

```text
新手引导 | 今日状态 | 最近分析 | 长期模式 | 规划目标
```

入口展示规则：

| 入口 | 展示条件 | 点击行为 |
|------|----------|----------|
| 新手引导 | 新用户、低数据用户、未完成基础使用闭环 | 展示 HoloAI 基础功能最佳实践，不直接替代核心入口 |
| 今日状态 | 默认常驻 | 根据数据充足度生成状态、有限结论或引导补充上下文 |
| 最近分析 | 默认常驻 | 默认分析最近 7 天，可切换维度 |
| 长期模式 | 有长期记忆或候选记忆时优先展示；无长期记忆时展示“开始形成模式” | 展示已确认长期记忆、候选模式和证据 |
| 规划目标 | 默认常驻 | 复用现有目标规划链路，后续接入记忆上下文 |

不提供“更多”。基础录入能力放到新手引导中教学，成熟用户通过自然语言直接使用。

### 2. 新手引导

新手引导的目标不是展示功能列表，而是告诉用户怎样输入才能让 Holo 更好地理解自己。

建议内容：

| 基础能力 | 最佳实践文案 |
|----------|--------------|
| 记账 | “直接说：午饭 35、妈妈红包 1666。Holo 会识别金额、收支、分类和备注。” |
| 创建任务 | “直接说：明天提醒我买牛奶，也可以带优先级、截止时间和子任务。” |
| 记录心情 | “比起只说开心/难过，更建议说：今天有点烦，主要是工作节奏乱。” |
| 习惯打卡 | “直接说：打卡跑步，或今天没喝奶茶。Holo 会区分正向习惯和控制型习惯。” |
| 多件事 | “一句话可以处理多件记录，但查询和执行最好分开发。” |

展示策略：

- 新用户默认展示。
- 当用户完成至少 3 类基础输入，或累计有效记录达到阈值后，从启动台隐藏；用户仍可在 HoloAI 设置中重新打开。
- 隐藏判断放在 `HoloAICapabilityProvider` 或 `ChatViewModel`，不要放进纯模型 `HoloAICapability`。
- “3 类基础输入”指财务、任务、习惯、想法/心情四类中的任意三类；“累计有效记录”V1 建议阈值为 10 条跨模块记录，后续再用真实数据调参。
- 成熟用户不展示“更多”，但输入框 placeholder 可以轮播基础示例，例如“试试：打卡跑步”“试试：午饭 35”。

### 3. 今日状态兜底

今日状态不能在数据不足时假装有洞察。V1 必须分三档。

#### 3.1 Rich: 数据充足

触发条件建议满足任意两类：

- 今日有任务或逾期任务。
- 今日有习惯记录或活跃习惯。
- 今日有消费/收入记录。
- 最近 7 天有可比较数据。
- 有活跃目标。
- 有健康信号。

输出：

- 今日风险或恢复迹象。
- 最值得关注的 1-3 件事。
- 可确认行动，例如“把这个整理成今天计划”。

#### 3.2 Partial: 部分数据

触发条件：

- 只有 1 类数据可用，或近期数据不足以做跨域判断。

输出原则：

- 明确说明参考范围有限。
- 只对已有数据做有限结论。
- 引导用户补充缺失上下文。

示例：

```text
我今天只看到消费记录，还没有任务和习惯数据。
目前只能从消费节奏看：今天已有 3 笔支出，晚些时候我可以在你补充任务或心情后再帮你判断整体状态。
```

#### 3.3 Empty: 几乎无数据

触发条件：

- 今日无任务、无习惯、无交易、无健康信号，且近期数据极少。

输出原则：

- 不做状态判断。
- 把今日状态降级为“建立今日上下文”。

示例入口：

- 今天最重要的一件事是什么？
- 今天想关注消费、任务、健康还是心情？
- 要不要我帮你规划今天？

示例回复：

```text
今天还没有足够记录，我先不判断你的状态。
你可以告诉我今天最重要的一件事，或者直接让我帮你规划今天。
```

## Memory Layer

### 1. 记忆分层

Holo 的记忆分为三层：

| 层级 | 名称 | 是否持久 | 是否需要用户确认 | 用途 |
|------|------|----------|------------------|------|
| L0 | 原始数据 | 持久，已有业务数据 | 用户创建数据时已确认 | 财务、任务、习惯、想法、健康等事实来源 |
| L1 | 短期记忆 | 当前流程内临时生成，可重算，会过期 | 不需要逐条确认 | 理解最近状态和当前上下文 |
| L2 | 长期记忆 | 持久保存 | 分级确认 | 记住稳定偏好、长期目标、反复模式、重要事实 |

关键原则：

1. **短期记忆不是长期画像**。它是近期状态摘要，默认自动生成、自动过期。
2. **长期记忆不是原始数据备份**。它只保存对未来有持续指导价值的结构化结论。
3. **AI 猜测不能直接成为长期记忆**。必须有证据、规则、确认或稳定反馈。
4. **所有长期记忆必须可查看、可删除、可纠正**。

### 2. 短期记忆

短期记忆回答的是：

> 最近发生了什么？现在有什么上下文？

#### 2.1 数据来源

短期记忆从现有业务数据和上下文构建器生成，但 **V1 不新建一套并行聚合逻辑**。

`HoloMemorySnapshotBuilder` 的职责是门面（facade），不是新的数据仓库：

```text
HoloMemorySnapshotBuilder
  -> 调用 UserContextBuilder.buildContext() 获取已有上下文
  -> 调用 DailySenseStateBuilder.buildToday() 获取今日状态
  -> 读取 ChatMessageRepository 最近对话摘要
  -> 读取 HoloLongTermMemoryStore 中相关长期记忆
  -> 计算 dataCoverage / evidenceSummary
  -> 输出 HoloMemorySnapshot
```

禁止在 `HoloMemorySnapshotBuilder` 里重新实现财务、任务、习惯、目标的完整聚合逻辑。已有字段继续以 `UserContextBuilder` 为准，避免双数据源。

| 来源 | 数据 |
|------|------|
| `UserContextBuilder` | 今日消费、今日任务、习惯完成、最近想法、账户、个人档案、近期趋势、目标上下文 |
| `DailySenseStateBuilder` | 今日 stable / atRisk / recovering、原因、置信度 |
| `AnalysisContextBuilder` | 最近 7/14/30 天各领域分析上下文 |
| `MemoryInsightRepository` | 最近周期洞察摘要和异常 |
| `GoalRepository` | 当前活跃目标、目标进度、关联任务/习惯 |
| `ChatMessageRepository` | 最近一次 HoloAI 对话意图和未完成规划 session |

#### 2.2 短期记忆类型

```swift
struct HoloShortTermMemory: Codable, Equatable {
    var generatedAt: Date
    var window: HoloMemoryWindow
    var dataCoverage: HoloMemoryDataCoverage
    var todayState: HoloTodayStateSummary?
    var recentSignals: [HoloMemorySignal]
    var activeGoals: [HoloGoalMemorySummary]
    var recentConversation: HoloConversationMemorySummary?
}

enum HoloMemoryWindow: String, Codable {
    case today
    case sevenDays
    case fourteenDays
    case thirtyDays
}

struct HoloMemoryDataCoverage: Codable, Equatable {
    var transactionCount: Int
    var taskCount: Int
    var activeHabitCount: Int
    var habitRecordCount: Int
    var thoughtCount: Int
    var healthSignalCount: Int
    var activeGoalCount: Int
    var hasRecentInsight: Bool
    var coverageLevel: HoloMemoryCoverageLevel
}

enum HoloMemoryCoverageLevel: String, Codable {
    case empty
    case partial
    case enoughForSingleDomain
    case enoughForCrossDomain
}
```

#### 2.3 获取规则

短期记忆由 `HoloMemorySnapshotBuilder` 每次进入 HoloAI 或点击能力入口时生成：

```text
进入 HoloAI
  -> HoloMemorySnapshotBuilder.build()
  -> 拉取今日状态、近期趋势、活跃目标、近期对话
  -> 生成 dataCoverage
  -> 返回 shortTerm snapshot
```

各窗口的获取口径：

| 窗口 | 获取内容 | 用途 |
|------|----------|------|
| today | 今日任务、今日交易、今日习惯记录、今日健康信号、今日目标提醒 | 今日状态、低数据兜底 |
| sevenDays | 最近 7 天交易、习惯表现、任务完成、想法/心情、健康趋势 | 最近分析默认窗口 |
| fourteenDays | 最近 14 天同类趋势和轻量对比 | 判断“是否只是偶发” |
| thirtyDays | 最近 30 天周期洞察、目标进展、重复异常 | 长期记忆候选的证据补充 |

短期记忆获取必须满足三条约束：

1. **只聚合，不推断人格**：短期记忆可以说“最近 7 天外卖支出增加”，不能说“用户自控力差”。
2. **每条 signal 都带来源**：例如来自交易、习惯记录、目标、洞察卡，而不是一段无来源总结。
3. **低覆盖度时降级**：`coverageLevel` 为 `.empty` 或 `.partial` 时，只能输出有限结论或引导补充上下文。

`coverageLevel` 建议规则：

| coverageLevel | 条件 |
|---------------|------|
| empty | 今日无任务、无交易、无习惯记录、无健康信号，且最近 7 天核心记录总量 < 3，活跃目标 = 0 |
| partial | 只有 1 个领域有数据，或最近 7 天核心记录总量在 3-5 之间但不足以做趋势判断 |
| enoughForSingleDomain | 单领域达到最小分析量：交易 >= 3 笔、任务 >= 2 个、习惯记录 >= 3 条、想法/心情 >= 2 条、健康信号 >= 2 天，满足任一 |
| enoughForCrossDomain | 至少 2 个领域达到最小分析量，或 1 个领域达到最小分析量且存在活跃目标 |

缓存策略：

- V1 不做跨会话缓存，也不做 5/30 分钟 service 级缓存。
- 一次用户点击流程内可以复用同一个 `HoloMemorySnapshot`，避免同一次生成重复查询。
- 如果后续性能不够，再基于明确的 Repository 变更通知或 Core Data object change 设计缓存失效机制。
- 短期记忆必须可重算，不能成为新的事实来源。

#### 2.4 过期规则

短期记忆不需要用户删除，因为它会自然过期：

| 类型 | 过期 |
|------|------|
| 今日状态 | 次日自动过期 |
| 最近 7/14/30 天摘要 | 滑动窗口自动更新 |
| 最近对话意图 | 默认保留最近 1 次或 24 小时 |
| 未完成目标规划 session | 按现有 session 状态保留 |

### 3. 长期记忆

长期记忆回答的是：

> 哪些信息值得 Holo 在未来继续参考？

长期记忆分为五类：

| 类型 | 说明 | 例子 |
|------|------|------|
| explicitPreference | 用户明确表达的偏好 | “以后建议直接给我行动清单” |
| stableFeedbackPreference | 多次反馈形成的偏好 | 连续两次标记“建议太泛” |
| recurringPattern | 跨周期重复出现的模式 | 睡眠不足后习惯完成下降 |
| longTermGoalContext | 长期目标相关背景 | 正在进行体能恢复、财务控制 |
| importantLifeFact | 用户确认的重要事实 | 搬家、换工作、备考、家庭安排变化 |

#### 3.1 长期记忆数据结构

```swift
struct HoloLongTermMemoryRecord: Codable, Identifiable, Equatable {
    let id: UUID
    var type: HoloLongTermMemoryType
    var title: String
    var summary: String
    var confidence: Double
    var sensitivity: HoloMemorySensitivity
    var status: HoloLongTermMemoryStatus
    var evidence: [HoloMemoryEvidence]
    var createdAt: Date
    var updatedAt: Date
    var lastUsedAt: Date?
    var expiresAt: Date?
    var userEditedSummary: String?
}

enum HoloLongTermMemoryStatus: String, Codable {
    case candidate
    case confirmed
    case silentlyAccepted
    case rejected
    case archived
}

enum HoloMemorySensitivity: String, Codable {
    case low
    case medium
    case high
}

struct HoloMemoryEvidence: Codable, Equatable {
    var sourceType: HoloMemoryEvidenceSourceType
    var sourceId: String?
    var sourceDate: Date?
    var excerpt: String
    var weight: Double
}
```

长期记忆必须有证据。没有证据的内容只能停留在候选或直接丢弃。

#### 3.2 候选生成来源

长期记忆不是从一个地方写入，而是由多个候选来源进入统一候选池：

| 来源 | 生成什么候选 |
|------|--------------|
| 用户明确表达 | explicitPreference、importantLifeFact |
| HoloProfile | longTermGoalContext、explicitPreference |
| 洞察反馈聚合 | stableFeedbackPreference |
| 周/月 MemoryInsight | recurringPattern、importantLifeFact |
| AnalysisContext 周期对比 | recurringPattern |
| GoalPlanning 保存结果 | longTermGoalContext |
| 行动闭环确认 | explicitPreference、longTermGoalContext |

统一流程：

```text
Candidate Source
  -> HoloMemoryCandidate
  -> HoloMemoryPromotionPolicy.evaluate()
  -> drop / observeMore / silentlyAccept / requireConfirmation
  -> HoloLongTermMemoryStore
```

候选模型：

```swift
struct HoloMemoryCandidate: Codable, Identifiable, Equatable {
    let id: UUID
    var proposedType: HoloLongTermMemoryType
    var title: String
    var summary: String
    var confidence: Double
    var sensitivity: HoloMemorySensitivity
    var evidence: [HoloMemoryEvidence]
    var source: HoloMemoryCandidateSource
    var firstObservedAt: Date
    var lastObservedAt: Date
    var occurrenceCount: Int
}

enum HoloMemoryCandidateSource: String, Codable {
    case userExplicitText
    case profile
    case insightFeedback
    case memoryInsight
    case analysisComparison
    case goalPlanning
    case actionConfirmation
}

enum HoloMemoryPromotionDecision: String, Codable {
    case drop
    case observeMore
    case silentlyAccept
    case requireConfirmation
}
```

候选合并规则：

| 场景 | 处理 |
|------|------|
| 新候选与已确认记忆语义相同 | 更新证据、`updatedAt`、`confidence`，不新建重复记忆 |
| 新候选与已拒绝记忆相同 | 30 天内丢弃，30 天后如果证据显著增强才重新进入候选 |
| 新候选与已有候选相同 | 合并证据，`occurrenceCount += 1` |
| 新候选与已确认记忆冲突 | 降为 `requireConfirmation`，由用户决定是否替换 |
| 新候选与 HoloProfile 内容重复 | 不新建长期记忆，只记录为 profile-backed evidence |
| 新候选与 HoloProfile 内容冲突 | 降为 `requireConfirmation`，不得自动覆盖用户档案 |
| 候选证据不足 | `observeMore`，不展示给用户 |

### 4. 什么情况下能存为长期记忆

长期记忆写入必须满足“类型 + 证据 + 风险 + 用户意图”四组规则。

#### 4.1 可以直接成为候选的内容

以下内容可以生成长期记忆候选：

| 内容 | 条件 |
|------|------|
| 明确偏好 | 用户直接说“以后/记住/我更喜欢/不要再” |
| 稳定反馈 | 30 天内同类反馈 >= 2 次 |
| 重复模式 | 至少跨 2 个周期出现，且每次有结构化证据 |
| 长期目标背景 | 用户创建或确认目标后生成 |
| 重要事件 | 用户明确记录，或在回顾中确认 |
| 行动偏好 | 用户多次确认同一类行动建议 |

晋升决策建议：

```text
if evidence.isEmpty:
    drop

if candidate.sensitivity == high:
    requireConfirmation

if candidate conflicts with HoloProfile explicit content:
    requireConfirmation

if candidate.confidence < 0.65:
    observeMore

if candidate.source == profile:
    silentlyAccept

if candidate.source == userExplicitText && text contains "记住":
    silentlyAccept

if candidate.proposedType == importantLifeFact:
    requireConfirmation

if candidate.proposedType == recurringPattern:
    if occurrenceCount >= 2 && evidence covers at least 2 periods:
        requireConfirmation
    else:
        observeMore

if candidate.proposedType == stableFeedbackPreference:
    if occurrenceCount >= 2 within 30 days:
        silentlyAccept
    else:
        observeMore

otherwise:
    requireConfirmation
```

优先级说明：

1. **高敏感优先于“记住”关键词**。用户说“记住我压力大时更爱外卖”仍然需要确认，因为它会固化敏感行为模式。
2. **证据和置信度先于类型判断**。没有证据直接丢弃，低置信度只观察。
3. **HoloProfile 是最高权重的用户主动档案**。长期记忆不得自动覆盖 HoloProfile；发生冲突时必须让用户确认。
4. **稳定反馈偏好不直接复制一份长期记忆**。它先进入 `InsightPreferenceProfile`，长期记忆只引用其中已稳定、可解释的摘要。

阈值说明：

| 阈值 | 建议值 | 说明 |
|------|--------|------|
| 最低候选置信度 | 0.65 | 低于此值不进入用户可见候选 |
| 重复模式最小次数 | 2 个周期 | 避免把偶发事件写成模式 |
| 稳定反馈最小次数 | 30 天内 2 次 | 与现有洞察偏好聚合阈值保持一致 |
| Chat/目标规划长期记忆注入数量 | 最多 5 条 | 控制上下文污染 |
| MemoryInsight 长期记忆注入数量 | 最多 1-2 条压缩偏好 | 避免挤占洞察上下文预算 |
| rejected 冷却期 | 30 天 | 尊重用户拒绝，避免反复提醒 |

#### 4.2 不能写入长期记忆的内容

| 内容 | 原因 |
|------|------|
| 单笔消费 | 原始事实，不是长期结论 |
| 单个普通任务 | 生命周期短，不应污染长期画像 |
| 一次心情波动 | 敏感且不稳定 |
| 一次健康异常 | 不能做长期健康判断 |
| AI 单次推断 | 缺少稳定证据 |
| 用户拒绝过的候选 | 尊重用户选择 |
| 低置信度洞察 | 避免错误固化 |
| 敏感身份/价值观猜测 | 必须用户明确提供或确认 |

#### 4.3 需要用户确认的长期记忆

以下内容必须显示确认卡，不能静默写入：

| 类型 | 例子 | 原因 |
|------|------|------|
| 敏感推断 | “你压力大时更容易外卖增加” | 涉及心理/健康/财务压力 |
| 行为模式推断 | “你睡眠不足后习惯完成率下降” | 虽有证据，但仍是解释性结论 |
| 价值观/身份 | “你更重视家庭陪伴” | 高影响、高敏感 |
| 长期目标方向 | “体能恢复是你的长期目标” | 会影响未来建议 |
| AI 推断出的建议风格偏好 | “你似乎更希望我少安慰，多给任务清单” | 由 AI 推断，会长期影响 AI 行为 |

确认 UI 文案：

```text
我观察到一个可能的长期模式：
你在睡眠不足后的第二天，更容易减少运动并增加外卖。这个模式出现过 3 次。

要让我记住吗？之后做计划时我会参考它。

[记住] [不记住] [调整一下]
```

#### 4.4 可以静默写入的长期记忆

以下内容可以不弹强确认，但必须出现在“记忆记录”中，允许撤销：

| 类型 | 条件 |
|------|------|
| 用户明确说“记住……” | 用户已经表达写入意图 |
| 用户在档案/设置中填写 | 用户主动配置 |
| 稳定反馈偏好 | 连续多次“这有用/没用/建议太泛” |
| 明确选择的偏好 | 用户点选“以后按行动清单给我”，或明确说“以后少安慰，多给任务清单” |

静默写入后可在 HoloAI 中提示：

```text
已记住：你更喜欢具体行动清单，而不是泛泛建议。
```

#### 4.5 继续观察，不写入

以下情况进入 observeMore：

- 同类现象只出现 1 次。
- 跨域关系证据不足。
- 置信度中等但不敏感。
- 与长期目标可能相关，但目标尚未确认。

observeMore 不展示给用户，最多用于下次候选评分。

### 5. 记忆确认策略

建议采用分级确认，而不是全部手动确认或全部自动写入。

| 层级 | 策略 | 用户体验 |
|------|------|----------|
| 短期记忆 | 自动生成，不确认 | 回答底部显示来源 |
| 低风险明确长期记忆 | 静默写入，可撤销 | “已记住...”轻提示 |
| 高影响/敏感/推断型长期记忆 | 必须确认 | 记忆候选卡 |
| 证据不足 | 不写入，继续观察 | 不打扰用户 |

这是 Holo 相比“全部手动确认”的折中：

- 不会每条都打扰用户。
- 不会偷偷记住敏感推断。
- 用户能理解 Holo 为什么记住。
- 长期记忆能持续积累，而不是全靠用户主动维护。

### 6. 记忆使用规则

长期记忆进入 Prompt 前必须压缩为短摘要，不能把所有证据原文塞进 Prompt。

```text
HoloMemorySnapshot
  - shortTerm: 当前状态摘要
  - longTerm: 最多 5 条相关长期记忆
  - evidenceSummary: 来源摘要
```

选择长期记忆时按相关性排序：

1. 与当前入口相关，例如“规划目标”优先目标和执行偏好。
2. 最近使用或最近确认。
3. 置信度高。
4. 用户未拒绝。
5. 敏感记忆只在必要场景使用，并避免直接暴露。

回答底部展示来源：

```text
基于：最近 7 天记录 · 当前目标 · 已确认长期偏好
```

点开来源面板展示：

- 参考了哪些短期数据。
- 参考了哪些长期记忆。
- 每条长期记忆的证据。
- 删除/纠正入口。

### 7. Prompt 注入策略与 Token 预算

记忆不能无差别注入所有 Prompt。不同链路的注入策略不同：

| 链路 | 是否注入长期记忆 | 策略 |
|------|------------------|------|
| `systemPrompt` / 普通 Chat | 可以 | 注入最多 5 条与当前问题相关的短摘要，不带完整证据 |
| `intentRecognition` | 默认不注入 | 只允许极短消歧上下文，例如当前活跃目标名称；不得注入行为模式 |
| `queryAnalysis` | 可以 | 通过 `AnalysisContext` 或附加 memory summary 注入最多 3 条相关记忆 |
| `memoryInsightGeneration` | 谨慎 | 默认不注入长期记忆；如需要，只注入 1-2 条稳定偏好摘要 |
| `GoalPlanningPromptBuilder` | 可以 | 注入目标相关长期记忆、执行偏好、已确认长期模式 |

Token 预算：

| 链路 | 预算建议 |
|------|----------|
| 普通 Chat | `memorySummary` 建议控制在 500-700 中文字符以内 |
| 目标规划 | 建议控制在 600 中文字符以内 |
| 分析查询 | 建议控制在 400 中文字符以内 |
| MemoryInsight daily | 建议不注入，或最多 120 中文字符 |
| MemoryInsight weekly/monthly | 最多 200 中文字符，并从上下文预算中明确扣除 |

实现要求：

- `PromptManager.renderTemplate` 当前只替换 `{{todayDate}}`、`{{currentYear}}`、`{{currentTime}}`。如使用模板变量注入记忆，必须先扩展为动态变量字典，例如 `renderTemplate(_ template: String, variables: [String: String])`。
- 如果修改后端托管 Prompt，需要同步更新 iOS `PromptManager`、后端 `defaultPrompts.json`、Prompt 版本号，并提醒后端发版/部署。
- MemoryInsight 的 token 预算已经由 `MemoryInsightContextBuilder` 管控；任何 memory summary 注入都必须纳入预算估算，而不是额外拼接。

### 8. 错误处理与降级

能力入口必须有独立降级策略：

| 场景 | 降级 |
|------|------|
| AI Provider 未配置 | 今日状态走本地规则；最近分析/长期模式提示“AI 服务不可用，可先查看基础状态” |
| 网络不可用 | 本地可计算内容照常展示；需要 LLM 的内容显示重试入口 |
| Core Data 查询失败 | 展示有限错误文案，不写入长期记忆候选 |
| 记忆 JSON 损坏 | 回退空长期记忆，保留原文件备份用于调试 |
| Prompt 加载失败 | 使用本地默认模板，不阻断基础状态展示 |

### 9. 来源展示分期

来源透明分两期实现：

| 阶段 | 展示 |
|------|------|
| V1 | 消息底部短标签，例如“基于：最近 7 天记录 · 当前目标 · 已确认长期偏好” |
| V2 | 可展开来源面板，展示短期数据、长期记忆、证据、删除/纠正入口 |

V1 不要求完整来源面板，但长期记忆本身必须已经保存证据，避免后续无证据可展示。

### 1. 新增模型

建议新增：

| 文件 | 责任 |
|------|------|
| `Models/AI/HoloShortTermMemoryModels.swift` | 短期记忆、数据覆盖度、来源摘要模型 |
| `Models/AI/HoloLongTermMemoryModels.swift` | 长期记忆、候选、证据、晋升状态模型 |
| `Services/AI/HoloMemorySnapshotBuilder.swift` | 门面层：复用 `UserContextBuilder`，补 dataCoverage、recentConversation 和相关长期记忆 |
| `Services/AI/HoloMemoryPromotionPolicy.swift` | 判断候选是丢弃、观察、静默写入还是需要确认 |
| `Services/AI/HoloLongTermMemoryStore.swift` | 长期记忆持久化、查询、更新、删除 |
| `Services/AI/TodayStateResolver.swift` | 今日状态 rich/partial/empty 判断 |
| `Models/AI/HoloAICapability.swift` | 能力启动台入口模型 |

### 2. 修改现有文件

| 文件 | 修改 |
|------|------|
| `Views/Chat/QuickActionBar.swift` | 改为展示 `HoloAICapability`，不再展示基础 CRUD |
| `Views/Chat/ChatViewModel.swift` | `sendQuickAction` 改为 `handleCapabilityTap` |
| `Services/AI/UserContextBuilder.swift` | 不直接依赖 `HoloMemorySnapshot`；保持当前上下文聚合职责 |
| `Services/AI/ConversationCoordinator.swift` | `queryAnalysis` 路径可接收 memory snapshot |
| `Services/AI/PromptManager.swift` | 如采用模板变量注入，扩展 `renderTemplate` 支持动态变量字典 |
| `Services/AI/GoalPlanningPromptBuilder.swift` | 规划目标时注入相关长期记忆摘要 |
| `Views/Settings/AISettingsView.swift` 或独立设置页 | 新增“记忆管理”入口 |

### 3. 持久化选择

V1 建议长期记忆使用 JSON 文件，不立刻上 Core Data：

- 数据量小。
- 模型迭代快。
- 容易导出、调试、审查。
- 已有 `InsightPreferenceProfileService` JSON 原子写入模式可复用。

路径：

```text
Application Support/Holo/HoloLongTermMemories.json
```

后续如果需要复杂筛选、全文搜索、同步，再迁移 Core Data 或 CloudKit。

### 4. 与现有系统关系

| 现有系统 | 新 Memory Layer 的关系 |
|----------|------------------------|
| `HoloProfile` | 用户主动档案，是长期记忆候选来源之一，但不被自动改写 |
| `InsightPreferenceProfile` | 继续作为洞察排序、语气、行动候选的运行时偏好画像；长期记忆只引用稳定摘要，不重复写同源 feedback |
| `MemoryInsight` | 周/月洞察是模式候选的重要来源；V1 在洞察生成成功后做本地规则提取，不新增额外 AI 调用 |
| `DailySenseSnapshot` | 今日状态是短期记忆来源 |
| `GoalPlanningSession` | 活跃规划是短期记忆；保存后的目标是长期候选来源 |

### 5. 并发与持久化安全

`HoloLongTermMemoryStore` 必须串行化写入：

- Store 使用 `@MainActor` 或内部 serial queue 保证读-改-写原子。
- `HoloMemoryPromotionPolicy` 只做纯规则判断，不直接写文件。
- 候选合并、rejected 冷却、冲突替换必须在 Store 的单一入口执行。
- JSON 原子写入只能防文件损坏，不能替代并发控制。

## Capability Workflows

### 今日状态

```text
用户点击 今日状态
  -> TodayStateResolver.resolve()
  -> rich: 使用 DailySense + shortTerm memory 生成状态；AI 不可用时走本地规则版
  -> partial: 明确数据范围有限
  -> empty: 引导建立今日上下文
```

### 最近分析

```text
用户点击 最近分析
  -> HoloMemorySnapshotBuilder.build(window: .sevenDays)
  -> AnalysisContextBuilder.build(crossModule)
  -> 生成最近变化、风险、恢复迹象
  -> 展示来源短标签；AI 不可用时展示本地摘要和重试入口
```

### 长期模式

```text
用户点击 长期模式
  -> 读取 confirmed / silentlyAccepted 长期记忆
  -> 读取 candidate 且 requireConfirmation 的候选
  -> 展示“已记住”和“待确认”
  -> 用户可记住/拒绝/调整
```

### 规划目标

```text
用户点击 规划目标
  -> HoloMemorySnapshotBuilder.build(relevance: goalPlanning)
  -> GoalPlanningCoordinator.start()
  -> Prompt 注入：当前目标、执行偏好、已确认长期模式
  -> 草案仍需用户确认保存
```

## Phased Rollout

### Phase 1: 能力启动台 MVP

- 新增 `HoloAICapability`。
- 新增 `HoloAICapabilityProvider`，负责新手/成熟用户入口展示条件。
- 替换 `QuickAction` 展示内容。
- 去掉“更多”和基础 CRUD 常驻入口。
- 新增新手引导入口和展示规则。
- 输入框 placeholder 轮播基础示例，承担“无更多入口”后的可发现性兜底。

验收：

- 输入框上方只展示新手引导/今日状态/最近分析/长期模式/规划目标。
- 基础录入能力只在新手引导中出现。
- 新手引导隐藏后，设置页可重新打开。

### Phase 2: 短期记忆与今日状态兜底

- 新增 `HoloShortTermMemoryModels`。
- 新增 `HoloMemorySnapshotBuilder`，作为 `UserContextBuilder` 门面，不重复聚合业务数据。
- 新增 `TodayStateResolver`。
- 实现 rich / partial / empty。
- 接入 `DailySenseStateBuilder`。
- 汇总今日、最近 7/14/30 天、活跃目标、最近对话意图。
- V1 只做当前点击流程内 snapshot 复用，不做跨会话缓存。

验收：

- 无任务、无习惯、无记录时不生成假状态。
- 部分数据时明确说明参考范围有限。
- 各能力入口能拿到统一 snapshot。
- 回答能展示“基于哪些来源”。

### Phase 3: Prompt 注入与来源短标签

- 扩展 `PromptManager.renderTemplate` 支持动态变量字典，或在 provider 层拼接受控 memory summary。
- 按链路实现不同注入策略：Chat/目标规划可注入，Intent 默认不注入，MemoryInsight 谨慎注入。
- 实现来源短标签，暂不做完整来源面板。
- 如修改后端 Prompt，同步更新后端 prompt 配置并安排后端发版。

验收：

- Chat/目标规划能接收受控 memory summary。
- MemoryInsight 不因 memory summary 挤占 daily token 预算。
- 来源短标签可展示。

### Phase 4: 长期记忆候选与晋升规则

- 新增 `HoloMemoryPromotionPolicy`。
- 新增 `HoloLongTermMemoryStore`。
- 新增 `HoloLongTermMemoryModels`。
- 接入用户明确表达、洞察反馈、周期洞察、目标规划保存结果。
- Store 串行化候选合并与 JSON 写入。

验收：

- 单次事件不会写入长期记忆。
- 敏感/推断型候选必须等待用户确认。
- 明确偏好和稳定反馈可静默写入并可撤销。
- `InsightPreferenceProfile` 的稳定偏好只被引用，不重复生成同源长期记忆。

### Phase 5: 记忆管理与来源透明

- 新增长期记忆查看、删除、纠正入口。
- 回答底部展示来源。
- 长期模式入口展示已记住和待确认候选。

验收：

- 用户能看见 Holo 记住了什么。
- 用户能删除或修改长期记忆。
- 每条长期记忆都有证据。

## Open Questions For Review

给 GLM 或人工审查重点看：

1. 长期记忆的“必须确认”和“可静默写入”边界是否足够安全。
2. `HoloMemorySnapshotBuilder` 作为 `UserContextBuilder` 门面是否足够清晰，是否还存在双聚合风险。
3. `InsightPreferenceProfile` 作为洞察运行时偏好画像、长期记忆只引用稳定摘要的边界是否成立。
4. Prompt 注入策略和 Token 预算是否足够保守，尤其是 MemoryInsight daily 场景。
5. JSON 文件存长期记忆是否足够，是否需要一开始就 Core Data。
6. 今日状态 empty 档的引导是否足够有价值。
7. 新手引导隐藏阈值是否合理。
8. 敏感记忆是否需要单独开关，例如“允许 Holo 记住行为模式”。

## Final Recommendation

建议先做本方案的 Phase 1-3，再进入 Phase 4-5。

原因：

- Phase 1-3 能快速验证“能力启动台 + 短期记忆”是否让 HoloAI 更有价值。
- Phase 4-5 涉及长期记忆信任和用户控制，必须在前面体验成立后谨慎推进。
- 暂不引入 Agent，可以保持系统可解释、可测试、可控。

最终形态应该是：

```text
能力启动台
  -> 选择高价值入口
  -> HoloMemorySnapshot
      -> 短期状态
      -> 相关长期记忆
      -> 来源证据
  -> 确定性工作流生成回答/分析/规划
  -> 用户确认后才落地行动或长期记忆
```
