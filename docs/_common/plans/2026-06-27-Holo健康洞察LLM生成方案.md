# Holo 健康洞察 LLM 生成方案

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 把健康页的“今日核心洞察”和“生活闭环”从固定规则文案升级为基于真实证据、由大模型生成、可校验可回退的个人健康洞察。

**Architecture:** 首页布局首版保持稳定，只替换洞察数据来源。iOS 本地负责收集证据、拉取健康洞察 system prompt、调用后端、校验模型输出、缓存结果和失败回退；后端负责新增健康洞察专用 prompt 管理项和 purpose route，并把 iOS 传入的 messages 透传给模型。实现上复用已注册的 Agent health 能力和现有 task / habit / finance / thought 工具，不新建平行数据系统。

**Tech Stack:** Swift / SwiftUI / Codable / Core Data / HealthKit / Holo Local Agent Tools / HoloBackend Node.js / Prompt Registry / OpenAI-compatible Chat Completions / XCTest / Node test runner.

## 1. 产品结论

### 1.1 是否改变健康页布局

首版不重做健康页布局。

保留当前页面结构：

```text
健康页
- 顶部标题、同步状态、日期切换
- 今日身体状态 / 三环指标
- Apple Health 数据源状态
- 今日核心洞察
- 生活闭环
- 近 7 天趋势
```

本次改造只替换两个区域的数据来源：

- `今日核心洞察`：从 `HealthDashboardSnapshot.coreInsight` 的规则文案，改为 LLM 生成结果。
- `生活闭环`：从 `HealthDashboardSnapshot.lifestyleInsights` 的固定三条，改为 0-3 条有证据支撑的跨模块洞察。

允许的轻量 UI 变化：

- 在洞察区展示生成状态：`生成中`、`今日已更新`、`数据不足`、`使用本地兜底`。
- 每条生活闭环可点开查看“为什么这么说”。
- 数据不足时展示诚实空态，不再伪装成真实跨模块关联。

不做的 UI 变化：

- 不改健康页主信息架构。
- 不新增大型 AI 聊天入口。
- 不把洞察区做成长文报告。
- 不让首页等待模型返回后才可用。

### 1.2 用户可见效果

改造后，健康页从“健康数据看板 + 固定建议”变成“HOLO 观察我的生活节奏”：

- 睡眠、步数、站立/活动不再只显示数值，而会和任务、习惯、消费、想法产生可验证关联。
- 如果数据里没有关联，系统不会硬说有关联。
- 每条洞察能解释证据来源，例如“过去 14 天里，有 3 个低睡眠日同时出现咖啡支出升高”。
- 用户可以反馈“有用 / 不准 / 少提醒这个”，后续生成会逐步收敛。（审查修订 P12：反馈闭环为 v2，见 Task 9，MVP 不含反馈入口）

### 1.3 成功标准

MVP 完成时必须满足：

- 健康页布局基本不变，用户不会感到页面被推倒重来。
- `今日核心洞察` 来自 LLM 结构化生成，失败时回退到现有规则。
- `生活闭环` 不再固定三条，而是由证据包和模型输出决定，最多展示 3 条。
- 每条 LLM 洞察都带 `evidenceIds`、`confidence` 和生成时间。
- 模型输出必须经过本地校验，不能直接渲染。
- 后端 prompt / purpose 可独立灰度和回滚。
- 后端改动部署后，线上 `api.holoapp.cn` 可验证真实请求生效。

### 2.3 不复用的方式

不建议让健康页直接调用通用 `agent_loop` 并渲染自由结果。

原因：

- 健康首页需要稳定卡片结构，不适合吃自由文本。
- 健康洞察需要更强的医疗安全边界。
- 首页不能每次打开都跑完整多轮 Agent。
- 通用 Agent 适合深度分析，健康页适合短、稳、可缓存的结构化结果。

本方案采用“专用生成链路 + 复用底层 Agent 工具”的折中路线。

## 3. 目标架构

```text
HealthView
  ↓
HealthInsightViewModel
  ↓
HealthInsightGenerationService
  ↓
HealthInsightContextBuilder
  ↓
HoloHealthTool + Task/Habit/Finance/Thought tools
  ↓
HoloBackendPromptService.loadPrompt(.healthInsightGeneration)
  ↓
HoloBackend health_insight_generation purpose
  ↓
HealthInsightResponseParser
  ↓
HealthInsightVerifier
  ↓
HealthInsightCache
  ↓
今日核心洞察 / 生活闭环 / 证据详情
```

### 3.1 iOS 分层

| 层 | 职责 |
| --- | --- |
| `HealthInsightViewModel` | 管理首页洞察加载状态、刷新、缓存命中、错误回退 |
| `HealthInsightContextBuilder` | 汇总健康、任务、习惯、财务、想法证据包 |
| `HealthInsightGenerationService` | 编排上下文构建、后端调用、解析、校验、缓存 |
| `HealthInsightResponseParser` | 解析后端 JSON，拒绝不完整结构 |
| `HealthInsightVerifier` | 校验证据存在、数字一致、表达安全、置信度阈值 |
| `HealthInsightCache` | 保存最近一次生成结果，避免每次打开都调模型 |
| `HealthInsightFallbackBuilder` | 复用现有规则文案，作为失败兜底 |

### 3.2 后端分层

| 层 | 职责 |
| --- | --- |
| Prompt Registry | 新增 `health_insight_generation` prompt，并同步 `PROMPT_VERSIONS` |
| Route config | 为该 purpose 配置 provider/model/temperature/maxTokens；不负责注入 prompt，也不配置 JSON mode |
| Provider | 复用现有 OpenAI-compatible chat completions，透传 iOS 请求体里的 `response_format` |
| Tests | 固定 prompt 存在、route 配置存在、`response_format` 可从请求透传到 provider |

### 3.3 数据边界

iOS 发送给后端的是摘要证据包，不发送完整生活数据库。

证据包应包含：

- 统计摘要：均值、达标天数、低值天数、趋势方向。
- 候选关联：本地规则预计算出的可能相关关系。
- 证据列表：脱敏后的短摘要和 ID。
- 反馈偏好：用户不喜欢的洞察类型、过度提醒类型。

### 4.4 Evidence ID 规范（审查修订 P3）

> 现有工具内部 event id 三套规则互不一致（`sleep-yyyyMMdd`、`{reqId}-keyword-{词}`、`task-trend-{index}`），不能直接用于 verifier 校验。本方案统一规范。

`HealthInsightContextBuilder` 构建的每条 `HealthInsightEvidence.id` 必须遵循：

```text
<domain>-<subKind>-<yyyyMMdd>
```

示例：

| domain | subKind | id 示例 |
| --- | --- | --- |
| health | sleep / steps / active | `health-sleep-20260624` |
| finance | keyword-coffee / meal-night | `finance-keyword-coffee-20260622` |
| task | completion / overdue | `task-completion-20260624` |
| habit | checkin / streak | `habit-checkin-20260624` |
| thought | stress / mood | `thought-stress-20260624` |

**同源约束**：这个 id 必须由 `HealthInsightContextBuilder` 一处生成，同时用于三处——

1. 发给 LLM 的上下文 JSON `evidence[].id`（4.3）。
2. iOS 本地 `evidence` 表的 key（4.1 `HealthInsightEvidence.id`）。
3. `HealthInsightVerifier` 的合法 id 集合（Task 6）。

LLM 回填的 `evidenceIds` 必须命中这个集合，否则视为编造并丢弃该洞察。不直接透传工具内部 event id。

## 5. 生成与校验规则

### 5.1 Prompt 约束

`health_insight_generation` prompt 必须明确：

- 只输出 JSON，不输出 Markdown。
- 不做医学诊断，不暗示疾病。
- 只能基于给定证据说话。
- 不允许把相关性说成因果。
- 不允许编造 evidenceId。
- 没有足够证据时，返回较少洞察，甚至返回空 `lifestyleLoops`。
- 标题要短，summary 面向普通用户，suggestedAction 要轻量可执行。
- 语言采用 HOLO 的观察者视角，不制造焦虑。

### 5.2 本地 Verifier

`HealthInsightVerifier` 校验：

- `coreInsight.evidenceIds` 至少 1 个真实存在（命中 4.4 规范的合法 id 集合）。
- 每个 `lifestyleLoop.evidenceIds` 至少 2 个真实存在，**且对应 evidence 的 domain 去重后至少覆盖 2 个**（审查修订 P5：判定对象是 evidenceIds → evidence.domain 的集合，不是 loop.domain 单值）。
- `confidence` 必须在 `0...1`。
- `lifestyleLoop.confidence < 0.55` 不展示。
- `title` 不超过 24 个中文字符。
- `summary` 不超过 90 个中文字符。
- 禁止词：诊断、疾病确定性、人格判断、过强因果。
- 数字类表述如果不在 evidence / summary 中出现，不允许渲染为事实。

校验失败处理：

- 单条 lifestyle loop 失败：丢弃该条。
- core insight 失败：使用 fallback core insight。
- 整体 JSON 解析失败：使用缓存；无缓存则使用 fallback。

### 5.3 Fallback 规则

首版继续保留现有规则作为兜底，但文案要更诚实：

- 数据不足：`健康数据正在积累。连续记录后，HOLO 会尝试把身体状态和任务、习惯、消费、想法串起来。`
- 后端失败：展示最近一次缓存，并标记 `使用上次洞察`。
- 无缓存且失败：展示本地规则洞察，不展示伪跨模块生活闭环。

## 6. 缓存与刷新策略

### 6.1 缓存

新增 `HealthInsightCache`，首版用 **JSON 文件**存储（审查修订 P7：snapshot 含 evidence 数组较大，不用 UserDefaults）。

缓存 key：

```text
healthInsight.generated.v1.<yyyy-MM-dd>
```

缓存内容：

- `GeneratedHealthInsightSnapshot`
- `contextHash`
- `promptVersion`
- `appVersion`

**contextHash 定义（审查修订 P6）**：只取稳定摘要的哈希——`period` + `healthSummary`（均值/达标天数/低值天数等汇总值）+ `candidateCorrelations` 的 id 与 confidenceHint。**不含**逐笔 evidence、不含当日实时指标值，否则会同一天多次进入都触发刷新，与 6.2「当天首次进入才生成」冲突。

**缓存清理（审查修订 P7）**：保留最近 7 天，超出 7 天的旧 key 在写入时顺带清理，避免无限堆积。

### 6.2 自动生成

触发条件：

- 当天首次进入健康页。
- 已授权 HealthKit，且至少有一个健康指标有数据。
- 今日缓存不存在，或 `contextHash` 明显变化，或缓存 `promptVersion` 与当前不一致（三轮修订 N4：prompt 升级后旧缓存失效）。

不触发条件：

- 未授权 HealthKit。
- 后端 feature flag 关闭。
- 过去 30 分钟内刚失败过，避免反复请求。

### 6.3 手动刷新

健康页洞察区提供轻量刷新入口。

限制：

- 每天最多手动刷新 3 次。
- 手动刷新与自动生成**共享 6.2 的「30 分钟失败节流」**（审查修订 P8）：失败后 30 分钟内手动刷新也跳过实际请求，仅提示。
- 刷新期间保持旧内容可见。
- 失败时提示轻量状态，不弹阻塞 alert。

## 9. 风险与取舍

### 9.1 主要风险

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 模型编造关联 | 看起来很聪明但证据不存在 | evidenceId 必须本地校验 |
| 医疗合规风险 | 输出诊断或过度健康建议 | prompt + verifier 双层限制 |
| 生成慢 | 打开健康页卡顿 | 缓存优先，后台刷新 |
| 数据稀疏 | 新用户看不到洞察 | 诚实空态 + 本地 fallback |
| 过度打扰 | 每天都生成相似提醒 | 反馈降权 + 手动刷新限额 |
| 后端未发版 | iOS 调不到新 purpose | 明确后端发版为验收门槛 |

### 9.2 首版刻意不做

- 不做医学风险评分。
- 不接 Apple Health 写入。
- 不把健康洞察自动写入长期记忆。
- 不做完整跨域相关性统计引擎；MVP 仅做 1 个跨域候选（低睡眠∩咖啡，算法见 Task 3）+ 单域趋势，其余跨域（步数∩运动习惯、低活动∩压力想法）推 v2。
- 不让模型主动访问工具；工具调用仍由 iOS 本地编排。
- 不改健康页整体视觉设计。
