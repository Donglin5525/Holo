## 一、最终产品结论

第一版新人引导不承担“激活体系”“首周留存”“首次洞察”或“完整新人成长路径”。

它只完成四件事：

1. 让用户知道 Holo 是什么。
2. 让用户知道 Holo 主要能做什么。
3. 让用户知道从哪里、用什么方式开始使用。
4. 把 HoloAI 必要的数据处理授权讲清楚，并让用户自行选择。

最终采用：

```text
3 个轻量引导页面
  +
进入首页后 1 次 HoloAI 入口提示
```

不在 onboarding 内要求用户完成真实记账、创建待办或生成成果卡。用户点击首页 HoloAI 后，继续使用现有 Chat、确认卡片和授权兜底，不为新人流程另造一条执行链。

这版的产品定位是：

> 先让用户看懂、会用、敢授权，再让真实使用自然发生。

## 二、对上一轮 Claude 审查的复审结论

### 2.1 应当接受的意见

以下意见成立，并已在本版中解决：

| 审查意见 | 复审结论 | 本版处理 |
|---|---|---|
| 第一版没有跨用户埋点能力，却定义激活率和 D1/D7 指标 | 正确 | 删除第一版北极星、留存和事件上报方案 |
| “第一次回顾”依赖 AI 授权、数据门槛和后台调度 | 正确 | 删除起步任务、首次回顾和首周回顾 |
| 首页视觉中央是今日看板，不是 HoloAI | 正确 | 明确 HoloAI 位于底部导航栏中央按钮 |
| 有效记录日不覆盖健康，不能直接作为全域新人进度 | 正确 | 第一版不使用有效记录日和跨域进度 |
| “关于 Holo 区域”目前并不存在 | 正确 | 第一版不新增设置页回看入口 |
| 真实保存回执分散在多条执行路径，接入成本被低估 | 正确 | 第一版不接保存回执、不修改 Chat 执行链 |
| 已发送未确认、编辑旁路和强杀恢复会让流程变重 | 正确 | onboarding 不承载真实操作，不需要恢复确认卡片 |
| 强制选择关注方向收益低、反而增加阻塞 | 正确 | 删除关注方向选择 |
| 90 秒完成真实操作缺少依据 | 正确 | 删除硬性 90 秒目标 |
| “首周回访”在没有通知时用词不准确 | 正确 | 删除整个首周阶段 |
| 多阶段 Phase 与实际交付顺序矛盾 | 正确 | 合并为一个轻量 V1 实施批次 |

### 2.2 部分成立但措辞过重的意见

#### 金额与本地回执

原方案“金额可以只存在本地回执、不能进入日志和上报事件”在技术上并不矛盾，问题是文档没有明确区分本地展示数据与遥测数据。Claude 指出的歧义成立，但将其定义为“致命矛盾”偏重。

本版删除成果回执，因此不再需要处理这条边界。

#### 卸载重装与 iCloud 恢复

“卸载重装后，UserDefaults 已丢失且 CloudKit 尚未导入时，老用户可能被识别为新人”这一事实成立；但“老用户 100% 被误判”只在特定重装时序下成立，不代表正常版本升级。

第一版不为这个边缘场景引入 Keychain 安装标记、CloudKit 等待器或复杂 eligibility resolver。控制方式是：

- 引导只有 3 页。
- 每页都可以跳过。
- 不创建、删除或修改业务数据。
- 普通升级用户继续通过旧昵称完成标记跳过引导。

#### Deep Link 打断

新用户通过通知冷启动的概率很低，但外部 URL、小组件等 Deep Link 确实存在，因此优先级仍需明确。本版规定：

> Deep Link 优先于新人引导；本次先完成用户主动发起的跳转，下次正常回到首页时再展示引导。

### 2.3 不再需要解决的重方案问题

以下问题并非无效，而是随着范围收缩自然消失：

- 首次成果卡如何生成。
- 财务、待办、观点保存回执如何统一。
- 编辑 Sheet 保存旁路如何回传。
- `recordMood` 分类映射是否影响 onboarding。
- 首次回顾如何降级。
- 起步任务卡如何完成和频控。
- D1/D7 如何统计。
- 设置页 preview 模式是否写真实数据。

这些都不进入轻量 V1。

## 三、V1 范围

### 3.1 必须交付

- 三页全屏新人引导。
- 首页旧昵称 Alert 合并到新人引导，不再单独弹出。
- Holo 核心功能和基本使用方式说明。
- HoloAI 数据处理授权说明与选择。
- 进入首页后一次性指向底部 HoloAI 按钮的轻提示。
- 新用户、旧用户、跳过、授权拒绝和 Deep Link 的基本规则。
- 浅色、深色、大字体和 VoiceOver 基础适配。

### 3.2 明确不做

- 不要求用户在引导内完成记账或创建待办。
- 不调用 LLM，不依赖网络。
- 不修改 Chat、IntentRouter、Repository 或确认卡片链路。
- 不生成首次成果卡、日报、周报或洞察。
- 不做首页起步任务卡。
- 不改造各模块空状态。
- 不做首日、三日、首周生命周期。
- 不新增服务端埋点、第三方分析 SDK 或后端接口。
- 不做关注方向问卷。
- 不做复杂状态机、Core Data onboarding 实体或 CloudKit 同步。
- 不在设置页新增“重新查看引导”。

## 四、核心产品表达

统一价值主张：

> 把生活告诉 Holo，它帮你记录、整理，并在时间里看见变化。

统一使用心智：

```text
想记录 → 告诉 Holo
想查询 → 直接问 Holo
想回顾 → 打开记忆长廊
```

新人不需要理解 Intent、Prompt、Agent、长期记忆、跨域融合、置信度或新鲜度等内部概念。

## 五、完整用户流程

```text
正常首次打开
  ↓
页面 1：认识 Holo，可选填写昵称
  ↓
页面 2：知道能做什么、怎样开始
  ↓
页面 3：了解并选择 AI 数据处理授权
  ↓
进入首页
  ↓
一次性提示底部中央 HoloAI 入口
  ↓
用户自行点击并进入现有 Chat 流程
```

任一页面点击“跳过”都会直接进入首页，并将轻量引导标记为已完成，不在下次启动重复打扰。

## 六、页面一：认识 Holo

### 6.1 产品目的

让用户在一屏内理解 Holo 的定位，同时把现有单独昵称 Alert 合并掉。

### 6.2 推荐文案

主标题：

> 你好，我是 Holo

副标题：

> 你的个人数据助理。帮你记录生活、安排事情，并慢慢看见自己的变化。

昵称输入：

> 希望我怎么称呼你？（选填）

主按钮：`继续`

右上角：`跳过`

### 6.3 交互规则

- 昵称选填，不阻塞继续。
- 完成整个 onboarding 时，若昵称非空则复用 `UserDisplayNameSettings.saveDisplayName()`。
- 未填写时继续使用现有回退称呼“你”。
- 完成或跳过整个引导时调用旧 `markOnboardingCompleted()`，避免旧昵称 Alert 再出现。
- 页面不申请任何系统权限。
- 键盘出现时页面可滚动，主按钮必须始终可到达。

## 七、页面二：功能与使用方式

### 7.1 产品目的

让用户知道 Holo 的三个核心使用动作，不介绍所有模块，不做控件级教学。

### 7.2 页面结构

标题：

> 你可以这样使用 Holo

三张能力卡：

#### 1. 一句话记录

图标：`square.and.pencil`

说明：

> 不用先找入口，直接告诉 Holo 发生了什么。

示例：

> “午饭花了 35 元”

补充小字：`还可以创建待办、记录想法、习惯打卡。`

#### 2. 直接问自己的数据

图标：`bubble.left.and.text.bubble.right`

说明：

> 想知道最近的情况，直接用自然语言提问。

示例：

> “这个月餐饮花了多少？”

#### 3. 回看生活变化

图标：`book.closed`

说明：

> 记忆长廊会把不同模块的记录串成时间线，方便回看。

示例：

> 财务、待办、习惯和想法会出现在同一段生活轨迹中。

主按钮：`下一步`

右上角：`跳过`

### 7.3 交互规则

- 所有示例都使用当前已支持的单动作或查询，不展示双动作演示。
- 卡片只做静态说明，不在 onboarding 内发送消息或创建数据。
- 页面不自动播放长动画；可使用轻微淡入。
- 不加入“关注方向选择”“你最想改善什么”等额外问题。

## 八、页面三：AI 数据处理授权

### 8.1 产品目的

在第一次使用 HoloAI 前，把数据如何处理说清楚，避免用户进入 Chat 后才被一段技术文案打断。

### 8.2 推荐文案

标题：

> 开始前，确认一项授权

正文：

> 当你使用 HoloAI、AI 洞察或语音转文字时，完成这项功能所需的问题和数据摘要，会经 Holo 后端发送给第三方 AI 或语音服务处理。

说明卡：

- 只在你使用相关 AI 功能时处理必要数据。
- 不授权也可以继续使用本地记账、待办、习惯和观点功能。
- 之后可以在 HoloAI 数据授权中随时开启或撤回。

链接：`查看隐私政策`

主按钮：`同意并开始使用`

次级按钮：`暂不授权，先进入 Holo`

### 8.3 授权行为

点击“同意并开始使用”：

1. 调用 `HoloAIDataProcessingConsent.shared.grant()`。
2. 标记轻量 onboarding 已完成。
3. 关闭 onboarding，进入首页。
4. 展示一次 HoloAI 入口提示。

点击“暂不授权”：

1. 不修改现有 consent 状态。
2. 标记轻量 onboarding 已完成。
3. 关闭 onboarding，进入首页。
4. 用户以后点击 HoloAI 发送消息时，继续使用现有授权提醒。

### 8.4 不在此页申请的权限

- 麦克风：用户点击语音输入时再申请。
- 通知：用户创建需要提醒的任务或主动开启通知时再申请。
- HealthKit：用户进入健康模块并主动选择连接时再申请。
- iCloud：继续沿用现有同步策略，不在 onboarding 新增系统弹窗。

## 九、进入首页后的一次性提示

### 9.1 为什么保留这一项

首页视觉中央的大按钮是今日看板，HoloAI 入口实际位于底部导航栏中央。只看三页说明后，用户仍可能不知道从哪里开始，因此需要一个一次性位置提示。

### 9.2 推荐样式

在底部导航栏中央 HoloAI 按钮上方显示轻量气泡：

> 从这里告诉 Holo 一件事

副文案：

> 例如：“午饭花了 35 元”

交互：

- 点击气泡或 HoloAI 按钮：关闭提示并进入现有 Chat。
- 点击气泡外区域：关闭提示。
- 提示关闭后不再自动出现；它只由本次 onboarding 完成回调触发。
- 不使用全屏遮罩，不阻止首页其他操作。
- 不自动填充或发送示例，避免意外创建数据。

### 9.3 未授权用户

未授权用户仍可以看到入口提示。进入 Chat 后可以浏览现有界面；当真正发送消息时，再由现有授权提示拦截。

## 十、状态与展示规则

### 10.1 最小状态

只增加一个 UserDefaults Bool：

```swift
enum LightweightOnboardingSettings {
    static let completedKey = "holo_onboarding_lightweight_v1_completed"
}
```

AI 入口提示使用 `HomeView` 当前会话内的 `@State` 控制，不增加持久化 key。因为它只在 onboarding 完成回调中触发，旧用户迁移和后续启动都不会自动看到。

不保存当前页、不保存输入草稿、不建立 Codable 状态机。

### 10.2 为什么不做步骤恢复

整个流程只有三页，不执行网络请求，也不写业务数据。用户中途杀掉 App 后从第一页重新开始，成本可接受；为此增加状态机、迁移和恢复矩阵不符合轻量 V1 的投入边界。

昵称只有在完成整个 onboarding 时才保存，避免用户在第一页输入一半后退出产生半完成状态。

### 10.3 新老用户规则

按以下顺序判断：

1. DEBUG 截图或模拟器验收模式：最高优先级显式跳过。
2. `holo_onboarding_lightweight_v1_completed == true`：不展示。
3. 旧 `hasCompletedUserNameOnboarding == true`：视为已有用户，写入新 completed key 并跳过。
4. 本次进入首页前存在待处理 Deep Link：本次不展示，也不写 completed key。
5. 其他情况：展示轻量 onboarding。

第一版不查询 Core Data，也不等待 CloudKit import。

### 10.4 卸载重装边界

卸载重装后 UserDefaults 可能丢失，CloudKit 数据也可能尚未导入，因此老用户可能再次看到轻量 onboarding。

V1 接受这一已知边界，原因是：

- 三页都可跳过。
- 不改变业务数据。
- 不申请一组系统权限。
- 不阻断 Deep Link。

如果未来真实反馈显示该问题频繁，再单独评估 Keychain 安装标记；不提前建设。

## 十一、展示时机与导航优先级

建议由 `HomeView` 承担展示，不新增 `HoloRootView`：

```text
HomeView.onAppear
  → 先记录本次是否携带 DeepLinkState.pendingTarget
  → 再处理 Deep Link
  → 再判断是否需要展示 onboarding
```

规则：

- Deep Link 优先，避免用户点击小组件或外部链接后被欢迎页拦截。
- 使用当前会话 `didDeferOnboardingForDeepLink` 标记，而不是在 `handleDeepLink()` 后重新读取可能已被消费的 `pendingTarget`。
- HomeView 已打开其他 fullScreenCover 时不再同时展示 onboarding。
- Deep Link 消费完成后，本次不补弹；等下次正常进入首页再展示。
- onboarding 使用 `fullScreenCover`，关闭后回到已经完成初始化的首页。
- 不修改 `ContentView` 的现有职责。

## 十二、视觉与交互规范

第一版复用现有 Holo Design Tokens，不建立 onboarding 专属设计系统。

| 元素 | 规范 |
|---|---|
| 背景 | `Color.holoBackground` |
| 水平边距 | `HoloSpacing.xl`，小屏最低 `lg` |
| 页面标题 | `.holoTitle` 或 `.holoHeading` |
| 正文 | `.holoBody` + `holoTextSecondary` |
| 能力卡背景 | `holoCardBackground` |
| 卡片圆角 | `HoloRadius.lg` |
| 卡片间距 | `HoloSpacing.md` |
| 主按钮 | 全宽、最小高度 50pt、`holoPrimary` |
| 次级按钮 | 文字按钮，不使用同等视觉权重 |
| 页码 | 3 个圆点，只表示当前位置，不显示百分比 |

### 12.1 加载、空状态与错误

- onboarding 本身无网络加载态。
- 昵称为空属于合法状态，不显示错误。

### 12.2 键盘与焦点

- 昵称输入不自动弹键盘，避免首屏一进入就被键盘占据。
- 用户点击输入框后使用 `@FocusState` 管理焦点。
- 点击“继续”或页面空白区域收起键盘。
- Dynamic Type 下页面允许纵向滚动，主按钮不被遮挡。

### 12.3 无障碍与深色模式

- 图标和文字共同表达含义，不以颜色作为唯一状态。
- 所有按钮提供明确 VoiceOver 标签。
- 正文、次级按钮和页码使用语义色，禁止硬编码白色或黑色。
- Reduce Motion 开启时关闭位移动画，只保留淡入淡出。
- 主按钮触控高度不低于 44pt，本方案统一使用至少 50pt。

## 十四、推荐实现方式

### 14.1 页面切换

在 `HoloLightweightOnboardingView` 内使用简单的本地 `@State currentPage`：

```swift
@State private var currentPage = 0
@State private var nicknameDraft = ""
```

使用显式按钮切换页面，不依赖复杂 coordinator。是否允许左右滑动可按实现便利决定，但授权页的两个选择必须通过明确按钮完成。

### 14.2 完成回调

```swift
enum OnboardingCompletionChoice {
    case grantedAIConsent
    case skippedAIConsent
    case skippedOnboarding
}
```

关闭前统一完成：

1. 如果昵称有效则保存。
2. 标记旧昵称 onboarding 已完成。
3. 根据选择决定是否 grant AI consent。
4. 写入 lightweight completed key。
5. dismiss 并安排一次 AI 入口提示。

### 14.3 跳过行为

“跳过”不是暂停：

- 不保存昵称草稿。
- 不授予 AI consent。
- 将轻量 onboarding 标记为已完成。
- 进入首页并展示一次 AI 入口提示。

这样不会在每次启动重复询问用户。

## 十五、实施步骤

第一版按一个批次实施：

1. 新增一个 UserDefaults key 与最小封装。
2. 实现三页静态 onboarding 和授权操作。
3. 将昵称输入迁移进第一页，删除旧 Alert。
4. 在 `HomeView` 接入展示判断和 Deep Link 优先级。
5. 增加底部 HoloAI 一次性入口提示。
6. 补最小 standalone 状态测试。
7. 完成 Debug/Release build 与模拟器人工流程验收。

不拆首周 Phase，不等待埋点系统，不依赖后端改动。

## 十六、测试与验收

### 16.1 逻辑测试

- 新 key 已完成时不展示。
- 旧昵称 onboarding 已完成时自动迁移并不展示。
- 新用户正常展示。
- 点击跳过后不再展示。
- 同意授权后 consent 为 true。
- 暂不授权后 consent 保持原值。
- Deep Link 存在时本次不展示、completed key 不被误写。
- 新用户完成或跳过 onboarding 后，本次会话出现一次 AI 入口提示。
- 旧用户迁移不会出现 AI 入口提示。

### 16.2 人工流程

- 新安装：完整走完三页。
- 新安装：第一页直接跳过。
- 新安装：不填昵称继续。
- 授权页：同意授权。
- 授权页：暂不授权。
- 老用户升级：不出现新人引导。
- 小组件或外部 URL 冷启动：先完成 Deep Link。
- 浅色、深色模式。
- 小屏、大字体和键盘弹出状态。
- VoiceOver 逐项读取。
- DEBUG 截图和模拟器验收模式不出现 onboarding。

### 16.3 验收标准

- [ ] 首次流程最多三页，每页都可跳过。
- [ ] 不发起任何网络请求或 LLM 调用。
- [ ] 不创建任何业务数据。
- [ ] 用户能看懂记录、查询和记忆长廊三种核心用法。
- [ ] AI 数据处理范围、关闭后的影响和隐私政策入口清晰。
- [ ] 同意或拒绝授权都能正常进入首页。
- [ ] 旧昵称 Alert 不再出现。
- [ ] 首页只出现一次 HoloAI 入口提示。
- [ ] Deep Link 不被 onboarding 阻断。
- [ ] 老用户正常升级不被强制展示。
- [ ] 未修改 Chat 执行链、后端和 Prompt。
- [ ] 浅色、深色、大字体和 VoiceOver 可用。

## 十七、已知边界

第一版有意接受：

- 无跨用户 onboarding 转化率数据。
- 卸载重装的老用户可能再次看到三页轻量说明。
- 不保证用户完成第一次真实记录。
- 不提供首周留存干预。
- 不提供从设置重新查看引导。

这些不是遗漏，而是轻量 V1 的范围控制。只有上线后出现明确问题，才分别立项，不在第一版提前建设。

## 十八、后续可选方向

1. 设置页增加“使用引导”回看入口。

## 十九、发布边界

本方案仅修改 iOS 客户端 UI 和本地 UserDefaults 状态：

- 不修改 `HoloBackend/`。
- 不修改 Prompt。
- 不增加新的网络接口。
- 不需要后端部署。

完成定义为：代码实现、状态测试、Debug/Release build、模拟器完整流程和旧用户迁移验证均通过。
