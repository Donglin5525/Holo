# Holo AI Sense Loop 公测方案

> 日期：2026-06-07
> 状态：草案，供 GLM / 第三方审查
> 目标版本：App Store 公测前后分阶段落地
> 方案定位：让 Holo 的 AI 从“能回答、能分析”升级为“能持续理解用户，并且表达得更有分寸”

## 1. 背景

Holo 即将进入 App Store 公测阶段。当前产品已经具备财务、习惯、待办、想法、健康、记忆长廊、HoloAI 聊天、Daily Sense 等基础能力，也已经开始接入个人档案和记忆洞察。

但从真实体验看，Holo 的 AI 智能感仍然不稳定：

| 现象 | 用户感受 | 根因判断 |
|------|----------|----------|
| 洞察文案偏冷 | “消费比日均高 xx%”没有情绪价值 | 还停留在数据异常，没有转成生活状态 |
| 上下文联动弱 | AI 有时像重新认识用户 | 用户档案、近期状态、长期记忆、当前输入没有统一调度 |
| 表达不够友好 | 有时泛泛、有时过度分析 | 缺少统一的表达原则和介入强度判断 |
| 记忆缺少主线 | 记录很多，但不一定值得回看 | 没有区分流水、异常、高光、里程碑 |
| 反馈无法学习 | 用户觉得“不准/没感觉”后，系统没有明显变好 | 缺少轻量、结构化的校准闭环 |

因此，本方案不把问题定义为“再加几个 AI 功能”，而是定义为：

> Holo 缺少一条把用户档案、日常状态、记忆洞察、聊天表达和用户反馈串起来的理解闭环。

## 2. 目标

### 2.1 产品目标

让公测用户逐渐感受到：

> Holo 不是在展示 AI 能力，而是在持续观察我的生活，并且越来越知道该怎样陪我。

### 2.2 用户可感知目标

公测阶段优先做到 5 个用户可感知变化：

| 目标 | 用户看到什么 |
|------|--------------|
| 更具体 | Holo 能结合用户最近目标、习惯、状态说话，而不是泛泛鼓励 |
| 更克制 | 不把猜测说成结论，不做人格判断，不频繁打扰 |
| 更有记忆 | Holo 能把今天的小事放回长期变化里理解 |
| 更有温度 | 文案从冷数据转成生活状态，比如“节奏有点乱”“正在找回节奏” |
| 更可校准 | 用户可以告诉 Holo 哪些洞察有用、哪些不准、哪些不要再提醒 |

### 2.3 非目标

本方案明确不做以下事情：

| 不做 | 原因 |
|------|------|
| 不把 Holo 包装成全自动 AI 管家 | 公测阶段风险高，容易过度承诺 |
| 不让 AI 自由写入长期记忆 | 容易污染用户画像，造成误解和漂移 |
| 不做医疗、心理、人格判断 | 健康和想法只能作为生活信号，不做专业诊断 |
| 不默认高频主动推送 | “更懂用户”不等于“更频繁打扰用户” |
| 不一次性重构所有 AI 链路 | 公测前应优先做高感知、低风险的闭环 |

## 3. 总体方案：Holo AI Sense Loop

本方案命名为 **Holo AI Sense Loop**。

它不是一个新页面，也不是一个单独功能，而是一套贯穿 HoloAI、Daily Sense、记忆长廊、洞察卡片和用户反馈的体验闭环。

```text
用户日常记录
  -> 状态观察
  -> 用户理解
  -> 表达决策
  -> 聊天 / Daily Sense / 记忆长廊 / 洞察卡片
  -> 用户轻反馈
  -> 偏好校准
  -> 下一次表达更贴近用户
```

核心原则：

1. Holo 先观察，再表达。
2. Holo 可以提出假设，但不能把假设说成事实。
3. Holo 优先记录阶段变化，而不是冷数据异常。
4. Holo 的个性化来自稳定信号，不来自某一次随手反馈。
5. Holo 应该知道什么时候少说话。

## 4. 四层能力设计

### 4.1 用户理解层：Holo 知道“这个人是谁”

这一层回答的问题是：

> Holo 在理解当前输入前，应该知道用户哪些长期背景？

第一版建议包含：

| 类型 | 示例 | 用途 |
|------|------|------|
| 长期关注 | 戒烟、存钱、减重、提高睡眠质量 | 帮助 AI 判断哪些信息对用户重要 |
| 当前目标 | 本月控制支出、连续打卡、完成某个计划 | 让建议贴近用户正在做的事 |
| 偏好表达 | 喜欢直接、喜欢温和、讨厌说教 | 控制文案语气 |
| 敏感边界 | 不做人格判断、不夸大健康问题 | 避免让用户不适 |
| 最近状态 | 压力偏高、恢复中、节奏稳定 | 决定此刻应不应该提醒 |

注意：这一层不是“越多越好”。公测阶段更重要的是准确、可解释、可回退。

### 4.2 状态观察层：Holo 知道“用户最近怎么了”

这一层回答的问题是：

> 今天或这一周，用户的生活状态有没有变化？

观察对象不是单个数字，而是多个信号之间的组合：

| 维度 | 观察方式 | 用户友好表达 |
|------|----------|--------------|
| 财务 | 消费是否偏离平时、是否与压力场景同时出现 | “这两天像是在用外卖和咖啡撑过忙碌状态” |
| 习惯 | 是否断连、恢复、连续稳定 | “节奏正在找回” |
| 待办 | 是否堆积、临近截止、完成回升 | “任务压力有点集中” |
| 想法 | 是否反复出现同一主题 | “你最近频繁提到职业方向” |
| 健康 | 睡眠、步数、运动是否与其他状态同向变化 | “休息偏少时，其他节奏也有点乱” |

这里要避免三类表达：

1. 避免只说数字：比如“消费偏离均值 3.2x”。
2. 避免强因果：比如“你因为压力才花钱”。
3. 避免人格判断：比如“你是一个缺乏自控的人”。

推荐表达方式：

> “我不确定原因，但这几个信号是一起出现的：咖啡变多、外卖变多、睡眠偏少。它更像是一个高压支撑状态，值得你留意一下。”

### 4.3 表达决策层：Holo 知道“此刻该怎么说”

这一层回答的问题是：

> 同样的事实，Holo 应该用什么强度表达？

建议把 Holo 的表达分成 5 档：

| 档位 | 使用场景 | 示例 |
|------|----------|------|
| 看见 | 信号弱，只做轻轻记录 | “我看到这几天咖啡多了一点。” |
| 归纳 | 多个信号同时出现 | “这更像是一个忙碌支撑状态。” |
| 提醒 | 与用户目标明显相关 | “如果你还想控制预算，今晚可以先别急着点外卖。” |
| 行动建议 | 用户明确求助，或信号稳定 | “要不要我帮你把这周的高压时段整理一下？” |
| 庆祝 | 达到阶段门槛 | “这是你连续控制抽烟的第 7 天，已经不是偶然了。” |

关键规则：

1. 当前用户明确输入永远优先。
2. 长期档案只能辅助理解，不能覆盖用户当下表达。
3. 低置信度只能用“可能/像是/值得留意”，不能用确定判断。
4. 当用户处于压力或恢复状态时，优先减少指责和任务感。
5. 当用户已经连续达成目标时，优先给具体确认，而不是空泛夸奖。

### 4.4 反馈学习层：Holo 知道“哪里没懂对”

这一层回答的问题是：

> 用户觉得 Holo 不准、没感觉、太打扰时，系统怎样变好？

公测第一版只做轻反馈，不做复杂编辑。

建议每条洞察或关键 AI 回答支持 4 个轻反馈：

| 反馈 | 含义 | 后续影响 |
|------|------|----------|
| 有用 | 这个判断对我有帮助 | 同类洞察权重略升 |
| 不准 | 事实或关联不对 | 降低同类判断置信度 |
| 没感觉 | 事实可能对，但没有价值 | 降低同类内容展示优先级 |
| 少提醒这个 | 用户不想频繁看到 | 增加冷却时间 |

反馈不能直接塞进 Prompt。推荐流程：

```text
单次反馈
  -> 结构化记录
  -> 同类反馈累计
  -> 达到阈值后形成稳定偏好
  -> 作为短摘要影响后续生成和排序
```

示例：

| 用户反馈 | 不应该做 | 应该做 |
|----------|----------|--------|
| 对消费洞察点了“没感觉” | 以后完全不讲消费 | 降低冷数据消费异常的优先级 |
| 对戒烟洞察点了“有用” | 每天都提醒戒烟 | 在阶段进展时优先展示戒烟相关高光 |
| 对某条关联点了“不准” | 删除所有跨模块洞察 | 降低相同关联模式的置信度 |

## 5. 公测阶段能力分期

### Phase 0：统一表达原则

目标：先让 Holo 不说错、不说重、不装懂。

改动范围：

| 项目 | 内容 |
|------|------|
| AI 文案原则 | 具体、有证据、克制、少评判、不把猜测说成结论 |
| 禁止表达 | 空泛人格判断、强因果判断、医疗/心理诊断、油腻鼓励 |
| 推荐表达 | “我看到/像是/可能/值得留意/如果你愿意” |
| Prompt 规则 | 聊天、洞察、Daily Sense 共用一套表达边界 |

验收标准：

1. 用户看不到“你是一个很自律的人”这类泛泛判断。
2. 洞察文案能说明证据来源。
3. 不再把消费、睡眠、习惯等信号直接说成因果关系。

### Phase 1：今日状态摘要

目标：让 Holo 有一个统一的“最近状态背景”。

建议状态：

| 状态 | 含义 | 示例文案 |
|------|------|----------|
| 节奏不错 | 多数信号稳定 | “今天整体节奏不错。” |
| 节奏有点乱 | 多个信号波动 | “待办和消费都有点偏离平时。” |
| 节奏在找回 | 近期曾波动，但出现恢复信号 | “你这两天在把节奏拉回来。” |
| 压力偏高 | 睡眠/消费/任务等同时偏紧 | “这更像是一个需要支撑的忙碌状态。” |
| 新阶段出现 | 长期指标跨过门槛 | “这不是单日变化，更像一个阶段进展。” |

这一状态应被三个地方共用：

1. Daily Sense 卡片。
2. HoloAI 聊天上下文。
3. 记忆长廊高光和里程碑筛选。

### Phase 2：洞察文案从“数据异常”改为“生活状态”

目标：把“看数据”变成“看见生活”。

文案改造规则：

| 旧方式 | 新方式 |
|--------|--------|
| 今日消费高于日均 42% | 今天的餐饮和咖啡一起变多，像是忙碌状态下的补偿型消费 |
| 2 个习惯断连 | 有两个习惯断了节奏，但其中一个已经开始恢复 |
| 待办逾期 3 条 | 任务压力集中在这两天，可能需要先清一个最小项 |
| 本周步数下降 | 这周活动量偏少，如果你在恢复节奏，可以先从轻量散步开始 |

每条洞察尽量包含：

1. 看见了什么。
2. 为什么值得留意。
3. 这是不是确定结论。
4. 用户可以做什么小动作，或者先不行动。

### Phase 3：记忆长廊筛选升级

目标：只把真正值得回看的内容放进主叙事。

记忆分层：

| 层级 | 定义 | 示例 |
|------|------|------|
| 流水 | 用户做过的普通记录 | 今天记了一笔咖啡 |
| 异常 | 数据明显偏离平时 | 今天餐饮明显高于平时 |
| 高光 | 某次不同选择或短期恢复 | 忙碌中仍完成了戒烟记录 |
| 里程碑 | 长期阶段跨过门槛 | 存款过一档、体重下一个台阶、连续控制抽烟 7 天 |

主叙事优先级：

```text
里程碑 > 高光 > 状态变化 > 异常 > 流水
```

里程碑判断四条件：

| 条件 | 说明 |
|------|------|
| 阶段性 | 不是单日波动，而是进入一个新阶段 |
| 稳定性 | 有连续或重复证据 |
| 用户相关性 | 与用户目标、长期关注、主动表达有关 |
| 可证据化 | 能说明来自哪些记录，而不是凭空总结 |

### Phase 4：轻反馈入口

目标：公测阶段开始收集“懂不懂用户”的真实信号。

入口建议：

| 场景 | 反馈方式 |
|------|----------|
| 洞察卡片 | 有用 / 不准 / 没感觉 / 少提醒这个 |
| 记忆高光 | 收藏 / 不值得记 |
| HoloAI 回答 | 有帮助 / 没答到 / 语气不喜欢 |

第一版不需要复杂设置页，重点是低摩擦。

反馈结果第一阶段只影响排序和展示，不直接改长期档案。

### Phase 5：生活模式模型

目标：从“记住事实”升级为“理解用户的生活节奏”。

生活模式模型建议包含：

| 模型内容 | 示例 |
|----------|------|
| 压力模式 | 用户压力大时，外卖、咖啡、晚睡是否一起出现 |
| 恢复模式 | 用户通常通过什么方式找回节奏 |
| 目标反复点 | 戒烟、减重、存钱等目标容易在哪些场景受影响 |
| 有效建议风格 | 用户更接受直接建议、温和提醒，还是数据回顾 |
| 少打扰场景 | 哪些主题用户反馈“没感觉”或“少提醒” |

这一阶段不建议放在公测前硬做。它更适合作为公测后的核心升级。

## 6. 与现有系统的关系

本方案不推翻已有模块，而是把已有模块串起来。

| 现有模块 | 在本方案中的角色 |
|----------|------------------|
| HoloProfile | 提供用户长期背景，比如目标、关注点、偏好 |
| Daily Sense | 提供今日状态摘要，是“最近怎么了”的入口 |
| Memory Insight | 生成周期性洞察，是“这段时间发生了什么”的入口 |
| Memory Gallery | 承载高光和里程碑，是“什么值得回看”的入口 |
| HoloAI Chat | 承载即时对话，是“现在如何回应”的入口 |
| Feedback / Rating | 承载用户校准，是“哪里没懂对”的入口 |

关键整合点：

```text
HoloProfile
  + Daily Sense
  + Recent Memory Insight
  + Current User Input
  + Feedback Preference
  -> Expression Decision
  -> HoloAI / Cards / Gallery
```

这里最重要的是优先级：

1. 用户当前明确输入最高优先。
2. 用户主动设置的目标和档案第二优先。
3. 最近状态第三优先。
4. AI 推断出的长期模式第四优先。
5. 低置信度猜测只能作为轻提示，不能主导回答。

## 7. Prompt 与文案策略

### 7.1 统一表达原则

建议所有 Holo AI 相关 Prompt 共用以下规则：

1. 说具体证据，不说空泛评价。
2. 表达假设时必须带不确定性。
3. 不做人格、医疗、心理判断。
4. 不把并发信号说成因果关系。
5. 优先使用用户能理解的生活语言。
6. 当用户状态偏乱时，减少任务感和说教感。
7. 当用户达成阶段进展时，给具体确认，而不是泛泛夸奖。

### 7.2 推荐文案模板

| 场景 | 模板 |
|------|------|
| 低置信观察 | “我只看到一个轻微信号：{证据}，先不用过度解读。” |
| 多信号并发 | “这几个信号是一起出现的：{A}/{B}/{C}，它可能代表 {状态}。” |
| 目标相关提醒 | “如果你还想靠近 {目标}，现在最小的一步可以是 {行动}。” |
| 恢复确认 | “这不是一下子变好，但你已经有 {证据}，节奏在往回走。” |
| 里程碑确认 | “这已经不是单次记录了，{证据} 说明你跨过了一个小阶段。” |

### 7.3 禁止文案示例

| 不建议 | 原因 |
|--------|------|
| “你是一个很自律的人” | 空泛人格判断 |
| “你最近压力大，所以花钱变多” | 强因果 |
| “你的睡眠问题很严重” | 医疗/健康风险表达过重 |
| “你又失败了” | 指责感强 |
| “根据复杂模型分析” | 用户无感，技术味太重 |

## 8. 数据与隐私边界

公测阶段需要特别注意：

| 边界 | 策略 |
|------|------|
| AI 推断 | 必须可回退，不直接写入用户主动档案 |
| 用户反馈 | 结构化保存，不能把原文直接塞进 Prompt |
| 健康数据 | 只作为生活节奏信号，不做诊断 |
| 想法内容 | 只提炼主题，不暴露隐私细节 |
| 主动提醒 | 默认低频，重要提醒也应给用户控制权 |
| 记忆生成 | 高光和里程碑需要证据链，避免凭空总结 |

## 9. 验收与公测指标

不要只看 AI 准确率。更重要的是用户是否觉得 Holo 更懂自己。

建议指标：

| 指标 | 说明 |
|------|------|
| 洞察有用率 | 用户点击“有用”的比例 |
| 没感觉率 | 用户点击“没感觉”的比例，越低越好 |
| 不准率 | 用户点击“不准”的比例，必须重点追踪 |
| 收藏率 | 用户是否愿意收藏高光或里程碑 |
| 回访问题 | 用户是否开始问“我最近怎么样” |
| 负反馈主题 | 哪类洞察最容易被用户认为没价值 |
| 打扰感 | 用户是否频繁选择“少提醒这个” |

质性问题：

1. 用户有没有觉得 Holo 说的话“像是基于我”？
2. 用户有没有觉得某条洞察值得回看？
3. 用户有没有因为 Holo 的提醒做了一个小动作？
4. 用户有没有觉得 Holo 太爱分析、太爱说教？
5. 用户有没有开始信任 Holo 的长期观察？

## 10. 风险与对策

| 风险 | 表现 | 对策 |
|------|------|------|
| 过度智能感 | AI 自作聪明，用户反感 | 明确不确定表达，不做强判断 |
| Prompt 污染 | 用户反馈原文越塞越乱 | 反馈结构化，聚合后才进摘要 |
| 隐私不适 | 用户觉得 Holo 读得太深 | 给证据、给关闭权、避免隐私细节外露 |
| 冷数据复燃 | 又回到“比日均高 xx%” | 文案验收必须检查生活语言 |
| 高频打扰 | 用户觉得 Holo 太烦 | 默认低频，只在阶段变化时增强存在感 |
| 误判阶段 | 把偶发异常当里程碑 | 里程碑必须满足阶段性、稳定性、相关性、可证据化 |

## 11. 建议落地顺序

### 公测前必须做

1. 统一 Holo AI 文案原则。
2. 重写关键洞察文案，减少冷数据表达。
3. 让 Daily Sense 成为聊天和记忆长廊可共用的“最近状态”。
4. 调整记忆长廊主叙事，减少普通异常，突出高光和里程碑。

### 公测期第一轮迭代

1. 增加轻反馈入口。
2. 统计“不准 / 没感觉 / 有用 / 少提醒”的分布。
3. 根据真实反馈调整洞察排序和文案。
4. 对高频负反馈主题做专项修正。

### 公测后核心升级

1. 建立生活模式模型。
2. 引入个人策略库。
3. 做低频、高相关的主动陪伴。
4. 让 Holo 逐步形成“知道什么建议对这个用户有效”的能力。

## 12. 需要 GLM 重点审查的问题

请重点审查以下问题：

1. 这个方案是否真的能让用户感受到“更懂我”，还是只是换了一套文案？
2. “今日状态摘要”是否适合作为 HoloAI、Daily Sense、记忆长廊的共同上下文？
3. 反馈不直接进入 Prompt、而是结构化聚合后再影响生成，这个边界是否足够安全？
4. 生活模式模型是否应该放到公测后，而不是公测前硬做？
5. 高光和里程碑的判断标准是否足够避免“冷数据异常”抢主叙事？
6. 当前表达原则是否能有效避免 AI 自作聪明、过度分析、说教和隐私不适？
7. 还有哪些公测前必须补上的“低成本、高感知”能力？

## 13. 推荐结论

本方案建议采用 **Conditional Go**：

> 可以进入公测前方案落地，但第一阶段只做“表达原则 + 状态摘要 + 洞察文案 + 记忆筛选”四件事；反馈闭环和生活模式模型应作为公测期与公测后的分阶段能力，不建议在 App Store 发布前一次性做完。

原因：

1. 公测前最重要的是稳定、克制、可感知。
2. Holo 目前不缺单点 AI 能力，缺的是统一理解链路。
3. 生活模式模型是长期壁垒，但需要真实用户反馈训练，不适合闭门一次性定死。
4. 先把“冷数据”变成“生活状态”，用户会最快感受到变化。

最终目标不是让用户觉得：

> “Holo 的 AI 很强。”

而是让用户在某些时刻自然产生：

> “它好像真的在慢慢懂我。”
