# Holo Agent 深度分析 80 分产品设计

## 1. 产品结论

当前深度分析卡片的问题不是单一文案或视觉缺陷，而是 Agent 将“模型和工具如何工作”直接暴露给了用户：内部指标名进入正文，标题按数据域粗放生成，摘要与观察重复，所有内容又被统一包装成“观察 01”。这让用户既无法快速得到答案，也无法判断结论与问题是否相关。

本次采用“结构化答案契约”方案：先把用户问题变成可验收的回答任务，再把工具数据整理成用户可读事实，最后由统一的展示模型决定卡片层级。Prompt 负责推理与表达，确定性代码负责完整性、可读性和兜底，UI 只展示已经整理好的语义，不再解析或修补内部字段。

目标不是把截图这一例改漂亮，而是让健康、财务、习惯、任务、目标、观点、记忆等所有 Agent 查询共享同一套用户友好规则。

## 2. 80 分产品定义

用户打开深度分析时，应当在三秒内回答三个问题：

1. 结论是什么？
2. 这个结论针对我刚才问的什么问题？
3. 它依据哪些数据，边界是什么？

对应的产品标准如下：

- **直接回答**：首屏第一结论必须直接回答用户的问题，不先展示过程、内部字段或泛化观察。
- **主题一致**：标题从本轮用户问题与已回答指标生成，不从“健康”“睡眠”等宽泛数据域猜测。
- **人类语言**：`health.steps.average`、`goal_met_days`、公式名、工具名和 JSON 字段永不进入用户可见区域。
- **逻辑完整**：每个明确子问题都要有对应结论；未回答完整时不能以成功态结束。
- **信息不重复**：外层入口卡片提供结论预览，详情页提供完整解释；同一内容不在标题、摘要、观察正文中重复三次。
- **证据可核对**：结论包含时间范围、有效数据覆盖和必要边界，详细来源默认折叠。
- **克制建议**：查询题只回答事实；用户询问建议或数据出现明确风险时，才展示行动建议。
- **稳定降级**：模型输出不可用时，确定性兜底仍能产出自然、完整、与主题一致的答案。

以“最近一个月平均步数是多少？”为例，首屏应呈现：

> 最近 30 天，日均 6,991 步
>
> 统计到 30 天中的 28 天有效步数记录。期间有 1 天达到 10,000 步。

标题应为“最近一个月的步数”，不能出现“睡眠与健康数据”；详情中的模块标题应为“平均步数”“达标情况”“数据覆盖”，不能出现“观察 01”。

## 3. 备选方案与决策

### 方案 A：仅调整卡片文案和视觉

优点是改动小、短期可改善截图。缺点是原始指标仍可能从新的数据域泄漏，标题漂移和模型不完整回答仍然存在，只能解决表面症状。

### 方案 B：仅加强 Agent Prompt

优点是覆盖面广。缺点是模型输出具有波动性，无法可靠保证所有指标都被翻译、所有子问题都被回答，也无法修复 UI 的重复层级。

### 方案 C：结构化答案契约 + 确定性语义层 + 统一展示模型

优点是把准确性、完整性、可读性和视觉层级分别放到明确职责中，所有数据域可复用，能够稳定达到本次目标。代价是需要同时调整 Agent runtime、渲染结果模型、Prompt 和卡片 UI。

**产品决策：采用方案 C。**

## 4. 用户答案契约

每次 Agent 查询在完成前必须形成以下语义结构：

### 4.1 问题上下文

- `question`：用户本轮原始问题。
- `topic`：本轮唯一主主题，例如“步数”“睡眠”“本月支出”。
- `timeRangeLabel`：用户可读时间范围，例如“最近 30 天”。
- `answerMode`：事实查询、趋势分析、关联分析或建议请求。

### 4.2 核心答案

- `headline`：短标题，只描述本轮主题。
- `directAnswer`：一到两句直接回答，可独立被用户理解。
- `heroMetric`：可选的主数值及单位，用于需要数字回答的查询。

### 4.3 分析段落

每段必须有语义标题和正文：

- 标题表达内容，例如“平均步数”“主要支出”“完成趋势”。
- 正文解释事实、比较或边界。
- 不使用“观察 01/02”“结果 01/02”等无语义编号。
- 同一事实只能承担一个主要展示位置。

### 4.4 证据与边界

- `coverageText`：如“最近 30 天中有 28 天有效记录”。
- `evidenceItems`：用户可读来源摘要，不包含内部字段名。
- `limitations`：缺失数据、口径或不能推断的事项。
- `suggestion`：可选；只有回答模式允许时才存在。

旧消息解码时，如果缺少新增字段，由本地展示适配器从现有 title、summary、sections 和 evidence 中安全降级，不破坏历史数据。

## 5. 系统职责划分

### 5.1 问题任务规划层

Agent 在取数前先识别：用户问了哪些明确子问题、主主题、时间范围和答案模式。最终完成判定按子问题清单检查，而不是按“模型输出了 claim”判断。

例如“最近一个月平均步数是多少”只有一个核心子问题“日均步数”，`goal_met_days` 可以作为辅助解释，但不能反客为主。

### 5.2 工具证据层

工具继续保留稳定的内部 metric key，供验证和计算使用；同时为每个指标提供统一的用户语义：

- 中文名称；
- 合理精度；
- 单位和量词；
- 数值句式；
- 是否适合作为主指标；
- 可读证据摘要。

健康步数汇总证据应写成“最近 30 天平均每天 6,991 步”，而不是“步数汇总：health.steps.average = 6990.80 步”。

该语义映射作为跨领域统一格式化入口。未知指标不得原样回显 metric key；只能降级为安全的通用描述，并在开发日志中记录缺失映射。

### 5.3 Agent 整理与完成层

Prompt 负责基于证据组织逻辑紧密的表达；runtime 负责确定性约束：

- 校验所有子问题是否已回答；
- 删除无证据或重复 claim；
- 将模型漏答的指标用用户可读事实补齐；
- 阻止内部字段、工具名和公式进入用户文本；
- 根据问题生成主题标题；
- 为事实查询关闭无意义建议；
- 模型失败时用同一答案契约生成兜底结果。

模型不再负责决定“观察 1/2”这类展示编号，结果 renderer 也不再自动添加编号。

### 5.4 展示模型层

渲染结果从“title + summary + numbered sections”升级为面向用户的答案模型。Renderer 只组装已校验的可读语义，不再把所有 claim 简单拼成 summary，也不再把 section.title 强制写成“观察 N”。

兼容策略：保留旧 Codable 字段和默认值，新增语义字段均可选；新卡片优先读取新字段，旧历史消息通过适配器整理后展示。

### 5.5 UI 层

#### 外层入口卡片

- 主题标题；
- 一句直接答案或主数值；
- 一句数据范围/覆盖说明；
- “查看分析”入口。

外层不再展示“核心观察 / 观察 1”，也不重复详情页第一段全文。

#### 深度分析详情

从上到下固定为：

1. 本轮主题与直接答案；
2. 最多三个有语义标题的分析段落；
3. 数据覆盖或能力边界；
4. 折叠的数据依据；
5. 可选的行动建议。

视觉上减少大标题重复和卡片套卡片；主答案使用最高层级，分析段落使用轻量分组，依据保持低干扰。详情不再显示“观察 01/02”。

## 6. 标题与主题规则

标题优先级：

1. 用户问题中的明确对象；
2. 已回答的主指标语义；
3. 数据域通用标题，仅作为最后降级。

示例：

| 用户问题 | 正确标题 | 禁止标题 |
|---|---|---|
| 最近一个月平均步数是多少 | 最近一个月的步数 | 最近的睡眠与健康数据 |
| 最近睡眠质量怎么样 | 最近的睡眠情况 | 健康数据结果 |
| 本月钱都花哪了 | 本月支出去向 | 深度分析 |
| 最近任务完成得怎么样 | 最近的任务进展 | 本期观察 |

当一个问题确实跨领域时，标题描述用户问到的关系，例如“睡眠与步数的同期变化”，而不是笼统写“健康数据”。

## 7. 异常与降级

- **无数据**：直接说明哪个时间范围没有有效数据，并给出数据权限或记录覆盖边界；不生成空“观察”。
- **数据覆盖不足**：仍可回答事实，但必须展示有效天数；趋势和关联结论降低置信度或不输出。
- **部分回答**：runtime 自动补齐；无法补齐时明确说缺少哪部分，不能伪装完成。
- **未知指标**：不回显内部 key；使用安全说明并记录开发告警。
- **主题冲突**：如果标题主题与问题主题不一致，使用确定性标题覆盖模型标题。
- **重复内容**：规范化文本后去重；标题、直接答案和分析段落不得重复相同句子。
- **模型不可用**：确定性 formatter 按工具结果生成直接答案、覆盖信息和证据，不退回机器格式。

## 8. Prompt 双端同步

Agent Loop Prompt 增加统一回答契约：

- 先列出用户明确子问题并逐项回答；
- `displayText` 必须是无需内部上下文也能理解的中文；
- 禁止输出 metric key、工具名、JSON 字段和“观察 N”；
- 标题必须绑定本轮主题；
- 查询题默认不输出 suggestion；
- 完成前检查直接答案、数据范围、证据与边界。

改动同时更新 iOS `PromptManager.swift` 后备模板与后端 `defaultPrompts.json` 生效模板，并提升 `promptVersions.agent_loop`。部署后通过生产元数据、Prompt 内容和真实 Agent 请求验收。

## 9. 验收与测试

### 9.1 关键截图用例

输入：“最近一个月平均步数是多少？”

必须满足：

- 主答案包含日均步数，数值使用整数千位格式；
- 标题包含“步数”，不包含“睡眠”；
- 页面不存在 `health.`、`goal_met_days`、`average =`；
- 页面不存在“观察 01/02”；
- 外层与详情不重复同一套标题和正文；
- 展示最近 30 天及有效记录天数；
- 达标天数仅作为辅助信息；
- 证据展开后仍是自然中文。

### 9.2 跨领域回归矩阵

至少覆盖：

- 健康：步数单值、睡眠分析、站立趋势、无数据、覆盖不足；
- 财务：总额、平均每天、分类去向、多子问题；
- 习惯：完成率、负向习惯、趋势；
- 任务与目标：完成情况、积压、关联进展；
- 跨领域：健康与习惯、任务与目标的相关性表达；
- 旧消息：缺少新字段时仍可读且不崩溃。

### 9.3 自动化断言

- 指标格式化映射覆盖所有工具声明的 outputMetrics；
- 任何用户可见文本不得匹配内部 key 形态；
- section 标题不得为通用编号；
- 单主题标题必须包含对应主题且不得包含无关主题；
- 多子问题必须全部有回答；
- 查询模式不得生成泛化建议；
- 新旧结果 JSON 均能正确解码。

### 9.4 交付验证

- standalone 纯逻辑测试真实执行并通过；
- 相关 Xcode 测试或 `test_sim` 真实执行，不接受 `Executed 0 tests`；
- iOS Simulator 完整构建通过；
- 后端测试通过；
- ECS 完成 Docker 重建并通过 `/v1/health`、`/v1/prompts/meta`、`/v1/prompts/agent_loop` 与 `verify:prod`；
- 使用真实问题完成一次端到端 Agent 结果检查。

## 10. 非目标

- 不在本次重做整个 Chat 信息架构。
- 不引入与理解无关的复杂图表或动画。
- 不让模型进行医疗诊断、无证据因果推断或人格判断。
- 不用截图特判、字符串替换某一个错误标题或仅删除“01”来冒充系统性修复。
