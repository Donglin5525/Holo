# Holo 生活轨迹 AI 洞察升级实施方案 V2

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将 Holo 从“个人数据查询与周期总结助手”升级为“基于真实记录的生活轨迹观察助手”，让财务、习惯、待办、想法数据形成有证据、有温度、有深度、不过度推断的 AI 洞察。

**Architecture:** 本方案不新建并行 AI 系统，继续基于现有 HoloBackend Prompt 托管、SQLite Prompt 版本体系、iOS `HoloBackendAIProvider`、`MemoryInsightContextBuilder`、`MemoryInsightService` 和 `CrossModuleCorrelator` 演进。Prompt 负责定义洞察方法、表达边界和输出格式；代码负责 Prompt 生效链路、Prompt 版本绑定、洞察缓存一致性、生活轨迹上下文、个人基线和跨模块规则。

**Tech Stack:** Holo iOS Swift / SwiftUI / Core Data，HoloBackend Node.js / Hono / better-sqlite3 / Prompt Registry / 管理后台，现有 AI Chat Completions 网关。

---

## 一、背景与目标

### 1.1 当前能力边界

本方案严格基于 Holo 当前已有或已接入的能力设计，不假设 App 拥有未接入的数据。

| 模块 | 当前可用能力 | 可用于洞察的边界 |
|------|--------------|------------------|
| 财务 | 收入/支出、分类、预算、日消费、分类聚合、上期对比、异常描述 | 可以观察支出结构、分类变化、预算异常、日消费波动 |
| 习惯 | 活跃习惯、打卡记录、完成率、连续打卡、断连检测 | 可以观察稳定习惯、断连、恢复、完成率变化 |
| 待办 | 创建、完成、更新、删除、逾期、优先级、完成统计、趋势 | 可以观察任务闭环、堆积、重要任务推进 |
| 想法 | 文本记录、近期片段、心情/标签分布、文本内容注入 | 可以总结用户自己写下的主题，不做心理诊断 |
| AI Chat | 意图识别、执行动作、分析查询、流式回答 | 可以回答主动分析问题，但必须基于上下文 |
| 记忆洞察 | 日报/周报/月报/年度回顾、结构化 JSON 输出、持久化 | 可以生成周期回放和卡片 |
| 后端 | AI 网关、Prompt 托管、SQLite Prompt 历史、diff、回滚、日志、模型路由 | 可以承载 Prompt 发布、版本、测试、元数据查询 |

Holo 不能洞察以下内容，除非用户在记录中明确提供：

- 睡眠质量。
- 地理位置。
- 屏幕使用时间。
- 真实工作压力。
- 社交关系。
- 医疗或心理诊断。
- 未接入的健康指标。

### 1.2 产品目标

升级后的 Holo 要能从用户已有记录中观察：

1. 财务上钱流向哪里，是否偏离个人常态。
2. 习惯上哪些承诺被兑现，哪些习惯断连，是否出现恢复。
3. 待办上任务是否闭环，是否堆积，重要任务是否推进。
4. 想法上用户反复写到什么主题，但不做心理诊断。
5. 跨模块上多个现象是否在同一时间段并发出现，但不做武断因果判断。

最终用户感受到的不是“AI 在写漂亮总结”，而是：

> Holo 不替用户编故事，但能把用户自己留下的数据串成一条温柔、准确、可回看的生活轨迹。

### 1.3 核心原则

| 原则 | 要求 |
|------|------|
| 证据优先 | 每条洞察必须来自输入上下文中的事实 |
| 克制表达 | 不诊断、不审判、不人格评价 |
| 分层洞察 | 区分事实、变化、模式、关联、假设、建议 |
| 弱因果约束 | 跨模块只能说并发、相关、值得留意，不能说导致/证明 |
| 小切口建议 | 建议必须具体、轻量、可执行 |
| 后端优先 | 正式 Prompt 以 HoloBackend 托管版本为准 |
| 单一事实源 | Prompt 版本历史以 SQLite `prompt_versions` 为主，JSON 只作降级兼容 |

## 二、现有链路与关键修正

### 2.1 Prompt 获取链路

```text
HoloBackend SQLite prompt_versions
  -> promptRegistry.getPrompt(type)
  -> GET /v1/prompts/:type
  -> HoloBackendPromptService.loadPrompt()
  -> HoloBackendAIProvider
  -> /v1/ai/chat/completions
  -> 上游模型
```

当前事实：

- `promptRegistry.js` 已优先从 SQLite `prompt_versions` 读取最新版本。
- `managedPrompts.json` 仍存在，但只应视为兼容降级，不应再扩展为新的主存储。
- 后端已具备版本递增、历史查询、diff、非破坏式回滚。
- iOS `HoloBackendPromptService` 当前只缓存正文，不缓存版本，也没有 TTL 或 metadata 检查。

### 2.2 记忆洞察生成链路

```text
MemoryInsightContextBuilder
  -> MemoryInsightContext JSON
  -> AIProvider.generateMemoryInsight()
  -> memory_insight_generation Prompt
  -> MemoryInsightResponseParser
  -> MemoryInsightRepository.saveReady()
  -> MemoryGallery 展示
```

必须修正的当前问题：

1. `generateMemoryInsight()` 没有强制 `responseFormat: .jsonObject`。
2. `MemoryInsightService.saveReady()` 当前写死 `promptVersion: 4`，无法准确记录真实 Prompt 版本。
3. `MemoryInsight.fetchAvailable()` 只按 `periodType + periodStart + status` 查询，不含 `sourceSnapshotHash` 和 `promptVersion`。
4. `MemoryInsightService.checkSimilarityWithRecent()` 的 Jaccard 去重不检查 `promptVersion`，Prompt 升级后可能仍复用旧洞察。
5. 旧记录 `promptVersion = 0` 需要兼容策略，不能直接破坏用户已有记忆内容。

### 2.3 升级后目标链路

```text
HoloBackend Prompt Registry
  -> SQLite prompt_versions
  -> /v1/prompts/meta 批量版本元数据
  -> /v1/prompts/:type 获取正文
  -> iOS CachedPrompt(content, version, loadedAt)
  -> AI 调用记录 promptType + promptVersion

MemoryInsightContextBuilder
  -> 原有四模块上下文
  -> DailyLifeSnapshot 每日快照
  -> LifeEvent 关键事件流
  -> PersonalBaseline 个人基线
  -> CrossModuleCorrelation 扩展后的跨模块模式
  -> 渐进式 token budget 裁剪

MemoryInsightService
  -> 以 sourceSnapshotHash + promptVersion 控制缓存
  -> Jaccard 去重只比较相同 promptVersion
  -> 生成结果保存真实 promptVersion
```

## 三、Prompt 改动

所有正式 Prompt 改动通过 HoloBackend 管理后台发布。iOS 本地 `PromptManager` 只作为后端不可用时的 fallback。

### 3.1 `system_prompt`

**目标：** 将 Holo 的基础身份从“个人数据管理助手”升级为“生活轨迹观察助手”。

**建议内容：**

```text
你是 Holo，用户的个人生活轨迹观察助手。你的职责不是评价用户，而是基于真实记录，帮助用户看见财务、习惯、待办、想法之间的节奏、变化、偏离和恢复。

今天是 {{todayDate}}。

核心原则：
- 只基于上下文中的真实数据回答，禁止编造数字、日期、分类、任务或习惯。
- 区分事实、观察、假设和建议。没有证据时只能说“不足以判断”。
- 不做心理诊断，不评价人格，不使用羞辱、审判或命令式表达。
- 跨模块关系只能表达为“并发现象”或“值得留意的关联”，不能说“导致、证明、说明一定是因为”。
- 对金额、次数、完成率、日期保持精确。
- 如果用户要执行记账、创建任务、打卡等操作，而当前链路无法确认执行成功，不要假装已完成。
- 回复使用中文，克制、温和、具体。少说空泛鼓励，多给一个可执行的小切口。
```

### 3.2 `analysis_prompt`

**目标：** 用户主动问“我最近怎么样”“帮我分析一下这周”时，输出有层次的分析，而不是普通报表。

**建议内容：**

```text
你是 Holo 的个人数据分析助手。你会收到结构化 JSON 上下文和用户问题，请基于数据生成分析。

分析顺序：
1. 事实：先列出上下文中明确存在的数据。
2. 变化：对比本期与上期、工作日与周末、不同分类或不同习惯的差异。
3. 模式：识别重复出现的时间、分类、任务节奏或习惯断连。
4. 关联：如果多个模块同时变化，只能描述并发关系，不做因果断言。
5. 建议：给出一个具体、轻量、可执行的下一步。

规则：
- 禁止编造数据。
- 禁止重新估算上下文没有给出的数字。
- 明确区分“数据支持的观察”和“个人建议”。
- 如果样本不足，请直接说明样本不足。
- 语气要有温度但不软化事实；锐利来自证据，不来自批评。
- 不输出 JSON，只输出 Markdown 文本。
- 控制在 300-500 字。
```

### 3.3 `memory_insight_generation`

**目标：** 将日报/周报/月报从周期总结升级为生活轨迹回放。

**硬性约束：**

- 保持现有 JSON Schema，避免前端解析崩。
- 每张卡片必须有 evidence。
- 数据稀疏时少输出，不硬凑。
- 跨模块只表达并发，不表达因果。

**建议内容：**

```text
你是 Holo 的生活轨迹洞察生成器。你的任务是基于用户一个周期内的结构化数据，生成可复看的记忆回放。

你要观察的不是单个模块，而是用户这一周期的生活轨迹：
- 财务：钱流向哪里，是否偏离个人常态，是否出现异常集中。
- 习惯：哪些承诺被兑现，哪些习惯断连，是否出现恢复。
- 待办：任务是否闭环，是否堆积，重要任务有没有推进。
- 想法：用户反复写到什么主题，情绪基调如何，但不得心理诊断。
- 跨模块：多个模块是否在同一时间段出现并发变化。

洞察层级：
- fact：明确事实。
- change：本期相对上期的变化。
- pattern：重复出现的模式。
- correlation：跨模块并发现象。
- hypothesis：低置信度假设，必须用“可能/值得留意”。
- suggestion：轻量建议。

硬性规则：
1. 只使用输入数据中的事实。
2. 不编造金额、日期、次数、完成率。
3. 不做人格判断或心理诊断。
4. 跨模块不得使用“导致、因为、证明、说明”。
5. 每张卡片必须有 evidence；没有 evidence 就不要生成。
6. 数据稀疏时少输出，不要硬凑。
7. 优先识别“偏离常态、连续变化、恢复迹象、任务堆积、预算异常、习惯断连”。

语言风格：
- 克制、温暖、具体。
- 不说教，不审判。
- 标题要有画面感，但正文必须落回证据。
- 建议只给一个小动作，不要泛泛规划人生。

输出严格 JSON，不要在 JSON 外添加任何文字。
```

### 3.4 `annual_review`

**目标：** 年度回顾不做逐月流水账，而是观察长期模式、转折点和恢复能力。

**建议内容：**

```text
你是 Holo 的年度生活轨迹回顾助手。你会收到用户一年内的月度洞察摘要和年度汇总数据。

你的目标不是逐月流水账，而是找出这一年里反复出现的模式、转折点和恢复能力。

请重点观察：
- 财务：全年支出结构、异常月份、预算压力与改善。
- 习惯：哪些习惯真正稳定，哪些反复中断，何时恢复。
- 待办：长期任务推进节奏，高优先级事项是否被完成。
- 想法：反复出现的主题和表达变化。
- 跨模块：多个模块在同一月份的并发现象，但不得做因果断言。

规则：
- 只基于输入数据。
- 不编造缺失月份。
- 不做心理诊断。
- 不评价人格。
- 识别转折点时必须引用月份或证据。
- 同时写出“值得记住的进展”和“下一年最小改进方向”。

输出严格 JSON，不要在 JSON 外添加任何文字。
```

### 3.5 `intent_recognition`

**目标：** 只做小补丁，避免影响记账、待办、打卡执行链路。

**新增规则：**

```text
额外规则：
- “复盘、总结、看看这周/本月状态、我最近怎么样”优先识别为 generate_memory_insight 或 query_analysis。
- 如果用户想要跨财务、习惯、待办一起分析，analysisDomain 使用 crossModule。
- 如果一句话同时包含执行动作和分析查询，返回 clarification，不要混合执行。
```

### 3.6 其他 Prompt

`clarification`、`response_template`、`data_extraction` 保持轻量改动，不作为本期主要风险点。

## 四、数据模型与上下文设计

### 4.1 不新增并行 `CrossModulePattern`

本期不新建 `CrossModulePattern`。直接扩展现有 `CrossModuleCorrelation`。

**Modify:** `HOLO/Holo/Holo APP/Holo/Holo/Services/AI/CrossModuleCorrelator.swift`

建议目标结构：

```swift
struct CrossModuleCorrelation: Codable, Equatable, Sendable {
    let modulePair: [AnalysisDomain]
    let observation: String
    let signalStrength: Double
    let summary: String
    let patternType: String?
    let evidenceDates: [String]
}
```

兼容策略：

- 现有规则可先传 `patternType: nil`、`evidenceDates: []`。
- 新规则必须填 `patternType` 和 `evidenceDates`。
- Prompt 只读取 `crossModuleCorrelations` 一个字段，避免新旧模型并存。

### 4.2 `LifeEvent`

**目标：** 给 AI 一条轻量的关键事件流，让它能看到周期内“哪天发生了什么”。

**新增位置建议：** 实际落地前先确认 `MemoryInsightModels.swift` 所在路径，优先放在现有 MemoryInsight 模型文件内，而不是新增孤立模型文件。

建议结构：

```swift
struct LifeEvent: Codable, Equatable, Sendable {
    let id: String
    let date: String
    let module: String      // finance | habit | task | thought
    let type: String        // expense | income | habitCompleted | habitMissed | taskCompleted | taskOverdue | thoughtCreated
    let title: String
    let valueText: String?
    let tags: [String]
    let sourceId: String?
}
```

设计说明：

- 不使用 `Double` 存金额，避免浮点精度风险。
- 第一版不引入 `intensity`，避免无定义字段被 AI 误用。
- `valueText` 只用于给 AI 阅读，例如 `¥35.20`、`完成 3 次`、`逾期 2 天`。

### 4.3 `DailyLifeSnapshot`

**目标：** 支持“这周不是全程混乱，而是某两天节奏被打断”这类局部节奏观察。

建议结构：

```swift
struct DailyLifeSnapshot: Codable, Equatable, Sendable {
    let date: String
    let expenseTotalText: String
    let taskCreatedCount: Int
    let taskCompletedCount: Int
    let overdueCount: Int
    let habitCompletionRate: Double?
    let thoughtCount: Int
    let topSignals: [String]
}
```

### 4.4 `PersonalBaseline`

**目标：** 避免用通用标准评判用户，让 Holo 能说“这不像你平常的节奏”。

建议结构：

```swift
struct PersonalBaseline: Codable, Equatable, Sendable {
    let baselineStart: Date
    let baselineEnd: Date
    let effectiveWeekCount: Int
    let expenseWeeklyAverageText: String?
    let categoryAverages: [CategoryBaseline]
    let taskCompletionRateAverage: Double?
    let habitCompletionRateAverage: Double?
    let usualHighExpenseWeekdays: [String]
}

struct CategoryBaseline: Codable, Equatable, Sendable {
    let categoryName: String
    let weeklyAverageText: String
}
```

基线规则：

- 基线期不与观察期重叠。
- 默认取观察期之前 4 周。
- 至少需要 2 个有效周才生成基线。
- 空周跳过，不做插值。
- 数据不足时不输出“偏离常态”类洞察。
- `usualHighExpenseWeekdays` 需要显著性门槛，建议某星期几日均支出 >= 基线日均 1.5 倍时才列入。

### 4.5 Token Budget 策略

当前 `enforceTokenBudget()` 使用 `encoded.count / 2` 粗估 token，且年度上下文不调用该方法。V2 必须改为更明确的渐进式裁剪。

建议预算：

| 周期 | 预算 |
|------|------|
| daily | 800 |
| weekly | 2200 |
| monthly | 3800 |
| annual | 5000 |

估算策略：

- 短期继续使用近似估算，但封装为独立方法 `estimateTokenCount(_:)`。
- 修正 CJK 粗估：可先用 `String.count * 1.5` 作为保守估算，而不是 `encoded.count / 2`。
- 后续如需要，再替换为真正 tokenizer。

渐进式裁剪顺序：

1. 裁剪 `thoughts.textContents` 到 5 条。
2. 裁剪普通 `LifeEvent`。
3. 裁剪 `dailyExpenses` / `dailyCompletionTrend` 到最近 14 天。
4. 保留 anomaly、budget overrun、high priority task、habit recovery、crossModuleCorrelation。
5. 每次裁剪后重新估算，直到低于预算。

年度上下文：

- `buildAnnualContext()` 必须调用预算裁剪，或新增年度专用裁剪。
- 当前年度 context 使用 `periodType: .monthly` 可能混淆，V2 实施时需要新增 `.annual`，或在代码注释中明确说明年度上下文复用 `.monthly` 的原因。

## 五、跨模块规则设计

### 5.1 现有规则处理策略

当前 `CrossModuleCorrelator` 已有 6 条规则。本期不是简单“新增 5 条”，而是替换、增强、去重。

| 目标规则 | 处理方式 | 说明 |
|----------|----------|------|
| 任务堆积 + 消费上升 | 替换/增强 `detectTaskFinanceCorrelation` | 不并行输出重复洞察 |
| 习惯断连 + 餐饮集中 | 增强 `detectHabitFinanceCorrelation` | 从全期完成率下降升级为连续断连 + 餐饮集中 |
| 重要任务完成 + 习惯恢复 | 新增 | 识别恢复迹象 |
| 周末支出集中 + 工作日低完成 | 增强 `detectWeekdayWeekendPattern` | 从纯财务差异升级为跨模块候选模式 |
| 恢复迹象 | 新增 | 优先展示，避免洞察只盯问题 |

### 5.2 去重规则

同一周期内，同一 `modulePair + evidenceDates + patternType` 不应重复输出。

建议实现：

```swift
private static func deduplicateCorrelations(_ items: [CrossModuleCorrelation]) -> [CrossModuleCorrelation]
```

去重优先级：

1. `patternType` 更具体者优先。
2. `signalStrength` 更高者优先。
3. 有 `evidenceDates` 者优先。
4. 恢复类模式优先于单纯风险类模式。

### 5.3 语言约束

规则引擎输出的 `observation` / `summary` 不得包含：

- 导致。
- 证明。
- 说明一定。
- 因为。
- 自控力差。
- 心理诊断词。

Prompt 层仍需二次约束，但规则层先降低风险。

## 六、后端实施计划

### Task 1: 新增 Prompt Metadata API

**Files:**

- Modify: `HOLO/HoloBackend/src/app.js`
- Modify: `HOLO/HoloBackend/src/prompts/promptRegistry.js`
- Test: `HOLO/HoloBackend/tests/chat.test.js` 或新增 `HOLO/HoloBackend/tests/prompts.test.js`

**目标：** 提供不含正文的批量 Prompt 元数据，供 iOS 判断缓存版本。

**接口：**

```http
GET /v1/prompts/meta
```

**Response:**

```json
{
  "prompts": [
    {
      "type": "memory_insight_generation",
      "version": 5,
      "source": "managed",
      "updatedAt": "2026-05-16 00:00:00"
    }
  ]
}
```

## 十一、风险与处理

| 风险 | 影响 | 处理 |
|------|------|------|
| Prompt 变好但上下文不足 | 输出仍像模板 | Phase 1 只承诺表达升级，深度洞察等 Phase 4 |
| 跨模块误判因果 | 用户信任受损 | Prompt + 规则 + 测试三层禁止因果化 |
| Prompt 不生效 | 后端改了真机仍旧 | Phase 2 修版本缓存和刷新入口 |
| 旧洞察挡住新版 | V2 Prompt 看不到效果 | Phase 3 修 hash/version/Jaccard |
| 新上下文过大 | 生成失败或超时 | Phase 4 渐进式裁剪，必要时评估 45-60 秒超时 |
| 年度上下文语义混乱 | 年度回顾逻辑不清 | 增加 `.annual` 或明确 `.monthly` 复用含义 |
| `habitCategoryCompletionSummaries` 为空 | 习惯分类洞察无法成立 | 本期不依赖该字段，后续如需要再实现 |
