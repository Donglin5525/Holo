# HoloAI 记忆闭环与显性感知重构方案

> 日期：2026-07-14
>
> 范围：旧格式清理、新格式唯一化、候选身份与证据合并、统一召回上下文、记忆写入/使用回执、洞察与 Agent 召回。
>
> 不在本轮：Agent Memory Curator 自动写入真实记忆 Store。

## 1. 产品结论

Holo 的长期记忆不应继续承担“保存洞察卡片副本”的职责。它只保存未来确实能帮助 Holo 理解用户的稳定信息，并且必须能回答三个问题：

1. Holo 记住了什么。
2. 这条记忆以后会在什么场景使用。
3. 用户如何撤销、纠正或拒绝它。

洞察负责发现可能值得记住的变化；深度分析负责用本地证据验证当前问题；记忆负责把已经确认或低风险稳定的信息带到未来相关场景。三者共享同一个召回入口，但保持不同职责。

## 2. 已确认的产品决策

1. 删除所有 `semanticType == nil` 的旧格式长期记忆，包括待确认、已确认、已拒绝和已归档。
2. 删除旧格式生成、召回和 UI 兼容入口，不保留双轨逻辑。
3. 删除记忆副本不影响原始账单、习惯、任务、想法、洞察和 Agent 结果。
4. 修正新格式候选身份，按语义身份跨周期合并证据，不再按洞察卡片序号覆盖。
5. 升级 `HoloMemorySnapshotBuilder` 为统一召回上下文入口。
6. 普通对话、目标规划、洞察和 Agent 按各自场景使用统一召回结果。
7. 增加写入回执、使用回执和可展开来源面板。
8. 洞察可以读取 1～2 条相关且已确认的稳定记忆；Agent 根据问题语义决定是否预取记忆与历史洞察。
9. `HoloMemoryCurator` 本轮只保留规则判断，不接入真实 Store。

## 3. 方案比较

### 方案 A：继续兼容旧格式

保留 `semanticType == nil` 数据和 fallback，逐步迁移。

- 优点：历史记忆不会立即消失。
- 缺点：流水账噪声、旧卡片文案和召回污染继续存在；每个消费端都要维持双轨逻辑。
- 结论：不采用。

### 方案 B：新旧 Store 并存

新建第二套新格式 Store，旧 Store 只读归档。

- 优点：回滚容易。
- 缺点：记忆管理页、召回、删除和 CloudKit/本地文件边界更复杂；用户仍会看到两套“记忆”。
- 结论：不采用。

### 方案 C：单一新格式 + 一次性清理 + 统一召回

启动时一次性删除旧格式记录，所有新写入必须经过语义 Mapper 和晋升策略；被拒绝的候选直接丢弃。所有 AI 场景通过统一召回入口取得可用记忆和来源元数据。

- 优点：产品语义单一，后续链路更容易验证；能够彻底结束旧格式污染。
- 风险：旧格式中可能包含少量有价值信息，但原始业务数据仍保留，可由后续洞察重新发现并按新格式生成。
- 结论：采用。

## 4. 数据模型与迁移

### 4.1 新格式唯一条件

一条可用长期记忆必须满足：

- `semanticType` 非空。
- `displaySummary` 非空，供用户审核。
- `aiUseSummary` 非空，供 AI 召回。
- `useScopes` 非空。
- `prohibitedInferences` 已按语义类型和敏感度生成。
- 至少一条可追溯证据。

不再允许 `title + summary + legacy type` 作为召回 fallback。

`HoloLongTermMemory` 领域模型同步收紧：删除 `HoloLongTermMemoryType`、`type` 和兼容用 `summary` 字段；`semanticType`、`displaySummary`、`aiUseSummary`、`useScopes`、`prohibitedInferences` 改为非 Optional。旧 JSON 不再直接解码成领域模型，而由一次性的宽松迁移 DTO 解码、筛选并转换，避免为了兼容旧数据继续把 nil 状态带进所有消费端。

### 4.2 一次性旧格式清理

在长期记忆 Store 增加带版本标记的清理：

1. 首次运行新版本时用 `HoloLongTermMemoryMigrationRecord` 宽松解码全部长期记忆。
2. 删除所有 `semanticType == nil` 的记录，不区分确认状态。
3. 保存剩余新格式记录。
4. 写入迁移版本标记，后续启动不重复执行。
5. 记录删除数量，不记录或上传记忆正文。

清理只作用于 `HoloLongTermMemories.json`，不访问各业务 Repository。

迁移在 `HoloApp` 启动、候选 Observer 注册前执行。文件采用临时文件原子替换：只有新格式文件保存成功后才写迁移完成标记；保存失败时保留原文件并在下次启动重试。

### 4.3 删除旧入口

- `HoloLongTermMemoryCandidateObserver`：只有 payload 含合法 `memoryCandidate` 时才进入 Mapper；没有新字段或 Mapper 拒绝时直接丢弃，不再构造 `.recurringPattern` 旧候选。
- `HoloMemoryPromotionPolicy`：删除 `semanticType == nil` 的 legacy 分支。
- `HoloMemorySummaryProvider`：删除按 legacy type 选择和 `title：summary` fallback。
- `HoloMemoryPromptSummary`：删除 `lines` 字段，只保留结构化 `entries`、`sourceIDs` 和覆盖度。
- `AIUserContextMessageBuilder`：只渲染结构化 `entries`。
- `HoloMemoryCenterView`：不再展示旧格式兼容状态或旧类型说明。

## 5. 候选身份、合并与晋升

### 5.1 稳定语义身份

不能继续使用 `candidate-\(card.id)`。`habit_1`、`finance_1` 等只是单份洞察内序号，会跨报告重复。

`MemoryCandidatePayload` 新增必填 `subjectKey`，表示单一、稳定的记忆主题，例如 `habit:running`、`finance:dining`、`task:execution_rhythm`。后端 Prompt 负责输出候选主题，本地只接受 2～48 个字符的中文、英文、数字、下划线和冒号，并做小写化、空白压缩和非法字符剔除。

新增本地 `memoryKey`，由以下字段规范化后生成稳定哈希：

- `semanticType`
- 主领域：habit / finance / task / milestone
- 本地校验后的 `subjectKey`

最终 ID 使用 `memory-v2-<hash>`。日期、金额和洞察 card ID 不进入稳定身份。

若 `subjectKey` 缺失或不合法，Mapper 直接拒绝，不使用展示摘要猜测身份，也不生成 fallback 候选。

### 5.2 跨周期证据合并

相同 `memoryKey` 再次出现时：

- 保留原 `createdAt`。
- 更新 `updatedAt`。
- 证据优先按 `source + sourceID` 去重；`sourceID` 缺失时按 `source + 规范化 excerpt + 业务日期` 去重。候选抽取时间不参与证据身份，避免每次生成洞察都制造一条“新证据”。
- 重新计算 confidence 和晋升决策。
- 已确认记忆不得被新候选降回待确认。
- `displaySummary` 和 `aiUseSummary` 允许用更新版本替换，但不得丢失用户编辑结果；本轮模型尚无用户编辑字段，因此先保留已确认记录的现有摘要。

### 5.3 晋升规则

- `stablePattern`：至少 3 条跨周期证据、普通敏感度时可静默接受。
- `phaseShift`：至少 2 条证据，必须确认。
- `driftSignal`：至少 2 条证据，必须确认并设置 21～30 天过期。
- `lifeEvent`：至少 1 条证据，必须确认。
- `statMilestone`：只展示和回顾，不进入核心对话召回，不要求用户确认。
- 新规则拒绝即结束，不允许回退生成旧候选。

## 6. 统一记忆召回上下文

### 6.1 `HoloMemorySnapshotBuilder` 升级

将当前无人使用的短期快照 Builder 升级为统一入口，输入至少包含：

- `purpose`：chat / goalPlanning / insight / agentAnalysis / retrospective
- `userText`：当前问题或任务描述
- `timeWindow`
- `limits`

输出统一 `HoloMemoryContextEnvelope`：

- 当前会话与近期结构化状态摘要。
- 相关情景记忆。
- 相关已确认/静默接受长期记忆。
- 可选的历史洞察摘要。
- 数据覆盖度。
- 每条来源的稳定 ID、展示标题和使用范围。

### 6.2 相关性排序

召回排序按以下顺序综合：

1. 当前问题与记忆领域、主题词的匹配度。
2. `useScopes` 是否允许当前场景。
3. Profile 与当前明确输入的冲突检查；当前输入永远优先。
4. 置信度。
5. 更新时间和过期时间。
6. 敏感记忆是否与当前话题直接相关。

普通聊天默认最多 5 条；目标规划最多 3 条；洞察最多 2 条；Agent 预取最多 3 条，仍可在后续轮次通过工具继续查询。

### 6.3 各链路接入

- 普通聊天：替换当前“无 userText 的最新 5 条”选择，使用本轮问题做相关召回。
- 分析查询：数字型确定查询不注入长期记忆；趋势、原因、建议和复盘型查询才允许召回。
- 目标规划：召回目标、执行偏好和已确认稳定模式。
- 洞察：只注入 1～2 条相关稳定记忆，作为个性化背景而非事实证据；当前周期数据与记忆冲突时以当前数据为准。
- Agent：新增本地 `HoloAgentMemoryPrefetchPolicy`。确定数字查询不预取；原因、建议、模式、复盘和“你还记得吗”类问题，在首轮 LLM 前确定性执行 Memory/Insight Tool，将结果按现有 canonical evidence 流程写入 checkpoint 和 Evidence Ledger。预取结果不是旁路 Prompt 文本，最终结论仍必须引用工具证据。

## 7. 用户显性感知

### 7.1 写入回执

新增持久化 `HoloMemoryReceiptCenter`。Store 写入成功后只投递结构化回执，不传递原始证据正文；前台全局一次展示，后台生成的回执在用户下次进入 HoloAI 或记忆长廊时消费。

仅在记忆状态真正发生变化时显示：

- 静默接受：`已记住：你更适合把计划拆成小步骤`，附“撤销”。
- 需要确认：展示“建议记住”卡片，说明以后会在哪些场景使用。
- 证据不足：不展示、不打扰。
- `statMilestone`：进入回顾收藏，不显示“AI 以后会据此判断你”。

### 7.2 使用回执

只有回答确实引用记忆时，才在回答或结果卡底部展示：

`参考了：2 条长期记忆 · 最近 7 天记录`

为了避免伪造“使用过”，普通流式回答使用受控尾标记协议：当模型确实使用某条记忆时，在回答末尾追加 `[[HOLO_MEMORY_IDS:id1,id2]]`。iOS 新增流式解析器，始终保留一个标记前缀长度的尾缓冲，不把控制标记显示给用户；完成后只接受本轮召回集合内的 ID，并持久化为消息的 `memoryReferencesJSON`。标记缺失、截断或含未知 ID 时不展示记忆回执。

Agent 不使用尾标记；它以最终 claim 实际引用的 Memory/Insight canonical evidence 反推使用回执。洞察以本次注入的稳定记忆 ID 记录“结合的背景”，但页面文案明确区分“背景”与“本期证据”。

### 7.3 来源面板

点击使用回执后展示：

- 本轮使用的记忆标题与事实摘要。
- 为什么与当前问题相关。
- 原始证据数量与日期范围。
- “准确”“不相关”“修改”“忘记”操作。

本轮先完成本地模型、消息元数据和可复用 UI 组件；不同结果卡按相同组件接入。

“修改”采用本地确定性规则：用户修改事实摘要后，记忆转为 `.confirmed`，`displaySummary` 使用用户文本，`aiUseSummary` 更新为“用户明确纠正：<文本>。仅在相关场景使用，不得扩展推断”，并保留原证据用于审计。无需额外 LLM 调用。

## 8. 洞察与深度分析边界

### 8.1 洞察

洞察是周期性生活解释，不是长期记忆本身。它可以：

- 读取少量相关稳定记忆，减少重复给出不适合用户的建议。
- 输出新格式 `memoryCandidate`。
- 继续使用上期洞察和偏好反馈维持内容连续性。

它不能：

- 把已有长期记忆当作本周期事实证据。
- 因为记忆中写过某个模式，就忽略当前周期相反数据。
- 把普通金额、单日记录或单次完成生成长期候选。

### 8.2 Agent 深度分析

Agent 是当前问题的证据执行器。它可以读取相关记忆和历史洞察，但每个事实结论仍必须由本地工具证据支持。记忆只提供“该查什么、怎样个性化表达”的背景，不能替代财务、健康、习惯等真实数据。

### 8.3 Memory Curator 本轮不接入

`HoloMemoryCurator` 当前会把已验证 claim 分类为：

- 仅本轮回答。
- 仅证据保存。
- 情景记忆。
- 长期候选。
- 仅展示。
- 抑制规则。

但它目前只返回路由结果，没有写入 Store。若直接接入，会与“洞察生成候选”形成第二个写入口，并引入以下未决问题：

- Agent claim 如何映射到新格式 `semanticType`。
- 如何避免同一结论被洞察和 Agent 重复写入。
- 用户主动提问产生的临时结论是否有资格变成长记忆。
- 高敏感 claim 的确认入口放在哪里。

因此本轮保留 Curator 纯规则能力，不执行写入。待统一候选写入服务稳定后，再单独设计“Agent 结论沉淀”闭环。

## 9. Feature Flag 与发布策略

- 移除旧格式相关 flag 分支。
- `semanticMemoryPromptEnabled`、`semanticMemoryTypesEnabled`、`semanticMemoryRecallEnabled` 收敛为正式新格式能力，不再允许用户形成互相矛盾的组合。
- `longTermMemoryEnabled` 继续作为用户是否允许自动沉淀长期记忆的总开关。
- 召回开关与写入开关分离：即使用户关闭自动写入，已有已确认记忆仍可在用户允许的情况下参与召回。
- `episodicMemoryObservationEnabled` 本轮不强制默认开启；先完成统一召回接入，再决定是否面向正式用户开放自动观察。

后端 Prompt 需要同步删除旧格式兼容表达，并保持只输出新 `memoryCandidate`。由于涉及 `HoloBackend/`，代码完成后必须部署 ECS 并验证生产 Prompt 版本与实际返回结构。

## 10. 错误处理与数据安全

- 迁移保存失败：保留原文件，不写迁移完成标记，下次重试。
- 新格式记录解码失败：沿用损坏文件备份机制，不能用空数组覆盖原文件。
- 召回失败：回答继续使用当前数据，不显示记忆使用回执。
- `usedMemoryIDs` 含未知 ID：本地过滤，未知 ID 不展示。
- 删除或忘记记忆：只删除记忆 Store 记录，不级联业务数据。
- 记忆正文与证据不上传日志；只记录数量、状态和匿名 request/job ID。

## 11. 验收用例

### 11.1 旧格式清理

- Store 同时存在旧/新记录时，迁移后只保留 `semanticType != nil` 的新记录。
- 删除旧“6 月 22 日支出 1999”候选后，原始交易仍可在财务模块查询。
- 迁移只运行一次，重复启动不重复写文件。

### 11.2 新候选

- 单笔支出、单日打卡、普通任务完成不产生候选。
- Mapper 拒绝的 `memoryCandidate` 不生成任何 fallback 候选。
- 两个周期都发现“跑步恢复”时合并到同一 memory ID，证据数量增长。
- 两个不同主题都使用 `habit_1` card ID 时不会互相覆盖。

### 11.3 显性感知

- 洞察确认了低风险稳定模式并静默写入后出现一次写入回执，并能撤销。
- 后续相关问题的回答自然使用该偏好，并显示记忆使用回执。
- 不相关问题不注入、不展示该记忆。
- 用户点击“忘记”后，下一轮不再召回。

### 11.4 洞察与 Agent

- 洞察最多注入 2 条相关稳定记忆，且不会把记忆当成本期证据。
- 数字查询不因长期记忆改变确定性结果。
- 原因/建议/复盘型深度分析可预取相关记忆和历史洞察。
- Agent 结果本轮不会自动写入长期记忆 Store。

## 12. 实施顺序

1. 旧格式迁移与旧入口删除。
2. 新候选稳定身份、证据合并与晋升规则修正。
3. 统一 `HoloMemoryContextEnvelope` 与相关性召回。
4. 普通聊天、目标规划、洞察和 Agent 接入。
5. 写入回执、使用回执、来源面板。
6. iOS standalone/单元测试、后端测试、Simulator build。
7. 后端 ECS 部署与生产 Prompt 验证。
