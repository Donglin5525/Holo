# 记忆长廊 AI 洞察回放实施方案

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将记忆长廊从规则生成的“回放 + 今日展柜 + 逐日封面”，升级为基于 AI 理解个人数据的“AI 周/月回放 + 可追问洞察 + 可复看记忆资产”。

**Architecture:** 新增独立的 `MemoryInsight` 洞察层：由 `MemoryInsightContextBuilder` 聚合周期数据快照，`MemoryInsightService` 复用现有 `AIProvider` 生成结构化洞察，`MemoryInsightRepository` 将结果缓存为本地资产，`MemoryGalleryViewModel` 只消费已生成洞察并负责展示状态。AI Chat 与记忆长廊共用 `AIProvider` / `PromptManager` / AI 配置，但不直接复用 `ChatViewModel`。

**Tech Stack:** SwiftUI、Core Data、MVVM、UserNotifications、可选 BGTaskScheduler、现有 `AIProvider.generateInsight`、现有 `OpenAICompatibleProvider`、现有 PromptManager

---

## 1. 背景

当前记忆长廊方案已经明确了最终交互形态：

```text
回放 Tab:
  本周回放卡
  今日展柜
  最近日子封面流

地图 Tab:
  统计概览
  热力图
  选中日期预览

明细 Tab:
  三层叙事时间线
```

原方案的回放文案主要依赖本地规则，例如习惯记录变化、支出变化、观点关键词命中。这可以作为离线兜底，但不应该是最终核心能力。

新的产品判断是：

> 记忆长廊的真正价值不是展示数据，而是让 AI 帮用户理解这些数据之间的关系，并把理解结果沉淀成可以复看的个人记忆资产。

因此，记忆长廊需要一个独立的 AI 洞察系统，而不是每次打开页面实时调用 AI。

## 2. 设计原则

### 2.1 AI 结果是资产，不是临时 UI 文案

AI 生成结果应该被保存，可复看、可过期、可重新生成、可继续追问。

```text
周期数据快照 -> AI 生成 -> 本地保存 MemoryInsight -> 记忆长廊展示
```

### 2.2 打开页面不等待 AI

记忆长廊打开时只读取本地缓存：

- 有 AI 洞察：展示 AI 回放。
- 没有 AI 洞察：展示生成入口 + 规则兜底预览。
- 洞察过期：展示旧结果 + “有新记录，刷新洞察”。
- 生成失败：展示失败状态 + 重试。

### 2.3 手动优先，定时作为增强

第一阶段优先做手动生成，因为它最可控：

- 用户明确知道何时调用 AI。
- 成本和隐私边界清晰。
- 方便验证 AI 输出质量。

每周/月定时建议先做“提醒生成”，再做后台 best-effort 自动生成。

### 2.4 AI Chat 与记忆长廊共享底层能力，不共享界面编排

```text
AI Chat
  └── ChatViewModel / ConversationCoordinator

Memory Gallery
  └── MemoryInsightService

Both
  └── AIProvider / PromptManager / AIProviderConfig
```

AI Chat 可以触发洞察生成，也可以继续追问洞察，但记忆长廊不应该依赖 ChatViewModel 的消息流状态。

## 3. 产品形态

### 3.1 回放 Tab

默认入口仍是回放 Tab，但首屏内容由 AI Insight 驱动。

```text

## 4. 触发策略

### 4.1 手动生成

最高优先级，MVP 必做。

UI 状态：

```text
无洞察:
  “本周已有 38 条记录，可以生成 AI 回放”
  Button: 生成本周回放

已生成:
  Button: 重新生成
  Button: 继续问 AI

已过期:
  “生成后新增了 7 条记录”
  Button: 刷新洞察

失败:
  “生成失败：网络不可用”
  Button: 重试
```

### 4.2 每周/月提醒生成

第二阶段做。

设置项：

```text
AI 回放
  [ ] 每周提醒我生成周回放
      周一 09:00
  [ ] 每月提醒我生成月回放
      每月 1 日 09:00
  [ ] 允许后台自动尝试生成
```

默认建议：

- 默认关闭。
- 用户开启时请求通知权限。
- 到点发本地通知：“本周记忆已准备好，可以生成 AI 回放”。
- 点击通知打开记忆长廊并显示生成入口。

### 4.3 后台自动尝试生成

第三阶段做，不能作为核心承诺。

iOS 限制：

- `BGAppRefreshTask` 不保证准时执行。
- 后台联网和执行时长受系统调度影响。
- 用户关闭后台刷新后不可用。

产品表述建议：

```text
允许 Holo 在后台尝试生成周/月回放。若系统未安排后台任务，将在你下次打开 App 时继续生成。
```

### 4.4 打开 App 补生成

即使开启后台，也需要前台兜底：

- 进入记忆长廊时检查是否有 due insight。
- 如果用户开启自动生成，并且当前网络/AI 配置可用，可自动开始生成。
- 如果用户未开启自动生成，只显示提示。

### 5.3 展示模型

Swift 展示层模型：

```swift
enum MemoryInsightPeriodType: String, Codable, CaseIterable {
    case daily
    case weekly
    case monthly
}

enum MemoryInsightStatus: String, Codable {
    case generating
    case ready
    case failed
    case stale
}

/// 洞察卡类型（约束 AI 输出范围）
enum MemoryInsightCardType: String, Codable, CaseIterable {
    case habit
    case finance
    case task
    case thought
    case milestone
    case crossDomain = "cross_domain"
    case overview
}

struct MemoryInsightCard: Codable, Identifiable, Equatable {
    let id: String
    let type: MemoryInsightCardType
    let title: String
    let body: String
    let evidence: [MemoryInsightEvidence]
    let suggestedQuestion: String?
}

/// AI 输出的 evidence 只有描述性字段，不直接引用 UUID。
/// 后处理映射在 MemoryInsightService.postProcessEvidence() 中完成。
struct MemoryInsightEvidence: Codable, Identifiable, Equatable {
    let id: String
    let label: String
    let date: String?          // yyyy-MM-dd，AI 从 context 中引用
    let sourceType: String?    // "habitRecord" / "transaction" / "task" / "thought"
    /// 仅供后处理使用：AI 输出时为 nil，Service 后处理填充真实 UUID
    var matchedSourceId: UUID?
}

struct MemoryInsightPayload: Codable, Equatable {
    let title: String
    let summary: String
    let cards: [MemoryInsightCard]
    let suggestedQuestions: [String]
}
```

#### 5.3.1 Evidence 后处理映射

AI 不可能知道数据库中的 UUID，因此 evidence 的 `matchedSourceId` 通过后处理填充：

```swift
// MemoryInsightService.postProcessEvidence()
// 策略：按 (date, sourceType) 查询当天该类型的 MemoryItem，
// 返回第一条匹配的 id。匹配不到则保持 nil。

MVP 不强求精确匹配，evidence 主要用于 UI 展示文字而非跳转。后续可升级为按 content 关键词匹配。

### 6.3 快照 Hash

生成 insight 前，将 context 编码为 JSON，并计算 hash。

用途：

- 判断旧洞察是否过期。
- 避免重复生成同一份报告。
- 失败重试时复用同一输入。

MVP 可用 SHA256。

```swift
let snapshotHash = SHA256.hash(data: encodedContext)

### 7.2 Prompt 要求

#### 7.2.1 Token 预算

| 周期 | 输入 Context 上限 | 输出上限 | 预估总 Token |
|------|-------------------|----------|-------------|
| 周回放 | ≤ 2000 token | ≤ 1000 token | ~3000 |
| 月回放 | ≤ 3500 token（聚合，不传每日明细） | ≤ 1500 token | ~5000 |

`MemoryInsightContextBuilder` 在序列化后检查 token 数（粗估：1 中文字 ≈ 2 token），超限时截断 `dailyExpenses` 到最近 14 天或聚合为周汇总。

#### 7.2.2 Prompt 模板

```text
你是 Holo 的个人记忆分析助手。
你的任务是基于用户提供的结构化周期数据，生成一份可复看的记忆回放。

## 必须遵守

1. 只基于输入数据中明确存在的事实，不要编造。
2. 不做心理诊断，不判断人格，不说"你很焦虑""你状态不好"。
3. 可以提出温和观察（如"支出集中在周末"），但必须有 evidence 支撑。
4. 金额、日期、数量直接从输入数据引用，不要重新计算或估算。
5. 输出严格 JSON，不要 Markdown，不要解释，不要在 JSON 外添加任何文字。
6. title 要像回放标题（口语化、有画面感），不要像报表标题。
7. summary 控制在 80 字以内。
8. cards 输出 3-5 张，每张聚焦一个维度。
9. type 只能取以下值：habit / finance / task / thought / milestone / cross_domain / overview。
10. 如果某个维度数据为空，不要强行生成该维度的 card。
11. suggestedQuestions 提供 2-3 个用户可能想追问的问题。

### 7.3 JSON 解析与降级策略

解析链（复用现有 `OpenAICompatibleProvider` 的 JSON 提取逻辑）：

```swift
func parseInsightResponse(_ raw: String) -> MemoryInsightPayload? {
    // 1. 直接解析
    if let data = raw.data(using: .utf8),
       let payload = try? JSONDecoder().decode(MemoryInsightPayload.self, from: data) {
        return validatePayload(payload) ? payload : nil
    }
    // 2. 提取 ```json ... ``` 代码块
    if let extracted = extractCodeBlock(raw),
       let data = extracted.data(using: .utf8),
       let payload = try? JSONDecoder().decode(MemoryInsightPayload.self, from: data) {
        return validatePayload(payload) ? payload : nil
    }
    // 3. 提取第一个 { ... }
    if let extracted = extractFirstBraces(raw),
       let data = extracted.data(using: .utf8),
       let payload = try? JSONDecoder().decode(MemoryInsightPayload.self, from: data) {
        return validatePayload(payload) ? payload : nil
    }
    return nil
}

/// Schema validation
private func validatePayload(_ payload: MemoryInsightPayload) -> Bool {
    guard !payload.title.isEmpty,
          !payload.summary.isEmpty,
          payload.summary.count <= 100,  // 宽松上限
          payload.cards.count >= 1,
          payload.cards.count <= 8 else {
        return false
    }
    // 验证 card.type 在枚举范围内
    for card in payload.cards {
        guard MemoryInsightCardType(rawValue: card.type.rawValue) != nil else {
            return false
        }
    }
    return true
}
```

降级链：

1. 解析成功 + validation 通过 → 保存 `ready`。
2. 解析失败 → 保存 `failed`，errorMessage 记录原始响应前 200 字符。
3. UI 显示规则兜底回放，不丢失体验。

#### 8.1.1 Provider 获取方式

`MemoryInsightService` 不依赖 `ChatViewModel`，独立获取 AIProvider：

```swift
@MainActor
final class MemoryInsightService {
    private var currentProvider: AIProvider?

    /// 从 Keychain 读取 AI 配置并创建 Provider
    /// 与 ChatViewModel.setup() 使用相同的配置源
    private func resolveProvider() throws -> AIProvider {
        if let current = currentProvider { return current }

        guard let config = KeychainService.shared.getAIConfig(),
              config.isConfigured else {
            throw MemoryInsightError.aiNotConfigured
        }

        let provider = OpenAICompatibleProvider(config: config)
        currentProvider = provider
        return provider
    }

    /// AI 配置变更时调用（监听配置更新通知）
    func resetProvider() {
        currentProvider = nil
    }
}
```

配置变更通知：在 `AIConfigViewModel` 保存配置成功后发 `NotificationCenter.default.post(name: .aiConfigDidChange)`，`MemoryInsightService` 监听并调用 `resetProvider()`。

#### 8.1.2 并发保护

```swift
func generateInsight(...) async throws -> MemoryInsight {
    guard !isGenerating else {
        throw MemoryInsightError.generationInProgress
    }
    isGenerating = true
    defer { isGenerating = false }

    // 带超时的生成
    let result = try await withThrowingTaskGroup(of: MemoryInsight.self) { group in
        group.addTask {
            // 实际生成逻辑
        }
        group.addTask {
            // 超时守卫
            try await Task.sleep(nanoseconds: UInt64(self.generationTimeout) * 1_000_000_000)
            throw MemoryInsightError.generationTimeout
        }

        let first = try await group.next()!
        group.cancelAll()
        return first
    }
    return result
}
```

#### 8.1.3 AI 未配置状态

`isAIConfigured` 通过 `KeychainService.shared.getAIConfig()` 检查。当返回 `false` 时，ViewModel 状态为 `.notConfigured`，UI 显示"配置 AI 后可生成周回放"并跳转设置页。

### 8.2 MemoryInsightRepository

职责：

- 查询周期 insight。
- 保存生成结果。
- 更新状态。
- 删除旧失败记录。

接口：

```swift
final class MemoryInsightRepository {
    /// 只返回 ready 或 stale 状态的记录
    /// generating/failed 由 Service 内部管理
    func fetchInsight(
        periodType: MemoryInsightPeriodType,
        start: Date,
        end: Date
    ) throws -> MemoryInsight?

    /// 生成前保存 generating 状态，同时清理同周期旧的 failed 记录
    func saveGenerating(
        periodType: MemoryInsightPeriodType,
        start: Date,
        end: Date,
        snapshotHash: String
    ) throws -> MemoryInsight

    func saveReady(
        insight: MemoryInsight,
        payload: MemoryInsightPayload,
        rawResponse: String,
        providerName: String?
    ) throws

    func saveFailed(
        insight: MemoryInsight,
        errorMessage: String
    ) throws

    /// 清理同周期同类型的 failed 记录
    func cleanupFailedRecords(
        periodType: MemoryInsightPeriodType,
        start: Date,
        end: Date
    ) throws
}
```

#### Repository 查询策略

- `fetchInsight` 过滤条件：`periodType == x AND periodStart == y AND status IN {"ready", "stale"}`，按 `generatedAt` 降序取最新一条。
- `saveGenerating` 执行前先调用 `cleanupFailedRecords` 删除同周期旧的 failed 记录，避免 `fetchInsight` 意外命中失败记录。
- `generating` 状态的记录只在生成过程中存在，超时或异常退出时由 Service 标记为 `failed`。

## 13. 验收标准

### 手动生成

- AI 未配置时显示”去配置 AI”入口，点击跳转设置页。
- 无洞察 + AI 已配置时显示”生成本周回放”。
- 点击后进入 generating 状态，按钮禁用，不可重复点击。
- 成功后展示 AI 标题、摘要、3-5 张洞察卡。
- 失败后展示错误信息和重试按钮。
- 重新打开 App 后仍显示已生成结果。

### 缓存与过期

- 同一数据快照重复生成不会重复调用 AI，除非 force refresh。
- 新增记录后 snapshotHash 变化，旧 insight 标记 stale。
- stale 状态仍展示旧内容，同时提示”有新记录，刷新洞察”。
- failed 记录不干扰 fetchInsight 查询。

### 周/月周期

- 周回放范围为周一到今天/周日。
- 月回放范围为当月 1 日到今天/月末。
- 周报和月报互不覆盖。

### 规则兜底

- AI 未配置时，Hero Card 展示改进后的规则回放。
- 连续 3 周看到不重复的标题和摘要。
- 全空数据时展示温和文案。

### AI Chat

- “帮我复盘这周”能生成周回放。
- Chat 生成后记忆长廊可读取同一份 insight。
- 记忆长廊点击”继续问 AI”能带上下文进入 Chat（预填不自动发送）。

### 定时提醒

- 开启每周提醒后能收到本地通知。
- 关闭后通知被取消。
- 每月提醒按设置日期触发。

## 14. 风险与处理

### 14.1 AI 编造事实

处理：

- Prompt 明确”只基于输入数据”。
- 输入 context 提供可引用 evidence。
- UI 中展示 evidence，不只展示结论。
- 对金额、日期、数量保持结构化，不让 AI 自由计算关键数字。
- 解析后做 schema validation（title 非空、cards.count >= 1）。

### 14.2 AI JSON 不稳定

处理：

- Prompt 末尾强调”只输出 JSON，不要添加其他文字”。
- 完整 JSON Schema + few-shot 示例嵌入 prompt。
- 三层 JSON 提取降级链（直接解析 → 代码块 → 花括号）。
- 解析后 schema validation 检查必填字段和 type 枚举范围。
- MockProvider 测试解析。
- 解析失败保存 `failed`，不污染 `ready` insight。
- 失败时展示规则兜底（7.4 节）。

### 14.3 隐私和成本

处理：

- 默认手动生成。
- 定时自动生成默认关闭。
- 设置页说明会把周期摘要发送给配置的 AI Provider。
- 只发送聚合摘要和必要 snippets，不发送全量原文。
- Token 预算控制（周 ~3000 token、月 ~5000 token）。

### 14.4 iOS 后台不可靠

处理：

- 本地通知作为主方案。
- 后台自动生成只作为”尝试”。
- 前台打开补偿生成。

### 14.5 现有 UserContext 不够用

处理：

- 新增 MemoryInsightContextBuilder。
- 不改 Chat 的 UserContextBuilder 主流程。
- 后续如稳定，再考虑抽公共聚合层。

### 14.6 snapshotHash 过期判断粒度

当前策略：任何数据变化都导致 hash 变化并标记 stale。这意味着用户加了 1 笔小额支出，已生成的周洞察也会被标记 stale。

MVP 接受这个粒度，原因：
- 实现简单，不会遗漏重要变化。
- stale 状态仍展示旧内容，不影响阅读。
- 用户可自主决定是否刷新。

后续可优化为”重要变化阈值”（如支出变化 > 20%、习惯数变化 > 3 条才标记 stale）。

### 14.7 并发生成

处理：

- `MemoryInsightService.isGenerating` 标记防重入。
- 重复点击返回 `MemoryInsightError.generationInProgress`。
- UI 在 generating 状态下禁用按钮。
