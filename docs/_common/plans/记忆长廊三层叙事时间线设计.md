## Problem Statement

当前「记忆长廊」以瀑布流展示原始记账/习惯/待办记录，信息密度高但缺乏叙事感。用户期望一条垂直时间线，展示跨模块的个人数据全景，同时兼顾三个场景：每日/每周回顾、查找关键信息、感受成长轨迹。

## What Makes This Cool

- **三层叙事结构**：日摘要（数据聚合）→ 高亮（算法筛选）→ 里程碑（重大成就），视觉权重递增
- **零成本数据输入**：所有节点数据来源于已有模块（记账、习惯、待办、健康、观点），无需用户额外操作
- **渐进式 AI 增强**：免费版 = 数据全景 + 存在感；Pro 版 = AI 洞察节点（异常检测、智能建议）

## Constraints

1. 数据来源仅限已有模块，不新增用户输入入口
2. 使用 Holo 设计系统（颜色、间距、圆角、字体）
3. 首屏必须有意义 — 不能是空状态或加载中
4. 支持按模块筛选（记账/习惯/待办/健康/观点）
5. 垂直时间线布局，替代当前瀑布流网格

## Premises

1. **筛选 > 堆砌**：时间线应展示经过筛选的有意义节点，而非所有原始数据 ✅ 已确认
2. **三层结构服务多场景**：日摘要服务回顾，高亮服务查找，里程碑服务成长感 ✅ 已确认
3. **垂直时间线 > 瀑布流**：时间线叙事感更强，适合「记忆」主题 ✅ 已确认

## Approaches Considered

### Approach A: 简化时间线（最小可行版）
- 仅保留日摘要节点（每天一行聚合数据）
- 无高亮、无里程碑
- 替换当前瀑布流为简单时间线
- **工作量**: 小（~5个文件改动）
- **复杂度**: 低
- **问题**: 缺少叙事深度，无法体现成长感和异常发现

### Approach B: 三层叙事时间线（已选择）
- 日摘要 + 高亮 + 里程碑三层节点
- 垂直时间线布局，节点视觉权重递增
- 算法自动检测高亮和里程碑
- 支持模块筛选
- **工作量**: 中（~12个文件改动）
- **复杂度**: 中

### Approach C: AI 驱动智能时间线
- 所有节点由 AI 生成自然语言描述
- 类似「每日一封日记」的体验
- 需要 LLM 调用成本
- **工作量**: 大（~20个文件 + 后端）
- **复杂度**: 高
- **问题**: 依赖 Pro 功能，不能作为免费版基础

## Recommended Approach

**方案 B: 三层叙事时间线**

理由：
1. 三层结构天然服务三个用户场景（回顾/查找/成长）
2. 数据全部来自已有模块，无需新增输入
3. 为 Pro 版 AI 功能预留扩展点（高亮节点可升级为 AI 洞察）
4. 视觉叙事感强，符合「记忆长廊」的隐喻

## Technical Design

### 数据模型

```swift
/// 时间线节点类型
enum MemoryNodeType: String {
    case dailySummary   // 日摘要 — 每天一个
    case highlight      // 高亮 — 算法筛选的值得注意的事件
    case milestone      // 里程碑 — 重大成就
}

/// 时间线节点
struct MemoryTimelineNode: Identifiable {
    let id: UUID
    let date: Date
    let type: MemoryNodeType
    let data: NodeData

    /// 节点在时间线上的视觉权重
    var visualWeight: Int {
        switch type {
        case .dailySummary: return 1
        case .highlight: return 2
        case .milestone: return 3
        }
    }
}

/// 节点数据（按类型区分）
enum NodeData {
    case summary(DailySummaryData)
    case highlight(HighlightData)
    case milestone(MilestoneData)
}

struct DailySummaryData {
    let totalExpense: Decimal?     // 当日总消费
    let habitsCompleted: Int       // 完成习惯数
    let habitsTotal: Int           // 总习惯数
    let tasksCompleted: Int        // 完成任务数
    let steps: Int?                // 步数（如有健康数据）
}

struct HighlightData {
    let category: HighlightCategory
    let title: String
    let subtitle: String?
    let icon: String
    let colorHex: String
    let sourceModule: MemoryItemType  // 来源模块
}

enum HighlightCategory: String {
    case streakAchievement  // 连续打卡成就
    case spendingAnomaly    // 消费异常（高于/低于均值）
    case taskCompletion     // 重要任务完成
    case habitPerfect       // 习惯全勤日
}

struct MilestoneData {
    let title: String
    let description: String
    let icon: String
    let milestoneType: MilestoneType
}

enum MilestoneType: String {
    case streakDays       // 连续打卡 N 天
    case cumulativeCount  // 累计记录 N 笔
    case habitMastery     // 习惯掌握（连续 30/90 天）
    // spendingGoal（消费目标达成）— 等预算功能上线后再加入
}
```

### 节点生成算法

**日摘要**：每天自动生成 1 个，聚合当日所有模块数据

**高亮检测规则**：
1. `streakAchievement` — 习惯连续打卡达到 3/7/14/21 天
2. `spendingAnomaly` — 当日消费 > 7 日日均消费 x 1.5
3. `taskCompletion` — 已完成的任务（MVP 阶段所有完成任务均计入摘要，不做优先级区分）
4. `habitPerfect` — 当日所有习惯全部完成

**里程碑检测规则**（MVP 仅实现前 3 项）：
1. `streakDays` — 习惯连续打卡达到 30/365 天（MVP 仅两个阈值）
2. `cumulativeCount` — 记账累计 100/500 笔
3. `habitMastery` — 单个习惯连续完成 30 天以上
4. ~~`spendingGoal` — 月消费低于预算~~ （等预算功能上线后再加入）

### 节点排序规则

同一天内的节点按以下顺序排列：
1. 日摘要（始终第一个）
2. 里程碑（如有，第二个）
3. 高亮节点（按时间戳倒序排列，最近的在前）

### 检测时机与缓存策略

**检测时机**：视图加载时实时计算（非事件驱动）

**性能策略**：
1. **ViewModel 层缓存**：检测结果缓存 5 分钟（与现有 ViewModel 缓存策略一致）
2. **批量获取**：Core Data 查询使用 `predicate` + `sortDescriptor` 批量获取，避免逐条遍历
3. **复用已有计算**：`HighlightDetector` 复用 `HabitRepository` 已有的 streak 计算方法，不重复实现
4. **懒加载**：高亮和里程碑检测仅在滚动到对应日期范围时触发
5. **后台线程**：所有检测在 `Task.detached` 中执行，不阻塞 UI

### 交互状态矩阵

```
FEATURE         | LOADING          | EMPTY                | ERROR       | SUCCESS    | PARTIAL
日摘要         | skeleton 卡片     | 引导文案 + CTA 按钮  | 重试按钮     | 正常渲染    | 仅显示有数据的模块字段
高亮/里程碑    | 不显示           | 不显示（不展示空区域）| 不显示       | 正常渲染    | 不显示
筛选切换      | 旧节点渐隐 + 转圈 | —                   | —          | 新节点列表渐显 | —
分页加载      | 底部转圈         | "已加载全部" 文字    | 重试按钮     | 新数据 append | —
下拉刷新      | 转圈             | —                   | —          | 刷新完成   | —
```

**Skeleton 卡片设计**（首次加载）：模拟日摘要卡片的骨架形状，用 Color.holoBorder 圆角矩形 + shimmer 动画，持续约 0.5s 后切换为真实内容。

**空状态**（完全无数据）：
- 插图：自定义时间线端点图示（一条虚线延伸到远方的简约线条画），40pt 宽，Color.holoTextPlaceholder
- 主标题："暂无记录"，.holoHeading，Color.holoTextPrimary
- 副标题："记录你的第一笔，开启记忆长廊"，.holoCaption，Color.holoTextSecondary
- CTA 按钮："去记账"，holoPrimary 背景，点击后跳转到记账模块

**部分数据**（仅有记账无习惯/任务）：日摘要卡片只显示有数据的字段（💰¥128），缺失字段不显示，不显示占位。

### 新用户空状态策略

约束 3 要求「首屏必须有意义」。策略：
- **不足 1 天数据**：显示引导卡片 + 今日已记录的数据（即使只有 1 笔记账或 1 个习惯）
- **无高亮/里程碑**：仅显示日摘要节点，不显示空的高亮/里程碑区域
- **完全无数据**：保留现有 emptyView，但文案改为「记录你的第一笔，开启记忆长廊」

### 用户情感旅程

```
STEP | USER DOES              | USER FEELS       | DESIGN SUPPORTS IT BY
1    | 从首页点"记忆"入口     | 好奇、期待       | fullScreenCover 全屏进入，仪式感
2    | 看到今日的摘要          | "哦，我花了¥128"  | 消费金额最大最醒目（信息层级）
3    | 看到连续打卡高亮        | 小成就感          | holoPrimary 橙色边框 + emoji + 数字
4    | 滚动看到里程碑          | "我居然坚持了30天!" | 金色光晕，全卡最醒目
5    | 滚动到上周              | 回顾感            | 日期头 "3月24日 周一" 提供时间锚点
6    | 切换筛选到"记账"        | 聚焦感            | 非记账节点渐隐，记账数据渐显
7    | 消费异常高亮出现        | "我昨天花多了?"   | 红色边框引起注意但不过度警告
8    | 下拉刷新                | 期待新数据        | 系统刷新 indicator
```

**5 秒印象**（visceral）：一条有节奏的时间线，橙色高亮点缀，金色里程碑醒目
**5 分钟行为**（behavioral）：筛选、滚动、回顾 — 每次交互有明确反馈
**5 年关系**（reflective）：用户回头看自己的成长轨迹，里程碑记录坚持

### 模块筛选语义

- 筛选「记账」→ 仅显示记账相关的日摘要字段 + 记账相关的高亮（消费异常、累计笔数里程碑）
- 筛选「习惯」→ 仅显示习惯相关的日摘要字段 + 习惯相关的高亮（连续打卡、全勤）和里程碑
- 筛选「待办」→ 仅显示待办相关的日摘要字段 + 任务完成的高亮
- 日摘要节点在所有筛选下都显示，但只展示对应模块的统计字段

### Pro 版扩展点（未来）

1. **AI 洞察节点**：在高亮节点基础上，用 LLM 生成自然语言洞察
   - 例：「你本周餐饮消费比上周高了 40%，主要因为周三那顿火锅」
2. **周报/月报摘要**：AI 生成的周期性总结节点
3. **智能回顾**：根据用户行为模式推荐回顾特定时期
4. **情绪标签**：结合使用数据推断用户状态（高频记账 = 焦虑？全勤习惯 = 积极向上？）

## What I noticed about how you think

- 你对「时间线上放什么」这个问题的思考方式很有意思 — 不是先想「有什么数据就展示什么」，而是从「用户在什么场景下看这个页面」出发。这是产品思维，不是工程思维。
- 你能清晰地区分免费版和 Pro 版的功能边界 — 「数据全景是基础，AI 洞察是增值」。这个判断是对的。
- 你对「结合三个场景」的坚持很好 — 很多人会选「先做一个场景」，但你意识到三个场景（回顾/查找/成长）可以在一个三层结构里自然共存，这是设计上的克制和智慧。
