## Context

HOLO 目前是纯本地 iOS 应用（记账/习惯/待办/观点），AI 对话 Tab 为占位状态。本方案将接入国内大模型 API（DeepSeek、通义千问、Moonshot、智谱），实现：意图识别、自然语言记账/建任务、流式对话、每日总结。这是项目的首个网络层。

**技术栈**：SwiftUI + MVVM + Core Data（编程式实体）+ URLSession + SSE
**核心决策**：国内模型优先（均兼容 OpenAI API 格式，一套代码多家复用）、无后台、本地 JSON 提示词

## Phase 0：CoreDataStack 重构（前置，独立提交）

> **原因**：CoreDataStack.swift 已有 1358 行（15 个实体全部内联在 `createDataModel()` 中），继续往里追加实体会导致文件膨胀到 1500+ 行，可维护性极差。必须先拆分再加新实体。

### 0.1 修改 `Models/CoreDataStack.swift`

将 `createDataModel()` 按模块拆分为私有方法：

- `createFinanceEntities()` → Transaction, Category, Account, HomeIconConfig（4 个实体）
- `createHabitEntities()` → Habit, HabitRecord（2 个实体）
- `createTodoEntities()` → TodoFolder, TodoList, TodoTask, TodoTag, CheckItem, RepeatRule（6 个实体）
- `createThoughtEntities()` → Thought, ThoughtTag, ThoughtReference（3 个实体）

每个方法返回 `[NSEntityDescription]`，`createDataModel()` 只剩方法调用 + `model.entities` 赋值，约 20 行：

```swift
private func createDataModel() -> NSManagedObjectModel {
    let model = NSManagedObjectModel()
    var entities: [NSEntityDescription] = []
    entities.append(contentsOf: createFinanceEntities())
    entities.append(contentsOf: createHabitEntities())
    entities.append(contentsOf: createTodoEntities())
    entities.append(contentsOf: createThoughtEntities())
    // Phase 3 时追加: entities.append(contentsOf: createChatEntities())
    model.entities = entities
    return model
}
```

**Phase 0 验证**：App 正常启动 + 已有数据完好 + 编译零警告 + 功能无变化

### 1.3 `Services/Network/APIError.swift` (~80 行)

### 1.4 `Services/Network/APIRequest.swift` (~80 行)

Builder 模式构建 URLRequest：baseURL + path + headers + JSON body

### 1.5 `Services/Network/APIClient.swift` (~200 行)

- `@MainActor class APIClient { static let shared }`
- `send<T: Decodable>()` 普通请求 + 指数退避重试（最多 2 次，仅 429/5xx 重试）
- `sendStreaming()` 返回 `AsyncThrowingStream<String, Error>`，用 `URLSession.bytes(for:).lines` 异步迭代
- **并发模型**：`sendStreaming()` 在后台 Task 中执行 URL 请求 + SSE 解码，只将解码后的纯字符串通过 AsyncThrowingStream 传递。ChatViewModel 在 MainActor 消费 stream 拼接 UI 文本。
- Logger 日志，无 print()

### 1.6 `Services/Network/SSEParser.swift` (~80 行)

- `struct SSEParser { mutating func parse(_ line: String) -> String? }`
- 跳过空行/注释行，解析 `data: {...}` 前缀，处理 `[DONE]` 结束标记
- 解码 `SSEChunk` 提取 `delta.content`

### 1.7 `Services/AI/AIProvider.swift` (~50 行)

```swift
protocol AIProvider {
    func parseUserInput(_ input: String, context: UserContext) async throws -> ParsedResult
    func generateInsight(type: InsightType, data: UserContext) async throws -> String
    func chat(messages: [ChatMessageDTO], userContext: UserContext) async throws -> String
    func chatStreaming(messages: [ChatMessageDTO], userContext: UserContext) -> AsyncThrowingStream<String, Error>
}
```

> **注意**：协议不加 `Sendable` 约束。`OpenAICompatibleProvider` 标记 `@MainActor`，但网络请求在后台 Task 执行，只有 UI 更新回 MainActor。

### 1.8 `Services/AI/OpenAICompatibleProvider.swift` (~280 行)

- `@MainActor class OpenAICompatibleProvider: AIProvider`
- 通过注入不同 `AIProviderConfig`（baseURL + apiKey + model）支持所有国内 LLM
- 4 个协议方法的完整实现

### 1.9 `Services/AI/MockAIProvider.swift` (~100 行)

- `@MainActor class MockAIProvider: AIProvider`
- 返回预定义的 ParsedResult 和固定回复文本
- 用于开发阶段调试 UI 和 IntentRouter，无需真实 API Key
- `chatStreaming()` 模拟逐字输出（Timer/Task.sleep 逐字返回）

**Phase 1 验证**：配置 API Key → `testConnection()` → 收到真实响应；MockAIProvider 无需 Key 即可返回模拟数据

### 4.2 `ViewModels/ChatViewModel.swift` (~300 行)

- `@MainActor class ChatViewModel: ObservableObject`
- 核心状态：messages, inputText, isStreaming, streamingText, errorMessage, isConfigured
- `sendMessage()` 流程：
  1. 保存用户消息到 ChatMessageRepository
  2. 创建 AI 占位消息
  3. 构建上下文（UserContextBuilder + 历史 ChatMessage）
  4. 先做意图识别（parseUserInput）
  5. 高置信度 (>0.7) → IntentRouter 直接执行本地操作
  6. 低置信度/闲聊 → SSE 流式对话（chatStreaming）
- 支持取消（Task.cancel 中断 for try await）
- **网络失败处理**：显示错误提示 + 保留用户消息在 ChatMessage 中，用户可重新发送（不做自动离线队列，避免过早优化）

## 实施节奏与分支策略

### 为什么用 feature 分支？

本方案新增 28 个文件 + 修改 3 个已有文件。其中 `CoreDataStack.swift`、`ContentView.swift`、`SettingsView.swift` 可能与日常功能迭代产生合并冲突。使用 feature 分支可以**让 AI 开发和正常迭代互不干扰**。

### 分支流程

```
main 分支（日常迭代：记账/习惯/待办/观点等）
  │
  ├── Step 1: Phase 0（CoreDataStack 重构）在 main 上直接做
  │          原因：纯重构，对所有后续开发都有好处
  │          提交后 push 到远程
  │
  ├── Step 2: 从 main 创建 feature/ai-chat 分支
  │
  ├── Step 3: 在 feature/ai-chat 上按顺序完成 Phase 1 → 2 → 3 → 4 → 5
  │          每个 Phase 独立提交，编译验证
  │          期间 main 上正常迭代，互不影响
  │
  └── Step 4: AI 功能完成后，合并 feature/ai-chat 回 main
```

### 冲突风险分析

| 文件 | 冲突概率 | 说明 |
|------|---------|------|
| `CoreDataStack.swift` | 已解决 | Phase 0 在 main 上完成，后续只在 feature 分支加 `createChatEntities()` |
| `ContentView.swift` | 低 | 改动仅 1 行（`.holo` case），合并时手动处理即可 |
| `SettingsView.swift` | 低 | 改动仅 +15 行（插入 AI 设置入口），位置固定 |
| 28 个新文件 | 无 | 全新目录和文件，不存在冲突 |
| 已有 Repository | 无 | AI 模块只调用现有方法，不修改 Repository 代码 |

### 每个 Phase 的提交规范

```
refactor(iOS): 拆分 CoreDataStack createDataModel 为私有方法     ← Phase 0
feat(iOS): 新增 AI 网络层基础（APIClient + SSE + Provider）       ← Phase 1

```
