## G2. P0 修订一：3 天轻量版必须有 stage/variant

GLM 的 F1 把 3 天轻量观察和 7 天完整观察都映射为同一个 `MemoryInsight(.weekly)`，但当前 MemoryInsight 查询与缓存语义主要由以下字段决定：

```text
periodType + periodStart + periodEnd + snapshotHash + promptVersion
```

如果 3 天轻量版直接保存为本周 `.weekly`：

- 7 天时 `fetchInsight(periodType:start:end)` 可能先拿到 3 天轻量版，导致完整观察不生成。
- `fetchAvailable(... snapshotHash:promptVersion:)` 虽然能区分数据变化，但 `checkSimilarityWithRecent` 可能把 7 天完整版误判为与 3 天版相似并复用旧结果。
- 首页胶囊无法区分「初步观察已读」和「完整本周观察未读」。
- 文案承诺不同，但实体无字段表达阶段，后续状态会混乱。

执行前必须补一个阶段字段。推荐：

```swift
enum MemoryInsightObservationStage: String {
    case light3d
    case full7d
}
```

落点建议：

| 位置 | 要求 |
| --- | --- |
| Core Data `MemoryInsight` | 增加 `observationStage` 可选字符串字段，旧数据 nil 视为 full/legacy |
| 查询 | 首页候选读取时区分 light3d/full7d；7 天完整观察不能被 light3d 挡住 |
| 去重 | `checkSimilarityWithRecent` 需要按 stage 或 generationPurpose 分组，至少 full7d 不应被 light3d 直接复用 |
| 展示 | 轻量版必须显示「初步观察」，完整版显示「本周观察」 |
| 已读 | readAt 需与 stage 绑定，不能看过 light3d 就吞掉 full7d |

如果不想改 Core Data 字段，备选是把 stage 写进 `cardsJSON` 或 `rawResponse`，但这会让查询、排序和已读逻辑变复杂，不推荐。

## G3. P0 修订二：consent gate 需要双层防线

GLM 要求在 4 处显式加 guard 是正确的，但不完整。实际调用 `MemoryInsightService.generateInsight` 的入口还包括：

| 入口 | 代码事实 | 风险 |
| --- | --- | --- |
| MemoryGallery 手动生成 | `MemoryGalleryViewModel.generateInsight` 调 `service.generateInsight` | 用户未授权时仍会进入生成流程，最终由 provider 抛错并显示失败 |
| Chat 触发回放 | `IntentRouter.handleGenerateMemoryInsight` 调 `service.generateInsight` | HoloAI 里可通过意图触发洞察生成，仍会漏 |
| 后台任务 | `MemoryInsightBackgroundService.handleBackgroundTask` | GLM 已覆盖 |
| 前台补偿 | `MemoryInsightBackgroundService.checkForegroundCompensation` | GLM 已覆盖 |
| schedule settings | `MemoryInsightScheduleSettings` | GLM 已覆盖一部分，但它主要是通知提醒设置，不等同所有自动生成触发 |

执行规则应升级为：

1. `MemoryInsightService.generateInsight` 开头必须有最终防线：

```swift
guard HoloAIFeatureFlags.aiDataProcessingConsentGranted else {
    throw MemoryInsightError.aiDataProcessingConsentRequired
}
```

2. UI/入口层再做友好兜底：

- MemoryGallery 手动生成：不进入 `.failed`，显示授权卡或授权提示。
- HoloAI/IntentRouter：返回「开启 HoloAI 后可生成本周观察」并带授权按钮/跳转。
- Background/foreground compensation：静默跳过，不写 failed，不反复重试。

这样才能同时满足两件事：

- 不漏任何未来新增入口。
- 用户主动点击时不是冷冰冰的失败，而是可理解的授权引导。

## G4. P1 修订：Core Data 字段扩展要写成迁移任务

GLM 建议扩展 `MemoryInsight` 三字段 `readAt/snoozedUntil/hiddenUntil` 是对的，但方案需要明确这不是简单加属性。

当前 Holo 使用程序化 Core Data 模型：

- 实体定义在 `CoreDataStack+MemoryInsightEntity.createMemoryInsightEntity()`。
- 持久化容器开启了 `NSMigratePersistentStoresAutomaticallyOption` 和 `NSInferMappingModelAutomaticallyOption`。
- `MemoryInsight+CoreDataProperties.swift` 需要同步 NSManaged 属性和 factory 默认值。

实施计划里必须列出：

| 文件 | 必做 |
| --- | --- |
| `CoreDataStack+MemoryInsightEntity.swift` | 新增 `readAt/snoozedUntil/hiddenUntil/observationStage` 属性，全部 optional 或有安全 default |
| `MemoryInsight+CoreDataProperties.swift` | 新增 NSManaged 属性，createGenerating 初始化 stage |
| `MemoryInsightRepository.swift` | 新增 markRead/snooze/hide 查询与更新方法 |
| 迁移验证 | 用旧 store 或 migration test 验证自动轻量迁移，不只跑 clean install |

否则会出现旧用户升级后 store 不兼容、属性不存在或已读状态不能持久化的问题。

## G5. P1 修订：生成内核升级不应一次性强切 Agent

GLM 写「Agent 深度分析作为 weekly MemoryInsight 的生成质量升级路径，替换/增强现有 provider.generateMemoryInsight 单次调用」，方向对，但执行上不能首发直接强切。

原因：

- 现有 `MemoryInsightService` 是单次 `provider.generateMemoryInsight`，有 snapshotHash、promptVersion、JSON parser、rerank、feedback、memory candidate 通知等完整链路。
- Agent runLoop 是多轮 LLM + 工具调用，成本、耗时和失败形态完全不同。
- Agent renderer 当前输出 `HoloRenderedAgentResult`，而 MemoryInsight UI 消费的是 `MemoryInsightPayload/cardsJSON`，中间需要稳定转换契约。

推荐执行顺序：

1. V1 先用现有 `MemoryInsightService` 生成「本周观察」卡片形态，补首页胶囊、stage、已读/稍后/授权兜底。
2. Agent 内核作为 V1.0b 灰度或 V1.1：只在 `agentMemoryGalleryEnabled` 或新 flag 开启时生成增强版。
3. 增加转换层：

```text
HoloAgentResult
  -> WeeklyObservationPayload
  -> MemoryInsightPayload
```

4. 成本预算按 GLM M4 补齐后再打开默认路径。

这样不会把首页胶囊和 Agent 多轮成本绑死在同一个首发风险里。

## G6. P2 修订：有效记录日的「跨 ≥2 模块」规则要避免误伤

GLM 增加「跨 ≥2 模块防刷」是合理护栏，但如果硬性要求每个有效记录日都跨 2 模块，会误伤真实新用户：很多人前 3 天可能只记账或只打卡。

建议口径改成：

```text
有效记录日：当天至少有 1 个有效记录。
轻量观察门槛：累计 3 个有效记录日，且累计覆盖至少 2 个模块。
```

也就是「跨 2 模块」作用于整个观察窗口，而不是每一天。这样既防刷，又不会让单模块真实用户永远看不到进度。

首页文案也要跟随：

- 如果天数够但模块不足：`再记录一种内容，观察会更准`
- 如果模块够但天数不足：`再记录 1 天生成观察`

## G7. P2 修订：Privacy Manifest 之外还要补审核叙事

当前 `PrivacyInfo.xcprivacy` 已包含 Health / Fitness / FinancialInfo / OtherUserContent 等数据类型，GLM 提醒合规是对的，但本次能力的审核风险不只在 manifest。

需要同步补：

| 项 | 要求 |
| --- | --- |
| App Store 隐私问卷 | 确认 AI 处理涉及的数据类型、是否 linked、是否第三方处理，与 manifest 一致 |
| Review Notes | 说明本周观察需要用户开启 AI 数据处理授权，未开启时不上传记录 |
| App 内 consent 文案 | 明确本周观察会使用记账/待办/习惯/想法/健康等记录生成 |
| Push 文案 | V1.1 前保持不含具体敏感内容 |

本项不阻断 V1 无 Push 版，但阻断 App Store 提审前的最终 release checklist。

## G8. 二次审查后的首发范围

建议把 GLM 的 V1 范围修订为：

**首发 V1：**

1. `MemoryInsightService.generateInsight` consent 最终防线 + MemoryGallery/IntentRouter/Background 入口友好兜底。

8. AI 授权兜底卡片。
9. PrivacyInfo + App Store 隐私问卷/Review Notes checklist 更新。

**V1.0b / V1.1：**

10. Agent 内核增强 weekly MemoryInsight，走 feature flag 灰度。
11. Push 全套：通知权限兜底、静默时段、幂等限频、本周观察开关。
12. 卡片级 DeepLink：`DeepLinkTarget.aiMessage` + ChatView 滚动定位。

## G9. 最终判断

**Conditional Go。**

可以继续推进，但不要直接按 GLM 的「修订后 Go」进入代码实现。先把 G2/G3 写进执行方案或实施计划，否则会出现两个高概率问题：

1. 用户 3 天看到轻量观察后，7 天完整版生成/展示/已读状态混乱。

## 1. 致命问题修订

### 1.1 F1 — weekly_observation 即 MemoryInsight(.weekly) 的升级形态

**定性**：对用户叫「本周观察」的那个东西，**就是** `MemoryInsight(periodType:.weekly)`，不是第二个平行产物。不新建 `weekly_observation` 实体、不新建调度、不新建独立状态机。

落地映射：

| 维度 | 决策 |
| --- | --- |
| 产物实体 | 沿用 `MemoryInsight`（Core Data），不新建实体 |
| 调度 | 沿用 `MemoryInsightBackgroundService`（BGTask + reschedule 自循环），不新建 `WeeklyObservationScheduler` |
| 生成内核 | Agent 深度分析（plan→tools→minePatterns→verifyClaims）作为 weekly MemoryInsight 的**生成质量升级路径**，替换/增强现有 `provider.generateMemoryInsight` 单次调用 |
| 渲染 | 升级 `MemoryInsight` 卡片，承载「1 主观察 + 2-4 条证据 + 1 个建议 + 下一步动作」 |
| 已读/未读/稍后/隐藏 | 扩展 `MemoryInsight` 模型加字段（见 M3），不新建独立状态机 |
| 去重 | 复用现有 `checkSimilarityWithRecent`，不再有跨产物冲突 |
| 3 天轻量观察 | 有效记录日达标 3 天时**提前生成一份 `MemoryInsight(.weekly)`**（标注「初步观察 / 降低承诺」）；7 天生成完整版。同实体、不同时机与承诺，**不产生第二条本周总结** |
| 首页胶囊 | 新增候选构建器，读 `MemoryInsight(.weekly)` 状态（notEnoughData / generating / ready / failed / read / unread） |

**连锁收益**：F1=a 使 H1 的工作量进一步下降——调度直接复用 MemoryInsight 线，无需新建调度器。

### 1.2 F2 — consent gate 显式落点（修后台/前台静默失败）

现状 weekly 洞察的「未授权保护」靠底层 `HoloBackendAIProvider.ensureDataProcessingConsent()` 抛错间接兜底，未授权时会反复尝试失败、写错误日志、标 failed——**这正是上周 commit `0c7f539`（2026-07-06）修的 consent 静默失败 bug 的同类未修实例**，那次只修了手动发送路径。

必须在以下 4 处显式加 `guard HoloAIFeatureFlags.aiDataProcessingConsentGranted`（主动跳过，非抛错）：

1. `MemoryInsightService.generateInsight` 入口（`MemoryInsightService.swift:102`）
2. `MemoryInsightBackgroundService.handleBackgroundTask`（`:69` 之后、`:76` 之前）
3. `MemoryInsightBackgroundService.checkForegroundCompensation`（`:171` 之后）
4. `MemoryInsightScheduleSettings` 调度后台任务前（`:39/72`）—— 否则调度了也是空跑

§8 的「未授权时展示本地进度」成立（有效记录日纯本地计数，不触发 AI），仍是方案最扎实的一块，但前提是 F2 落地——未授权时**绝不调用** `generateInsight`。

### 1.3 F3 — Push 整体 defer 到 V1.1

原方案 §1「Push 首发一起做」推翻。Push 有 4 个硬阻断项，任一未解决，首发 Push 都比不做更伤：

| # | 阻断项 | 代码证据 |
| --- | --- | --- |
| 1 | 系统通知权限未授权无兜底 | `MemoryInsightNotificationService.swift:49-78` 不查 `isAuthorized` 就 add，iOS 静默丢弃，开关=摆设 |
| 2 | 凌晨静默完全未定义 | 全代码库零「安静时段」逻辑；「记完账即触发」会在半夜 push 用户（投诉级 bug） |
| 3 | 全局幂等+限频存储不存在 | `TodoNotificationService` 只靠 `identifier` 去 pending 队列重，对「已投递历史」零记忆，无法保证「每周最多 1 次」「新手 Push 各发 1 次」 |
| 4 | DeepLink 只能页面级 | `DeepLinkTarget` 无 message/card case（`DeepLinkState.swift:32-49`），ChatView 有 `scrollTo` 但未接 DeepLink |

defer 范围：本周观察 Push 全套（含新手 2 次 + 常规每周）、`DeepLinkTarget.aiMessage` + ChatView 卡片级滚动、通知权限兜底 UI、凌晨静默窗口、幂等存储、AI 设置里「本周观察提醒」开关。

## 2. 高危问题修订

### 2.1 H1 — §10.4 复用清单重写

- **删除** `HoloAgentScheduler`（无定时能力，误导）。
- F1=a 下不新建 `WeeklyObservationScheduler`，改为**扩展 `MemoryInsightBackgroundService`**：支持「有效记录日达标触发」+「3 天轻量版提前生成」。

### 2.2 H2 — weeklyObservation 需独立业务优先级

现有 insight urgency 硬编码 `.pending`（最低，`HomeScheduleService.swift:245`），排序 `sorted{ $0.urgency > $1.urgency }.first`（`:142`）导致**有任何未完成任务时 insight 永不展示**。这是方案要解决的现状根因。

落地：在 `HomeScheduleService` 的排序层引入**业务优先级维度**（§6.1 的 P0-P5），给「新观察未读」P0（高于 `.pending`），不再复用 `.pending`。否则加完了等于没加。

### 2.3 H3 — 有效记录日 builder 净新增

- **禁用** `effectivePeriodRange.minDays`（语义是日历跨度，非有效记录日，`MemoryInsightContextBuilder.swift:337-352`）。
- 新建统一 builder：跨 Finance / Todo / Habit / Thought（V1 四类）按天聚合 + `Set<startOfDay>` 去重。
- **自带缓存层**（仿 `HoloWidgetSnapshotService` 快照模式），避免每次首页 `onAppear`（`HomeView.swift:291/461`）全表扫描 4 张 Core Data 表。

### 2.4 H4 — DeepLink defer

见 1.3。首发点击胶囊/Push 落到 HoloAI 根页（Push 已 defer，故首发仅胶囊点击，落到 HoloAI）。

## 1. 产品结论

本方案把首页顶部现有「有 N 个任务待完成」提醒位升级为统一的 **首页提醒胶囊**。它不再只承载任务提醒，也承载 HoloAI 的「本周观察」唤起与新用户洞察养成提示。

核心路径：

```text
首页提醒胶囊 / Push
  -> HoloAI
  -> 本周观察卡片
  -> 证据、建议、下一步动作
```

关键决策：

| 议题 | 决策 |
| --- | --- |
| 首页入口 | 复用首页顶部提醒胶囊，不新建站内信中心 |
| 最终承载 | 统一落在 HoloAI 聊天流里的「本周观察」卡片 |
| 新用户路径 | 3 个有效记录日生成轻量观察，7 个有效记录日生成完整观察 |
| Push | 首发一起做；用户可在 AI 设置里关闭「本周观察提醒」 |
| 卡片名称 | 对用户统一叫「本周观察」 |
| 埋点 | 本期不做，当前没有承接系统 |

本方案的产品目标不是再增加一个分析入口，而是让用户形成明确预期：

> 持续记录后，Holo 会越来越懂我，并主动告诉我生活里正在出现什么模式。

## 2. 背景与现状

### 2.1 当前首页提醒位

当前首页顶部胶囊由 `HomeScheduleService` 聚合提醒并展示一条最高优先级信息。现状主要展示待办任务：

```text
有 12 个任务待完成
```

代码现状已经具备跨模块扩展雏形：

- `ReminderModule` 已有 `task` 和 `insight`。
- `ScheduleReminderState` 已包含 `urgency / message / module / deepLinkTarget`。
- `HomeScheduleService.refresh()` 已按候选提醒排序取一条展示。
- 现有 insight 提醒目前更偏向记忆长廊洞察入口，缺少新用户进度、已读未读、Push、稍后提醒和 AI 授权兜底。

### 2.2 当前 Agent / 洞察能力

现有 Agent 深度分析已经有可复用基础：

- `HoloLocalAgentRuntime.shared` 生产装配已覆盖 `memory / habit / health / finance / goal / thought / task` 七类工具。
- Agent loop 已有 `plan -> executeTools -> minePatterns -> verifyClaims -> render` 的骨架。
- 结果模型已有 `claim / evidence / confidence / renderer`。
- Chat/HoloAI 已能承载 Agent 深度分析结果。

因此，本方案不重新造一套跨域 Agent。首发应复用现有 Agent 能力，新增一个产品场景：

```text
weekly_observation
```

它的目标是主动生成一条「最值得看的生活观察」，而不是等待用户提出一个深度分析问题。

### 2.3 与旧 AI 洞察方案的关系

旧 `AI 智能洞察 Phase B+C` 方案明确把首页胶囊、系统通知、异常触发提醒延期处理。本方案正是补齐这块产品链路：

- 不替代现有 `MemoryInsightService` 周期洞察。
- 不新增独立异常资产链路。
- 不把首页做成洞察正文承载页。
- 只把首页胶囊作为唤起入口，把内容统一收敛到 HoloAI。

## 3. 产品体验

### 3.1 首页提醒胶囊

可选副语义：

```text
再记录 2 天生成观察
```

若胶囊空间有限，优先展示短文案；详情进入 HoloAI 后展开。

#### 接近轻量观察

```text
再记录 1 天生成观察
```

#### 轻量观察已准备好

```text
第一条观察已准备好
```

#### 完整本周观察已准备好

```text
本周观察已准备好
```

#### 有强洞察待查看

```text
Holo 发现了一个新模式
```

### 3.3 点击行为

| 胶囊类型 | 点击后 |
| --- | --- |
| 任务提醒 | 保持现有逻辑，跳任务详情或任务页 |
| 新用户养成进度 | 跳 HoloAI，展示本周观察进度卡 |
| AI 授权未开启 | 跳 HoloAI 或 AI 设置授权引导卡 |
| 本周观察已生成 | 跳 HoloAI，并打开或插入「本周观察」卡片 |
| 观察失败兜底 | 跳 HoloAI，展示可重试卡片 |

首页胶囊不依赖长按。长按可发现性弱，不作为核心交互。

### 3.4 新手引导

第一次出现洞察类胶囊时，允许出现一次轻提示：

```text
点开后，Holo 会告诉你最近记录里出现了什么模式
```

规则：

- 只出现一次。
- 不常驻。
- 不遮挡首页主功能。
- 用户点过、关闭过或当天已看过 HoloAI 后不再展示。

### 3.5 HoloAI 承载卡片

点击首页胶囊或 Push 后，统一进入 HoloAI 聊天流。HoloAI 内生成或展示一张卡片：

```text
本周观察
```

卡片结构：

| 区块 | 内容 |
| --- | --- |

| 状态判断 | Holo 观察到的一个主要生活模式 |

轻量版卡片必须注明：

```text
这是基于最近 7 个有效记录日生成的本周观察。
```

## 4. 新用户洞察养成

### 4.1 有效记录日定义

「有效记录日」不是自然日，也不是打开 App 的天数。满足以下任一条件的日期算 1 个有效记录日：

- 新增记账记录。
- 新增或完成待办。
- 完成习惯打卡。
- 新增想法/观点记录。
- 有健康数据同步成功记录，且用户已授权健康数据读取。
- 与 HoloAI 发生一次有效记录型交互，例如创建任务、记账、记录想法、生成行动项。

同一天多次记录只计 1 个有效记录日。

### 4.2 触发标准

| 阶段 | 条件 | 产物 |
| --- | --- | --- |
| 养成期 | 0-2 个有效记录日 | 首页胶囊提示 Holo 正在认识你 |
| 轻量观察 | 达到 3 个有效记录日 | 可生成第一条轻量「本周观察」 |
| 完整观察 | 达到 7 个有效记录日 | 可生成完整「本周观察」 |

如果用户连续使用 7 个自然日但有效记录不足，应展示数据不足兜底，不伪造洞察。

### 4.3 新用户胶囊文案

| 阶段 | 胶囊文案 | HoloAI 详情文案 |
| --- | --- | --- |
| 0 天 | Holo 正在认识你 | 开始记录记账、待办、习惯或想法后，Holo 会逐渐发现你的生活模式 |
| 1 天 | 再记录 2 天生成观察 | Holo 已有第一批线索，继续记录会让观察更准 |
| 2 天 | 再记录 1 天生成观察 | 再有 1 个有效记录日，就能生成第一条初步观察 |
| 3 天 | 第一条观察已准备好 | Holo 可以基于最近记录生成一条轻量观察 |
| 7 天 | 本周观察已准备好 | Holo 可以基于一周记录生成更完整的生活观察 |

## 5. Push 方案

### 5.1 设置入口

在 AI 设置里新增：

```text
本周观察提醒
```

说明文案：

```text
Holo 会在记录足够后提醒你查看新的本周观察。关闭后仍可在 HoloAI 中手动查看。
```

开关影响范围：

- 关闭后不发送系统 Push。
- 不影响首页提醒胶囊。
- 不影响 HoloAI 内手动查看或生成本周观察。

### 5.2 新手 Push

新手 Push 首发允许发 2 次。

| 次数 | 触发时机 | Push 标题 | Push 正文 |
| --- | --- | --- | --- |
| 第 1 次 | 达到 2 个有效记录日，且未达到 3 天轻量观察 | Holo 已经有一些线索了 | 再记录 1 天，就能生成第一条本周观察 |
| 第 2 次 | 达到 3 个有效记录日，且第一条轻量观察可生成 | 第一条本周观察已准备好 | Holo 从最近的记录里整理出了一条生活线索 |

约束：

- 两次新手 Push 每个用户最多各发一次。
- 如果用户已在首页胶囊点击生成或查看，则不再发送对应 Push。
- Push 不写具体敏感内容。

### 5.3 常规本周观察 Push

7 天完整观察不算新手 Push，归入常规提醒。

建议文案：

```text
本周观察已准备好
Holo 从最近的记录里发现了一个生活模式
```

约束：

- 每个自然周最多发送 1 次完整本周观察 Push。
- 用户已读后不再重复推送同一条观察。
- 若 AI 授权未开启，不发送「已准备好」类 Push。

## 6. 首页胶囊优先级与曝光保护

### 6.1 基础优先级

首页胶囊按「用户当下价值」排序：

| 优先级 | 类型 | 说明 |
| --- | --- | --- |
| P0 | 新观察已生成且未读 | 给 24 小时保护期，优先展示 |
| P1 | 逾期任务 | 长期优先，保护任务主线 |
| P2 | 第一条观察养成进度 | 只在新手期展示，不压过逾期任务 |
| P3 | 今日到期任务 / 大量今日任务 | 任务主线提醒 |
| P4 | 普通任务数量 | 现有「有 N 个任务待完成」 |
| P5 | 普通本周观察提醒 | 没有更高优内容时展示 |

### 6.2 保护期规则

新观察生成后，胶囊享有 24 小时保护期：

- 24 小时内优先展示「本周观察已准备好」或「Holo 发现了一个新模式」。
- 保护期内仍不应覆盖 P1 级别的严重任务提醒，如果任务已经逾期，任务可抢占。
- 24 小时后，如果用户未读且有逾期任务或今日任务，任务重新优先。

这样可以短期推 Aha，又不长期伤害任务效率主线。

### 6.3 每日曝光保护

| 类型 | 曝光规则 |
| --- | --- |
| 新观察未读 | 可持续展示，直到用户已读、稍后提醒或今天不再显示 |
| 新手养成进度 | 每天最多展示 2-3 次 |
| AI 授权引导 | 每天最多展示 1 次，用户明确关闭后当天不再出现 |
| 普通任务数量 | 无洞察高优内容时展示 |

首页胶囊不做随机轮播，避免用户觉得状态不稳定。每次刷新只展示一条最高优先级提醒。

## 7. 已读、稍后提醒与关闭

必须区分以下状态：

| 状态 | 含义 | 影响 |
| --- | --- | --- |
| 已读 | 用户看过这条观察 | 不再作为「新观察」顶置 |
| 稍后提醒 | 用户暂时不想看 | 延后到今天晚些时候或明天再次展示 |
| 今天不再显示 | 用户今天不想看到这条胶囊 | 当天不再展示该提醒，第二天可恢复 |
| 关闭 Push | 用户关闭系统通知 | 只影响 Push，不影响首页胶囊和 HoloAI |

交互位置：

- 首页胶囊本身不展示关闭按钮，保持简洁。
- 点击进入 HoloAI 后，在本周观察卡片中提供 `稍后提醒`、`今天不再显示`。
- 若 HoloAI 卡片顶部有更多菜单，也可把这些动作放入菜单中。

### 8.1 首页胶囊文案

```text
开启 HoloAI 生成本周观察
```

### 8.2 HoloAI 内授权卡

文案：

```text
开启 AI 数据处理后，Holo 可以基于你的记录生成本周观察。你的记录仍由你控制，可随时关闭。
```

按钮：

```text
开启 HoloAI
```

次级说明：

```text
不开启也可以继续使用记账、待办、习惯和想法记录。
```

### 8.3 行为规则

- 未授权时不生成 AI 本周观察。
- 未授权时不发送「本周观察已准备好」Push。
- 未授权时可展示本地进度，但必须说明开启 HoloAI 后才能生成观察。
- 用户点击 `开启 HoloAI` 后跳转到现有 AI 数据处理授权入口。

## 9. 生成失败与数据不足兜底

### 9.1 AI 生成失败

如果 AI 调用失败、网络失败或 Agent job 失败，HoloAI 内展示：

```text
记录已经足够，Holo 暂时没能生成本周观察，请稍后再试。
```

按钮：

```text
稍后再试
重新生成
```

首页胶囊可展示：

```text
本周观察待重试
```

### 9.2 数据不足

如果用户自然使用 7 天但有效记录不足，不能伪造洞察。展示：

```text
Holo 还需要更多记录
```

详情：

```text
再记录几次记账、待办、习惯或想法后，Holo 才能生成更可靠的本周观察。
```

### 9.3 轻量观察置信度

3 天轻量观察必须降低承诺：

- 使用「初步观察」「可能」「线索」。
- 不使用「结论」「证明」「导致」。
- 不做心理诊断、健康诊断或因果断言。

## 10. 技术复用建议

### 10.1 首页胶囊

建议基于现有 `HomeScheduleService` 演进，而不是新建并行服务。

需要增强：

- `ReminderModule` 增加更细粒度的 insight/weeklyObservation 类型，或在 state 内增加 subtype。
- `ScheduleReminderState` 增加稳定 ID、展示状态和动作信息。
- 引入本周观察提醒候选构建器，负责把养成进度、未读观察、授权兜底、生成失败转成候选提醒。
- 排序逻辑不能只依赖 `ReminderUrgency.rawValue`，需要引入业务优先级和保护期。

建议概念模型：

```swift
enum HomeReminderKind {
    case taskOverdue
    case taskToday
    case taskPending
    case weeklyObservationUnread
    case weeklyObservationOnboarding
    case weeklyObservationAuthRequired
    case weeklyObservationRetry
}
```

### 10.2 本周观察状态

建议独立维护轻量状态，不直接塞进普通任务提醒。

最小状态字段：

```swift
struct WeeklyObservationReminderState {
    let id: String
    let status: Status
    let effectiveRecordDays: Int
    let generatedAt: Date?
    let readAt: Date?
    let snoozedUntil: Date?
    let hiddenUntil: Date?
    let pushStage1SentAt: Date?
    let pushStage2SentAt: Date?
    let weeklyPushSentAt: Date?
}
```

状态：

```swift
enum Status {
    case notEnoughData
    case authRequired
    case lightReady
    case fullReady
    case generating
    case readyUnread
    case readyRead
    case failed
}
```

### 10.3 有效记录日统计

有效记录日应由统一 builder 计算，避免各入口口径不一致。

候选来源：

- FinanceRepository
- TodoRepository
- HabitRepository
- ThoughtRepository
- HealthRepository 或 HealthKit 同步状态
- Chat/Agent 记录型交互

V1 可以先支持前四类确定性记录，健康和 Chat 交互作为增强项；但产品文案口径应保持一致。

### 10.4 Agent 复用

推荐新增一个产品场景，而不是新建运行时：

```text
weekly_observation
```

可复用：

- `HoloAgentScheduler`
- `HoloLocalAgentRuntime.shared`
- 七类 Holo data tools
- `HoloEvidenceLedger`
- `HoloClaimVerifier`
- `HoloAgentResultRenderer`
- Chat/HoloAI 的 Agent 卡片展示链路

需要增强：

- 给 Agent question 或 job type 明确传入「生成本周观察」场景。
- Prompt 要求只输出 1 条主观察，最多 2 条备选证据。
- Renderer 需要支持「本周观察」卡片格式，而不是多条 claim 的长报告。
- 结果需要持久化已读/未读，供首页胶囊和 Push 去重。

### 10.5 Push

可复用现有 `TodoNotificationService` 的本地通知能力，但需要新增：

- `WeeklyObservationNotificationCategory`
- AI 设置中的开关。
- 新手 Push 两阶段发送记录。
- 常规每周 Push 限频。
- DeepLink 到 HoloAI 本周观察卡片。

Push 内容不得包含具体敏感结论，只能提示有观察可查看。

## 12. 验收标准

### 12.1 新用户养成

- 0-2 个有效记录日：首页胶囊能提示 Holo 正在认识你。
- 达到 2 个有效记录日：若 Push 开启且未查看，允许发送第 1 次新手 Push。
- 达到 3 个有效记录日：首页胶囊能提示第一条观察已准备好；Push 开启时允许发送第 2 次新手 Push。
- 达到 7 个有效记录日：可生成完整本周观察。

### 12.2 首页胶囊

- 新观察未读时可优先展示。
- 逾期任务长期优先，不被养成进度压掉。
- 新观察 24 小时保护期后，任务提醒可重新优先。
- 用户选择今天不再显示后，当天不再展示该洞察胶囊。

### 12.3 HoloAI 卡片

- 点击首页胶囊进入 HoloAI。
- 能展示本周观察卡片。
- 卡片包含状态判断、证据、建议和下一步动作。
- 3 天轻量版明确标注初步观察。
- 已读后首页不再把同一条观察当成新观察顶置。

### 12.4 Push

- AI 设置里可关闭本周观察 Push。
- 关闭 Push 后首页胶囊仍可展示。
- 新手 Push 每阶段最多发送一次。
- 常规本周观察 Push 每周最多发送一次。
- Push 点击能进入 HoloAI 本周观察。

### 12.5 AI 授权与失败兜底

- 未授权时不生成 AI 本周观察。
- 未授权胶囊点击后能看到授权说明和开启按钮。
- AI 失败时有重试或稍后再试，不出现空白页。
- 数据不足时不伪造洞察。

## 13. 建议实施顺序

### Phase 0：方案审查与口径冻结

- 让 GLM 审查本方案。
- 明确首页胶囊优先级是否与任务主线冲突。
- 明确 3 天轻量观察是否足够安全。
- 明确 Push 文案和频率是否过打扰。

### Phase 1：首页胶囊状态层

- 扩展 `HomeScheduleService` 的候选提醒模型。
- 新增本周观察提醒候选构建器。
- 补已读、稍后提醒、今天不再显示。
- 补每日曝光保护和 24 小时保护期。

### Phase 2：新用户有效记录日

- 实现有效记录日统计。
- 接入首页胶囊文案。
- 接入 HoloAI 进度卡。

### Phase 3：HoloAI 本周观察卡

- 复用 Agent runtime 生成 `weekly_observation`。
- 新增或扩展卡片 renderer。
- 支持 3 天轻量版和 7 天完整版。
- 支持已读状态回写。

### Phase 4：Push 与设置

- AI 设置增加「本周观察提醒」。
- 新增新手 Push 两阶段规则。
- 新增常规本周观察 Push。
- 接入 DeepLink 到 HoloAI 本周观察。

### Phase 5：兜底与验收

- 未授权兜底。
- 生成失败兜底。
- 数据不足兜底。
- 补单测、目标场景手测和构建验证。

## 14. GLM 审查重点

请重点审查以下问题：

1. 首页胶囊从任务提醒升级为统一提醒后，会不会损伤任务主线？
2. 「新观察 24 小时保护期」是否会过度抢占任务提醒？
3. 3 个有效记录日生成轻量观察是否太早，是否容易给出低质量结论？
4. 有效记录日口径是否合理，是否会鼓励用户为了触发洞察而低质量记录？
5. 新手 Push 两次是否过打扰，文案是否足够克制？
6. 未授权 AI 时的引导是否合规、清晰且不过度承诺？
7. 现有 Agent 深度分析结果模型是否足够承载「一条主观察 + 证据 + 行动」？
8. 首页胶囊、Push、HoloAI 卡片之间的已读/稍后/关闭状态是否会产生状态不一致？
9. 是否需要在 V1 就加入更严格的隐私提示或本地处理说明？
10. 本方案是否遗漏了 App Store 审核可能关注的 AI 数据处理与通知权限问题？

---

## 15. 当前结论

本方案建议 Go，但需要 GLM 对三个点做重点挑战：

1. 首页任务提醒与洞察提醒的优先级是否足够稳。
2. 3 天轻量观察是否能在质量和新人留存之间取得平衡。
3. Push 频率是否克制，是否会被用户感知为打扰。

若审查通过，可以进入实施计划；若审查指出风险，优先调整首页优先级、有效记录日阈值和 Push 频率，不建议推翻「首页胶囊唤起、HoloAI 卡片承载」这一主结构。
