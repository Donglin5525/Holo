> 核验日期：2026-07-22  
> 核验对象：本地工作区 `bc4832f` + 当前未提交改动；生产公开状态为 `f473c294`（2026-07-20 构建）

> 本文目标：区分事实、风险推断和误判，并给出可独立交付、可验证、可部署的整改顺序。

## 一、产品结论

ChatGPT 的总判断基本成立：Holo 的主要问题不是业务架构方向错误，而是安全、数据语义、测试执行和文档真相没有形成闭环。

但原评审把不同性质的问题都压成了 P0，容易导致整改范围失控。当前应聚焦三个真实风险：

1. **成本接口缺少可信调用身份和输入硬限制**：AI/ASR 会信任客户端声明的 device ID，App Attest 仍是占位实现。
2. **Agent 可能把“读取失败、范围截断、缺失值”解释为真实空数据或完整数据**：会直接损害回答可信度。
3. **测试文件存在不等于测试执行（已于 Phase 0 修复）**：原始状态是 70 个 XCTest 文件中仅 30 个进入 HoloTests Sources，且仓库没有 CI workflow；现已建立统一清单、执行入口和 CI 门禁。

不建议全面重写。先完成安全止血和测试门禁，再修数据语义，最后做启动层、历史快照和模块化治理。

## 二、核验方法与边界

- 逐项对照当前工作区源码、Xcode `project.pbxproj`、npm scripts、README/TODO/CLAUDE 文档。
- 本地后端测试：`134/134` 通过；这证明现有测试没有回归，不代表未覆盖的缺陷不存在。
- Phase 0 完成后，本地 iOS 完整 XCTest 实际执行 `690/690` 通过，另有不适合桥接进 XCTest 的脚本测试 `6/6` 通过。
- 生产公开只读检查：`/v1/release/status` 和 `/v1/health` 可访问；公开状态不能证明 App Attest、生产密钥和内部配置已启用。
- 未执行会产生外部副作用的 AI/ASR 生产请求，也未修改 GitHub 分支保护。
- 当前工作区已有大量用户未提交改动；后续实施必须按整改包独立暂存和提交，不能混入现有业务改动。

### B. 部分成立，但原评审优先级或表述不准确

| 原评审判断 | 修正后的结论 |
|---|---|
| “限流检查和增加不是原子的，因此现在会超发” | 检查与增加确实不是条件式单 SQL；但当前单 Node 进程 + 同步 better-sqlite3 下不是眼前 P0。多进程/多容器前必须升级。 |
| “大量测试不执行，standalone 测试本身是错误做法” | 测试假覆盖事实成立；但 standalone 测试是 Holo 对纯逻辑的既定补充机制，不应一律改 XCTest。真正缺口是：没有统一 manifest/runner，XCTest 文件也大量漏 Target。 |
| “HoloApp `.task` 一定重复执行并造成故障” | 有重复触发风险，但目前没有运行时证据证明必然发生。应先给启动任务加幂等/阶段观测，再拆 Coordinator。 |
| “金额 Double 是 P0 数据错误” | 对 AI 趋势和展示通常不是 P0；对账、预算余额、资产差额等精确链路必须保持 Decimal/分。 |
| “Docker 缺 SBOM/只读文件系统就是当前重大故障” | 属供应链与部署硬化，不是当前功能故障；固定基础镜像和非 root 优先，SBOM/扫描随后补。 |
| “所有 DataSource 都应立刻改成新泛型 Result” | 错误语义必须修，但应复用现有 Health strict outcome / Tool status 模型，避免再造一套平行协议。 |

- XCTest Target 对齐、standalone manifest、CI workflow、README/架构/测试/部署文档。
- 官网路由和演示文案。

### 需要东林确认或外部权限

1. **匿名 AI 策略**：推荐继续允许匿名用户使用，但身份改为 App Attest 验证的 app instance session；登录用户再叠加 Apple user session。
2. **App Attest 真机条件**：Apple Developer entitlement、真实 bundle/environment、真机灰度和生产 key 注册。
3. **生产部署**：所有 HoloBackend 改动本地完成后必须发布 ECS，并用真实域名验证；不能只 commit/push。
4. **GitHub 管理设置**：main branch protection、required checks、CODEOWNERS 审查策略。
5. **基础设施访问策略**：Admin IP allowlist/VPN、反向代理 body limit、镜像仓库与 digest。
6. **历史快照数据模型**：Goal/Task/Habit/预算是否长期保留日快照及其保留周期。
7. **法律内容**：Privacy、Terms、Support、Account Deletion、Data Export 的正式文本和联系信息。

## Phase 4：真实 App Attest 身份闭环（3～6 天 + 真机灰度）

### 推荐产品方案

- 匿名用户：App Attest 验证 app instance，后端签发短期 Holo instance session。

### 外部验收缺口

- 手写 Apple attestation verifier 尚未喂入真实 Apple attestation object；测试使用可控 fake verifier 验证了编排与安全边界，不能替代真机密码学验收。
- 需要 Apple Developer 中为对应 App ID/Provisioning Profile 启用 App Attest，并向 ECS 安全提供 Apple 可信根证书路径。
- 需要覆盖首次安装、重装、key 丢失、升级、离线、Apple 服务不可用、Debug/TestFlight/Release；通过后才能在生产打开 enforcement。

### 验收

- 伪造 device ID 无法获得新配额。
- assertion 重放、counter 回退、错误 bundle/environment 全部拒绝。
- 正常用户升级不丢失 AI 能力；失败路径有可理解提示和服务端可观测错误码。

## 七、完成定义

整改不能以“代码已改”或“本地 build 成功”作为完成。最终完成需要同时满足：

- 安全：生产 AI/ASR 不信任自报 device ID，输入和成本有硬上限。
- 数据：读取失败、真实空数据、零值、截断和权限不可用可被 Agent 明确区分。
- 测试：所有 XCTest/standalone 测试都有可证明的执行入口，CI 失败可阻止合并。
- 隐私：默认日志和普通 Agent 查询不保存/暴露健康、财务、记忆、语音正文。
- 发布：后端修改已部署，release commit/digest、ready、拒绝路径和关键行为均有生产证据。
- 文档：README、架构、安全、测试、部署和线上真实行为一致。
