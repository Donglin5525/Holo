# HoloAI 可信记忆与细粒度洞察改造方案

> 日期：2026-06-12  
> 范围：HoloAI 短期记忆、长期记忆、Memory Insight、Daily Sense、UserContext 注入、习惯/财务细粒度趋势。  
> 结论：当前 HoloAI 已有记忆和洞察基础设施，但主链路仍偏“报表摘要 + 洞察卡候选”，还没有形成“结构化事实信号 -> 持久短期记忆 -> 多周期命中 -> 可解释长期记忆 -> 按需召回”的可信闭环。

---

## 1. 用户问题复述

用户真实不满不是“记忆条数太少”，而是：

1. 记住的内容不重要：例如“任务清零”“任务全部闭环”“支出偏高”只是汇总结果，不是洞察。
2. 缺少可验证细节：例如“宵夜吃得多”必须能落到“晚上餐饮消费占比上升 + 夜宵/宵夜科目频次增加 + 相比上期变化明显”。
3. 对坏习惯理解不够：例如“抽烟 1 号 8 根、2 号 12 根、3 号 20 根”应该被识别为坏趋势，而不是“习惯完成更多”。
4. AI 没有形成持续理解：用户用了一周后，感受不到短期记忆、长期记忆在回答和洞察中真正生效。

目标不是让 HoloAI 说得更像 AI，而是让它先有可靠的事实判断，再表达。

---

## 2. Codex / ChatGPT 记忆设计可借鉴点

参考 OpenAI 官方记忆说明，成熟 AI 记忆系统有几个关键原则：

1. **记忆不是全量聊天记录**  
   系统会在回答时按需使用相关上下文，而不是每次把所有历史都塞进 prompt。

2. **可见、可改、可删除**  
   用户应该能看到 AI 记住了什么，能纠正、删除、要求不要再提。

3. **分层记忆**  
   有显式保存的记忆，也有从历史中综合出来的上下文。稳定事实和近期状态不能混在一起。

4. **来源可解释**  
   用户需要知道“这次回答为什么用了某条记忆”，否则记忆越强，越容易不可信。

5. **只在相关时使用**  
   记忆不是每次都强行插入，而是当它能改善当前回答时才召回。

HoloAI 可以借鉴这套思想，但不能照搬。原因是 HoloAI 不是纯聊天产品，它拥有财务、任务、习惯、目标、健康、想法等结构化生活数据。HoloAI 更应该先做“证据型记忆”，再做“语言型解释”。

## 3. 当前代码事实

### 3.1 已经具备的能力

当前代码已经有这些基础：

| 能力 | 当前路径 | 评价 |
| --- | --- | --- |
| 长期记忆存储 | `HoloLongTermMemoryStore.swift` | 有 JSON Store，支持 candidate / confirmed / rejected / silentlyAccepted |
| 长期记忆注入 | `UserContextBuilder.swift` + `AIUserContextMessageBuilder.swift` | 聊天链路会读取 confirmed / silentlyAccepted 记忆并注入“记忆摘要” |
| 记忆候选提取 | `HoloLongTermMemoryCandidateObserver.swift` | 从 MemoryInsight 生成事件里抽取有 patternType 和 evidence 的卡片 |
| 语义记忆字段 | `HoloLongTermMemoryModels.swift` | 已有 phaseShift / stablePattern / driftSignal / lifeEvent / statMilestone |
| 情景记忆 Store | `HoloEpisodicMemoryStore.swift` | 已有短期情景记忆模型、过期、拒绝、删除、suppression |
| Memory Observer | `HoloMemoryObserverService.swift` | 已有“信号 -> LLM -> parser -> validator -> applier -> store”雏形 |
| 负向习惯语义 | `HabitPerformanceModels.swift` + `HabitFocusModels.swift` | 已能区分坏习惯、控制率、超标天数、发生总量 |
| 财务餐饮分类基础 | `CategoryCandidateResolver.swift` + 分类 catalog | 已能按时间把餐饮归到早餐/午餐/晚餐/夜宵 |

这说明 HoloAI 不是从零开始，问题是这些能力没有组成一个稳定、日常自动运行、细粒度可信的闭环。

### 3.2 当前主要断点

#### D1. 短期记忆快照太浅

`HoloMemorySnapshotBuilder` 目前只是基于 `UserContextBuilder` 生成“今日习惯”“今日消费”等轻量信号，且 `relevantLongTermMemorySummary` 仍为 nil。它不是用户期待的“最近状态记忆池”。

影响：用户看到的会是“今天消费 ¥xx”“完成 x/y”，而不是“最近宵夜频次上升”“抽烟发生量连续增加”。

#### D2. 情景记忆观察不是日常主链路

`HoloMemoryObserverService.runObservation()` 设计正确，但触发入口主要在 AI 设置页的“手动触发观察”。`episodicMemoryObservationEnabled` 默认关闭，且目前只覆盖习惯和目标。

影响：用户正常使用一周，短期记忆很可能没有持续自动沉淀。

#### D3. 财务信号没有进入 Memory Observer

财务侧已经有“夜宵/宵夜”分类和分析上下文的子分类、消费模式能力，但 Memory Observer 的 observation package 目前只有 `habitSignals` 和 `goalSignals`。

影响：“宵夜吃得多”这种用户最关心的生活模式不会进入短期记忆链路。

#### D4. MemoryInsight 财务上下文缺少“时段 + 二级分类 + 趋势”结构

`MemoryInsightContextBuilder.buildFinanceContext()` 有总支出、Top 分类、日支出、预算、交通语义，但没有输出：

- 晚间餐饮次数
- 夜宵/宵夜二级分类次数
- 夜宵金额占餐饮比例
- 相比上期的频次变化
- 具体日期和金额证据序列

影响：AI 只能写“餐饮偏高”或“支出偏高”，难以写出“6 月 1、2、3 日夜间餐饮连续增加”这种可信洞察。

#### D5. 负向习惯有总量，但缺少日序列证据

`HabitPerformanceSnapshot` 有 totalValue、overLimitDays、controlledDays、completionRate；`HabitFocusSummary` 能说“发生总量增加”“超标天数增加”。但它没有把每日值序列带给洞察层。

影响：系统可以知道“抽烟变差”，但很难展示“1 号 8 根、2 号 12 根、3 号 20 根”这种用户一眼信服的证据。

#### D6. 旧格式卡片会污染用户信任

截图里的“旧格式”标签说明当前用户看到的候选很可能来自旧 payload、旧 promptVersion 或缺少新版语义字段的卡片。虽然 `MemoryInsightService` 已经用 snapshotHash + promptVersion 做缓存检查，但 UI 仍允许用户看到旧语义质量的记忆候选。

影响：用户以为“AI 记忆正在工作”，实际看到的是旧规则/旧 prompt 产物。

## 4. 什么才算“值得记住的洞察”

HoloAI 应该把洞察分成四级，不要把所有 summary 都当记忆。

| 等级 | 名称 | 例子 | 是否可进入记忆 |
| --- | --- | --- | --- |
| L0 | 原始事实 | 6 月 3 日夜宵 ¥38；抽烟 20 根 | 不直接进入长期记忆 |
| L1 | 结构化信号 | 最近 7 天夜宵 4 次，较上期 +3 次 | 可进入观察包 |
| L2 | 短期情景记忆 | 最近一周夜间餐饮明显变多，值得留意 | 可过期、可续期 |
| L3 | 长期记忆 | 用户长期在控制宵夜/戒烟，需要回答时尊重这个目标 | 用户确认后进入 |

“任务清零”通常只是 L0/L1 统计，不应该默认进入 L2/L3。  
“抽烟次数连续上升，且与戒烟目标冲突”才是 L2 候选。  
“用户正在戒烟，希望 AI 用控制摄入的角度理解抽烟记录”才适合进入 L3。

## 5. 建议架构

```text
原始记录
  -> Domain Signal Builders
      - FinanceMemorySignalBuilder
      - HabitMemorySignalBuilder
      - TaskMemorySignalBuilder
      - GoalMemorySignalBuilder
      - ThoughtMemorySignalBuilder
  -> Observation Package Builder
  -> LLM Memory Observer
  -> Episodic Memory Store
  -> 多周期命中 / 衰减 / 用户反馈
  -> Long-term Memory Candidate
  -> 用户确认 / 编辑 / 拒绝
  -> UserContext / MemoryInsight / Daily Sense 按需召回
```

关键原则：

1. **规则负责事实，不负责人生判断。**
2. **LLM 负责解释重要性，但必须引用已有 evidenceRefs。**
3. **短期记忆默认会过期，长期记忆必须更稳。**
4. **所有记忆都要能回答：为什么记住、依据什么、怎么纠正。**
5. **当前输入最高优先级，不能被记忆覆盖。**

## 6. 需要新增的核心信号

### 6.1 财务：夜宵 / 晚间餐饮趋势

新增 `FinanceMemorySignalBuilder`，首版至少输出：

```swift
LateNightDiningSignal(
    period: "last7Days",
    lateNightMealCount: 4,
    previousLateNightMealCount: 1,
    lateNightMealAmount: 186,
    diningAmountRatio: 0.42,
    evidence: [
        "2026-06-01 22:43 夜宵 ¥38",
        "2026-06-02 23:10 外卖 ¥56",
        "2026-06-03 21:50 夜宵 ¥45"
    ]
)
```

触发建议：

- 最近 7 天夜间餐饮 >= 3 次，且较上期增加 >= 2 次。
- 或夜宵/宵夜二级分类金额占餐饮 >= 30%，且较上期明显增加。
- 或连续 3 天 21:00 后餐饮支出。

表达方式：

> 最近夜间餐饮在变密集。过去 7 天有 4 次 21 点后的餐饮记录，其中夜宵/外卖占餐饮支出的 42%。这不是“支出偏高”，更像是晚间饮食节奏的变化。

### 6.2 习惯：负向习惯升级趋势

扩展 `HabitPerformanceSnapshot` 或新增 `HabitDailyValuePoint`：

```swift
HabitDailyValuePoint(
    date: "2026-06-03",
    value: 20,
    targetValue: 8,
    isOverLimit: true
)
```

触发建议：

- 负向习惯最近 3 天连续上升。
- 当前窗口总量较上期增加 >= 30%。
- 超标天数较上期增加 >= 2 天。
- 与目标/档案中的“戒烟、控烟、少喝酒、少熬夜”语义冲突。

表达方式：

> 抽烟记录不是在改善。1 号 8 根、2 号 12 根、3 号 20 根，连续三天上升，并且 3 号超过你的控制目标。这应该作为警戒，而不是普通打卡完成。

### 6.3 任务：只记录真正影响节奏的任务模式

不要默认记录“任务全部闭环”。只有满足以下之一才值得进入短期记忆：

- 高优任务反复延期。
- 同类任务连续多次创建但未完成。
- 与目标相关的任务长期无推进。
- 任务堆积和其他模块在同一天出现明显并发。

“本周完成 5 个任务、逾期 0 个”更适合做页面摘要，不适合进记忆中心。

## 7. 记忆召回策略

### 7.1 UserContext 应分三段注入

当前 `AIUserContextMessageBuilder` 只有一个“记忆摘要”。建议拆成：

1. **用户主动档案**：HoloProfile，稳定、最高可信。
2. **长期记忆**：用户确认/静默接受的稳定事实和偏好。
3. **近期情景记忆**：最近 7-30 天的状态，必须带过期和置信度。

示例：

```text
--- 用户主动档案 ---
用户正在关注戒烟。

--- 长期记忆 ---
[coreContext] 用户希望 HoloAI 用减少尼古丁摄入的视角理解抽烟记录。

--- 近期情景记忆 ---
[recentInsight, expires in 9 days] 最近三天抽烟数量连续上升，属于控制变弱信号。
规则：近期情景记忆只能辅助判断，不代表长期事实。
```

### 7.2 召回不应只按“最新”

`HoloMemorySummaryProvider` 当前主要按 useScopes、置信度、更新时间排序。后续应加入：

- 当前用户问题的模块意图。
- 记忆 sourceModules。
- 用户最近是否拒绝/隐藏过类似内容。
- 当前回答是否真的需要它。

问“帮我记一笔咖啡”时，不需要召回“任务清零”。  
问“我最近是不是戒烟状态变差了”时，必须召回抽烟趋势、戒烟目标、近期情景记忆。

## 8. UI 与信任机制

记忆管理中心需要从“卡片列表”变成“可解释控制台”：

1. 每条记忆展示：记忆正文、置信度、有效期、来源模块、证据条数。
2. 点开后展示具体证据：日期、金额/次数、分类/习惯名、来源记录。
3. 提供三个低摩擦反馈：
   - “不准”：事实或关系错。
   - “没感觉”：没错但不值得记。
   - “少提醒这个”：有用但太频繁。
4. 旧格式卡片应允许“一键刷新为新版”，不要继续和新记忆混在一起。
5. AI 回答里如果用了记忆，应该能显示“参考了哪些记忆”。

## 9. 实施路线

### Phase 0：先止血

1. 对“旧格式”洞察候选加降权或刷新提示。
2. 明确 Release/TestFlight 下哪些 memory feature flag 默认开启。
3. 确认生产后端 `memory_insight_generation`、`memory_observer` prompt 版本是否已部署。
4. 加一个 debug 入口导出当前：
   - confirmed long-term memories
   - suggested episodic memories
   - latest observation run
   - latest memory insight payload

### Phase 1：习惯可信洞察

1. 为负向习惯补每日序列数据。
2. 让 MemoryInsight 和 Memory Observer 都能拿到“1 号 8 根、2 号 12 根、3 号 20 根”这种证据。
3. 增加 golden tests：
   - 抽烟连续上升必须生成 warning/critical。
   - 负向习惯发生更多不能写成完成更多。
   - 无目标值时不能编造超标。

### Phase 2：财务夜宵/细分类洞察

1. 新增 `FinanceMemorySignalBuilder`。
2. 在 Observation Package 中加入 financeSignals。
3. MemoryInsightContext 增加 `mealTimingSummary` 或 `spendingTimePatternSummary`。
4. 覆盖夜宵、烟酒、打车、外卖等高价值模式。

### Phase 3：自动观察主链路

1. App 启动/前台恢复时，如果距离上次观察超过 12-24 小时，触发轻量 observation。
2. 记忆中心打开时允许刷新。
3. MemoryInsight 生成后触发轻量 observation，但只传该洞察和相关 signal，避免自我强化。

### Phase 4：长期记忆升级

1. 短期记忆被 2-3 个独立周期命中后，才进入长期候选。
2. 高影响内容必须用户确认。
3. 用户拒绝后写 suppression rule，至少 30 天内不再换个说法重复出现。

### Phase 5：回答时可解释召回

1. 将 UserContext 中的 profile / long-term / episodic memory 分段注入。
2. 让聊天、query_analysis、memory_insight_generation 都能按 purpose 选择记忆。
3. 在调试日志或 UI 中显示“本次用了哪些记忆”。

## 10. 验收标准

### 10.1 宵夜用例

输入数据：

- 最近 7 天 21:00 后餐饮 4 次。
- 其中 3 次二级分类为夜宵/外卖。
- 上期只有 1 次。

期望：

- 生成 finance signal。
- 记忆洞察不能只写“支出偏高”。
- 必须引用夜间餐饮频次、金额或日期证据。
- 可以进入短期记忆候选，但不应自动升级长期。

### 10.2 抽烟用例

输入数据：

- 6 月 1 日抽烟 8 根。
- 6 月 2 日抽烟 12 根。
- 6 月 3 日抽烟 20 根。
- 用户目标或档案里有戒烟/控烟。

期望：

- 生成 negativeHabitTrend。
- 严重度至少 warning，若超过目标明显则 critical。
- 表达为“控制变弱/摄入增加”，不能表达为“完成不错”。
- 进入短期情景记忆，等待用户确认是否长期记住。

### 10.3 冷数据过滤

输入数据：

- 本周完成 5 个任务，逾期 0 个。
- 没有高优任务、没有目标关联、没有明显变化。

期望：

- 可以出现在页面摘要。
- 不进入长期记忆候选。
- 不应在记忆中心反复出现“任务清零”。

## 11. 风险与边界

1. **隐私风险**：财务和坏习惯属于高影响内容，不能默认长期保存。
2. **误归因风险**：夜宵增加不能推断压力大，只能说饮食时段变化。
3. **自我强化风险**：MemoryInsight 生成的卡片不能直接作为下一次洞察事实，必须保留原始 evidenceRefs。
4. **旧缓存风险**：旧 promptVersion 产物可能继续显示，需要刷新或标记。
5. **后端发版风险**：涉及 HoloBackend prompt、registry、Docker 或路由改动时，必须部署后端，否则生产不会生效。

---

## 12. 建议优先结论

建议不要先继续调 prompt，而是先做两条端到端薄切片：

1. **抽烟/负向习惯薄切片**  
   因为当前习惯语义基础最好，最快能做出“AI 确实懂坏习惯趋势”的体验。

2. **夜宵/晚间餐饮薄切片**  
   因为这是用户最直观能验证的财务洞察，也能证明 HoloAI 不再只是说“支出偏高”。

这两条跑通后，再扩大到任务、目标、健康、想法。否则继续堆大方案，用户体感不会改变。
