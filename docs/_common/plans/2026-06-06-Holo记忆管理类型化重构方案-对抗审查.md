# Holo 记忆管理类型化重构方案 — 对抗性审查

审查日期：2026-06-06
审查依据：逐文件追踪完整调用链，而非仅依赖方案文档描述

---

## 🚨 CRITICAL：发现 3 个方案未识别的致命缺陷

### C1. 长期记忆召回链路完全断路 — 方案 Phase 4 建在空中楼阁上

**事实**：逐链路追踪发现：

```
HoloMemorySummaryProvider.selectRelevantSummary()  ← 已实现，但零调用者
        ???
UserContextBuilder.buildContext()                   ← memorySummary 硬编码 nil（第 67 行）
        |
AIUserContextMessageBuilder.build(.chat)            ← memorySummary 为 nil，注入逻辑永远不执行
        |
HoloBackendAIProvider → LLM                        ← LLM 从未收到任何长期记忆上下文
```

**影响**：
- 方案 Phase 4 写着"优先注入 `aiUseSummary`""按 `useScopes` 与当前 `purpose` 选择记忆"——但 **当前根本没有任何记忆被注入到任何 prompt 中**。
- Phase 4 应该从"调整召回策略"改为"**修复并激活召回链路**"，否则后面所有关于 `aiUseSummary`、`useScopes`、`prohibitedInferences` 的设计都无从验证。
- 这是一个 **前置阻塞问题**，必须先解决，否则方案交付后仍无法证明 HoloAI 确实在使用长期记忆。

**建议**：在 Phase 1 之前增加 **Phase 0: 召回链路修复**：
1. `UserContextBuilder.buildContext()` 接入 `HoloLongTermMemoryStore.queryPromptSummary()`
2. 验证 `AIUserContextMessageBuilder` 的注入逻辑实际触发
3. 通过后端 `/admin/logs` 确认 LLM 请求中包含记忆摘要
4. 此阶段不改类型、不改摘要格式，只把已有链路接通

---

### C2. 情景记忆体系与方案存在概念冲突

**事实**：代码中存在两套独立的记忆体系：

| 体系 | 数据模型 | 存储 | 来源 | 过期 | 展示位置 |
|------|----------|------|------|------|----------|
| 长期记忆 | `HoloLongTermMemory` | `HoloLongTermMemoryStore` (JSON) | 洞察卡片提取 | 无强制过期 | 记忆管理页 "已记住" |
| 情景记忆 | `HoloEpisodicMemory` | `HoloEpisodicMemoryStore` (JSON) | `HoloMemoryObserverService` (LLM) | 90 天默认过期 | 记忆管理页 "最近记住" |

方案提出的"轻量记录"（`statMilestone`）在概念上与情景记忆高度重叠：
- 两者都用于"不强影响 AI 判断的轻量数据"
- 两者都有过期机制
- 两者都展示在记忆管理页

但方案完全没有提及情景记忆体系。具体冲突：

1. **UI 层**：方案提议将页面改为"待确认 / 长期记忆 / 轻量记录"三段，但现有 UI 已经是"最近记住(情景) / 待确认 / 已记住"三段。"轻量记录"是否替代"情景记忆"？还是并存？用户会看到四个 Section？

2. **召回层**：`HoloEpisodicMemoryStore` 同样没有任何召回注入链路。方案没有提及是否要同时修复情景记忆的召回。

3. **`statMilestone` 的归宿**：如果 `statMilestone` 进入"轻量记录"Section，那它是存在 `HoloLongTermMemoryStore` 还是 `HoloEpisodicMemoryStore`？如果是前者，它和情景记忆的关系是什么？

**建议**：方案必须明确：
- `statMilestone` 与情景记忆的关系（替代？并存？合并？）
- 是否需要同时重构情景记忆的分类逻辑
- UI Section 最终是 3 个还是 4 个

### C3. `MemoryCandidateSemanticMapper` 的实现方式完全未指定 — 可能引入重大成本/复杂度

**事实**：方案第 6 节提出桥接层 `MemoryCandidateSemanticMapper`，负责把洞察卡片转换成长期记忆语义。但方案没有回答：

1. **这是 LLM 调用还是规则映射？**
   - 如果是 LLM 调用：每次洞察生成 3-5 张卡片，每张卡片一次调用 = 每次洞察多 3-5 次 LLM 请求。按当前架构（后端网关代理），这意味着显著增加延迟和成本。
   - 如果是规则映射：仅凭关键词匹配无法可靠区分 `phaseShift` vs `stablePattern` vs `driftSignal`。现有 `patternType` 的关键词匹配只有 4 种（`spending_increase`/`habit_break`/`task_backlog`/`recovery`），远不够覆盖 5 种语义类型。

2. **与洞察生成 prompt 的关系是什么？**
   - 如果洞察生成 prompt 本身就能输出语义化字段，那 `MemoryCandidateSemanticMapper` 就不需要是独立的一层。
   - 如果需要独立映射，那 insight prompt 的输出生成和语义映射就是两个串行步骤。

**建议**：方案必须明确选择以下之一：
- **方案 A（推荐）**：修改洞察生成 prompt，让 LLM 在生成卡片时同时输出 `semanticType`/`displaySummary`/`aiUseSummary` 等字段。这样 `MemoryCandidateSemanticMapper` 只做轻量格式转换，不做 LLM 调用。
- **方案 B**：在 `MemoryCandidateSemanticMapper` 中增加一次 LLM 调用。此方案需要评估成本影响，且需要处理 LLM 调用失败时的降级策略。
- **方案 C**：纯规则映射。此方案限制大，不推荐。

## ⚠️ HIGH：5 个高风险问题

### H1. 方案未涉及 Prompt 双端同步

**事实**：根据 CLAUDE.md 的 Prompt 双端同步规则，任何 Prompt 变更必须：
1. 修改 iOS 端 `PromptManager.swift` 内嵌模板（后备）
2. 同步修改后端 `HoloBackend/src/prompts/defaultPrompts.json`（生效端）
3. 升级 `PromptManager.promptVersions` 版本号
4. 重建 Docker 镜像并部署

方案 Phase 1 提到"更新 `memoryInsightGeneration` 或新增长期记忆候选提取 prompt"，但：
- 没有提及需要同时修改后端 `defaultPrompts.json`
- 没有提及 Docker 重建部署
- 如果新增独立 prompt 类型（如 `memory_candidate_semantic_mapping`），需要在后端 `src/config.js` 中新增路由配置

**建议**：Phase 1 的实施步骤中必须包含后端 prompt 同步和部署流程。

---

### H2. 候选类型硬编码 `.recurringPattern` — 方案未提及如何消除

**事实**：`HoloLongTermMemoryCandidateObserver` 第 47 行：
```swift
type: .recurringPattern  // 所有从洞察卡片提取的候选都硬编码为这个类型
```

方案引入了 `HoloMemorySemanticType`（5 种），但：
- 旧 `type` 字段在晋升策略中仍被使用（`HoloMemoryPromotionPolicy.evaluate()` 有 11 条分支，按旧 `type` 决策）
- 方案说"旧 type 保留做 fallback"，但 `HoloLongTermMemoryCandidateObserver` 硬编码了 `.recurringPattern`——即使新增了 `semanticType`，旧 `type` 仍是错误的
- 晋升策略的 11 条分支中，只有 `.recurringPattern` 相关的 3 条会被命中，其他 8 条形同虚设

**建议**：
- Phase 1 必须修改 `HoloLongTermMemoryCandidateObserver`，将硬编码的 `.recurringPattern` 改为根据 `semanticType` 映射到合理的旧 `type`（如 `phaseShift` → `.profileBackedFact`、`stablePattern` → `.recurringPattern`、`driftSignal` → `.recurringPattern`、`lifeEvent` → `.longTermGoal`、`statMilestone` → `.recurringPattern`）
- Phase 3 重写晋升策略时，应基于 `semanticType` 而非旧 `type` 做决策

---

### H3. `expiresAt` 有字段无执行 — `driftSignal` 过期机制不可靠

**事实**：`HoloLongTermMemory` 模型已有 `expiresAt: Date?` 字段，但：
- `HoloLongTermMemoryStore` 没有任何清理过期记忆的方法
- 没有定时任务（cron/timer）检查过期
- `queryPromptSummary()` 和 `queryConfirmed()` 不过滤 `expiresAt`
- 用户查看记忆管理页时也不会自动隐藏过期记忆

方案要求 `driftSignal` 默认 14-30 天过期，但如果只是设置字段值，没有消费端逻辑，过期机制形同虚设。

**建议**：Phase 3 必须增加：
- `HoloLongTermMemoryStore.cleanupExpired()` 方法
- `queryPromptSummary()` 和 `queryConfirmed()` 中过滤 `expiresAt < now`
- 在记忆管理页加载时触发过期清理
- 或在 App 生命周期中（如 `scenePhase` 变化时）触发清理

---

### H4. `prohibitedInferences` 和 `useScopes` 是死数据 — 没有消费端

**事实**：
- 当前 `useScopes` 在整个代码库中 **零存在**（搜索确认）
- `prohibitedInferences` 同样不存在于当前模型中
- 唯一的记忆注入格式是 `"title：summary"`（`HoloLongTermMemoryStore.queryPromptSummary()`），没有任何场景使用这两个字段

方案在模型中新增这两个字段，但：
- Phase 4 只说"按 `useScopes` 与当前 `purpose` 选择记忆"，没有给出选择算法
- `prohibitedInferences` 在方案中没有任何消费端描述——写入后谁读它？如果注入到 prompt，格式是什么？
- 当前记忆注入最多 5 条（`prefix(5)`），用 `useScopes` 筛选后可能不足 5 条，降级策略是什么？

**建议**：
- `useScopes` 和 `prohibitedInferences` 的消费逻辑必须在 Phase 4 中有明确的伪代码/算法描述
- 或者先在 Phase 1 中只写不读，Phase 4 中才接通，但要标注为"预期未验证"

---

### H5. 数据迁移策略不完整 — 旧数据 fallback 路径模糊

**事实**：方案说"旧数据没有 `semanticType` 时按旧 `type` 做 fallback"，但：
- 旧 `type` 的 5 个枚举值（`explicitUserPreference`/`stableFeedbackPreference`/`recurringPattern`/`longTermGoal`/`profileBackedFact`）与新 `semanticType` 的 5 个枚举值语义完全不同
- 没有 fallback 映射表（如 `.recurringPattern` → `stablePattern`？还是 `driftSignal`？）
- 旧 `summary` 字段与新 `displaySummary` 的格式完全不同（旧的是洞察卡片 body，新的是用户可审核的事实摘要）。方案说"旧 summary 继续展示"，但用户看到的是"体重习惯连续中断"这种系统话术——这正是方案要解决的问题

**建议**：Phase 2 必须包含：
- 旧 `type` → 新 `semanticType` 的映射表
- 是否需要一次性迁移脚本重写旧 summary
- 旧数据在 UI 上如何标注（如"旧格式记忆，建议重新确认"）

## 📋 MEDIUM：4 个中等问题

### M1. 方案的 Phase 顺序有依赖冲突

- Phase 3（晋升策略）依赖 `semanticType` 字段 → 需要 Phase 2（模型）完成
- Phase 4（召回）依赖 `aiUseSummary` 和 `useScopes` → 需要 Phase 1 和 Phase 2 完成
- 但 Phase 4 **同时依赖 C1 的召回链路修复**（当前断路）
- Phase 5（UI）依赖所有前序 Phase

建议调整为：
```
Phase 0: 修复召回链路（C1）→ 可独立验证
Phase 1: Prompt + 桥接层（含后端同步）
Phase 2: 模型扩展 + 迁移映射
Phase 3: 晋升策略重写
Phase 4: 召回策略升级（基于 Phase 0 的已验证链路）
Phase 5: UI 调整
```

### M2. 没有回滚方案

方案涉及 7+ 个文件的修改（模型/Store/Observer/Policy/Provider/Builder/UI），如果新语义类型产生 worse 结果：
- 没有 feature flag 机制切回旧行为
- 旧数据迁移后无法自动恢复
- 后端 prompt 变更需要重新部署

**建议**：每个 Phase 增加 feature flag 守卫，允许逐 Phase 灰度。

### M3. 验证用例缺少自动化测试

方案第 11 节的验证用例都是手工检查项。对于核心逻辑（类型分类、晋升决策、过期清理、召回筛选），应该有单元测试：
- `MemoryCandidateSemanticMapper` 的映射正确性
- `HoloMemoryPromotionPolicy` 按新类型的决策
- `HoloLongTermMemoryStore` 过期清理逻辑
- `HoloMemorySummaryProvider` 按 `useScopes` 筛选逻辑

### M4. 没有考虑并发安全

`HoloLongTermMemoryStore` 是 `enum`（无状态命名空间），所有操作直接读写 JSON 文件。如果：
- 洞察生成完成时 `CandidateObserver` 在后台线程写入
- 用户在 UI 上同时确认/删除记忆
- 两个操作同时触发 `save()` → JSON 文件可能损坏

这不是方案引入的新问题，但方案增加了字段和写入频率，加剧了风险。

## 📎 LOW：3 个小问题

### L1. 方案的示例改写质量高但覆盖不足

3 个示例覆盖了 `driftSignal`/`stablePattern`，但没有：
- `phaseShift` 的完整端到端示例
- `lifeEvent` 的示例
- 跨域卡片（`cross_domain` 类型）如何拆分语义的示例
- 卡片证据不足时的降级处理

### L2. `confidence` 字段重复

方案第 6 节 prompt 输出结构中有 `confidence: "low | medium | high"`，但现有模型已有 `HoloMemoryConfidence` 枚举（`low`/`medium`/`high`）。方案没有说明这是同一个字段还是新增字段。如果是同一个，为什么还要在 prompt 中让 LLM 输出？当前 `CandidateObserver` 已经用 `evidence.count` 硬编码 confidence。

### L3. 敏感度分类过于简单

当前 `classifySensitivity()` 仅靠 10 个关键词匹配。方案引入的 `sensitivity: "normal | highImpact | sensitive"` 如果仍用关键词匹配，会漏掉大量敏感场景（如收入数字、家庭变故、健康诊断）。但这个问题的优先级低于上述 CRITICAL/HIGH。

## 总结：修改建议优先级

| 优先级 | 编号 | 建议 |
|--------|------|------|
| **P0** | C1 | 增加 Phase 0：修复召回链路断路 |
| **P0** | C3 | 明确 `MemoryCandidateSemanticMapper` 的实现方式（推荐方案 A：改 prompt 让 LLM 直接输出） |
| **P0** | C2 | 明确 `statMilestone` 与情景记忆体系的关系 |
| **P1** | H1 | Phase 1 加入后端 prompt 同步和部署步骤 |
| **P1** | H2 | 修改 CandidateObserver 硬编码的 `.recurringPattern` |
| **P1** | H3 | 实现过期清理机制 |
| **P1** | H4 | 明确 `useScopes`/`prohibitedInferences` 的消费逻辑 |
| **P1** | H5 | 完善旧数据迁移映射 |
| **P2** | M1-M4 | Phase 顺序调整、回滚方案、单元测试、并发安全 |

**结论**：方案的问题诊断准确（"记忆管理页的问题是系统话术而非用户语言"），核心设计思路正确（语义类型 + 双摘要 + 晋升矩阵）。但方案建立在一个**未经验证的假设**上——"当前 HoloAI 已经在使用长期记忆"。实际上当前链路是断的，必须先修复基础链路，再叠加类型化重构。建议按上述优先级调整后重新评审。
