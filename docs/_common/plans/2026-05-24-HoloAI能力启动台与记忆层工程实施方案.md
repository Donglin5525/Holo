# HoloAI 能力启动台与记忆层工程实施方案

> **For agentic workers:** REQUIRED SUB-SKILL: Use `subagent-driven-development` or `executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 HoloAI 输入框上方无效 Tab 改造成“能力启动台”，并在不引入 Agent 的前提下，落地短期记忆、长期记忆、今日状态兜底、记忆注入与用户可控机制。

**Architecture:** V1 采用确定性工作流，不引入 Agent 编排。普通聊天和意图识别通过 `UserContext -> AIUserContextMessageBuilder` 注入受控上下文；MemoryInsight 生成链路单独通过 `MemoryInsightContext/contextJSON` 或生成后候选提取处理。长期记忆采用本地 JSON Store + 规则晋升 + 用户确认/撤销，不直接改写 `HoloProfile`。

**Tech Stack:** SwiftUI, Swift Concurrency, XCTest, JSON file persistence, existing Holo AI service layer.

---

## 0. 最终裁定

### 做什么

- 替换输入框上方当前无价值 Tab，形成固定的 HoloAI 能力启动台。
- 新增新人引导，解释记账、创建任务、记录心情等基础能力的最佳实践。
- 新增今日状态入口，并实现 rich / partial / empty 三档兜底。
- 新增短期记忆模型，但 V1 不做跨会话缓存，只做单次流程内快照复用。
- 新增长期记忆模型、候选池、晋升规则、确认规则和本地 JSON Store。
- 扩展 `UserContext`，让普通聊天和意图识别能拿到 `dataCoverage` 与 `memorySummary`。
- 给 MemoryInsight 生成完成链路增加解耦事件，用于异步提取长期记忆候选。
- 修复 `InsightFeedbackAggregator` 过期检查失效问题，但不把它当作本功能主链路的前置阻塞。

### 不做什么

- V1 不引入 Agent。
- V1 不做“更多”入口。
- V1 不把基础 CRUD 操作作为启动台主入口。
- V1 不做跨会话短期记忆缓存。
- V1 不让 AI 自由写入长期画像。
- V1 不把所有长期记忆都静默写入。
- V1 不把 MemoryInsight 生成链路强行接入 `AIUserContextMessageBuilder`。
- V1 不把长期记忆直接塞进后端 PromptManager 默认模板。

### 核心设计原则

| 原则 | 工程含义 |
|------|----------|
| 启动台服务于引导用户更好使用 HoloAI | 入口必须指向高价值 AI 工作流，而不是普通 CRUD |
| 短期记忆自动生成、可重算、会过期 | 不需要用户逐条确认，不作为事实源持久化 |
| 长期记忆必须有证据和用户控制 | 必须能查看、删除、纠正、撤销 |
| 敏感推断不能静默写入 | 涉及健康、财务压力、关系、人格、身份、心理状态等，必须确认 |
| 普通对话和洞察生成分开处理 | Chat 走 `AIUserContextMessageBuilder`，MemoryInsight 走独立上下文或后处理 |
| 先测价值，再扩复杂度 | Phase 1-3 先验证启动台和短期记忆价值，Phase 4-5 再谨慎上长期记忆 |

## 1. 信息架构

### 1.1 能力启动台入口

V1 固定展示 4 个核心入口，另加新人引导入口的条件展示。

| 入口 | 展示条件 | 目的 | 行为 |
|------|----------|------|------|
| 新人引导 | 新用户、数据少、未完成引导 | 教用户如何更好使用 HoloAI | 打开轻量引导面板 |
| 今日状态 | 常驻 | 让用户理解“今天 Holo 看到什么” | 生成今日状态摘要或空态引导 |
| 最近分析 | 有最近数据时优先，否则弱化展示 | 总结近期变化、风险、恢复迹象 | 基于 7/14/30 天短期快照分析 |
| 长期模式 | 有长期记忆或候选时优先，否则展示形成中 | 展示 Holo 已经学到的稳定偏好/模式 | 展示已确认记忆和待确认候选 |
| 规划目标 | 常驻 | 帮用户把状态转成行动计划 | 启动目标规划，不自动保存 |

删除以下常驻入口：

- 记一笔消费
- 创建任务
- 记录心情
- 习惯打卡
- 更多

这些基础能力放入新人引导和输入框 placeholder 示例中，不再占据启动台核心空间。

### 1.2 新人引导内容

新人引导不是教程长文，只展示 3-4 个实践示例：

| 基础能力 | 推荐表达 | Holo 应该做什么 |
|----------|----------|-----------------|
| 记账 | “午饭 38，和同事吃的” | 识别金额、类别、上下文，追问缺失字段 |
| 创建任务 | “明天下午提醒我交房租” | 创建任务，识别时间和提醒 |
| 记录心情 | “今天有点焦虑，可能是因为项目延期” | 记录心情和原因，不做人格判断 |
| 规划目标 | “我想这个月把作息调回来” | 进入目标规划，不直接生成长期记忆 |

隐藏规则：

- 用户完成任意 3 次有效 AI 行动后默认隐藏。
- 用户可在设置中重新打开。
- 数据覆盖度为 empty 时，可短暂重新展示新人引导入口。

## 2. 目标架构

```text
QuickActionBar
  -> HoloAICapabilityProvider
  -> Capability tap
      -> ChatViewModel.handleCapabilityTap(...)
      -> HoloMemorySnapshotBuilder.build(...)
          -> UserContextBuilder.build()
          -> DataCoverageEvaluator.evaluate(...)
          -> HoloMemorySummaryProvider.select(...)
      -> Deterministic workflow
          -> 今日状态 / 最近分析 / 长期模式 / 规划目标
      -> AIUserContextMessageBuilder.render(...)
      -> Provider.chat(...)

MemoryInsightService.generate(...)
  -> provider.generateMemoryInsight(type, contextJSON)
  -> save insight
  -> post .memoryInsightDidGenerate
  -> HoloLongTermMemoryCandidateObserver
      -> extract candidates off main path
      -> HoloMemoryPromotionPolicy.evaluate(...)
      -> HoloLongTermMemoryStore.upsertCandidate(...)
```

### 2.1 分层职责

| 层 | 组件 | 职责 |
|----|------|------|
| UI | `QuickActionBar`, `HoloAICapability` | 展示能力入口，不承载业务判断 |
| ViewModel | `ChatViewModel` | 响应入口点击，调度确定性工作流 |
| Context | `UserContextBuilder`, `DataCoverageEvaluator` | 聚合上下文和数据覆盖度 |
| Memory Facade | `HoloMemorySnapshotBuilder` | 单次流程内组装短期快照和相关长期记忆摘要 |
| Prompt Context | `AIUserContextMessageBuilder` | 将 `UserContext` 渲染为受控 system context |
| Long-term Store | `HoloLongTermMemoryStore` | 长期记忆 JSON 持久化、查询、更新、删除 |
| Promotion | `HoloMemoryPromotionPolicy` | 判断候选丢弃、观察、静默写入、要求确认 |
| Observer | `HoloLongTermMemoryCandidateObserver` | 消费事件，异步生成长期记忆候选 |

### 2.2 关键 ADR

#### ADR-1: 不引入 Agent

**决定：** V1 不引入 Agent。

**原因：**

- 现阶段需求是能力入口和记忆层工程化，不是多步自主任务执行。
- Agent 会引入任务规划、工具权限、失败恢复、状态回放等复杂度。
- HoloAI 当前更需要确定性、可解释、可测试的产品能力。

**后续触发条件：** 只有当用户需要跨应用、多步骤、长时间执行并需要恢复/重试时，再单独设计 Agent。

#### ADR-2: 短期记忆不做跨会话缓存

**决定：** V1 短期记忆只在单次点击/发送流程内生成和复用。

**原因：**

- 当前已有 `UserContextBuilder` 能聚合大量上下文。
- 过早做缓存会引入失效规则、通知订阅、数据一致性成本。
- 先通过耗时日志判断是否需要缓存。

**后续触发条件：** 单次上下文构建 P95 超过 500ms，且被用户频繁触发，再引入带通知失效的短期缓存。

#### ADR-3: 长期记忆先用 JSON Store

**决定：** V1 长期记忆使用 Application Support 下的 JSON 文件。

**原因：**

- 数据量小，结构仍会迭代。
- JSON 易审查、易导出、易回滚。
- 可复用 `InsightPreferenceProfileService` 的原子写入思路。

**后续触发条件：** 出现同步、复杂查询、全文搜索、多设备一致性要求，再迁移 Core Data 或 CloudKit。

#### ADR-4: 记忆注入走 UserContext，不改 PromptManager 主模板

**决定：** 普通聊天、意图识别、目标规划优先通过 `UserContext` 和 `AIUserContextMessageBuilder` 注入记忆摘要。

**原因：**

- 当前 Provider 已经在 Chat 和 Intent 路径使用 `AIUserContextMessageBuilder`。
- 记忆是用户上下文，不应该散落在 Prompt 模板里。
- 可以按 `purpose` 控制注入强度，降低上下文污染。

**例外：** MemoryInsight 生成不走这条链路，必须单独处理。

## 4. 短期记忆规则

### 4.1 获取来源

短期记忆来源于已有上下文聚合，不新建并行业务查询体系：

| 来源 | 现有组件 | 用法 |
|------|----------|------|
| 用户档案 | `HoloProfileService` | 只读取用户主动档案 |
| 今日状态 | `DailySenseStateBuilder` | 作为 today 状态信号之一 |
| 普通上下文 | `UserContextBuilder` | 复用已有任务、习惯、消费、目标摘要 |
| 周期洞察 | `MemoryInsightService` / 存储结果 | 只读最近已生成洞察，不额外触发生成 |
| 长期记忆 | `HoloLongTermMemoryStore` | 选择与当前目的相关的少量摘要 |
| 最近对话 | `ChatViewModel` 当前会话 | 只用于当前流程，不持久化 |

### 4.2 什么能成为短期记忆

可以：

- 最近任务数量、今日是否有未完成任务。
- 最近习惯完成状态。
- 最近消费趋势的事实摘要。
- 用户当前正在规划的目标。
- 最近对话中明确表达的短期意图。

不可以：

- 人格判断。
- 情绪诊断。
- 健康或财务压力推断。
- 单次行为得出的长期标签。
- 没有数据支撑的“你总是/你经常”。

### 4.3 是否需要用户确认

短期记忆不需要用户逐条确认，因为它满足三点：

- 不持久化为长期事实。
- 可重算。
- 会随窗口过期。

但回答底部必须能展示来源短标签，例如：

```text
基于：今日任务 · 最近 7 天习惯 · 已确认偏好
```

### 4.4 数据覆盖度

**Create:** `Holo/Holo APP/Holo/Holo/Services/AI/DataCoverageEvaluator.swift`

规则：

| level | 条件 | 展示策略 |
|-------|------|----------|
| rich | 至少 2 个核心数据源有近 7 天数据，或今日有明确任务/习惯/目标 | 正常总结 |
| partial | 至少 1 个数据源有近 30 天数据 | 明确说明参考范围有限 |
| empty | 无任务、无习惯、无记录、无目标、无近期对话意图 | 不生成假总结，转为引导 |

注意：

- 健康信号作为加分项，不作为 rich 的硬条件。
- 今日没有任务不等于 empty；如果有习惯、消费、目标或心情记录，仍可能是 partial/rich。
- 没有任何数据时，文案应引导用户建立上下文，而不是伪造洞察。

## 5. 长期记忆规则

### 5.1 候选来源

| 来源 | 能否生成候选 | 说明 |
|------|--------------|------|
| 用户明确表达 | 可以 | “以后提醒我用更短的计划” |
| 用户确认的目标 | 可以 | 保存后的长期目标 |
| 稳定反馈偏好 | 可以引用 | 来自 `InsightPreferenceProfile.isStable`，不重复写同源偏好 |
| MemoryInsight 模式 | 可以 | 需要 `patternType`、evidence 和重复证据 |
| HoloProfile | 可以作为证据 | 不自动覆盖用户主动档案 |
| 单次消费/任务/心情 | 不可以 | 只能作为短期上下文 |
| AI 猜测 | 不可以直接写入 | 只能进入候选或丢弃 |

### 5.2 晋升规则

**Create:** `Holo/Holo APP/Holo/Holo/Services/AI/HoloMemoryPromotionPolicy.swift`

输出：

```swift
enum HoloMemoryPromotionDecision: Equatable {
    case discard(reason: String)
    case observe(reason: String)
    case silentlyAccept(reason: String)
    case requireConfirmation(reason: String)
}
```

规则：

| 场景 | 决策 |
|------|------|
| 用户明确说“记住/以后都/我偏好”且非敏感 | `silentlyAccept` |
| 用户明确保存长期目标 | `silentlyAccept` 或 `requireConfirmation`，由 UI 行为决定 |
| 来自 MemoryInsight 的重复模式，有 2 个以上证据，非敏感 | `requireConfirmation` |
| 来自稳定反馈偏好，`isStable == true` | 不新建同源记忆，只生成引用摘要 |
| 单次事件 | `discard` |
| 情绪、健康、财务压力、关系、身份、人格推断 | `requireConfirmation` |
| 与 `HoloProfile` 冲突 | `requireConfirmation` |
| 证据不足 | `observe` 或 `discard` |

### 5.3 用户确认策略

| 类型 | 是否确认 | 用户可控 |
|------|----------|----------|
| 短期记忆 | 不确认 | 来源短标签 |
| 明确低风险偏好 | 可静默写入 | 轻提示 + 撤销 |
| 长期目标 | 保存动作即确认 | 可编辑/删除 |
| 重复模式 | 必须确认 | 记忆候选卡 |
| 敏感/高影响推断 | 必须确认 | 默认不写入 |
| HoloProfile 冲突 | 必须确认 | 用户选择覆盖/忽略 |

### 5.4 Store 规则

**Create:** `Holo/Holo APP/Holo/Holo/Services/AI/HoloLongTermMemoryStore.swift`

要求：

- 使用单一写入口，避免并发读改写冲突。
- JSON 写入使用临时文件 + 原子替换。
- JSON 损坏时回退为空 Store，并保留损坏文件副本用于调试。
- 删除只改变状态为 `archived` 或物理删除由设置项控制；V1 可先物理删除。
- 查询 Prompt 摘要时最多返回 5 条。
- MemoryInsight 链路最多返回 1-2 条稳定偏好摘要。

文件路径：

```text
Application Support/Holo/HoloLongTermMemories.json
```

## 6. Prompt 与上下文注入

### 6.1 普通聊天

**Modify:** `Holo/Holo APP/Holo/Holo/Services/AI/AIUserContextMessageBuilder.swift`

在 `.chat` 渲染中加入：

- 数据覆盖度：只描述可用/缺失来源，不输出内部结构。
- 短期摘要：最多 5 行。
- 长期记忆：最多 5 条相关摘要。
- 规则：长期记忆只能辅助理解，不得覆盖用户当前明确指令。

### 6.2 意图识别

`purpose: .intentRecognition` 只允许使用：

- 用户档案做消歧。
- 当前目标做消歧。
- 数据覆盖度做风险提示。

不得使用长期记忆覆盖用户当前输入。

### 6.3 目标规划

**Modify:** `Holo/Holo APP/Holo/Holo/Services/AI/GoalPlanningPromptBuilder.swift`

只在“生成目标草案/行动建议”阶段注入：

- 目标相关长期记忆。
- 用户执行偏好。
- 已确认长期模式。

不在追问阶段强注入，避免把用户带偏。

### 6.4 MemoryInsight

MemoryInsight 生成链路不走 `AIUserContextMessageBuilder`。

处理方式：

- V1 默认不注入长期记忆。
- 如确实需要，只在 `MemoryInsightContext` 或 `contextJSON` 中加入 1-2 条稳定偏好摘要。
- 生成成功后通过事件提取长期候选，而不是在生成 prompt 中做复杂记忆推理。

## 7. 文件级实施计划

### Phase 1: 能力启动台 MVP

## 8. 测试策略

### 8.1 单元测试优先级

| 优先级 | 测试 |
|--------|------|
| P0 | `DataCoverageEvaluatorTests` |
| P0 | `HoloMemoryPromotionPolicyTests` |
| P0 | `AIUserContextMessageBuilderStandaloneTests` |

| P1 | `TodayStateResolverTests` |
| P1 | `HoloAICapabilityProviderTests` |
| P1 | `HoloLongTermMemoryCandidateObserverTests` |
| P1 | `InsightFeedbackAggregatorTests` |

### 8.2 必测用例

Data coverage:

- 没有任何数据 -> empty。
- 今天没有任务，但有习惯记录 -> partial 或 rich，不能 empty。
- 只有 HealthKit 为空 -> 不影响其他来源判断。
- 多个核心来源有 7 天数据 -> rich。

Promotion policy:

- “以后提醒我用简短计划” -> silentlyAccept。
- “我可能是一个很懒的人” -> discard 或 requireConfirmation，不能静默写入。
- 单次“今天点了外卖” -> discard。
- MemoryInsight 重复模式且有 evidence -> requireConfirmation。
- 与 HoloProfile 冲突 -> requireConfirmation。

Prompt context:

- `.chat` 包含 memory summary。
- `.intentRecognition` 不允许长期记忆覆盖明确输入。
- 空 memory 不输出空字段标题。
- Memory summary 超过限制会被截断。

Store:

- JSON 首次不存在时返回空数组。
- 写入后可读取。
- 删除后不进入 prompt summary。
- 损坏 JSON 会备份并回退空 Store。

基础构建命令：

```bash
xcodebuild -project 'Holo/Holo APP/Holo/Holo.xcodeproj' -scheme Holo -configuration Debug -destination 'generic/platform=iOS' build
```

说明：

- 新增 XCTest 后，如 scheme 已配置测试，再补跑对应测试。
- 如果涉及真机展示，最终需要真机点开 HoloAI 对话框验证启动台可见和页面不崩溃。

## 9. 风险与回滚

| 风险 | 缓解 | 回滚 |
|------|------|------|
| 启动台入口不符合用户预期 | Phase 1 独立上线，入口由 Provider 控制 | 回滚 `QuickActionBar` 数据源 |
| UserContext 扩字段导致构造点编译失败 | 字段设为 Optional 并提供默认值 | 移除字段，保留模型文件 |
| Memory summary 污染意图识别 | `.intentRecognition` 单独保守模板 | 关闭 intent memory section |
| 长期记忆误写入 | PromotionPolicy 默认偏保守 | 禁用 Store 写入，只保留候选 |
| MemoryInsight 候选提取拖慢生成 | 通知解耦，后台处理 | 关闭 observer 订阅 |
| JSON 文件损坏 | 原子写入 + 损坏备份 | 回退空 Store |
| 后端 prompt 不同步 | V1 不改后端 prompt | 若改后端，必须发版 |

## 11. 实施顺序检查

按以下顺序实施，不能跳步：

- [ ] Phase 1 完成后再开始短期记忆，避免 UI 和记忆层一起变动导致回归难定位。
- [ ] Phase 2 完成后再做 Prompt 注入，确保注入内容有可靠 coverage。
- [ ] Phase 3 完成后再做长期记忆，避免长期记忆尚未可解释就进入 prompt。
- [ ] Phase 4 Store 和 Policy 测试通过后，再接 MemoryInsight 候选来源。
- [ ] Phase 5 候选来源稳定后，再做记忆管理 UI。
