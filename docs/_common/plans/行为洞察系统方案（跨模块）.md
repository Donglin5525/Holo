# 行为洞察系统 — 跨模块增强方案（v2.2 — MVP 版）

## Context

原方案 `docs/finance/plans/财务行为洞察系统方案.md` 只覆盖财务模块。东林要求扩展为**四模块融合**（财务/习惯/待办/想法），支持单模块分析 + 整体洞察报告。

v2.1 合并了第三轮审查意见。本版本（v2.2）聚焦 MVP —— **只做路径 B 周期洞察报告，暂不接入即时对话（路径 A）**，用户触发洞察后一次性输出报告，不允许多轮追问。核心变化：
- **不再新建并行引擎**，增强现有 MemoryInsightContextBuilder
- 修复 Codable 陷阱（元组、关联值枚举）
- 补全空数据降级策略 + 低样本门槛
- 明确 prompt 增强指向 + 数据/指令分离
- 统计口径明确定义，避免歧义
- **砍掉路径 A 接入**（InsightSummaryBridge / UserContext 扩展 / buildContextMessage 扩展），延后到后续迭代
- **想法模块改为文本驱动**：直接读取想法原文送 AI 分析情感/主题，不再仅依赖用户手动标记的 mood/tag 标签
- **洞察持久化 + 年度回顾**：每条洞察作为数据资产持久化，年度回顾聚合多月洞察生成跨时间元洞察

## 一、现有系统分析

### 系统现状：两条并行的上下文路径

```
路径 A — 即时交互（聊天/意图识别）:
  UserContextBuilder → UserContext（今日数据）
    → OpenAICompatibleProvider.buildContextMessage()
    → 文本注入 system message

路径 B — 周期洞察（记忆回放）:
  MemoryInsightContextBuilder → MemoryInsightContext（周期聚合 + 上期对比）
    → JSON 直传
    → AI 生成结构化洞察报告（MemoryInsightPayload）
```

两条路径的数据源完全重叠（都读四模块 Repository），但聚合粒度不同：
- 路径 A：只有"今日"快照
- 路径 B：周期范围 + 上期对比，但想法/待办的聚合维度很薄

### 各模块 Repository 聚合能力（已验证）

| 模块 | 公开方法数 | 聚合方法数 | 现状 |
|------|-----------|-----------|------|
| 财务 | 37 | 6 | 分类/趋势/预算/日历汇总，完善 |
| 习惯 | 32 | 11 | 完成率/连续打卡/排名/日粒度趋势，丰富 |
| 待办 | 42 | 5 | 仅全局 total/completed/overdue，**极弱** |
| 想法 | 20 | 0 | 纯 CRUD，**零聚合** |
| 预算 | 13 | 4 | 全局状态/分类预警，完善 |

### 路径 B 中各维度的数据丰富度

| 维度 | MemoryInsightContextBuilder 已实现 | 缺失 |
|------|-----------------------------------|------|
| 财务 | 消费/收入/分类TOP5/日消费/上期对比 | 预算表现、异常检测 |
| 习惯 | 活跃数/完成记录数/连续打卡/上期对比 | 完成率趋势、最佳/最差习惯 |
| 待办 | 完成数/逾期数/重要完成项 | **完成率、时间趋势、优先级分布** |
| 想法 | 总数/5条30字片段 | **心情分布、话题标签、活跃度模式** |
| 里程碑 | 已有 | — |

**结论**：核心短板在待办（缺时间序列）和想法（缺全部聚合）。补齐后路径 B 的数据已足够支撑跨模块洞察。

## 二、架构决策

### 为什么不新建 BehavioralInsightEngine

v1 设计了独立的 `BehavioralInsightEngine`（含 ModuleInsightProvider 协议、4 个 Provider、独立缓存），与 MemoryInsightContextBuilder 数据源完全重叠但互不共享：

| 问题 | 影响 |
|------|------|
| 两套系统读同样的 Repository | 重复计算、缓存不一致 |
| MemoryInsightService 用 Core Data 缓存，新引擎用 JSON 文件 | 两套失效逻辑 |
| MemoryInsightContextBuilder 是无状态 struct，新引擎是 @MainActor 单例 | 并发保护需协调 |
| 洞察摘要注入到 UserContext（路径 A），但路径 A 只用于即时交互 | 周期洞察场景用不到 |

### 最终架构：增强路径 B，单次生成洞察报告

MVP 阶段**不接入即时对话（路径 A）**。用户触发洞察 → AI 生成报告 → 展示即结束，不允许多轮追问。

```
┌─────────────────────────────────────┐
│ 路径 B — 周期洞察（增强，单次输出）   │
│                                     │
│ MemoryInsightContextBuilder         │
│   buildFinanceContext()   ← 增强    │
│   buildHabitContext()     ← 增强    │
│   buildTaskContext()      ← 增强    │
│   buildThoughtContext()   ← 增强    │
│        ↓                           │
│ CrossModuleCorrelator.detect()     │
│   （跨模块关联注入到 context）       │
│        ↓                           │
│ MemoryInsightService               │
│   generateInsight()                │
│   缓存: Core Data + snapshotHash    │
│        ↓                           │
│ AI → MemoryInsightPayload          │
│   （洞察报告，一次性展示）            │
└─────────────────────────────────────┘
```

路径 A（即时聊天）暂不接入洞察。后续迭代时再通过 Bridge 将周期洞察摘要注入聊天上下文。

### 改动范围

| 层 | 改动 | 新建文件 |
|-----|------|---------|
| Repository 层 | TodoRepository +2 方法，ThoughtRepository +4 方法 | — |
| 数据模型层 | MemoryInsightModels 补充字段 | — |
| 聚合层 | MemoryInsightContextBuilder 增强四个维度 | — |
| 跨模块关联 | CrossModuleCorrelator | 1 个文件 |
| Prompt | PromptManager 增强 memoryInsightGeneration prompt | — |

**总计新建 1 个文件，修改 5 个文件**（路径 A 接入延后到后续迭代）。

## 三、详细设计

### Phase 1: Repository 聚合补齐

#### 1.1 TodoRepository 新增 2 个方法

**文件**: `Models/TodoRepository.swift`

```swift
// 按日分组的完成趋势
struct DailyTaskCount: Codable {
    let date: Date
    let completedCount: Int
}

func getCompletionTrend(from start: Date, to end: Date) -> [DailyTaskCount]

// 指定时间范围内的完成统计
struct TaskPeriodStats: Codable {
    let completedInPeriod: Int         // completedAt 在 [start, end] 内的任务数
    let dueInPeriod: Int               // dueDate 在 [start, end] 内的任务数
    let overdueInPeriod: Int           // dueDate 在 [start, end] 内且未完成的任务数
    let completionRate: Double         // completedInPeriod / max(dueInPeriod, 1)
    let highPriorityCompletionRate: Double?  // nil 表示无高优任务
}
```

**统计口径**：

| 指标 | 时间字段 | 过滤条件 | 空值处理 |
|------|---------|---------|---------|
| `completedInPeriod` | `completedAt` | `completedAt >= start && completedAt <= end` | 返回 0 |
| `dueInPeriod` | `dueDate` | `dueDate >= start && dueDate <= end` | 无截止日期的任务不计入分母 |
| `overdueInPeriod` | `dueDate` | `dueDate < end && status != completed && dueDate < end` | 返回 0 |
| `completionRate` | — | `completedInPeriod / dueInPeriod` | `dueInPeriod=0` 时返回 0 |
| `highPriorityCompletionRate` | `dueDate` + `priority` | priority >= 2 且 dueDate 在范围内 | 无高优任务返回 nil |

**去重规则**：一个任务如果 `dueDate` 和 `completedAt` 都在范围内，在 `dueInPeriod` 中只计一次（按 dueDate 归属），在 `completedInPeriod` 中只计一次（按 completedAt 归属）。两者是不同维度的计数，不需要互相去重。

**实现思路**：
- `getCompletionTrend`: 查 `completedAt` 在 [start, end] 范围内的 TodoTask，按日分组 count
- `getCompletionStats`: 分别按 `dueDate` 和 `completedAt` 查询，统计各自维度

#### 1.2 ThoughtRepository 新增 4 个方法

**文件**: `Data/Repositories/ThoughtRepository.swift`

现有 `search(query:filters:)` 已支持 `ThoughtFilters.startDate`/`endDate` 过滤，新增方法复用 predicate 构建逻辑：

```swift
func getThoughtCount(from start: Date, to end: Date) -> Int

func getMoodDistribution(from start: Date, to end: Date) -> [String: Int]

func getTopTags(from start: Date, to end: Date, limit: Int) -> [ThoughtTag]

// 读取想法原文，供 AI 直接分析文本内容（情感、主题、模式）
// 每篇截取前 200 字符，最多返回 20 篇
func getThoughtTexts(from start: Date, to end: Date, limit: Int) -> [String]
```

**设计理由**：大多数用户不会主动给想法打 mood 标签或 tag，仅靠元数据统计会导致想法维度空洞。将想法原文送入 AI，让 AI 直接从文本中识别情感倾向、反复出现的主题、写作模式——这比依赖用户手动标记的标签可靠得多。mood 和 tag 作为补充信号（如果用户打了），不作为唯一数据源。

**实现思路**：
- 提取 `search()` 中的 predicate 构建逻辑为 `private func buildPredicates(startDate: endDate:) -> NSPredicate`
- `getThoughtCount`: 用 predicate fetch 后 `.count`
- `getMoodDistribution`: fetch 后按 `mood` 分组统计（`mood == nil` 的记录跳过）
- `getTopTags`: 通过 `ThoughtTag.thoughts` 关系反查，过滤 createdAt 在范围内的想法，按关联数排序
- `getThoughtTexts`: fetch 后按 `createdAt` 倒序，取前 `limit` 篇的 `content` 字段，每篇截断至 200 字符

**统计口径**：

| 指标 | 过滤条件 | 空值处理 |
|------|---------|---------|
| `getThoughtCount` | `createdAt >= start && createdAt <= end && !isSoftDeleted && !isArchived` | 返回 0 |
| `getMoodDistribution` | 同上，且 `mood != nil` | 返回空字典 `[:]` |
| `getTopTags` | 关联的 Thought 满足上述条件 | 返回空数组 `[]` |
| `getThoughtTexts` | 同上，`content != nil && content != ""` | 返回空数组 `[]` |

**空数据降级**：
- `getThoughtCount` 返回 0 → 上层判断跳过想法维度
- `getMoodDistribution` 返回空字典 → AI 从文本推断情感（见 Prompt 引导）
- `getTopTags` 返回空数组 → AI 从文本提取主题（见 Prompt 引导）
- `getThoughtTexts` 返回空数组 → 想法维度仅输出"周期内无想法记录"

## 上期对比口径定义（全局约定）

"上期"在所有维度和方法中遵循统一口径，避免各模块各自解释导致 AI 输出矛盾。

### 周期定义

| 周期类型 | 本期范围 | 上期范围 | 示例 |
|---------|---------|---------|------|
| daily | 当天 00:00 — 23:59 | 前一天 | 本期 5/5，上期 = 5/4 |
| weekly | 周一 00:00 — 周日 23:59 | 上一个完整自然周 | 本期 5/5-5/11，上期 = 4/28-5/4 |
| monthly | 1日 00:00 — 月末 23:59 | 上一个完整自然月 | 本期 5月，上期 = 4月 |
| annual | 1/1 00:00 — 12/31 23:59 | 上一个自然年 | 本期 2026，上期 = 2025 |

### 原则

- **统一用自然周期**（非滚动窗口），符合用户心智模型。"上周"= 上个周一到周日，不是"过去 7 天"
- **上期始终等长**：上一个完整自然周期，不会出现"半个周"或"不完整月"
- **跨月/跨年自动锚定**：5月1日（周四）触发周洞察 → 本期 = 4/28-5/4（跨月），上期 = 4/21-27，自然对齐
- **当前周期不完整时**：如果今天是周三，周洞察的"本期"仍是周一到周日（含未来 4 天），统计仅用已有的数据，上期用完整的上周数据做对比

### 与各 Phase 的关系

| Phase | 影响 |
|-------|------|
| Phase 1 (Repository) | `from:to:` 参数按此口径传入，Repository 不关心周期语义 |
| Phase 2 (Builder) | `buildXxxContext()` 的 `periodStart/End` 和 `previousStart/End` 按此计算 |
| Phase 3 (Correlator) | 变化幅度基于本期 vs 上期的指标差值 |
| Phase 5 (年度回顾) | 年度范围固定为 1/1-12/31，月度 digest 按自然月匹配 |

#### Decimal Codable 处理

`FinanceContext` 和 `BudgetPerformanceSummary` 中的 `Decimal` 类型不遵循 Swift 原生 `Codable` 协议。统一用 `Decimal` 的 `Double` 桥接编码：

```swift
// MemoryInsightModels.swift 中统一使用此扩展
extension Decimal {
    var codableValue: Double { (self as NSDecimalNumber).doubleValue }
    init(codableValue: Double) { self = Decimal(codableValue) }
}
```

所有涉及 `Decimal` 的 Codable 结构体，字段声明为 `Double`，通过计算属性或 `CodingKeys` 映射。如果精度敏感（财务金额），改用字符串传输 `"\(self)"` → `Decimal(string:)`，避免 `Double` 浮点误差。**MVP 阶段用 Double 桥接**，后续按需升级。

#### 2.6 Token 预算调整

当前预算：weekly 2000 tokens、monthly 3500 tokens。新增字段预估：

| 新增内容 | weekly 估算 tokens | monthly 估算 tokens |
|----------|-------------------|-------------------|
| 预算表现 | ~60 | ~60 |
| 异常描述 | ~40 | ~40 |
| 习惯完成率 + 排名 | ~80 | ~80 |
| 待办统计 | ~40 | ~40 |
| 待办趋势（weekly 7天 / monthly 14天上限） | ~80 | ~140 |
| 想法心情 + 标签（补充） | ~40 | ~40 |
| 想法文本内容（200字 × 10-20篇） | ~400 | ~600 |

**weekly 增幅约 740 tokens**，**monthly 增幅约 1040 tokens**。文本内容是增量大头，需做截断控制：

- weekly：最多 10 篇文本（~400 tokens）
- monthly：最多 20 篇文本（~600 tokens）
- 超出的按创建时间倒序截断（保留最新的）

**dailyCompletionTrend 粒度控制**：
- weekly：按日输出（~7 个点），直接保留
- monthly：聚合为周粒度或最多 14 个点，避免 30 个点撑爆 JSON

**`enforceTokenBudget()` 截断优先级**（如果超限）：
1. 基础指标（总额/完成率） — **不可截断**
2. 上期对比数据 — 保留
3. 跨模块关联 — 保留
4. 想法文本内容（textContents） — **优先减少篇数至 5 篇**
5. 趋势明细（dailyExpenses / dailyCompletionTrend） — **截断至 14 天**
6. 标签/心情（moodDistribution / topTags） — 保留（数据量小）
7. snippets（recentSnippets） — 最后保留 3 条

### Phase 3: 跨模块关联 — CrossModuleCorrelator

**新建 `Services/AI/CrossModuleCorrelator.swift`**

纯规则检测，接收增强后的 `MemoryInsightContext`，输出关联观察列表。

```swift
struct CrossModuleCorrelation: Codable {
    let modulePair: [InsightModule]   // 恰好 2 个元素
    let observation: String           // "本周习惯完成率下降，同时餐饮占比上升"
    let signalStrength: Double        // 0-1，基于变化幅度，不是统计置信度
    let summary: String               // 精简描述（30 字内）
}

enum InsightModule: String, Codable {
    case finance, habit, task, thought
}

struct CrossModuleCorrelator {
    /// 输入增强后的 MemoryInsightContext，检测已知关联模式
    static func detect(
        context: MemoryInsightContext,
        previousContext: MemoryInsightContext?
    ) -> [CrossModuleCorrelation]
}
```

**检测规则**：

| 关联 | 条件 | signalStrength | 最低样本量 |
|------|------|---------------|-----------|
| 习惯↔财务 | 习惯完成率降 >10% 且餐饮分类占比升 >5pp | min(1.0, 变化幅度 / 20%) | 活跃习惯 ≥ 3 个，餐饮笔数 ≥ 5 笔 |
| 任务↔财务 | 逾期率 >20% 且支出升 >15% | min(1.0, 超出阈值 / 20%) | 周期内任务 ≥ 5 个 |
| 想法↔习惯 | 仅当 `moodDistribution` 非空时生效：负面心情占比 >40% 且完成率降 >10% | min(1.0, 负面占比 / 60%) | 想法 ≥ 5 条，活跃习惯 ≥ 3 个，且 moodDistribution 非空 |
| 任务↔习惯 | 同向变化 >10%（正相关） | min(1.0, 同向幅度 / 30%) | 任务 ≥ 5 个，习惯 ≥ 3 个 |

> **想法↔习惯 规则的局限**：该规则依赖用户手动标记的 mood 标签，大多数用户不会标记。当 moodDistribution 为空时，此规则不触发——但 AI 仍可通过阅读 `textContents` 文本内容自行判断情感与习惯的并发现象（见 Phase 4 Prompt 增强）。

**空数据降级**：任一维度数据为空（totalCount=0 / activeHabitCount=0 / 想法数=0）或未达最低样本量 → 跳过涉及该维度的关联检测。

**语言护栏**：
- `observation` 只描述并发现象，不得使用因果词汇（"导致/因为/说明/证明/所以"）
- Prompt 中强制要求："跨模块关联只能表达为并发现象，不得推断原因"
- 输出示例："本周习惯完成率下降，同时餐饮占比上升" ✓ / "习惯没坚持导致餐饮花多了" ✗

**集成方式**：在 `MemoryInsightContextBuilder.build()` 末尾调用 `CrossModuleCorrelator.detect()`，将结果追加到 `MemoryInsightContext` 的新字段 `crossModuleCorrelations: [CrossModuleCorrelation]`。

这样跨模块关联数据**随周期洞察一起持久化**，由 MemoryInsightService 的 snapshotHash 机制统一管理缓存失效（`computeHash()` 对整个 `MemoryInsightContext` 做 JSON encode + SHA256，新增字段自动纳入 hash 计算），不需要独立的缓存层。

### Phase 5: 洞察持久化与年度回顾

每条洞察报告作为用户的**数据资产**持久化保存。在此基础上，年度回顾场景将多份月度洞察聚合，生成跨时间的"元洞察"。

#### 5.1 持久化基础（已有）

`MemoryInsightPayload` 已通过 `MemoryInsightService` → `MemoryInsightRepository` 持久化到 Core Data，每条包含：

| 字段 | 说明 |
|------|------|
| `periodType` | daily / weekly / monthly |
| `periodStart` / `periodEnd` | 覆盖的时间范围 |
| `summary` | AI 生成的摘要文本 |
| `cards` | 各维度卡片内容 |

| `snapshotHash` | 数据指纹，同周期数据变化后触发重新生成 |

**结论**：持久化底座已就绪，年度回顾只需新增聚合逻辑 + Prompt。

#### 5.2 年度回顾数据模型

在 `MemoryInsightModels.swift` 中新增：

```swift
// 月度洞察的精简摘要，供年度回顾聚合
struct MonthlyInsightDigest: Codable {
    let periodStart: Date          // 月起始
    let periodEnd: Date            // 月结束
    let summary: String            // 当月洞察摘要（~200 字）
    let keyFindings: [String]      // 当月关键发现（2-3 条）
    let moduleSnapshots: [ModuleSnapshot]  // 各模块核心指标快照
}

struct ModuleSnapshot: Codable {
    let module: InsightModule
    let headline: String           // "餐饮支出 ¥3,200，占 42%"
}
```

#### 5.3 buildAnnualContext() 新增

在 `MemoryInsightContextBuilder` 中新增 `buildAnnualContext()` 方法：

```swift
func buildAnnualContext(year: Int) -> MemoryInsightContext {
    // 1. 从 MemoryInsightRepository 读取该年所有月度洞察（periodType = .monthly）
    let monthlyInsights = insightRepo.fetchMonthlyInsights(for: year)
    
    // 2. 对每条月度洞察提取摘要 + 关键指标快照
    let digests: [MonthlyInsightDigest] = monthlyInsights.compactMap { insight in
        guard let payload = insight.payload else { return nil }
        return MonthlyInsightDigest(
            periodStart: insight.periodStart,
            periodEnd: insight.periodEnd,
            summary: payload.summary,
            keyFindings: extractKeyFindings(from: payload),
            moduleSnapshots: extractModuleSnapshots(from: payload)
        )
    }
    
    // 3. 同时取该年原始数据做当年汇总（复用现有 build 方法的统计逻辑）
    let annualFinance = buildFinanceContext(range: yearRange)
    let annualHabit = buildHabitContext(range: yearRange)
    // ...
    
    // 4. 组装 context，monthlyInsightDigests 是核心新增字段
    return MemoryInsightContext(
        // ... 年度汇总数据 ...
        monthlyInsightDigests: digests,
        crossModuleCorrelations: CrossModuleCorrelator.detect(...)
    )
}
```

**关键设计决策**：
- 年度回顾**不重新拉全年原始明细数据**（token 会爆炸），而是聚合**月度洞察摘要**。每条月度摘要 ~200 字，12 个月 ~2400 字（~1200 tokens），加上年度汇总指标，总量可控。
- 如果某月没有洞察记录（用户未触发），对应位置留空，不编造。
- `extractKeyFindings` / `extractModuleSnapshots` 从 `MemoryInsightPayload.cards` 中提取，是纯文本解析，不调用 AI。
