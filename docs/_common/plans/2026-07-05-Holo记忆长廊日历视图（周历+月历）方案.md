# Holo 记忆长廊日历视图（周历 + 月历）方案

> 日期：2026-07-05
> 模块：记忆长廊（MemoryGallery）
> 状态：东林已 review（A 路线：并存 + 延后删），按 Codex 审查意见修订 → 转 writing-plans
> Mockup：`docs/superpowers/mockups/2026-07-01-calendar-mockup-v2.html`

---

## 一、背景与目标

生活星图（`MemoryConstellationCard`）作为记忆画廊「洞察 Tab」首卡，实际使用价值低。用 **周历 + 月历** 替换它，把记账/习惯/待办/想法聚合到一个统一日历视图，像市面日程软件平铺，让用户系统看到每天/最近做了什么。

**核心诉求**
- 一个视图看清每天做了什么
- 多模块数据（想法/习惯/待办/记账）平铺到时间维度
- 像日程软件一样可读、信息密集

**非目标**
- 不做新建/编辑入口（日历是只读回看）
- 不做 AI 洞察生成（AI 洞察保留为下钻补充）

## 二、现状结论（代码探索）

### 2.1 生活星图
- 位置：`MemoryGalleryView` 洞察 Tab 首卡，非独立页面
- 实现：SwiftUI `Canvas` 绘制 5 模块（习惯/财务/任务/思考/健康）星座连线
- 数据：寄生 `MemoryGalleryViewModel`，不独立取数（4 模块从 `MemoryItem` 聚合 + 健康从 `HealthRepository`）
- 文件：`MemoryConstellationCard.swift`(365) + `MemoryConstellationModels.swift`(163) + ViewModel 内 ~290 行

### 2.2 各模块日历聚合就绪度

| 模块 | 聚合字段 | day 维度现成查询 | hour 维度 |
|------|---------|----------------|----------|
| 记账 `Transaction` | `date`（非空） | `getTransactionsForDay` / `getDailySummaries` ✓ | 无现成，字段支持 |
| 习惯 `HabitRecord` | `date`（非空，可多次/天） | `getRecords(in:)`（按单习惯）✓ | 无现成，字段支持 |
| 待办 `TodoTask` | `completedAt` / `dueDate` / `plannedDate`（皆可选） | `getCompletionTrend` ✓ | 无现成 |
| 想法 `Thought` | `createdAt`（非空） | `getThoughtCountByDay` ✓ | 无现成，字段支持 |
| 观点 `Topic` | 无日历语义 | 无 | 不直接聚合 |

**关键**：所有时间字段带时分秒，hour 维度需新写 predicate，**不改 schema**。

## 三、设计决策（已与东林通过 mockup 确认）

1. **定位（修订·Q1）**：日历成为记忆画廊新首屏，**旧 `洞察 / 明细` 入口保留并存**（不直接换 Tab、不下沉 AI 回放/深度分析/时间线等既有能力）。进入画廊默认到日历，旧入口作为次级 Tab 仍可达。生活星图 P1 **隐藏不渲染、代码与测试保留**，等 P1A/P1B 闭环且东林确认后于 P1C 删除。
2. **周历**：默认 **列表视图**（7 天 × 横向事件流，信息全、字清晰，12px），保留 **网格视图**（7 列 × 24h 时间轴）作为切换。顶部「网格 / 列表」切换。
3. **月历**：31 天热力色块（复用 `MemoryHeatmapView` 5 档暖色阶 `#F5F2ED→#FF8C66`）+ 底部 4px 模块色条；选中当天下方平铺全部事件，按模块分组。
4. **模块配色**：
   - 记账 橙 `#F46D38`
   - 习惯 绿 `#22C55E`
   - 待办 蓝紫 `#6366F1`（**改动**：项目里待办原 = 橙，日历分色与记账撞，改蓝紫，取自记账分类色「购物」）
   - 想法 粉 `#EC4899`
   - 健康 蓝 `#3B82F6`（P3 月历可选图层）
5. **待办时间维度**：默认 `completedAt`（完成时间）；P2 支持切换 `dueDate` / `plannedDate`。
6. **观点 Topic**：不直接进日历。经其关联的 `Thought.createdAt` 间接体现。
7. **健康数据（修订·Q3）**：P1 **保底入口**——月历顶部「健康状态 chip」（未授权 / 已连接无数据 / 已连接·本周步数/睡眠摘要），复用从星图迁移出的 `loadConstellationHealthState`。完整 HealthKit 图层放 P3，不混进 5 模块色条。

8. **事件详情（修订·Q2·新增决策）**：新建 `CalendarEventDetailSheet`（只读），不复用 `MemoryDetailView`（其绑定 `MemoryItem` 单模型，跨 4 模块适配成本高）。`originID` 用 `NSManagedObjectID`（跨线程安全、可回查实体跳原模块），UI 层不暴露。各模块字段契约见 §4.1。

9. **失败可见性（修订·Q5·新增决策）**：单模块加载失败**不静默为空**。`CalendarEventProvider` 返回 `CalendarEventsResult { events, moduleStates }`，UI 弱提示「部分数据暂未载入」+ retry。

## 四、架构

### 4.1 新增类型

**`CalendarEvent`（统一事件模型）**
```swift
struct CalendarEvent: Identifiable {
    let id: UUID
    let module: CalendarModule        // .finance/.habit/.todo/.thought/.health
    let date: Date                    // 带时刻，定位锚点
    let title: String                 // 列表/详情主标题
    let detail: String?               // 金额/数值/摘要
    let originID: NSManagedObjectID   // 跳转详情用（跨线程安全）
}

enum CalendarModule: CaseIterable {
    case finance, habit, todo, thought, health
    var color: Color { ... }          // 上述 5 色
    var iconName: String { ... }      // SF Symbol（新增前先验证存在）
}
```

`detail` 各模块字段契约（`CalendarEventDetailSheet` 据此渲染，对应 Q2）：
- 记账：金额（`compactCurrency`）/ 支出或收入 / 分类名 / 账户名 / 备注
- 习惯：习惯名 / 记录值+单位 / 完成状态 / 备注
- 待办：标题 / 完成时间 / 截止时间 / 计划日期 / 优先级 / 描述
- 想法：正文摘要（前 N 字）/ 心情 / 标签 / 创建时间

**`CalendarEventDetailSheet`（新建·只读详情）**：按 `module` 渲染上述字段；底部「在 X 模块打开」按钮用 `originID` 回查实体跳原模块编辑页。

**`CalendarEventsResult` + `CalendarModuleLoadState`（修订·Q5）**
```swift
struct CalendarEventsResult {
    let events: [CalendarEvent]                              // 按 date 排序
    let moduleStates: [CalendarModule: CalendarModuleLoadState]
}
enum CalendarModuleLoadState {
    case loaded, empty, failed(message: String)
}
```

**`CalendarEventProvider`（统一聚合层·修订）**
- 职责：按区间从 4 个 Repository fetch，归一化为 `CalendarEventsResult`
- 无状态，由 View `.task` 驱动取数（`init()` 零 I/O）
- 各模块单独 do-catch，失败记 `Logger` 并在 `moduleStates` 标 `.failed`，**不静默丢模块**
- hour 定位：直接用 `event.date` 时刻

```swift
struct CalendarEventProvider {
    func fetchEvents(in range: DateInterval) async -> CalendarEventsResult
    func fetchDaySummary(_ date: Date) async -> CalendarEventsResult   // 月历当天详情
}
```

│   ├── MonthCell.swift               // 单格（色深 + 底部色条）
│   ├── DayDetailCard.swift           // 选中当天平铺（按模块分组）

### 4.3 数据流

```
CalendarRootView
  └─ .task → CalendarEventProvider.fetchEvents(当前周/月区间 DateInterval)
       ├─ FinanceRepository.fetchTransactions(in:)   ← 新增事件级（[start,end)）
       ├─ HabitRepository.fetchAllRecords(in:)       ← 新增（不带 habitId，返回 [(record, habit)]）
       ├─ TodoRepository.fetchCompletedTasks(in:)    ← 新增事件级（按 completedAt）
       └─ ThoughtRepository.fetchThoughts(in:)       ← 新增事件级（现有 fetchAll 无 range）
     区间统一 [start, end)，由 CalendarRangeBuilder 生成（周/月/日）
     → 归一化 CalendarEventsResult（events + moduleStates）
     → WeeklyListView / MonthlyCalendarView 渲染

月历点格 → fetchDaySummary(date) → DayDetailCard 按模块分组
周历点事件块 / 月历点事件行 → CalendarEventDetailSheet（新建，按 module 渲染字段契约）
失败模块 → 顶部「部分数据暂未载入」横条 + 详情内 retry
```

## 七、错误处理

- `CalendarEventProvider` 每个模块单独 do-catch，失败记 `Logger`（模块名+区间），在 `moduleStates` 标 `.failed(message)`，**不静默丢模块**（修订·Q5：原"返回空数组"会让待办仓储失败伪装成"今天没待办"，破坏事实回看信任）
- UI 层弱提示：顶部「部分数据暂未载入」横条（仅当存在 `.failed`）；日期详情内失败模块显示轻提示 + retry；全模块成功且无数据才显示「这段时间没有记录」空态
- hour 定位用 `event.date`；兜底放当天 `00:00`

## 十、风险与边界

1. **Repository `init()` 零 I/O**：`CalendarEventProvider` 不在 init 取数，由 View `.task` 驱动（CLAUDE.md 约定）。
2. **hour predicate 性能**：大区间（整月）跨 4 模块 fetch 需关注。若慢，给 `Transaction.date` / `HabitRecord.date` / `Thought.createdAt` 加 `isIndexed`（`CoreDataStack+*Entities.swift`），不改 schema。
3. **星图删除的连带（修订·Q6 + Codex 11.6）**：P1 隐藏不删，P1C 删前必须先拆解耦合——`loadConstellationHealthState`（健康真实入口，P1B 迁移到 HealthStatusChip）、`featuredNarrativeNodes`（高光/里程碑选择，评估迁移到 AI 回放/日期详情）、AI 洞察刷新（保留原入口）。原方案"安全删"过于乐观，已改为拆解清单（§六）。
4. **数据精度**：事后补录的记账/想法时刻可能不准 —— 列表视图显式展示时间戳让用户自知；网格视图按时刻定位、空时段留白不强填。
5. **不改 schema**：所有改动基于现有 `date` / `createdAt` / `completedAt` 字段，无 Core Data 迁移。
6. **遵循编码约定**：禁止 `!`、用 `Logger` 不用 `print`、`try-catch`、ScrollView 隐藏滚动条、日期用 `DateFormatter` + `zh_CN`、SF Symbol 新增前先 `NSImage(systemSymbolName:)` 验证。

## 十一、第一轮 Codex 对抗性审查意见

> 审查结论：**Conditional Go，不建议直接转 writing-plans。**
>
> 产品方向成立：周历 + 月历确实比生活星图更接近“每天/最近做了什么”的高频回看需求。但当前方案把“替换主信息架构”“删除生活星图”“新增跨模块聚合层”“新增事件详情弹层”合并进同一个 P1，风险过集中。建议先收缩 P1 范围，补齐数据契约和详情交互，再进入实施计划。

### 11.1 P0：P1 范围过大，直接替换主信息架构风险偏高

方案第 49 行要求把顶部 Tab 从 `洞察 / 明细` 改为 `周历 / 月历`，并把 AI 洞察、热力图、时间线下沉为“选中日期详情”的补充。这不是一个局部组件替换，而是记忆长廊主信息架构重构。

当前实现里：
- `洞察 Tab` 承载生活星图、Daily Sense、深度分析、AI 回放和精选故事。
- `明细 Tab` 承载热力图、选中日期预览、模块筛选和完整时间线。

如果 P1 直接替换为 `周历 / 月历`，用户可见风险是：日历变清楚了，但 AI 回放、深度分析、时间线这些既有能力会突然变难找，甚至在第一版里被事实下线。

**建议修订：**
- P1 不直接删除旧 `洞察 / 明细` 结构。
- 先做 `日历主视图 + 原 AI/时间线保留入口`。
- 等日历能稳定承载主回看后，再在 P2 或 P3 做信息架构切换和旧结构删除。

### 11.2 P0：事件详情契约没有成立

方案新增：

```swift
struct CalendarEvent: Identifiable {
    let id: UUID
    let module: CalendarModule
    let date: Date
    let title: String
    let detail: String?
    let originID: NSManagedObjectID
}
```

并在数据流里写“周历点事件块 / 月历点事件行 → 轻量事件详情弹层”。但当前记忆长廊详情 sheet 是 `MemoryDetailView(memory: MemoryItem)`，不是基于 `NSManagedObjectID` 的统一详情系统。

这里存在两个未决选择：

1. **把 CalendarEvent 映射回 MemoryItem**  
   好处是复用当前 `MemoryDetailView`；坏处是 `CalendarEvent` 需要保留足够字段，且健康事件无法自然适配。

2. **新建 CalendarEventDetailSheet**  
   好处是适合跨模块只读回看；坏处是 P1 必须设计 4 个模块的详情呈现规则、空字段、跳转行为。

当前方案没有选定其中一个，因此“日历只读回看”和“事件可点击详情”之间有断层。最坏结果是用户看到事件卡片，但点进去只能弹出很浅的摘要，不能确认来源。

**建议修订：**
- P1 明确事件详情形态。
- 如果选择 `CalendarEventDetailSheet`，需补充每个模块字段：
  - 记账：金额、收支类型、分类、账户、备注。
  - 习惯：习惯名、记录值、单位、备注、完成状态。
  - 待办：标题、完成时间、截止时间、计划日期、优先级、描述。
  - 想法：正文摘要、心情、标签、创建时间。
- `originID` 是否真的需要暴露到 UI 层也要重新评估；只读日历可能用 `sourceId + module` 更贴近现有模型。

### 11.3 P1：数据就绪度被高估

方案表格把多个模块标成“day 维度现成查询 ✓”，但这些查询大多只能支持统计，不足以支撑“事件级日历”。

具体问题：

- 待办 `getCompletionTrend(from:to:)` 返回的是 `[DailyTaskCount]`，只有每天完成数，不返回任务实体，不能渲染事件列表。
- 该方法使用 `completedAt <= end`，而其他周期统计多用开区间 `< end`；日历按天/月取数时容易出现边界重复。
- 想法 `ThoughtRepository.fetchAll` 当前没有 range 参数，只支持全量/分页/排序。
- 习惯 `getRecords(for:in:)` 是按单个 habit 查询，日历需要“不带 habitId 的全量区间记录 + habit 映射”。

**建议修订：**

P1 明确新增 4 个事件级 fetch 契约，而不是复用统计 API：

```swift
FinanceRepository.fetchTransactions(in range: DateInterval) async throws -> [Transaction]
HabitRepository.fetchAllRecords(in range: DateInterval) throws -> [(record: HabitRecord, habit: Habit)]
TodoRepository.fetchCompletedTasks(in range: DateInterval) throws -> [TodoTask]
ThoughtRepository.fetchThoughts(in range: DateInterval) throws -> [Thought]
```

同时统一时间边界：
- 全部使用 `[start, end)`。
- 周/月/日都由同一个 range builder 生成。
- 测试覆盖“月末 23:59”“次日 00:00”“跨周起止”。

### 11.4 P1：健康从一等入口降到 P3，和既有产品原则冲突

方案把健康数据放到 P3，并写“HealthKit 步数/睡眠作为月历可选图层，不混进 5 模块色条”。这在工程上可以降低 P1 复杂度，但在产品叙事上有倒退。

记忆长廊之前已经把健康作为生活星图第五维，且原则是：即使后端/Agent 链路没有完全打通，也要先给健康保留一等入口。现在直接移除生活星图，同时 P1 不给健康任何位置，用户会感到健康维度消失。

**建议修订：**
- P1 可以不做完整 HealthKit 日历图层。
- 但 P1 应保留轻量健康入口：
  - 月历/周历顶部健康状态 chip；
  - 或日期详情里的“健康占位卡”；
  - 或“本周步数/睡眠摘要入口”，状态包含未授权、已授权无数据、已连接。
- P3 再做完整步数/睡眠图层。

### 11.5 P2：错误处理会让数据悄悄消失

方案写 `CalendarEventProvider` 每个模块单独 do-catch，失败记 Logger、返回空数组，UI 空区间显示“这段时间没有记录”。

这个策略对聚合日历有明显误导：如果待办仓储失败，用户看到的是“今天没有待办”，而不是“待办暂未载入”。对日历这种事实回看页面，静默丢模块会破坏信任。

**建议修订：**

`CalendarEventProvider` 不应只返回 `[CalendarEvent]`，而应返回带模块状态的结果：

```swift
struct CalendarEventsResult {
    let events: [CalendarEvent]
    let moduleStates: [CalendarModule: CalendarModuleLoadState]
}

enum CalendarModuleLoadState {
    case loaded
    case empty
    case failed(message: String)
}
```

UI 层可弱提示：
- 顶部显示“部分数据暂未载入”；
- 日期详情内对失败模块显示轻提示；
- 保留 retry，而不是把失败伪装为空。

### 11.6 P2：删除生活星图相关代码前，需要先确认哪些能力不是星图专属

方案认为 `constellationSnippets` 仅被 `MemoryConstellationCard` 消费，安全删。但当前 ViewModel 中星图相关代码和 AI 洞察、健康状态、精选故事存在语义耦合：

- `refresh()` 会刷新本地数据、加载洞察，并在配额允许时触发 AI 洞察刷新。
- `constellationSummary` 直接读取 `currentInsight`。
- `constellationSignals` 读取当前洞察周期内的聚合摘要。
- `loadConstellationHealthState()` 是当前记忆长廊健康状态的真实 HealthKit 入口之一。
- `featuredNarrativeNodes()` 虽然服务星图片段，但它表达的是高光/里程碑故事选择逻辑，不一定应该随星图删除。

**建议修订：**
- 不要在 P1 删除所有星图相关 ViewModel 逻辑。
- 先拆分：
  - 纯星图 UI：可删除。
  - 健康状态聚合：迁移到日历健康入口或独立 service。
  - featured narrative 选择：评估是否迁移到日期详情/AI 回放入口。
  - AI 洞察刷新：保留原入口，不挂在日历刷新上。

### 11.7 P2：P1 同时包含“周历列表 + 月历 + 删除星图 + Tab 重组”，验收边界不清

当前 P1 的验收目标太多，容易出现“都做了一点，但没有一条体验闭环足够稳”的情况。

**建议把 P1 拆成更可验收的版本：**

**P1A：数据与周历列表闭环**
- 新增 `CalendarEvent`。
- 新增事件级 fetch 契约。
- 周历列表可显示 7 天事件。
- 事件可打开只读详情。
- 旧洞察/明细入口保留。

**P1B：月历闭环**
- 月历色块。
- 选中日期详情。
- 模块色条。
- 空态/失败态。

**P1C：入口重组候选**
- 基于 P1A/P1B 体验再决定是否替换 `洞察 / 明细`。
- 不在同一个 PR 里删除生活星图和所有测试。

### 11.8 建议给 GLM 第二轮重点审查的问题

请 GLM 重点判断下面几个决策，而不是泛泛再评一遍 UI：

1. P1 是否应该直接把顶部 Tab 改成 `周历 / 月历`，还是先保留旧 `洞察 / 明细` 并新增日历入口？
2. `CalendarEvent` 的详情应该复用 `MemoryItem + MemoryDetailView`，还是新建 `CalendarEventDetailSheet`？
3. 健康是否允许 P1 完全缺席？如果不允许，最低成本的健康保底入口应该是什么？
4. 事件级 fetch 契约是否应该独立于现有统计 API？是否全部统一为 `[start, end)`？
5. 单模块加载失败时，UI 是否可以静默显示为空，还是必须暴露“部分数据未载入”？
6. 生活星图删除应在 P1 做，还是等日历闭环被验证后再做？

### 11.9 修订后的推荐结论

推荐结论不是否定日历方案，而是把它从“大重构”改成“可验证替换”：

- **Go**：周历 + 月历作为记忆长廊新主线方向。
- **No-Go**：当前 P1 直接删除生活星图、替换 Tab、下沉旧洞察/时间线。
- **Conditional Go**：先补数据契约、详情契约、健康保底、失败态，再拆成 P1A/P1B/P1C 实施。
