# HoloAI 领域记忆与跨域融合升级方案

> 日期：2026-07-14
>
> 状态：本地实现与自动化验收已完成；默认 shadow；等待后端发版与真机灰度验收
>
> 范围：领域记忆、跨域记忆、统一召回、静默更新、用户控制、开发者验证
> 前置基础：现有严格语义长期记忆、情景记忆、证据引用、记忆回执和 Agent/洞察召回链路

## 1. 产品结论

Holo 的下一阶段记忆能力，不是继续增加一批通用 AI 摘要，而是建立一套“各模块分别理解、跨模块共同理解、按问题精准使用”的记忆体系。

最终用户应感受到：

> Holo 不仅记得我记录过什么，还能理解我在财务、健康、习惯、目标和观点上的阶段变化，并在合适的问题里把这些理解连接起来。

系统采用“稳定结论持久化、即时状态动态组合”的混合模式：

- 财务、观点、健康、习惯、任务、目标等模块分别形成领域记忆。
- 两个及以上领域共同支持的稳定结论，可以形成阶段性跨域记忆。
- “我最近状态如何”这类即时判断默认动态组合，不自动永久保存。
- 同一跨域结论连续多个周期成立，或经用户确认后，才允许持久化。
- 明确数字问题仍查询原始明细，记忆不能替代事实计算。

## 2. 已确认的产品决策

1. 采用“独立模块萃取器 + 统一记忆底座 + 跨域融合层”。
2. 模块之间逻辑隔离，但不为每个模块复制一套 Store、状态机和用户反馈系统。
3. 用户看到结论、来源和控制操作；内部置信度、新鲜度、相关度和排序分数不向普通用户展示。
4. 跨模块可以发现关联和冲突，但不得在证据不足时写成确定因果。
5. Profile 是用户明确填写或确认的高优先级事实，不由后台 AI 静默改写。
6. 用户当前明确输入优先于 Profile 和任何历史记忆。
7. “自动形成记忆”和“记忆辅助回答”分开控制。
8. 静默萃取失败不得影响记账、任务、习惯等主流程。
9. 用户选择“忘记”后，系统不得换个说法重新生成同一结论。
10. Debug 必须提供“AI 记忆实验室”，显性展示从数据变化到回答使用的真实链路。
11. 领域记忆只能解释本领域数据；涉及目标、Profile 或其他领域的关系，必须进入跨域层。
12. 所有记忆读取统一经过 `HoloMemoryAccessPolicy`，任何 Chat、洞察、规划、Agent 或 Tool 都不得旁路“记忆辅助回答”开关。
13. 普通聊天不在前台关键路径实时执行领域萃取或跨域 LLM 融合；使用最近有效记忆回答，并异步刷新。
14. “清除全部 AI 记忆”默认建立新的学习起点，旧明细不会在下一次后台任务中立即重建旧记忆。
15. 自动记忆默认保持现有用户选择，不因升级静默开启；首次开启需要明确说明会在使用期间后台形成记忆。

## 3. 当前实现基础与真实缺口

> 本节记录 2026-07-14 实施前基线，用于解释本次升级解决了什么；当前实现状态见第 25 节。

### 3.1 已有基础

当前工程已经具备：

- 严格语义格式的 `HoloLongTermMemory`。
- `semanticType`、`useScopes`、`aiUseSummary` 和 `prohibitedInferences`。
- 情景记忆 `HoloEpisodicMemory` 及过期、拒绝和抑制逻辑。
- 稳定记忆身份和跨周期证据合并。
- `HoloMemorySummaryProvider` 与相关性召回。
- 普通聊天、洞察和 Agent 的部分记忆接入。
- 记忆写入回执和使用回执。
- HoloAI 数据处理授权总闸。

### 3.2 主要缺口

1. `HoloObservationPackage` 和 `HoloMemoryObserverService` 目前实际只覆盖习惯和目标。
2. 财务、观点、健康、任务和对话没有稳定的领域记忆萃取器。
3. 现有记忆以长期/情景分类，没有形成统一的“领域/跨域”语义底座。
4. 当前召回把置信度、证据数和更新时间混合排序，没有独立表达“成立程度”和“当前相关性”。
5. 没有跨域融合服务，也没有共同目标、主题和时间阶段索引。
6. “情景记忆观察”只在 Debug AI 设置中可见，且当前只有手动触发入口。
7. Release 设置只有“长期记忆”和“记忆辅助对话”，尚不能完整管理静默模块萃取。
8. 长期记忆和情景记忆仍是本机 JSON Store，没有跨设备版本与删除冲突策略。
9. 当前 30 天 suppression 不足以实现用户明确要求的永久“忘记”。
10. 当前 Agent 启动和 Memory Tool 可以直接读取 Store，没有统一检查“记忆辅助回答”开关，存在消费端旁路。
11. 当前 `HoloEpisodicMemoryStore.save()` 只调用 `replaceItemAt`，目标文件首次不存在时没有 move 分支，首写可能无法形成正式 Store。
12. 当前长期/情景 Store 的 upsert、confirm、reject 都是分离的 load-modify-save，单次写入有 barrier，但完整事务不是原子的，并发更新可能互相覆盖。
13. 当前 Observer 的 `runID` 基于时间戳，同一份源数据在多设备或重复调度时不能幂等去重。
14. 当前数字查询和相关性判断主要依赖关键词，不足以承担本方案中的完整 Query Router 与跨域语义召回。
15. 当前 `HoloMemoryTool` 会把 suppression 的 `originalSummary` 转成 Agent evidence；这与“拒绝/忘记后不再被 AI 使用”的产品承诺直接冲突。

## 3.3 对抗性审查结论

方案方向有效，但原版存在五个会导致产品承诺落空的结构性风险，进入实施计划前必须按正文修订后的规则处理：

1. **领域层不够纯**：财务“消费与目标关系”等表述已经跨域，会造成同一结论在领域层和融合层重复生成。
2. **开关不能只在 UI 或 Prompt 注入处判断**：现有 Agent/Tool 可直接读 Store，必须在统一访问层强制授权。
3. **忘记不等于删除一条记录**：原始明细仍在时，后台会重新学习；现有 Tool 甚至会把 suppression 原文返回给 Agent。必须增加语义墓碑和全局学习起点，并禁止 suppression 正文进入模型。
4. **统一 Repository 不能只是协议门面**：现有 Store 已暴露首写和非原子事务风险，目标实现必须拥有原子 mutation、幂等 observation 和恢复语义。
5. **“按需刷新”不能阻塞普通聊天**：否则开放式问题会变成多次 LLM 串行调用，延迟、费用和失败率不可控。

审查后新增的架构决策：

| ADR | 决策 | 主要取舍 |
|---|---|---|
| ADR-001 | 领域记忆保持单域纯度，目标/Profile/其他领域只作为 anchor，不作为领域结论的关系证据 | 减少领域层表达能力，换取可解释、可去重的跨域边界 |
| ADR-002 | 统一 `HoloMemoryAccessPolicy` 位于 Repository/Query Service 之上，所有消费端强制经过 | 增加一层访问抽象，换取开关和敏感度策略无旁路 |
| ADR-003 | 记忆身份使用类型化 anchor + alias，而不是只依赖 LLM 自由生成 `subjectKey` | 需要维护 anchor registry，换取跨模块可连接和近义去重 |
| ADR-004 | 普通聊天采用 stale-while-revalidate，禁止在回答关键路径运行后台融合 LLM | 回答可能使用稍旧但仍有效的记忆，换取稳定延迟和成本 |
| ADR-005 | 自动记忆显式开启；忘记使用语义墓碑，清空使用 `learningBaselineAt` | 历史数据不会自动重学，用户如需回溯必须主动重新扫描 |
| ADR-006 | 健康派生记忆在隐私与同步边界单独评审前保持本机存储 | 跨设备体验暂不完整，换取上线前的隐私安全边界 |

## 4. 方案比较

### 方案 A：每个模块独立记忆系统

分别建立 Finance/Thought/Health 等 Store 和状态机。

- 优点：模块物理隔离。
- 缺点：重复实现存储、过期、置信度、用户反馈和召回；跨域融合还要重新统一数据。
- 结论：不采用。

### 方案 B：统一记忆底座，按领域逻辑隔离

每个模块拥有独立萃取器和时间窗口，但输出统一记忆契约；跨域融合只读取通过校验的领域记忆。

- 优点：领域语义清晰，用户控制和召回统一，能够复用现有记忆基础。
- 风险：统一 Schema 必须同时容纳近期状态、稳定模式和人生事实。
- 结论：采用。

### 方案 C：不保存模块记忆，查询时动态总结

- 优点：初期改动少。
- 缺点：每次重复读取明细、成本高、结果漂移，无法形成持续理解。
- 结论：不采用。

第一版不引入向量数据库。结构化领域、主题键、时间范围、记忆类型和关键词足以支撑初期召回；未来只有在真实规模证明结构化召回不足时，才增加 embedding 作为辅助排序。

## 5. 总体分层

```text
L0 原始数据
财务明细、观点原文、健康读数、习惯记录、任务、目标、对话
        ↓
L1 模块信号
确定性统计生成的高密度事实，不直接作为记忆
        ↓
L2 领域记忆
财务记忆、观点记忆、健康记忆、习惯记忆等
        ↓
L3 跨域记忆
由两个及以上领域共同支持的阶段状态、关联和冲突
        ↓
L4 HoloAI 运行时上下文
按当前问题组合 Profile、领域记忆、跨域记忆和必要明细
```

各层职责：

- 原始数据负责证明。
- 模块信号负责压缩明细。
- 领域记忆负责描述单一领域内的稳定理解。
- 跨域记忆负责连接多个领域，但不创造新的事实。
- Profile 负责提供用户明确声明的身份、偏好、目标和边界。
- HoloAI 负责根据当前问题选择信息，不把所有记忆全部注入。

## 6. 统一记忆契约

目标模型建议收敛为统一的 `HoloMemoryRecord`，由 `scope` 和 `persistenceClass` 表达领域、跨域和时间性质，而不是继续扩展多套 JSON 模型。

核心字段：

```swift
struct HoloMemoryRecord {
    var id: String
    var scope: HoloMemoryScope
    var primaryDomain: HoloMemoryDomain?
    var sourceDomains: [HoloMemoryDomain]
    var subjectKey: String
    var anchorRefs: [HoloMemoryAnchorRef]
    var claimKind: HoloMemoryClaimKind
    var persistenceClass: HoloMemoryPersistenceClass

    var displaySummary: String
    var aiUseSummary: String
    var prohibitedInferences: [String]

    var evidenceRefs: [HoloMemoryEvidenceRef]
    var upstreamMemoryIDs: [String]
    var counterEvidenceRefs: [HoloMemoryEvidenceRef]

    var validFrom: Date?
    var validTo: Date?
    var lastSupportedAt: Date?
    var expiresAt: Date?

    var confidenceScore: Double
    var freshnessScore: Double
    var scoringVersion: Int
    var scoreComputedAt: Date
    var extractorVersion: Int
    var promptVersion: Int
    var lastObservationKey: String?
    var state: HoloMemoryState
    var sensitivity: HoloMemorySensitivity
    var userDecision: HoloMemoryUserDecision

    var createdAt: Date
    var updatedAt: Date
    var schemaVersion: Int
}
```

### 6.1 Scope

- `domain`：只描述一个模块。
- `crossDomain`：引用两个及以上领域记忆。

### 6.2 Claim Kind

- `observedFact`：多个记录或一个明确重要事件直接支持、且对未来理解有持续价值的事实摘要；普通单笔流水不进入记忆。
- `recurringPattern`：多个周期重复出现的模式。
- `phaseShift`：相较此前阶段发生变化。
- `association`：多个变化同时出现，但不声明因果。
- `tension`：当前行为与明确目标或偏好存在冲突。
- `hypothesis`：证据有限、需要继续验证的推测。
- `explicitPreference`：用户明确表达或确认的偏好。
- `lifeEvent`：用户明确记录的重要人生事实。

### 6.3 Persistence Class

- `currentState`：短期状态，快速衰减。
- `phase`：阶段性记忆，可持续数周至数月。
- `durable`：跨周期稳定模式。
- `permanentFact`：明确人生事件或用户确认事实。

### 6.4 稳定身份

记忆 ID 使用稳定语义身份：

```text
scope + primaryDomain/sourceDomains + claimKind + canonical anchor IDs
```

日期、报告序号、卡片 ID、自由生成文案和未归一化 `subjectKey` 不得进入稳定身份。

### 6.5 类型化 Anchor Registry

仅依赖 `subjectKey` 无法稳定连接“减脂”“减肥”“体重管理”等近义主题。新增 `HoloMemoryAnchorRegistry`，优先使用可验证的类型化锚点：

- `goal:<goalID>`。
- `habit:<habitID>`。
- `financeCategory:<categoryID>`。
- `merchant:<normalizedMerchantKey>`。
- `thoughtTopic:<topicID>`。
- `healthMetric:<metricType>`。
- `userTheme:<canonicalKey>`。

LLM 可以提出 alias，但不能直接创建最终 canonical anchor。新 alias 需要本地归一化、已有 anchor 匹配或用户确认。`subjectKey` 继续作为可读辅助字段，不再单独承担跨模块身份。

### 6.6 评分版本

`confidenceScore` 和 `freshnessScore` 是可重算派生值，不是永久事实。每次保存必须同时记录 `scoringVersion` 和 `scoreComputedAt`；算法版本变化时后台重新计算，不允许把旧算法分数和新算法分数直接比较。

## 7. 领域记忆生产链

```text
模块数据变化
→ 标记对应领域 dirty
→ 确定性统计生成候选信号
→ 检查 material change 与 observation 幂等键
→ AI 解释候选含义
→ 本地验证证据、时间和表达边界
→ 按稳定身份合并或生成领域记忆
→ 更新支持证据、反例、时间状态和用户回执
```

每次领域观察使用确定性幂等键：

```text
domain + timeWindow + sourceRevisionDigest + extractorVersion + promptVersion
```

同一幂等键已成功执行时不得重复调用 AI。频繁新增或 CloudKit 批量同步先合并 dirty 事件，只有统计变化跨过 material threshold 后才允许生成观察任务。

### 7.1 规则与 AI 的边界

确定性代码负责：

- 金额、数量、频率、完成率、时间窗口和对照周期。
- 原始记录引用。
- 数据覆盖度和缺失状态。
- 置信度基础项、时间衰减、状态机和用户操作优先级。

AI 负责：

- 将多个信号解释成克制的领域结论。
- 判断是模式、阶段变化、关联、冲突还是待验证推测。
- 生成用户可读摘要和 AI 使用摘要。

AI 不负责：

- 自由计算金额和次数。
- 发明不存在的证据。
- 把一次事件固化为人格或长期事实。
- 绕过本地 Validator 直接写入 Store。

### 7.2 证据类型

证据不能统一假设为可长期存在的 Core Data 实体 ID：

- `entityRef`：财务、任务、习惯、目标、观点等持久记录的稳定 ID 与版本指纹。
- `aggregateSnapshot`：健康或统计聚合结果，保存查询定义、时间范围、样本覆盖、结果摘要和 `sourceRevisionDigest`，不复制原始样本正文。
- `explicitUserStatement`：用户明确表达，保存消息稳定 ID、必要摘录和确认状态。

跨域融合判断“两个领域独立支持”时，必须检查底层 evidence lineage；两条领域记忆若最终来自同一条用户表达或同一业务事件，只算一个独立来源。

### 7.3 Prompt Injection 边界

观点原文、任务标题、交易备注和聊天内容都属于不可信数据，可能包含类似系统指令的文本。Observation Package 必须：

- 把用户内容放在明确的数据字段中，不拼接成系统指令。
- 对长度、控制字符和角色标记做限制。
- 在系统 Prompt 中声明数据字段不可执行。
- Validator 只接受白名单 Schema、已存在 evidence ID 和允许的 claim kind。
- 不允许模型输出改变开关、调度、Store 或工具权限的指令。

## 8. 各领域萃取职责

| 领域 | 重点萃取 | 明确禁止 |
|---|---|---|
| 财务 | 消费结构变化、重复商户/品类、固定支出、预算偏离 | 不在领域层判断消费与目标的关系；不根据消费推断收入、阶层或人格；单次消费不固化为习惯 |
| 观点 | 反复主题、明确立场、立场变化、未解决问题、长期价值取向 | 不把 AI 总结当用户原话；一次随想不升级成人格标签 |
| 健康 | 睡眠、活动、站立、运动和恢复趋势、阶段变化 | 不做医疗诊断；不把缺失数据当异常 |
| 习惯 | 稳定节奏、中断场景、恢复方式、成功或失败条件 | 不使用“自律差”等道德评价 |
| 任务 | 完成节奏、积压变化、常见延期类型、工作负荷、执行偏好 | 不在领域层判断任务与目标冲突；不因少量逾期推断能力或心理问题 |
| 目标 | 当前重点、推进速度、目标调整 | 目标之间或目标与其他领域的协同/冲突进入跨域层；不把系统建议自动写成用户目标 |
| 对话 | 用户明确表达的偏好、纠正、承诺和重要上下文 | 普通聊天不自动长期保存；AI 自己的回复不能成为证据 |
| Profile | 用户主动填写或确认的身份、偏好、目标和边界 | 后台 AI 不得静默改写 |

MemoryInsight 只用于发现值得重新检查的主题，不能作为新的事实证据。领域记忆必须最终追溯到原始模块记录。

Profile 和其他领域的内容可以帮助领域萃取器附加 anchor 或遵守表达边界，但不能成为该领域结论的支持证据。任何“财务行为与减脂目标冲突”之类关系判断必须由跨域融合层生成。

## 9. 时间窗口与内部评分

### 9.1 领域时间窗口

- 健康、习惯：7 天、28 天、90 天。
- 财务：本周、本月、近 3 个月，并尊重账单周期等自然周期。
- 观点：30 天、90 天、半年，重点观察主题和立场演变。
- 任务：7 天、14 天、30 天，重点观察负荷和执行节奏。
- 目标：跟随目标自身生命周期。
- 对话：只处理明确表达，并设置较短候选观察期。

### 9.2 置信度与新鲜度分离

置信度回答“这条结论有多可靠”，新鲜度回答“它是否仍代表当前状态”。

```text
成立置信度
= 来源可靠性
× 证据覆盖度
× 跨周期一致性
× 用户确认加权
× 反例惩罚

当前召回分
= 语义相关度
× 新鲜度
× 成立置信度
× 当前场景适用度
```

衰减速度：

- 近期状态：7～30 天快速衰减。
- 行为模式：90～180 天缓慢衰减。
- 阶段目标：跟随目标状态，到期或完成后归档。
- 用户明确偏好：不自动衰减，直到用户修改。
- 人生事件：事实不衰减，但日常召回相关性可以降低。

### 9.3 用户可见表达

内部不得向普通用户展示百分比、评分条或排序分数。用户只看到：

- 最近形成。
- 持续了一段时间。
- 近期正在变化。
- 过去曾经如此。
- 目前观察到，仍需更多记录确认。

## 10. 反例、冲突与版本

新证据不直接覆盖旧记忆：

- 支持：增加独立周期支持并续期。
- 冲突：加入反例，降低当前有效性。
- 明确反转：旧记忆标记为 `superseded`，生成新的阶段记忆。

Holo 应能表达“以前是这样，但最近发生了变化”。历史版本只用于解释变化和开发者审计，不默认进入当前回答。

用户编辑或确认后的 `displaySummary`、`aiUseSummary` 和禁止推断边界不得被后台候选直接覆盖。新证据与用户文本冲突时，只能标记“需要复核”并展示变化来源，由用户决定是否更新明确表述。

优先级按问题类型分别处理，不能用一条全局顺序覆盖所有场景。

身份、偏好和用户意图类：

```text
用户当前明确输入
> 用户纠正、拒绝和忘记记录
> 用户明确填写的 Profile
> 已确认记忆
> 自动推断的未确认记忆
```

当前状态和可计算事实类：

```text
用户当前明确输入
> 当前原始数据和确定性计算
> 用户明确目标与边界
> 已确认记忆
> 自动推断的未确认记忆
```

因此，Profile 可以约束“用户想实现什么”，但不能覆盖本期真实金额、睡眠、完成率和任务状态。

## 11. 跨域融合

跨域候选必须存在共同锚点：

- 同一目标。
- 同一主题。
- 同一时间阶段。
- 同一生活事件。
- 用户当前明确问题。

示例：

```text
目标记忆：正在持续减脂
健康记忆：最近四周入睡较晚
财务记忆：高热量快餐消费增加
习惯记忆：运动完成率下降
        ↓
跨域候选：
“减脂目标仍在持续，但睡眠、饮食和运动执行近期出现了同时偏离。”
```

### 11.1 四道门

1. 候选发现：两个及以上领域记忆具有共同锚点和时间重叠。
2. AI 综合解释：输出阶段状态、共同变化、协同、冲突或假设。
3. 本地验证：验证 canonical anchor、上游 ID、时间重叠、独立证据 lineage、来源数量和因果表达。
4. 持久化判断：当前问题临时使用，或连续多个周期成立后保存。

### 11.2 防污染规则

- 跨域记忆只能引用领域记忆。
- 领域记忆只能引用原始模块证据。
- 跨域记忆不能作为新的事实来源递归生成更多记忆。
- 上游领域记忆过期、删除或被替代时，下游跨域记忆必须重新评估。
- 跨域记忆继承上游最高敏感等级和最严格同步策略。
- 两条领域记忆共享同一底层证据时，不得伪装成“双领域独立支持”。

### 11.3 因果边界

系统区分：

- 已记录事实。
- 观察到的关联。
- 待验证推测。
- 因果结论。

默认只允许前三层。没有足够证据或用户确认时，不得生成或保存确定因果。

## 12. HoloAI 查询路由

记忆路由必须成为现有 `IntentRouter`、受控 Planner 和 Agent Tool 规划的一部分，不新增一套与现有查询分类并行竞争的独立 Router。当前关键词规则只可作为快速否决或测试兜底，不能承担最终语义分类。

所有消费端通过统一 `HoloMemoryQueryService` 获取记忆；该服务先执行 `HoloMemoryAccessPolicy`：

- 检查“记忆辅助回答”开关。
- 检查当前 purpose 和 use scope。
- 检查敏感度、当前问题直接相关性和用户 suppression。
- 返回本轮允许集合及拒绝原因。

Chat、洞察、目标规划、Agent 预取和 `HoloMemoryTool` 不得直接访问底层 Store。

“记忆辅助回答”关闭只阻断 AI 消费 purpose，不阻断用户在记忆管理页查看、纠正、忘记或清除已有记忆。管理 purpose 和 Debug 只读检查使用独立权限分支。

Suppression 和语义墓碑只用于本地过滤、去重和用户管理，禁止作为 Memory Tool 结果、Evidence Ledger 记录或 Prompt 上下文返回给任何模型。模型最多接收“该候选不可使用”的无正文状态，不得看到被拒绝或遗忘的内容。

### 12.1 明确数字和事实问题

示例：“最近 14 天吃了多少麦当劳？”

- 直接查询原始数据并确定性计算。
- 记忆不能代替数字证据。
- 除非用户继续追问原因或趋势，否则不注入长期记忆。

### 12.2 单领域理解问题

示例：“我最近消费习惯怎么样？”

- 先读取相关领域记忆。
- 记忆过旧、覆盖不足或需要解释具体原因时，定向补查该模块明细。

### 12.3 开放式状态问题

示例：“我最近状态如何？”

```text
当前明确输入
→ Profile 中的目标与边界
→ 相关跨域记忆
→ 各领域近期记忆
→ 仅对缺口定向补查明细
```

系统按问题选择相关领域，不把全部记忆塞入 Prompt。

普通聊天使用 stale-while-revalidate：如果现有记忆仍在有效期内，直接本地召回并回答；如果发现领域过旧，只标记异步刷新并按需查询当前明细，不能在普通流式回答前串行等待新的领域萃取或跨域 LLM。只有用户明确启动“深度分析”时，才允许将刷新作为可见、可恢复的 Agent job 执行。

### 12.4 规划和建议问题

记忆用于个性化，实时数据用于约束。实际日期、金额、任务状态和日程必须读取当前数据。

### 12.5 用户回执

普通用户只看到简洁回执：

> 这次判断参考了你近期的健康、习惯和财务记录。

展开后显示：

- 使用了哪些记忆结论。
- 来源模块。
- 参考时间范围。
- 修改、忘记和不相关操作。

如果关键模块没有数据，Holo 必须说明判断范围有限，不能把“没有读取到”说成“没有问题”。

## 13. 静默更新与开关模型

### 13.1 用户侧只保留两个记忆开关

#### 自动形成记忆

控制：

- 领域记忆萃取。
- 跨域融合。
- 新记忆写入和自动更新。

关闭后停止形成和更新记忆，但不删除已有记忆。模块数据变化可以留下轻量 dirty 标记，供用户以后重新开启时补算；关闭期间不得生成新的派生内容，也不得调用记忆萃取或跨域融合 AI。

#### 记忆辅助回答

控制已有记忆是否进入：

- HoloAI 对话。
- 洞察。
- 目标规划。
- Agent 深度分析。

关闭后记忆仍保留并可管理，但不参与回答。

### 13.2 上层总闸

“HoloAI 数据授权”继续作为所有外部 AI 调用的总闸：

```text
数据授权关闭
→ 外部 AI 调用停止
→ 静默领域萃取和跨域融合停止
→ 原始本地记录不受影响
```

数据处理授权不自动等于同意后台形成记忆。用户首次开启“自动形成记忆”时，需要一次简洁说明：Holo 会在使用期间静默提炼模块记忆、结论保存在何处、用户可随时纠正或忘记。

### 13.3 内部 Feature Flags

现有 `longTermMemoryEnabled` 可升级为记忆写入总闸。`episodicMemoryObservationEnabled`、各模块开关和跨域融合开关作为内部灰度能力，不向普通用户暴露。

升级迁移规则：

- 原 `longTermMemoryEnabled == true` 且 AI 数据授权有效：迁移为“自动形成记忆”开启。
- 其他用户保持关闭，不因版本升级静默开启。
- “记忆辅助回答”沿用原 `memorySummaryInjectionEnabled` 选择。
- Debug 手动执行默认也遵守总闸；若开发者选择 dry-run override，页面必须持续显示“仅本次调试绕过”，且不得写入正式 Store。

### 13.4 更新节奏

- 本地统计信号：数据变化后低成本更新。
- 领域 AI 萃取：每天最多一次，只处理发生变化的模块。
- 跨域融合：每周一次；开放式问题只能标记异步刷新，或在用户明确启动深度分析时作为可见 Agent job 执行，不阻塞普通回答。
- 重大目标变化和用户纠正：立即使相关结论失效并重新评估。
- 后台机会未执行：下次进入 App 后轻量恢复。

关闭期间只合并一个领域级 dirty 状态，不累积逐条任务。重新开启后默认只补算各领域规定的最近窗口，不无上限回放整个历史库。

### 13.5 成本与资源预算

“每天每个变化模块一次”只是上限，不是调用目标。调度器还必须满足：

- 没有 material change 不调用 AI。
- 同一 observation 幂等键只调用一次。
- 每个领域有独立 cooldown。
- 每日有总调用次数和 token 预算，超出后按活跃目标、用户当前问题和数据变化强度排序延后。
- 低电量、低数据模式或系统后台时间不足时只保留 dirty 标记。
- 领域任务失败使用退避重试，不进行紧密循环。

具体额度进入实施计划并通过真实模型成本压测确定，不能在没有成本基线时默认七个领域每天全部调用。

## 14. 用户控制与记忆生命周期

### 14.1 用户操作

- `不相关`：降低当前召回，未来有新证据可重新判断。
- `这不准确`：保存用户纠正，后续萃取必须把它作为反例。
- `忘记这件事`：删除记忆，并以 canonical anchor + claim kind + scope 建立持久语义墓碑，禁止通过近义文案或新 subjectKey 重生；用户可在记忆管理中主动解除。
- `清除全部 AI 记忆`：清除领域记忆、跨域记忆、反馈和语义墓碑，原子写入 `learningBaselineAt = now`，默认只从清除后的新数据继续学习，不删除原始业务数据。

“清除全部”只有在新的学习起点成功写入后才能删除 suppression；否则会出现用户刚清空、后台又全部重建的产品违约。若用户以后希望重新利用历史记录，需要在记忆管理中主动选择“重新扫描历史记录”，并明确选择时间范围。语义墓碑只保存 canonical identity、作用范围和用户决定，不保留被忘记的原始正文。

### 14.2 原始数据联动

- 新增记录：对应领域标记 dirty。
- 修改记录：旧证据失效并重新计算。
- 删除记录：移除证据；证据不足时记忆减弱或归档。
- 模块同步失败：不更新结论，也不能把缺失解释成行为变化。

每条证据应携带来源稳定 ID 和版本指纹，以便判断原始记录是否变化。

## 15. 存储与跨设备策略

统一底座通过 `HoloMemoryRepository` 暴露单一接口，底层可按敏感等级选择存储策略。

Repository 必须由 actor 或等价串行事务边界保护，提供原子的 `mutate/upsert/confirm/reject/forget`，禁止消费端继续执行分离的 load-modify-save。保存成功后才能发布回执和清除 dirty 标记。

目标策略：

- 普通领域记忆、跨域记忆和用户反馈采用结构化、可版本化存储。
- 普通记忆可进入用户私有 iCloud，实现跨设备一致体验。
- 使用稳定 ID、版本号和删除墓碑处理多设备并发。
- 用户删除、纠正和关闭自动记忆的操作优先于后台生成更新。
- 健康及其他敏感派生记忆在合规边界确认前默认只保存在本机。
- 含敏感领域的跨域记忆继承最严格存储和同步策略。

稳定 ID 和版本号本身不能解决多设备重复生成。同步层还必须定义：

- 同一 observation 幂等键只接受一个成功结果。
- 自动生成结果冲突时合并 evidence lineage，并按 extractor/prompt/scoring 版本选择较新可验证版本。
- 用户操作使用单独的不可逆优先级，删除墓碑不得被后台生成覆盖。
- 没有完整源数据的设备不得因为本地缺失而削弱其他设备生成的记忆。
- CloudKit 导入期间先合并变更并等待短暂稳定窗口，再触发领域重算。

“自动形成记忆”“记忆辅助回答”、`learningBaselineAt` 和语义墓碑属于用户级控制状态。已启用 iCloud 时应跨设备同步；调度器在开始和提交 observation 前都要重新读取最新控制状态。多设备冲突时以时间上更新的用户操作为准，后台任务结果永远不能反向覆盖用户关闭或清空决定。未启用 iCloud 时只承诺本机生效，UI 不得暗示已经全设备同步。

健康派生记忆和包含健康内容的跨域记忆，在进入任何 CloudKit Schema 或正式版跨设备同步前，必须完成单独的隐私、App Store 和数据保留评审；Phase 2 可以先以本机模式验证，不把“未来可同步”写成当前已支持事实。

即使是普通派生记忆，在首次进入 iCloud 和隐私文案前，也必须补充“AI 生成的领域/跨域记忆”这一数据类别、用途、删除方式和同步范围，不能只沿用原始业务数据的同步说明。

现有长期/情景 JSON Store 不继续扩展为最终形态。迁移必须支持：

- 原子迁移。
- 失败回滚。
- Schema 版本。
- 旧记忆稳定 ID 保留或映射。
- 用户确认、拒绝和忘记状态不丢失。

迁移开始前必须先补现有 Store 回归测试，包括情景 Store 首次写入、损坏恢复、并发 mutation 和迁移失败回滚，不能把尚未证明可靠的 JSON 文件直接当作无损迁移源。

## 16. 失败与降级

- 网络或 AI 失败：保留 dirty 标记，下次重试。
- 单模块失败：其他模块继续更新，本轮覆盖度标记为部分。
- 输出格式错误：拒绝候选，不覆盖旧记忆。
- 引用不存在：拒绝候选。
- 时间窗口不重叠：拒绝跨域候选。
- 存储失败：保留上一个有效版本。
- 长时间未更新：只作为历史背景，不描述为最近状态。
- 数据授权或自动记忆关闭：不得旁路调用 AI。

这些失败默认不打扰普通用户。只有当用户的问题依赖缺失模块时，才用自然语言说明本轮参考范围有限。

## 17. Debug：AI 记忆实验室

在当前 Debug “AI 设置”中新增独立入口“AI 记忆实验室”，Release 不编译、不展示。

### 17.1 总览面板

- 数据授权、自动形成记忆、记忆辅助回答状态。
- 各领域最近成功更新时间。
- dirty 模块和计划运行时间。
- 领域、跨域、待确认、归档记忆数量。
- 最近失败原因与重试状态。
- Schema 和 Prompt 版本。

### 17.2 模块验证

逐步执行并查看真实链路：

```text
查看数据覆盖
→ 生成本地统计信号
→ 预览观察包
→ 执行领域萃取
→ 查看结构化返回
→ 查看 Validator 结果
→ 查看最终写入或拒绝原因
```

模块验证默认使用 dry-run，不写正式 Store；开发者显式切换到“应用结果”并二次确认后，才允许走真实 mutation。dry-run 和正式执行复用同一 Builder、AI Client、Parser 与 Validator，只在最终 Repository commit 边界分叉。

### 17.3 跨域验证

- 选择领域记忆手动执行融合。
- 查看共同锚点、选入和排除原因。
- 查看时间重叠、claim kind 和持久化判断。
- 查看完整上游领域记忆引用。

### 17.4 HoloAI 查询模拟器

输入问题后展示：

- 问题类型。
- 选择的领域。
- memory/detail 数据路径。
- 定向补查项。
- 过期、不相关和敏感内容排除原因。
- 内部置信度、新鲜度、相关度和排序分数。
- 最终上下文预览。

用于验证数字问题没有错误召回、开关能够阻断调用、敏感记忆不被无关召回、覆盖不足正确降级。

### 17.5 单条记忆检查器

- 当前结论和历史版本。
- 支持证据和反例。
- 原始记录引用有效性。
- 内部评分组成。
- 最近使用记录。
- 上游领域记忆和下游跨域记忆。
- 用户纠正、拒绝和忘记记录。

### 17.6 本地 Trace

保留本机环形追踪：

```text
原始数据变化
→ 模块标脏
→ 信号生成
→ AI 调用
→ 校验
→ 记忆写入
→ 查询召回
→ 回答使用
```

Trace 不进入 CloudKit、不上传后端、不持久保存完整敏感 Prompt。导出时只允许生成脱敏 JSON。

Debug Trace 采用最多 500 条或 7 天的本机环形上限，任一条件先到即清理。Release 不只是隐藏入口，而是不得编译观察包原文预览、内部评分详情和 Trace 导出实现。

开发者页面必须调用真实生产 Service 和 Validator，不能维护另一套只用于展示的模拟逻辑。

## 18. 非功能要求

### 18.1 性能与前台体验

- 本地记忆开关检查、筛选和 top-K 召回设计目标为 100ms p95 内完成。
- 普通聊天关键路径不运行领域或跨域 LLM，不增加额外串行网络往返。
- 每轮普通回答最多注入 8 条记忆，记忆上下文预算默认不超过约 2,000 tokens；不同 purpose 可以进一步收紧。
- 后台生成同时只运行一个记忆 AI job，避免与用户主动 HoloAI 请求争抢资源。

### 18.2 容量与保留

- MVP 每个领域最多保留 50 条 active/phase/durable 记忆。
- active 跨域记忆最多 100 条。
- 超限时优先归档低新鲜度、低使用率且无用户确认的记录，用户确认记忆不能因容量自动删除。
- 历史版本和 evidence metadata 需要独立保留上限，不能无限增长；具体字节预算在实施计划中通过真实编码体积校准。

### 18.3 可靠性

- 记忆系统故障不得阻塞原始记录 CRUD、HoloAI 明细查询或 App 启动。
- 用户纠正、忘记、开关和学习起点属于不可轻易重算的数据，写入必须原子完成；自动生成摘要允许从原始证据重建。
- dirty、observation 幂等键和迁移状态需要持久化，App 被系统终止后可恢复。
- Store 或 Schema 迁移失败时保留上一版本，不允许以空库覆盖。

### 18.4 安全与隐私

- 不可信用户文本按数据处理，不能执行为 Prompt 指令。
- 普通日志不记录记忆正文、原始观点、交易备注、健康摘要或完整观察包。
- 后端 AI 调用日志只记录 purpose、版本、耗时、token 和错误码，不持久化领域记忆正文或完整融合上下文。
- 敏感度从上游向跨域记录单向提升，不允许融合后降级。
- Debug override、原文预览和 Trace 导出全部在 Release 编译边界外。

### 18.5 成本与系统资源

- 调用预算、token 预算、重试和 cooldown 必须由统一 Scheduler 管理。
- 遵守低电量模式、低数据模式和 iOS 后台时间限制。
- 没有 material change 时后台成本应接近零。
- 上线前以影子模式测得单活跃用户日均调用和 token 成本，再决定默认调度额度。

## 19. 分阶段实施

### Phase 1：统一记忆底座

- 统一领域/跨域模型和 Repository。
- 类型化 Anchor Registry。
- 统一 `HoloMemoryAccessPolicy` 和 Query Service，先修复 Agent/Tool 旁路。
- 置信度、新鲜度、召回相关度分离。
- 原子 mutation、幂等 observation 和 scoring/extractor 版本。
- 证据、反例、版本、纠正、忘记和删除墓碑。
- `learningBaselineAt` 与历史重新扫描入口。
- 用户开关收口。
- 首次开启说明、现有用户开关迁移和隐私文案更新。
- dirty 调度和失败恢复。
- 现有长期/情景记忆迁移。
- AI 记忆实验室基础总览和单条检查器。

### Phase 2：结构化领域

- 财务。
- 习惯。
- 健康（先本机验证，通过独立隐私/同步评审后再决定跨设备策略）。
- 目标。
- 模块实验室分步执行与 Trace。

### Phase 3：语义型领域

- 观点。
- 任务。
- 对话明确表达。
- 防止临时表达、普通聊天和一次事件错误固化。

### Phase 4：跨域融合与 HoloAI 路由

- 共同目标、主题和时间阶段索引。
- 动态状态回答。
- 跨域阶段记忆持久化。
- Chat、洞察、规划和 Agent 接入。
- 来源回执和查询模拟器完整链路。

## 20. 上线策略

```text
内部影子运行
→ 记忆中心可见
→ HoloAI 仅使用用户确认记忆
→ 开放低风险自动记忆
→ 开放跨域阶段记忆
```

影子模式正常生成候选，但不进入正式回答、不主动展示给用户。重点检查准确性、重复、越界、证据可追溯性和数据缺失语义。

## 21. 测试与验收

### 21.1 纯逻辑测试

使用 standalone Swift 测试覆盖：

- 稳定身份。
- 置信度和新鲜度计算。
- scoring/extractor/prompt 版本变化后的重算与比较边界。
- 过期、替代和反例状态机。
- 用户操作优先级。
- suppression 和近义重生拦截。
- Query Router 分类。
- 跨域时间重叠和因果表达 Validator。
- 敏感等级继承。
- Anchor alias 归一化与近义去重。
- observation 幂等键和 material change gate。
- `HoloMemoryAccessPolicy` 对 Chat、洞察、规划、Agent 和 Tool 的统一阻断。
- `learningBaselineAt` 与语义墓碑。
- 用户编辑摘要不被后台候选覆盖。
- suppression/墓碑正文永不进入 Tool、Evidence Ledger 或模型上下文。

### 21.2 Fixture 测试

每个领域准备固定输入与预期结构，覆盖：

- 数据充足。
- 数据不足。
- 突然变化。
- 反例出现。
- 原始数据删除。
- 模型返回非法 evidence ID。
- 模型返回诊断、人格标签或确定因果。
- 用户数据中包含伪系统指令或 Prompt Injection 文本。
- 两条领域记忆共享同一底层 evidence，不能计为独立跨域支持。

### 21.3 集成验收场景

1. “最近 14 天吃了多少麦当劳”只查明细，不用记忆代替计算。
2. “我最近状态如何”组合多个有效领域，并说明覆盖范围。
3. 原始记录修改或删除后，领域和跨域记忆重新评估。
4. 用户选择“忘记”后，同一结论不以近义表达重生。
5. 关闭自动形成记忆后，不再发生后台 AI 萃取调用。
6. 关闭记忆辅助回答后，已有记忆不进入回答或规划。
7. 单模块失败不影响其他模块，不把缺失数据描述为正常。
8. 跨域结论每部分都能追溯到领域记忆和原始证据。
9. 关联不被表达成确定因果，健康内容不形成诊断。
10. Holo 能表达“以前如此，但最近正在变化”。
11. Debug 查询模拟器展示的路由、召回和最终上下文与真实回答链路一致。
12. Release 构建不包含 AI 记忆实验室入口和敏感 Trace 展示。
13. Agent 和 Memory Tool 在“记忆辅助回答”关闭时无法读取记忆。
14. 清除全部记忆后，旧历史不会在下一次静默任务中自动重建。
15. 情景 Store 首次写入、统一 Repository 并发 mutation 和迁移回滚全部通过。
16. 普通聊天发现记忆过旧时不会同步等待新的融合 LLM。
17. 重复 observation 和多设备同源任务不会产生重复记忆或重复 AI 调用。
18. 被拒绝或忘记的原始摘要不会通过 suppression Tool 查询重新进入 Agent。

### 21.4 验证要求

- 纯逻辑策略优先使用 standalone `swiftc` 测试。
- 如果使用 Xcode 测试，必须确认测试实际执行，`Executed 0 tests` 不算通过。
- 后端 Prompt/registry 变更必须运行后端测试，并在 ECS 部署后验证生产 Prompt 版本和输出契约。

## 22. 内部质量指标

- 用户纠正率、忘记率和不相关率。
- 领域记忆被实际使用的比例。
- 开放式问题的跨域覆盖度。
- 重复记忆率。
- 过期记忆误召回率。
- 静默更新耗时、AI 成本和失败率。
- 明细问题被错误路由到记忆的比例。
- 单活跃用户日均后台 AI 调用、token 和成本。
- 前台记忆召回 p95 延迟。
- 同源重复 observation 拦截率。

这些指标只用于内部质量改进，不向普通用户展示。

## 23. 非目标

本轮不做：

- 不让向量相似度直接决定事实。
- 不把所有原始数据复制进记忆库。
- 不把 MemoryInsight 或 AI 历史回答当事实证据。
- 不让跨域记忆递归强化自身。
- 不自动生成医疗、心理、人格或阶层判断。
- 不要求普通用户理解内部评分和状态机。
- 不让静默记忆任务阻塞主流程。

## 24. 后端与发布边界

领域萃取和跨域融合需要新增或升级后端 Prompt purpose、registry、默认 Prompt 和 Validator 契约。实施时必须：

1. 同步 iOS 后备模板与后端生效 Prompt。
2. 升级 Prompt 版本。
3. 完成后端契约测试。
4. 部署 HoloBackend 到 ECS。
5. 验证生产 Prompt 元数据和真实返回结构。

只完成本地 iOS 或本地后端代码不代表功能已经生产生效。

## 25. 实施完成状态（2026-07-15）

### 25.1 已完成

- 八个领域已接入独立 Signal Builder 和统一记忆契约；Profile 只接受明确填写、确认或迁移，不做后台人格推断。
- 静默运行已接入 App 启动、回前台和数据变化事件，真实执行领域萃取、校验、写入与跨域候选融合；无 material change 时不重复调用 AI。
- 查询统一经过 `HoloMemoryQueryService`：明细问题回源，开放式问题读取已整理记忆；召回时按最后证据时间动态衰减，新记忆获得更高内部权重。
- 用户可查看来源、确认、纠正、停用和忘记；语义墓碑阻断近义重生，清空操作建立新的学习起点。
