## 一、Step 0：范围挑战（最高影响）

### 原方案

手写 CloudKit 同步层：8 个新文件 + 6 个修改文件 = 14 个文件，4+ 新类（CloudKitSyncService, SyncMetadataRepository, GoalCloudSyncAdapter, SyncMetadata）。

### 审查发现

`CoreDataStack.swift:8-24` 已有 `NSPersistentCloudKitContainer` 切换指南注释，且 `CoreDataStack.swift:92` 已启用 `NSPersistentStoreRemoteChangeNotificationPostOptionKey` —— 代码库已为 CloudKit 同步做了预备。

原方案选择手写同步层的理由是「切 NSPersistentCloudKitContainer 会影响所有实体和迁移策略」。但这个理由被挑战：

1. **「影响所有实体」不是问题** — 私有数据库，所有数据本来就是用户自己的。Goal、Task、Habit、Finance 全部同步到用户私有 iCloud 不是风险，是功能。
2. **「迁移策略」不存在** — Core Data model 不变，没有 model migration。只是换容器类型。
3. **NSPersistentCloudKitContainer 自带** — 冲突解决、变更追踪、增量同步、离线队列、自动重试。手写方案要在 Task 7 里一行行实现这些。
4. **手写方案的复杂度成本** — 8 个新文件、4 个新类、手动 metadata 管理、两阶段 save 一致性问题、线程安全风险。用 NSPersistentCloudKitContainer 只需改 1 行代码（`NSPersistentContainer` → `NSPersistentCloudKitContainer`）+ entitlements 配置。

### 决策：换 NSPersistentCloudKitContainer

14 文件 / 4+ 新类 → 约 3-4 文件改动，0 新类。

## 二、架构审查（基于 NSPersistentCloudKitContainer 方案）

### Issue #1 — CloudKit Container ID 选择

方案指定 `iCloud.com.tangyuxuan.Holo`，但项目存在两个 Bundle ID：`com.tangyuxuan.Holo` 和 `com.tangyuxuan.holo-app`。

**决策**：使用 `iCloud.com.tangyuxuan.Holo`，与主 Bundle ID `com.tangyuxuan.Holo` 一致。

### Issue #2 — Schema 部署策略

当前 Core Data model 是纯代码创建的（非 .xcdatamodeld）。NSPersistentCloudKitContainer 在 Development 环境会自动根据 NSManagedObjectModel 生成 CloudKit schema 并部署。

**决策**：使用自动 schema 部署。Development 环境可随时重置 schema，风险极低。切 Production 时在 CloudKit Dashboard 手动 deploy。

### Issue #3 — 首次同步 UX

切换到 NSPersistentCloudKitContainer 后，所有现有本地数据会自动上传到 iCloud。

**决策**：
- 静默后台同步，不阻塞 UI
- 支持手动刷新入口（显示 iCloud 账号状态 + "同步中"提示）
- NSPersistentCloudKitContainer 没有暴露"立即同步" API，手动刷新只能触发 iCloud 账号状态检查

### What already exists（现有代码已具备的能力）

| 文件 | 行号 | 已有配置 |
|------|------|---------|
| `CoreDataStack.swift` | 92 | `NSPersistentStoreRemoteChangeNotificationPostOptionKey` 已启用 |
| `CoreDataStack.swift` | 110 | `automaticallyMergesChangesFromParent = true` |
| `CoreDataStack.swift` | 111 | `NSMergeByPropertyObjectTrumpMergePolicy`（Apple 推荐用于 CloudKit） |
| `CoreDataStack.swift` | 8-24 | NSPersistentCloudKitContainer 切换指南注释 |

## 三、代码质量审查

### Issue #4 — `refreshAllObjects()` 违规（现有代码）

`CoreDataStack.swift:255` 的 `batchInsert` 扩展中调用了 `refreshAllObjects()`。CLAUDE.md 明确规定「禁止 `refreshAllObjects()`，改为重新 fetch」。

**影响**：切换到 NSPersistentCloudKitContainer 后，远程变更会频繁触发上下文刷新，`refreshAllObjects()` 在高频场景下可能导致性能问题。

**决策**：不在本次 PR 中修复，标记为技术债。REPO_MODE=solo，已记录。

### Issue #5 — 手动刷新 UI 实现方式

NSPersistentCloudKitContainer 没有内置的"立即同步"按钮 API。

**决策**：在设置页显示 iCloud 账号状态 + "同步中"提示。实际同步由 CloudKit 自动完成，UI 只是状态展示。

## 六、外部视角（Outside Voice）

**执行方式**：Claude subagent（Codex 不可用）

### 返回的 5 个发现

| # | 发现 | 验证结果 | 严重性 |
|---|------|---------|--------|
| 1 | Thought.imageData 二进制数据会触发 CloudKit 记录大小限制 | **部分正确** — NSPersistentCloudKitContainer 自动将大型二进制数据转为 CKAsset（独立存储，不受 1MB 记录限制）。但图片多时仍消耗 iCloud 配额。 | 低 |
| 2 | ChatMessage rawLogJSON/analysisContextJSON 不该同步 | **低优先级** — ChatMessage 只有 `extractedDataJSON: String?`（JSON 字符串），体量可控。 | 低 |
| 3 | NSMergeByPropertyObjectTrumpMergePolicy 在多设备同步中是错误的 | **错误** — Apple 官方文档明确推荐此策略用于 NSPersistentCloudKitContainer。它解决的是进程内冲突，跨设备冲突由 CloudKit last-write-wins 处理。 | N/A |
| 4 | fatalError 在 iCloud 错误时会让 App 闪退 | **有效但属现有代码** — `CoreDataStack.swift:97` 的 fatalError 是现有问题，不在本次范围内。但 CloudKit 配置错误可能触发它。 | 中 |
| 5 | Transaction 历史数据会撑爆 CloudKit 配额 | **数据不支撑** — 自身计算显示 3 年仅 2.7MB，远低于 1GB 免费配额。 | 低 |
