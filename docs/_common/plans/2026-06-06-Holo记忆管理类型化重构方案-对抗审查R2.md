## 🚨 CRITICAL：2 个新发现的致命问题

### C1. `useScopes` / `prohibitedInferences` 非 Optional 但旧 JSON 中不存在 — 解码会崩

**事实**：方案 §5 的数据模型定义：

```swift
struct HoloLongTermMemory {
    var type: HoloLongTermMemoryType
    var semanticType: HoloMemorySemanticType?       // Optional ✅
    var displaySummary: String?                      // Optional ✅
    var aiUseSummary: String?                        // Optional ✅
    var useScopes: [HoloMemoryUseScope]              // 非 Optional ❌
    var prohibitedInferences: [String]               // 非 Optional ❌
    ...
}
```

当前代码中 `HoloLongTermMemory` 依赖 **Swift 自动合成 Codable**（无自定义 CodingKeys）。自动合成的 `init(from:)` 对非 Optional 属性会要求 JSON 中必须存在该 key。

旧 JSON 长这样：
```json
{
  "id": "candidate-xxx",
  "type": "recurringPattern",
  "title": "体重习惯连续中断",
  "summary": "...",
  "confidence": "low",
  "confirmationState": "candidate",
  "sensitivity": "normal",
  "evidence": [...],
  "createdAt": "2026-06-01T00:00:00Z",
  "updatedAt": "2026-06-01T00:00:00Z"
}
```

没有 `useScopes` 和 `prohibitedInferences` 字段。Swift `Decoder` 在解码时会抛出 `DecodingError.keyNotFound`，**导致 `HoloLongTermMemoryStore.load()` 返回空数组**（当前 load() 实现 catch 后返回 `[]`），**所有已有长期记忆静默丢失**。

**影响**：
- 用户已确认的长期记忆全部消失
- `HoloMemoryCenterView` 的 "已记住" Section 变空
- 更危险的是 `load()` 的 catch 分支会把损坏的 JSON 备份为 `_corrupted.json`，然后返回空数组 — 用户下次打开 App 看到记忆没了，数据文件还被移走，**没有 UI 反馈、没有恢复路径**

**建议**：必须做以下之一：
- **方案 A（推荐）**：将这两个字段声明为 Optional 并给默认值：
  ```swift
  var useScopes: [HoloMemoryUseScope]? = nil
  var prohibitedInferences: [String]? = nil
  ```
  nil 表示"未指定，按 fallback 逻辑处理"。
- **方案 B**：自定义 `init(from decoder:)` 实现，缺失 key 时使用默认值。参考 `HoloEpisodicMemory` 的 `schemaVersion: Int = 1` 模式（但注意 Swift 自动合成对非 Optional + 默认值的处理仍有坑）。
- **方案 C**：在 `load()` 中检测旧格式并做一次迁移，给旧数据补上默认的 `useScopes` 和 `prohibitedInferences`。

### C2. 方案 §5 的旧数据 fallback 映射与 §6 的 Mapper "不做文本分析" 矛盾

**事实**：方案 §5 的 fallback 映射表写着：

> `.recurringPattern` → `stablePattern` — 仅作为弱 fallback；**如果标题/摘要包含中断、偏离、超支等词，应降级为待确认旧格式**

这是一个**基于文本内容的分类决策**。但方案 §6 明确说：

> Mapper 不负责"理解文本并重新分类"，也不对每张卡片再次调用 LLM。

这两条直接矛盾。如果 Mapper 要根据 title/summary 中的关键词做分支判断，那它就是在做文本分析 — 即使是关键词匹配，也需要定义完整的关键词列表和映射规则。

**影响**：
- 如果严格遵守"不做文本分析"，所有旧 `.recurringPattern` 记忆都会 fallback 为 `stablePattern`，包括那些本质是 `driftSignal` 的记忆（如"体重习惯连续中断"）。这恰好是方案要修复的问题。
- 如果允许文本分析，需要定义完整的关键词→semanticType 映射表，且这个表会随着新语义的加入不断膨胀，维护成本高。

**建议**：
- 对于**旧数据 fallback**，不要试图自动分类。统一标记为"旧格式，待重新确认"更诚实、更安全。
- 对于**新数据**（从 prompt 输出），LLM 已经给出了 `semanticType`，Mapper 只需校验枚举值合法性。
- 把"关键词→semanticType"的文本分析从 Mapper 中完全移除，fallback 逻辑统一为"旧数据标记为 `semanticType = nil` + UI 提示需重新确认"。

## ⚠️ HIGH：4 个高风险问题

### H1. Phase 4 筛选算法的 purpose 类型对不上

**事实**：方案 Phase 4 的筛选算法第 3 步说"根据当前 `purpose` 匹配 `useScopes`"，但代码中有**两套不同的 purpose 枚举**：

| 枚举 | case | 使用位置 |
|------|------|---------|
| `HoloAICapabilityID` | `onboarding` / `todayState` / `recentAnalysis` / `longTermPatterns` / `goalPlanning` | `HoloMemorySummaryProvider.selectRelevantSummary(purpose:)` |
| `AIUserContextPurpose` | `.chat` / `.intentRecognition` | `AIUserContextMessageBuilder.build(from:purpose:)` |

记忆注入发生在 `AIUserContextMessageBuilder`，这里的 purpose 只有 `.chat` 和 `.intentRecognition`。但 `useScopes` 的定义是 `[.coreContext, .recentInsight, .goalPlanning, .retrospective, .displayOnly]`。

问题是：**当 purpose 是 `.chat` 时，哪些 useScopes 应该被匹配？** 方案没有给出 purpose → useScopes 的映射表。

当前的 `HoloMemorySummaryProvider.selectRelevantSummary(purpose:)` 使用 `HoloAICapabilityID`，但这个枚举在 `AIUserContextMessageBuilder` 的调用链中**从未被传入** — `UserContextBuilder.buildContext()` 不接收 purpose 参数。

**建议**：Phase 4 必须明确定义：
1. `AIUserContextPurpose.chat` → 允许的 `useScopes` 子集（如 `[.coreContext, .recentInsight]`）
2. `AIUserContextPurpose.intentRecognition` → 不注入记忆（当前已如此）
3. 是否需要将 `HoloAICapabilityID` 传递到 `UserContextBuilder.buildContext()` 中，以便更精细地按场景筛选

### H2. confidence 和 sensitivity 的赋值主体冲突

**事实**：方案 §6 的 prompt 输出结构要求 LLM 输出：

```json
{
  "confidence": "low | medium | high",
  "sensitivity": "normal | highImpact | sensitive",
  ...
}
```

但当前代码中，这两个字段**不是由 LLM 决定的**：

| 字段 | 当前赋值方式 | 代码位置 |
|------|------------|---------|
| `confidence` | `evidence.count >= 2 ? .medium : .low`（永远不产生 `.high`） | `CandidateObserver` 第 73 行 |
| `sensitivity` | 10 个关键词匹配，只有 `.normal` 和 `.sensitive`（永远不产生 `.highImpact`） | `CandidateObserver.classifySensitivity()` |

方案没有说明：
- LLM 输出的 `confidence` 和 Mapper 校验后的 `confidence` 哪个优先？
- 如果 LLM 说 `confidence: high` 但 evidence 只有 1 条，以谁为准？
- `sensitivity` 也同理：LLM 说 `normal` 但标题包含"压力"，以谁为准？
- 如果 LLM 不输出 `confidence`（因为 prompt 改动可能不稳定），Mapper 应该 fallback 到 evidence-count 策略还是丢弃候选？

**建议**：
- **confidence**：以 LLM 输出为准，Mapper 只做枚举合法性校验。如果 LLM 未输出，fallback 到 evidence-count 策略。删除 `evidence.count >= 2 ? .medium : .low` 硬编码。
- **sensitivity**：保留关键词匹配作为**下限**（LLM 说 normal 但关键词命中 → 升级为 sensitive），LLM 可以升级（normal → highImpact），但不允许降级（LLM 说 sensitive 但关键词未命中 → 保持 sensitive）。
- 在 Mapper 中写明合并规则。

### H3. Prompt 复杂度爆炸风险 — 增加 memoryCandidate 子对象对输出质量的影响

**事实**：当前 `memory_insight_generation` prompt 已经有：
- 235 行、~6,284 字符
- 8 种卡片类型、6 种洞察层级
- 财务/待办/习惯/异常/跨模块/想法/趋势等各自独立的规则段
- 严格的 JSON Schema 要求（title ≤15 字、body ≤60 字、evidence 含日期）

方案要在每张卡片内新增 `memoryCandidate` 子对象，含 10 个字段（semanticType/title/displaySummary/aiUseSummary/useScopes/prohibitedInferences/confidence/sensitivity/expiresInDays/evidence）。

**风险**：
1. LLM 需要在一次生成中同时输出洞察卡片 + 记忆候选，认知负荷显著增加
2. `memoryCandidate` 并非每张卡片都需要 — 方案说"可能进入长期记忆的卡片"，但 LLM 需要判断哪些卡片"可能"，这本身就是一个分类任务
3. 输出 token 增加意味着成本增加和超时风险增加（当前已有 30 秒超时）
4. 更复杂的 Schema → 更高的格式错误率 → `MemoryInsightResponseParser.parse()` 的三层降级会更频繁触发

**建议**：
- 首版让 `memoryCandidate` **只在特定卡片类型**（`habit`/`finance`/`task`/`milestone`）上输出，`overview`/`anomaly`/`cross_domain`/`thought` 类型不要求输出。这能显著降低复杂度。
- 或者考虑：让 `memoryCandidate` 只包含 3 个核心字段（semanticType/displaySummary/aiUseSummary），其余字段（confidence/sensitivity/useScopes/prohibitedInferences/expiresInDays）由 Mapper 根据语义类型默认填充。这样 LLM 只需做"分类 + 两段摘要"，不需要理解 Holo 内部的召回体系。
- 在 Phase 1 验收中增加：生成成功率不低于当前水平（如果当前 3-5 张卡片的成功率是 X%，新增 memoryCandidate 后不应低于 X% - 5%）。

### H4. Phase 0 被 `longTermMemoryEnabled` 默认关闭阻塞

**事实**：`HoloMemorySettings.longTermMemoryEnabled` 默认值为 `false`。这意味着：

1. `HoloAIFeatureFlags.memoryInsightCandidateExtractionEnabled` → `false`
2. `HoloAIFeatureFlags.longTermMemoryWriteEnabled` → `false`
3. `HoloLongTermMemoryCandidateObserver.extractCandidates()` 在第一步就被 guard 挡掉
4. 长期记忆候选永远不会被创建
5. 即使 Phase 0 接通了 `UserContextBuilder → HoloMemorySummaryProvider`，也没有记忆可供注入

**用户视角**：需要先手动在设置中打开"长期记忆"开关 → 触发洞察生成 → 候选被提取 → 用户确认 → Phase 0 才能验证注入。

方案没有提及这个前置条件。Phase 0 的验收标准"有已确认长期记忆时，聊天 prompt 能看到最多 5 条记忆摘要"依赖于：
- 用户开启了长期记忆开关
- 至少触发了一次洞察生成
- 至少有一个候选被提取
- 至少有一个候选被确认

这不是一个"改一行代码就能验证"的 Phase。

**建议**：
- Phase 0 的实施步骤中应包含"确认 `longTermMemoryEnabled` 开关状态，如果为 false 则在验证期间临时打开或创建测试数据"
- 或者：Phase 0 的 Store 层增加一个"写入测试记忆"的方法，不依赖完整洞察链路，直接写入几条确认状态的长期记忆用于验证注入
- 验收标准中应区分"开发者验证"和"用户场景验证"

## 📋 MEDIUM：4 个中等问题

### M1. Mapper 在管线中的位置未明确

方案 §6 的目标链路图：

```
memoryInsightGeneration prompt
  → MemoryInsightCard + optional memoryCandidate
  → MemoryCandidateSemanticMapper.validateAndMap(...)
  → HoloLongTermMemory candidate payload
  → HoloLongTermMemoryCandidateObserver
```

但 Observer 是**通知的接收方**，它从 `userInfo` 提取数据。Mapper 应该在 Observer 内部被调用（替代当前的直接构造逻辑），还是在 Observer 之前被调用（在 `MemoryInsightService` 发通知之前）？

如果是前者（Observer 内部）：Observer 需要理解 `memoryCandidate` 子结构，当前 Observer 的 guard 逻辑需要大改。

如果是后者（Service 层）：`MemoryInsightService` 需要在发送通知前调用 Mapper，这改变了通知的 payload 格式。

建议在方案中画出更精确的调用链：

```
MemoryInsightService.generateInsight()
  → 解析 JSON → MemoryInsightCard[]
  → 发送通知（payload 含 card + memoryCandidate 子结构）
    ↓
HoloLongTermMemoryCandidateObserver.extractCandidates()
  → 从 payload 提取 memoryCandidate
  → MemoryCandidateSemanticMapper.validateAndMap(memoryCandidate)
  → 构造 HoloLongTermMemory
  → PromotionPolicy.evaluate()
  → Store.upsertCandidate()
```

### M2. 旧数据 fallback 映射表的 `.recurringPattern` → `stablePattern` 过于粗糙

方案 §5 映射表将 `.recurringPattern` 默认映射为 `stablePattern`，但当前所有从洞察卡片提取的候选都是 `.recurringPattern`，其中包括：
- "体重习惯连续中断" → 本质是 `driftSignal`
- "任务全部闭环" → 本质是 `stablePattern` ✅
- "支出偏高" → 本质是 `driftSignal` 或不该晋升

映射为 `stablePattern` 会把"体重习惯连续中断"标记为"稳定习惯"，语义完全错误。

这与 C2 的结论一致：旧数据不应自动映射 semanticType，应统一标记为 nil + UI 提示。

### M3. cross_domain 卡片的语义映射缺失

`crossDomain` 是 8 种卡片类型之一，代表跨模块关联洞察（如"习惯打卡与支出变化并现"）。但方案 §4 的 5 种语义类型没有为 `crossDomain` 定义映射规则：

- `crossDomain` 应该映射为什么？`phaseShift`？还是不晋升？
- 如果不晋升，需要在 Mapper 中明确跳过 `crossDomain` 类型的卡片
- 如果晋升，混合两个模块的语义是否违反"一条记忆只表达一个语义"原则

建议在 §6 的生成约束中增加一条："`crossDomain` 类型卡片默认不晋升为长期记忆候选，除非能明确归入单一语义类型。"

### M4. Token 预算影响未评估

Phase 4 的注入格式从：
```
身体记录最近不连续：近 14 天内，体重、平板支撑等记录多次中断。
```
变为：
```
- [recentInsight, goalPlanning] 身体记录最近不连续：用户近期身体状态追踪不稳定。适合健康或习惯洞察参考。
  避免推断：不要判断用户已放弃目标；不要归因为懒惰或失败。
```

单条记忆从 ~30 字膨胀到 ~70 字，5 条记忆 ~350 字。对于 system message 来说这是可接受的，但方案没有做这个分析。

建议在 Phase 4 验收标准中增加：注入记忆后的 system message 总长不应超过 `N` tokens（需根据目标模型上下文窗口和实际对话长度确定 N）。

## 📎 LOW：3 个小问题

### L1. Phase 0 的 `memorySummaryInjectionEnabled` 开关检查缺失

方案说 Phase 0 要"在 `HoloAIFeatureFlags.memorySummaryInjectionEnabled` 开启时调用 Provider"。但当前 `AIUserContextMessageBuilder` 的注入代码（第 94 行）**不检查这个 flag**。需要明确是在 `UserContextBuilder` 还是 `AIUserContextMessageBuilder` 中加检查。

### L2. `.observe` 和 `.requireConfirmation` 行为等价

当前 `HoloLongTermMemoryCandidateObserver` 对 `.observe` 和 `.requireConfirmation` 的处理完全一样（都调用 `upsertCandidate` 保持 `.candidate` 状态）。方案在 Phase 3 重写晋升策略时，应该让这两个决策有实际行为差异（如 `.observe` 不在 UI 展示，或 `.observe` 有自动过期）。

### L3. `HoloMemoryUseScope` 枚举 case 与 `HoloAICapabilityID` 重叠但不一致

`HoloMemoryUseScope` 有 `goalPlanning`，`HoloAICapabilityID` 也有 `goalPlanning`。但 `coreContext`/`recentInsight`/`retrospective`/`displayOnly` 与 `todayState`/`recentAnalysis`/`longTermPatterns` 对不上。建议统一或明确映射关系。

## 总结

| 优先级 | 编号 | 建议 |
|--------|------|------|
| **P0** | C1 | `useScopes`/`prohibitedInferences` 必须 Optional 或自定义 init(from:) |
| **P0** | C2 | 旧数据 fallback 不做文本分析，统一标记为 nil + UI 提示 |
| **P1** | H1 | 定义 `AIUserContextPurpose.chat` → `useScopes` 的映射表 |
| **P1** | H2 | 明确 confidence/sensitivity 的 LLM vs 本地合并规则 |
| **P1** | H3 | 首版 memoryCandidate 只在部分卡片类型输出，减少 prompt 复杂度 |
| **P1** | H4 | Phase 0 补充测试数据创建方法，不依赖完整洞察链路 |
| **P2** | M1-M4 | Mapper 位置、crossDomain 处理、token 预算 |

**整体评价**：R1 修订质量很高，所有 CRITICAL/HIGH 问题都有实质性回应。R2 的问题集中在**修订引入的新技术细节**（数据模型兼容性、管线位置、purpose 类型映射）和**Prompt 工程层面的现实约束**（复杂度、token 预算、LLM 输出稳定性）。建议在进入实施前解决 C1 和 C2，其余问题可在对应 Phase 开始时细化。
