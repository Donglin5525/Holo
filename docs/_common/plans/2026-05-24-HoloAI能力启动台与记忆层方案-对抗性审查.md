| A10 | 新手引导隐藏后用户仍可通过设置找回 | **可行** — `AISettingsView` 已存在，可新增开关 | 简单 |

## 架构问题

### ARCH-1: 短期记忆层与 UserContextBuilder 的关系未厘清 [P1] (confidence: 9/10)

方案定义了 `HoloMemorySnapshotBuilder` 来聚合短期记忆，但数据来源与 `UserContextBuilder` 100% 重合：

| 方案短期记忆字段 | UserContextBuilder 已有字段 |
|---|---|
| `todayState` | `DailySenseSnapshot`（DailySenseStateBuilder 产出）|
| `recentSignals` 中的交易/习惯/任务/想法 | `UserContext.transactionSummary` / `habitSummary` / `taskSummary` / `thoughtSummary` |
| `activeGoals` | `UserContext.goalContext`（GoalRepository 聚合）|
| `recentConversation` | `ChatMessageRepository` 最近会话（UserContextBuilder 未用，但 Repository 已有）|
| `dataCoverage` | **新增** — UserContextBuilder 没有这个概念 |

唯一真正新增的是 `dataCoverage`（数据覆盖度评估）和缓存/过期策略。

如果新建 `HoloMemorySnapshotBuilder`，会出现两个问题：

1. **两套并行聚合逻辑**（UserContextBuilder vs SnapshotBuilder），任何数据源变更需要同步改两处
2. `ConversationCoordinator.process()` 当前接收 `UserContext`，新增记忆后是同时传两个参数还是合并？

**建议选项：**

- **A) 扩展 UserContextBuilder** — 在现有 `buildContext()` 中增加 `dataCoverage` 和 `recentConversation` 字段，增加可选缓存。修改量小，避免并行逻辑，但 UserContextBuilder 会变大。
- **B) 新建 HoloMemorySnapshotBuilder 作为门面** — 内部调用 UserContextBuilder 获取基础数据，只新增 dataCoverage 计算和缓存逻辑。职责隔离清晰，但增加一层间接调用。
- **C) 保持原方案新建独立 builder** — 完全独立的聚合逻辑，与 UserContextBuilder 并行。职责最清晰但维护成本最高。

### ARCH-2: 长期记忆与 InsightPreferenceProfile 的职责边界模糊 [P1] (confidence: 8/10)

`InsightPreferenceProfileService` 已有的结构化数据：

```
InsightPreferenceProfile {
    moduleWeights: [ModuleWeight]        // 用户关注哪些模块
    patternPreferences: [PatternPref]    // 喜欢什么类型的洞察
    tonePreference: TonePreference       // 偏好语气
    actionTypePreference: ActionTypePref // 偏好行动类型
    avoidanceSignals: [String]           // 不想看到的主题
}
```

方案长期记忆的 `stableFeedbackPreference`：

```
HoloLongTermMemoryRecord {
    type: stableFeedbackPreference
    summary: "用户不喜欢泛泛建议"
    evidence: [来自洞察反馈]
}
```

这两个系统都在回答同一个问题："用户喜欢什么"。方案说"可继续保留"但没有回答：

- 当两个系统对同一偏好有不同记录时以谁为准？
- `stableFeedbackPreference` 候选的来源是 `InsightFeedbackAggregator`，而 `InsightPreferenceProfile` 的来源也是同一个 Aggregator — 数据同源但结论存两处

**建议选项：**

- **A) InsightPreferenceProfile 作为长期记忆的子集** — `stableFeedbackPreference` 类型的长期记忆直接从 `InsightPreferenceProfile` 的稳定信号中提取，不另建候选来源。Profile 是"实时偏好画像"，长期记忆是"已固化的偏好结论"，两者是不同抽象层级。
- **B) 合并为统一偏好存储** — 废弃 `InsightPreferenceProfile`，把偏好全部统一到长期记忆系统。迁移成本高但消除了职责重叠。
- **C) 保持现状但明确分工** — `InsightPreferenceProfile` 只服务洞察排序（`InsightCardReranker`），长期记忆偏好服务 Chat 和能力入口。写入时去重。

### ARCH-3: Prompt 集成路径缺失 [P1] (confidence: 9/10)

方案第 6 节"记忆使用规则"说长期记忆需要压缩为短摘要注入 Prompt，但没有说明：

1. **注入哪个 Prompt？** `systemPrompt`？`intentRecognition`？还是 `memoryInsightGeneration`？每个 Prompt 的目的不同，记忆注入的价值也不同。
2. **怎么注入？** `PromptManager.renderTemplate` 只支持 3 个变量（`{{todayDate}}`、`{{currentYear}}`、`{{currentTime}}`）。需要扩展为支持动态变量列表。
3. **Token 预算如何分配？** 现有 token 预算已经很紧：
   - `MemoryInsightContextBuilder`：日 800 / 周 2200 / 月 3800
   - Chat system prompt 已包含 HoloProfile（最多 8KB Markdown）+ UserContext
   - 额外 5 条长期记忆摘要（估计 200-500 token）从哪里扣？
4. **后端 Prompt 同步** — CLAUDE.md 明确写了 Prompt 双端同步规则。修改 Prompt 模板意味着：改 iOS PromptManager → 改后端 defaultPrompts.json → 版本号升级 → 重建 Docker 镜像 → 部署。

**这是一个需要独立设计的子问题，不能留到实施阶段再决定。**

### ARCH-4: Phase 排序存在循环依赖 [P2] (confidence: 8/10)

方案 Phase 顺序：1 → 2 → 3 → 4 → 5

但 Phase 2（今日状态兜底）中的 `TodayStateResolver` 需要判断 dataCoverage（`empty` / `partial` / `enoughForSingleDomain` / `enoughForCrossDomain`），而 `HoloMemoryDataCoverage` 模型定义在 Phase 3（短期记忆 Snapshot）中。

Phase 2 无法独立完成 — 要么把 dataCoverage 模型提前到 Phase 2，要么把 Phase 2 和 3 合并。

**建议选项：**

- **A) 合并 Phase 2 和 3** — 今日状态兜底和短期记忆 snapshot 是同一组数据的不同消费方，合并后一次交付。
- **B) 把 dataCoverage 模型提前到 Phase 2** — Phase 2 只依赖 dataCoverage 结构，不依赖完整的 snapshot builder。

### ARCH-5: MemoryInsightContextBuilder（1945 行）与新记忆系统的交互未规划 [P2] (confidence: 7/10)

`MemoryInsightContextBuilder` 是项目中最重的文件，覆盖财务/习惯/任务/想法/里程碑/跨模块关联/异常/生活轨迹/个人基线/健康等维度。

方案计划从 MemoryInsight 生成的洞察中提取长期记忆候选（`recurringPattern`、`importantLifeFact`），但没有说明：

- **提取时机**：是在 `MemoryInsightService.generateInsight()` 成功后回调，还是后台定期扫描？
- **提取方式**：如果 MemoryInsight 的上下文本身已经很大（1945 行 builder），从中提取记忆是否需要额外的 AI 调用？
- **已有逻辑**：`InsightFeedbackAggregator` 是否已经隐含了"稳定偏好"的提取逻辑？

### ARCH-6: 长期记忆候选合并规则的竞态条件 [P2] (confidence: 7/10)

方案第 3.2 节定义了候选合并规则（新候选与已有候选合并、与已拒绝记忆 30 天冷却等），但没有考虑并发场景：

- `HoloMemoryPromotionPolicy.evaluate()` 是同步还是异步？
- 如果两个来源同时生成相同候选（如洞察反馈和周期洞察同时产出），合并逻辑是否线程安全？
- JSON 文件持久化的原子写入可以保证文件不损坏，但不能保证两个并发的 `evaluate()` 不会同时读取旧状态、各自计算、后写覆盖前写

## 代码质量问题

### CODE-1: HoloMemoryModels.swift 可能成为巨型模型文件 [P2] (confidence: 7/10)

方案把所有模型放在一个文件中：

```
HoloMemoryModels.swift（11 个类型）
  - HoloShortTermMemory, HoloMemoryWindow
  - HoloMemoryDataCoverage, HoloMemoryCoverageLevel
  - HoloLongTermMemoryRecord, HoloLongTermMemoryStatus
  - HoloMemorySensitivity
  - HoloMemoryEvidence
  - HoloMemoryCandidate, HoloMemoryCandidateSource
  - HoloMemoryPromotionDecision
```

项目编码规范要求 "200-400 行典型，800 max"、"高内聚低耦合"。这个文件包含 11 个类型，估算 400-600 行，且在 Phase 4-5 还会增长。

短期记忆和长期记忆模型分属不同 Phase（Phase 3 vs Phase 4），放一起反而让 Phase 1-3 必须编译全部模型。

**建议：** 拆分为 `HoloShortTermMemoryModels.swift`（Phase 3 用）和 `HoloLongTermMemoryModels.swift`（Phase 4 用）。

---

### CODE-2: TodayStateResolver 的三档判断规则缺少量化阈值 [P3] (confidence: 6/10)

方案说 rich 需要"满足任意两类"，但：

- "有活跃目标"中的"活跃"是什么定义？`GoalRepository` 中 `isActive` 的判断标准是什么？
- partial 说"只有 1 类数据可用"，但 `HoloMemoryDataCoverage` 的 `coverageLevel` 中有 `enoughForSingleDomain`（单领域数据足够）— 这和 partial 的"只有 1 类数据"怎么区分？
- "核心数据总量都很低"中的"很低"是多少？需要给出具体数字或百分比

项目只有 6 个 AI 专项测试文件，其中 2 个是 `@main struct` 独立可执行模式（非 XCTest）。核心编排逻辑零测试覆盖：

| 核心组件 | 测试覆盖 |
|----------|----------|
| `ConversationCoordinator` | ❌ 零覆盖 |
| `UserContextBuilder` | ❌ 零覆盖 |
| `DailySenseStateBuilder` | ❌ 零覆盖 |

| `MemoryInsightService` | ❌ 零覆盖 |

方案要新增的组件（`PromotionPolicy`、`LongTermMemoryStore`）是纯规则逻辑，非常适合单元测试。但如果基础层（`UserContextBuilder` / `DailySenseStateBuilder`）没有测试，修改它们时的回归风险无法检测。

**建议：** 在修改任何现有组件之前，先为核心方法补最低限度的 smoke test（不是追求覆盖率，而是保证修改不破坏现有行为）。

### TEST-2: 记忆晋升规则需要大量边界测试 [P2] (confidence: 8/10)

方案的晋升决策伪代码有多个分支，规则之间的优先级没有显式定义。例如：

- 用户明确说"记住我压力大时更爱外卖"（source=userExplicitText, sensitivity=high）→ 先匹配 `silentlyAccept` 还是 `requireConfirmation`？
- 伪代码是 if-else 顺序执行的，但实际实现可能是 switch/enum matching，优先级不同
- `confidence < 0.65` 在最后，意味着它会在所有类型判断之后才检查，但用户明确说的内容置信度也可能低于 0.65（例如语音识别不准确时）

**建议：** 把伪代码改为显式优先级表（而不是隐含的 if-else 顺序），并为每个优先级冲突场景写一个测试用例。

## 性能问题

### PERF-1: 短期记忆缓存策略与现有系统冲突 [P2] (confidence: 7/10)

方案提出：
- today 窗口缓存 5 分钟
- sevenDays / fourteenDays / thirtyDays 缓存 30 分钟
- 用户新增/编辑/删除数据时相关缓存失效

但缓存失效的触发机制没有说明。现有系统的缓存策略：
- `UserContextBuilder`：**无缓存**，每次实时查询
- `MemoryInsightService`：hash + version 缓存 + 相似度去重
- `InsightPreferenceProfileService`：内存缓存 + 原子写入 JSON
- `HoloBackendPromptService`：2 分钟 metaTTL

如果给短期记忆加缓存，但底层 Repository 数据变化时没有通知机制，缓存会返回过时数据。

当前 `UserContextBuilder` 选择"无缓存"可能不是偷懒 — Core Data 的 `NSFetchedResultsController` 和 `@Published` 已经在 View 层提供了数据响应性。在 Service 层加缓存需要额外的失效机制。

---

### PERF-2: 长期记忆 JSON 文件的读写频率 [P3] (confidence: 5/10)

方案说长期记忆用 JSON 文件，但：
- 每次进入 HoloAI 都会触发 snapshot build，需要读取长期记忆
- 每次候选评估都写入 JSON
- 如果用户频繁进出 HoloAI 页面，JSON 的读写频率会很高

`InsightPreferenceProfileService` 用了类似的 JSON 模式，但写入频率很低（只在反馈时）。长期记忆的读写频率可能高得多。

## 方案遗漏的关键问题

### MISSING-1: 记忆注入的 Token 成本预算

方案说"最多 5 条长期记忆"，但没有估算每条多少 token。如果每条摘要 40-80 token + 证据摘要 20-40 token，5 条就是 300-600 token。

这在 Chat 的 system prompt 中可能可接受，但在 MemoryInsight 的 token 预算（日 800）中占比极高。

**建议：** Chat 和 Insight 的记忆注入策略应该分开 — Chat 可以注入更多，Insight 需要更精简或跳过。

---

### MISSING-2: HoloProfile 与长期记忆的关系

`HoloProfile` 是用户主动编写的个人档案（8KB Markdown），它是最高权重的"用户说的关于自己的话"。方案的长期记忆中也包含 `explicitPreference` 和 `importantLifeFact`，这些和 HoloProfile 可能重叠。

当用户在 HoloProfile 中写了"我正在备考"，同时又对 AI 说"记住我正在备考"，会生成两条记忆吗？去重规则中没有覆盖这种跨系统重叠的场景。

---

### MISSING-3: 新手引导的展示判断逻辑不够具体

方案说"完成至少 3 类基础输入"或"累计有效记录达到阈值"后隐藏新手引导，但：

- "3 类"是哪 3 类？记账 + 习惯 + 任务？还是任意 3 种？
- "阈值"是多少？需要根据实际用户数据分布来确定
- 判断逻辑放在哪里？`HoloAICapability` 模型中还是 ViewModel 中？

---

### MISSING-4: 能力入口的点击行为缺少错误处理

方案的"Capability Workflows"只描述了 happy path。没有说明：

- AI Provider 未配置时点击"今日状态"怎么办？
- 网络不可用时点击"最近分析"怎么办？
- Core Data 查询失败时怎么降级？

现有 `ConversationCoordinator.process()` 有完整的错误处理（`shouldStreamChat` fallback、低置信度检查），但新入口是独立工作流，需要自己的错误处理。

---

### MISSING-5: 回答底部的"来源面板"如何实现

方案提到回答底部展示来源（"基于：最近 7 天记录 · 当前目标 · 已确认长期偏好"），但现有 `MessageBubbleView` 没有"可展开的来源面板"。这需要新的 UI 组件，且方案中没有放在任何 Phase 中。

---

### MISSING-6: "不提供更多"的可用性风险

方案明确不提供"更多"入口，把基础录入能力只放在新手引导中。但成熟用户如果忘记了自然语言指令（"直接说打卡跑步"），且新手引导已隐藏，他们如何回到基础录入？

是否需要在某个地方保留一个可发现但非核心的入口？例如在输入框的 placeholder 文案中轮播提示。

## 关键发现 1：记忆注入的最优路径是 AIUserContextMessageBuilder，不是 PromptManager

**代码证据** (`HoloBackendAIProvider.swift:198-206`)：

```swift
private func buildChatMessages(messages: [ChatMessageDTO], userContext: UserContext) async -> [ChatMessageDTO] {
    let systemPrompt = await loadManagedPrompt(.systemPrompt)       // 第一层：后端 Prompt
    var allMessages: [ChatMessageDTO] = [
        .system(systemPrompt),
        .system(AIUserContextMessageBuilder.build(from: userContext, purpose: .chat))  // 第二层：用户上下文
    ]
    allMessages.append(contentsOf: messages)                        // 第三层：对话历史
    return allMessages
}
```

system prompt 由三层构成。记忆摘要应该注入**第二层**（用户上下文消息），原因：

1. **不改 PromptManager 模板** — 避免了 Prompt 双端同步（iOS + 后端 defaultPrompts.json + Docker 重建）
2. **不改 ConversationCoordinator 签名** — 记忆通过 UserContext 自然流入，`process(text:userContext:provider:)` 零改动
3. **自动覆盖所有 Provider 调用点** — `parseUserInputBatch`、`chat`、`chatStreaming`、`buildChatMessages` 四处全部受益
4. **语义更正确** — 记忆是用户上下文的一部分，不是 system instruction

**具体改动范围**：
- `AIModels.swift` — UserContext 加 `dataCoverage: DataCoverageLevel?` 和 `memorySummary: String?`（Optional + 默认 nil，不破坏现有初始化）
- `UserContextBuilder.swift` — `buildContext()` 中构建新字段
- `AIUserContextMessageBuilder.swift` — `build(from:purpose:)` 中增加记忆 section 渲染

这三个文件的改动是**纯加法**（新增字段/逻辑，不修改现有代码），总改动约 40-60 行。

## 关键发现 3：DailySenseStateBuilder 不应被扩展

第一轮建议扩展 DailySenseStateBuilder 加 rich/partial/empty 三档。**第二轮推翻这个建议。**

**代码证据** (`DailySenseStateBuilder.swift`)：

```swift
// 当前逻辑：衡量的是"行为状态"，不是"数据丰富度"
// 任务信号: riskScore += 1.0 (3+ 逾期)
// 习惯信号: riskScore += 0.8 (2+ 断连)
// 恢复信号: recoveryScore += 1.0 (1+ 恢复打卡)
// 消费信号: riskScore += 0.6 (偏离均值 > 1.5x)
// 最终: atRisk / recovering / stable
```

DailySenseStateBuilder 衡量的是**用户的行为风险状态**（"今天是否出了问题"），而 rich/partial/empty 衡量的是**数据覆盖度**（"今天有没有足够数据做判断"）。这是两个正交维度：

```
               数据充足        数据不足
行为稳定     stable + rich    stable + empty (无意义判断)
行为风险     atRisk + rich    atRisk + partial (有限结论)
```

**正确做法：** 新建独立的 `DataCoverageEvaluator`，在 `UserContextBuilder.buildContext()` 中同步调用，产出 `DataCoverageLevel` 存入 UserContext。DailySenseStateBuilder 保持不变。

这也解决了 ARCH-4（Phase 排序依赖）的问题 — `DataCoverageLevel` 可以在 Phase 2 中独立定义，不需要等 Phase 3。

## 关键发现 4：InsightPreferenceProfile 的 isStable 机制可直接复用

**代码证据** (`InsightPreferenceProfile.swift` + `InsightFeedbackAggregator.swift`)：

```
InsightModulePreference {
    module: String
    weight: Double
    isStable: Bool          // ← 已有"是否稳定偏好"标记
    evidenceCount: Int      // ← 已有证据计数
```

- `InsightPreferenceProfile` 继续服务洞察排序（`InsightCardReranker`）
- 长期记忆系统在初始化或定期同步时，从 Profile 的 stable 字段提取候选
- 不需要两套独立的反馈聚合管道

## 已有 Bug 发现

### BUG-1: InsightFeedbackAggregator 的过期检查永远不生效 (confidence: 9/10)

**代码位置：** `InsightFeedbackAggregator.swift:161-166`

```swift
// 计算"现在到上次更新"的天数差
let components = calendar.dateComponents([.day], from: now, to: now)  // ← BUG: from 和 to 都是 now
let daysSinceUpdate = components.day ?? 0  // 永远是 0
if daysSinceUpdate > 30 { ... }  // 永远不触发
```

`dateComponents([.day], from: now, to: now)` 中 `from` 和 `to` 都是 `now`，结果永远是 0 天。弱信号永不过期，会随时间积累。

**影响：** `InsightPreferenceProfile` 中的弱信号只增不减，虽然不影响长期记忆方案，但会在 Phase 4 提取 stable 候选时引入噪声。

**修复建议：** 改为 `from: preference.lastUpdatedAt, to: now`。在长期记忆方案实施前或 Phase 4 前修复。

### BUG-2: DailySenseStateBuilder 的健康信号始终为空 (confidence: 8/10)

**代码位置：** `DailySenseStateBuilder.swift`

健康信号通过 Feature Flag 控制，当前返回空数组。这意味着今日状态的健康维度始终缺席。

**影响：** 方案 Phase 2 的 rich 档判断中"有健康信号"这一条件始终不成立。需要在实施时决定是否启用这个 Feature Flag，或从 rich 条件中移除健康信号。

### BUG-3: UserContextBuilder 每次调用都重新查询所有 Repository (confidence: 7/10)

**代码位置：** `UserContextBuilder.swift`

`buildContext()` 每次进入 HoloAI 对话都触发 5-7 个 Repository 查询（交易、习惯、任务、想法、账户、趋势、目标），无任何缓存。

**影响：** 如果能力入口每次点击都调用 `buildContext()`（方案的工作流暗示如此），频繁进出或切换入口会导致大量重复查询。方案的 5 分钟缓存策略有道理，但实现方式需要谨慎选择。

**当前设计的合理性：** `UserContextBuilder` 无缓存可能是刻意的 — Core Data 查询在内存中很快（数据量小），缓存引入的复杂性（失效、一致性）可能不划算。建议实测 `buildContext()` 耗时后再决定是否加缓存。

## 未解决的新问题

### NEW-1: MemoryInsightService 的 @MainActor 限制

`MemoryInsightService` 标记为 `@MainActor`，长期记忆候选评估如果涉及重量级操作，会阻塞主线程。

**建议：** 候选评估逻辑放在独立 Service 中（`HoloMemoryPromotionPolicy` 不需要 @MainActor），从 `generateInsight` 的 return 路径上通过 `Task.detached` 触发。

### NEW-2: 通知缺口的修复方式

需要新增 `.memoryInsightDidGenerate` 通知或等效机制。两种方案：
- **A)** 在 `MemoryInsightRepository.saveReady()` 后 post Notification
- **B)** 在 `generateInsight` return 前直接调用 `HoloMemoryPromotionPolicy.evaluate()`

方案 B 更简单，但会让 `MemoryInsightService` 依赖 `HoloMemoryPromotionPolicy`，增加耦合。方案 A 通过通知解耦，但增加一行 Notification 代码。

### NEW-3: GoalPlanningPromptBuilder 的记忆注入语义

`GoalPlanningPromptBuilder.questionPrompt(session:userContext:)` 和 `draftPrompt(session:userContext:)` 都接收 `UserContext`。如果 UserContext 加了 `memorySummary` 字段，可以直接在模板中引用。

但需要注意：目标规划是"引导式对话"（追问 → 草案 → 确认），记忆注入应该出现在**草案生成**阶段而不是**追问**阶段。追问阶段应聚焦于收集用户信息，不受已有记忆干扰。
