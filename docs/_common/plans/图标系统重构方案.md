# 图标系统重构方案

> 日期：2026-06-14
> 范围：财务侧（支出）分类图标 + 圆形底色占比 + 渲染组件统一
> 状态：待 review

---

## 1. 背景与问题

### 1.1 用户反馈
财务与习惯图标"丑、看不懂"，具体枚举（用户确认是系统性问题，不止这几个）：
- 餐饮图标整体太小、占比低；早/午/晚餐识别度差；夜宵用月亮
- 居住像"锁"（实为房租 `key.fill` 钥匙在小尺寸下被看扁）
- 烟酒像"零/云"（`smoke.fill` 烟雾）

### 1.2 全量审计结论（86 个支出分类图标）
| 风险等级 | 数量 | 说明 |
|---|---|---|
| 🔴 高风险（必须重做） | 17 | 4 自绘 + 8 语义错位 + 5 重复混淆 |
| 🟡 中风险 | 10 | 易混/重复/抽象 |
| 🟢 低风险 | 59 | 语义清晰 |

### 1.3 根因（用户顾虑"SF Symbols 效果差"的真正原因）
经实测渲染对比验证，**根因不是 SF Symbols 本身**，而是两个工程问题：

1. **占比仅 30%**（头号元凶）：`Category+Icon.swift:24/27` 的 `size * 0.6` 折算 + 各调用点 Circle 尺寸硬编码。分类网格中 48pt 圆配 24pt 入参 → 经 `*0.6` 后图标仅 14.4pt，占比约 30%。
   - **验证**：同一 `fork.knife`，占比 30% vs 60% 并排渲染，用户确认"60% 确实没那么丑了"。
2. **选词错误**：夜宵用月相、烟酒用烟雾、旅行用走路人——不是 SF Symbols 没有好图标，是 symbol 名选错了。

### 1.4 SF Symbols 的真实局限（已穷举验证）
`tooth` / `tooth.fill` / `cigarette` / `cigarette.fill` 系列 **系统均不存在**。
→ 仅"牙齿保健""烟酒"这两个分类受此局限（本轮摘出，见 §8）。
→ 其余 15 个高风险分类均能在 SF Symbols 找到贴切替代。

---

## 2. 方案总览

**技术路线**：纯 SF Symbols 重选（用户已选定，顾虑已通过 §1.3 验证消除）。

三件事：
1. **抽统一 `IconBadge` 组件** —— 解决占比不一致（§3）
2. **15 个分类图标重映射** —— SF Symbol 重选 + 自绘换 + 重复差异化（§4-6）
3. **数据迁移** —— 图标名是持久化字符串，改名必须 migration（§7）

> §4-6 按"问题类型"组织，部分分类会跨类（如房租既是重复混淆、也需重选）。总目标是 §1.2 的 17 个高风险中、除 §8 摘出的「牙齿保健」「烟酒」外的全部。

**本轮不合并财务/习惯两套系统**：本轮只抽财务侧统一组件 + 重选财务图标；习惯侧整体重构留作第二阶段。

## 3. 核心架构：统一 IconBadge 组件

### 3.1 现状问题
无统一组件，8+ 调用点各自 `ZStack { Circle().fill(); icon }` 硬编码：

| 调用点 | Circle | 图标 | 占比 |
|---|---|---|---|
| `TransactionCategoryGrid.swift:218/253` | 48 | 14.4 | **30%** ← 偏小 |
| `CategoryPicker.swift:307` | 64 | 26.4 | 41% |
| `IconPickerGrid.swift:79` | 64 | 22（未乘 0.6） | 34% |
| `HabitCardView.swift:129` | 56 | 24 | 43% |
| ... | | | 不一致 |

底色 opacity（0.1/0.12/0.15/0.25）也不统一。

### 3.2 目标组件
新增 `Views/Common/CategoryIconBadge.swift`（文件名待定，遵循 200-400 行规范）：

```swift
struct CategoryIconBadge: View {
    let iconName: String          // SF Symbol 名 或 holo.category.* 自绘名
    let colorHex: String          // 分类颜色
    let size: CGFloat             // 圆形直径（逻辑点）
    
    private let iconRatio: CGFloat = 0.58   // 单一占比常量（替代 size*0.6）
    private let bgAlpha: Double = 0.12      // 单一底色透明度
    
    var body: some View {
        Circle()
            .fill(Color(hex: colorHex).opacity(bgAlpha))
            .frame(width: size, height: size)
            .overlay {
                CategoryIconRenderer.icon(iconName, size: size * iconRatio, color: colorHex)
            }
    }
}
```

- **单一占比常量 0.58** 替代散落的 `size * 0.6` 和硬编码
- 内部调用统一的 `CategoryIconRenderer`（由现有 `transactionCategoryIcon` / `categoryIconGlyph` 整理而来），屏蔽 SF Symbol vs 自绘的差异
- 替换 §3.1 全部调用点

### 3.3 自适应
SF Symbols 自动支持深色模式、Dynamic Type、Hierarchical 渲染——统一组件继承这些能力，无需额外适配。

占比统一 0.58 后，这 8 个语义错位的分类换成贴切 symbol：

|---|---|---|---|
| 夜宵 | `moonphase.waning.crescent` | `cup.and.saucer.fill` | 月相→咖啡杯，有热饮/小吃感 |

| 过路费 | `building.columns.fill` | `road.lanes` | 希腊建筑→车道，与"路"挂钩 |
| 保健品 | `leaf.fill` | `pill.fill` | 叶子→药丸，更贴切 |
| 美妆 | `sparkles` | `wand.and.stars` | 抽象闪光→魔法棒，具象 |
| 娱乐（一级） | `music.note.list` | `theatermasks.fill` | 音乐偏窄→戏剧面具，娱乐范围广 |

> 以上 8 个 symbol 均已通过 `NSImage(systemSymbolName:) != nil` 验证存在。

## 5. B 类 · 自绘换 SF Symbol（4 个，实施时视觉定）

当前 4 个 `holo.category.*` 是手画 SwiftUI Path（`Category+Icon.swift:43-219` 的 BreakfastFallbackShape 等），识别度差。换成 SF Symbol。

| 分类 | 旧（自绘） | 新 SF Symbol（候选） | 备注 |
|---|---|---|---|
| 早餐 | `holo.category.breakfast` | `takeoutbag.and.cup.and.straw.fill` | 外带杯，早餐外食感 |
| 午餐 | `holo.category.lunch` | _待定_ | 难点：SF Symbol 无"午餐"细分，候选 `fork.knife.circle.fill` 需验证 |
| 晚餐 | `holo.category.dinner` | _待定_ | 难点：同上 |
| 水果 | `holo.category.fruit` | `carrot.fill` 或 `leaf` | 果蔬通用 |

**难点说明**：早/午/晚餐本质都是"吃饭"，SF Symbols 没有细分符号（这也是当初用自绘的原因）。午/晚餐的最终符号需在实施阶段用浏览器/模拟器视觉迭代确定，可能接受"几个用相近的餐具符号 + 靠分类名区分"。

## 6. C 类 · 重复图标差异化（4 组，实施时视觉定）

多个分类共用同一 symbol 导致无法区分：

| 组 | 分类（当前共用） | 差异化方案 |
|---|---|---|
| 酒杯三连 | 饮品 / 酒水 / 请客（均 `wineglass`） | 饮品 `cup.and.saucer.fill`（热饮，保持）、酒水 `wineglass.fill`（保持）、请客 → `person.2.fill` 或聚餐符号 |
| 闪光二连 | 美妆 / AI工具（均 `sparkles`） | 美妆已换 `wand.and.stars`（§4），AI工具独占 `sparkles` → 自动解决 |
| 礼物二连 | 购物·礼物 / 人情·送礼（均 `gift.fill`） | 购物·礼物 `gift.fill`（保持）、送礼 → `shippingbox.fill` 或 `bag.fill` |
| 警告二连 | 交通·违章罚款 / 其他·罚款（均 `exclamationmark.triangle.fill`） | 违章保持、罚款 → `yensign.circle.fill` 或 `creditcard.fill` |

> 部分新 symbol 需实施时验证存在性 + 视觉确认。

## 7. 数据迁移（关键风险点）

**图标名是 Core Data 持久化字符串**（`Category.icon`）。改名/换图标必须走迁移，否则老用户图标变空白。

### 7.1 迁移机制
复用现有 `migrateLegacyIcons` 模式（v1/v2/v3，见 `Category+CoreDataProperties.swift`），新增 **v4 迁移**：
- 旧 icon 名 → 新 icon 名 的映射表
- 仅对**系统默认分类**（`isSystem == true` 且 icon 匹配旧名）执行，**不动用户自定义分类**
- 幂等：已迁移的不再处理（靠 migration flag）

### 7.2 v4 映射表（草案）
```
moonphase.waning.crescent → cup.and.saucer.fill     // 夜宵
figure.walk → airplane.departure                      // 旅行
building.columns.fill → road.lanes                    // 过路费（注意：building.columns 也是其他分类，需按 parentName 匹配）
leaf.fill → pill.fill                                 // 保健品（注意：leaf 可能多处用）
...
holo.category.breakfast → <新 SF Symbol>              // 自绘迁移
holo.category.lunch → <新>
holo.category.dinner → <新>
holo.category.fruit → <新>
```

**注意**：`building.columns.fill`、`leaf.fill`、`sparkles` 等被多个分类使用，迁移必须 **按 (parentName, name) 组合匹配**，不能只按 icon 名全局替换。

### 7.3 自绘名清理
4 个 `holo.category.*` 迁移到 SF Symbol 后：
- `Category+Icon.swift` 的自绘分支（`categoryIconGlyph` 中的 `HoloFallbackCategoryIcon`）保留作为兜底，但默认分类不再使用这些名
- 4 个 FallbackShape 可标记 `@available(*, deprecated)`，下一阶段清理

## 10. 风险与对策

| 风险 | 对策 |
|---|---|
| 迁移误伤用户自定义分类 | v4 仅匹配 `isSystem == true` + (parentName, name) 组合 |
| 重名 icon（building.columns 等）全局替换出错 | 按 (parentName, name) 匹配，非按 icon 名 |
| SF Symbol 版本要求（部分需 iOS 16+） | 实施时确认 deployment target，新 symbol 验证最低支持版本 |
| 8+ 调用点替换遗漏 | 实施时 grep 所有 `ZStack.*Circle` + `transactionCategoryIcon` 调用，逐一替换为 IconBadge |
