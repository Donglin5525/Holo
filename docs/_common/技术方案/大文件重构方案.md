# 大文件重构方案：风险分析与执行计划

## Context

HOLO 项目有 10 个 Swift 文件超过 800 行阈值，最大的 `CoreDataStack.swift` 达 1,784 行。大文件降低可维护性、增加认知负担、拖慢代码审查。本次重构将它们拆分为 <800 行的职责化文件。

**核心约束**：纯重构，不改变任何业务逻辑。每次拆一个文件，编译通过后立刻提交。

---

## 风险分析

### 风险 1：`private` 跨文件扩展失效 (严重性：中)

**问题**：Swift 允许同一文件内的 extension 访问 `private` 成员，但拆到不同文件后只能访问 `internal` 及以上。

**影响范围**：
- `CoreDataStack.swift` — `private func create*Entities()` 等 9 个方法（含 `createDataModel()`、`buildContainer()`）
- `FinanceRepository.swift` — `private lazy var context` + 5 个 private 方法（`getTransactionCount`、`guessCategoryIconAndColor`、`guessAccountType`、`reloadCaches`、`notifyDataChange`）
- `HabitRepository.swift` — `private lazy var context` + 统计 extension 内 9 个 private 辅助方法（`convertToHabitDateRange`、`calculateCheckInCompletionRate`、`calculateNumericCompletionRate`、`orderedHabitsForStats`、`makeMonthCells`、`makeBadHabitMonthCells`、`makeWeekSlices`、`statsCardKind`、`statsSummary`）
- `TodoRepository.swift` — `private var context` + `private func notifyDataChange`

**缓解**：拆分前先将相关 `private` 改为默认访问级别（`internal`）。项目是单 target app，无框架边界，`internal` 是合理选择。逐项清单见下方执行计划。

### 风险 2：Core Data Schema 回归 (严重性：高)

**问题**：`CoreDataStack.swift` 包含整个 Core Data 模型的编程式定义（1,600+ 行实体描述）。拆分时任何遗漏或拼写错误都会导致运行时崩溃或数据丢失。

**缓解**：拆分后必须验证：
1. App 正常启动（无 "store incompatible" 错误）
2. 每个模块能正常读写数据
3. 可选：对比拆分前后的 `entityVersionHashesByName` 确认 schema 一致

**额外注意**：`createMemoryInsightEntity()` 返回**单个** `NSEntityDescription`（非数组），`createDataModel()` 调用它时**不带** `contentsOf`。拆分时不要改成数组签名。

### 风险 3：@State 跨 View 绑定出错 (严重性：中)

**问题**：`AddTransactionSheet`（20 个 @State）和 `AddTaskSheet`（33 个 @State）的表单区块拆分时，需要把 @State 作为 @Binding 传给子 View，容易遗漏导致状态不同步。

**缓解**：View 层拆分放在最后（Phase 3），只做最简单的提取（独立类型如 `TaskCardView`、`KeypadButton`）。复杂的表单区块拆分需要逐个追踪每个 @State 的读写关系。

### 风险 4：类型跨文件引用断裂 (严重性：低)

**问题**：`FinanceView.swift` 中定义的 `TransactionRowView` 被 `FinanceSearchView`、`DetailTabView` 等外部文件引用。`TaskListView.swift` 中的 `TaskCardView` 被 `TagListView`、`TaskSearchView` 引用。

**缓解**：项目使用 `fileSystemSynchronizedGroups`，新文件放入目录自动加入编译目标，无需手动编辑 pbxproj。拆分后全局编译即可捕获所有引用错误。

### 风险 5：无功能收益的时间投入 (严重性：低)

**问题**：重构不产生用户可见功能，但需要 2-3 小时投入。

**缓解**：按风险从低到高分 3 个阶段执行。每个阶段可独立暂停/恢复。优先处理收益最大的文件。

## 每次拆分的操作流程

1. 读取原文件，确认结构
2. 修改访问级别（按上方清单逐项 `private` → `internal`）
3. 创建新扩展文件，剪切对应代码
4. 全局编译：`xcodebuild -project Holo.xcodeproj -scheme Holo -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`
5. 确认所有文件 < 800 行
6. 提交：`git add` → `git commit` → 更新 CHANGELOG → `git push`

## 总结

| 阶段 | 文件数 | 风险 | 预估耗时 |
|------|--------|------|---------|
| Phase 1 基础层 | 4 → 16 文件 | 低 | ~60 min |
| Phase 2 简单 View | 3 → 10 文件 | 低 | ~45 min |
| Phase 3 复杂 View | 3 → 8+ 文件 | 中高 | ~60 min |
| **合计** | **10 → 34+ 文件** | — | **~2.5 hr** |
