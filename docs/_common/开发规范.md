## 📝 核心开发原则

### 0️⃣ 全局中文化（必读）

**原则**：Holo 是一款面向中文用户的 APP，**所有用户可见的文本内容必须使用简体中文**。

**强制要求**：
- ✅ **界面文本**：所有按钮、标签、标题、标题、占位符、提示语、错误信息必须为简体中文
- ✅ **日期格式**：使用 `zh_CN` locale，例如「2026 年 3 月 7 日 星期六」
  - **禁止** 使用 `Text(date, style: .date)` / `Text(date, style: .time)` — 依赖系统语言，英文设备会显示英文
  - **禁止** 使用 `date.formatted(.dateTime.xxx())` — 同上，依赖系统 locale
  - **必须** 使用 `DateFormatter` + `formatter.locale = Locale(identifier: "zh_CN")`
- ✅ **货币格式**：默认人民币（¥），`NumberFormatter` 使用 `currencyCode: "CNY"` + `locale: zh_CN`
- ✅ **代码注释**：使用中文注释，符合项目注释规范
- ❌ **禁止英文 UI 文本**：不得在界面中出现 "Expense"、"Income"、"Today's Ledger" 等英文文案
- ❌ **禁止混合语言**：同一界面中不得中英混用（品牌名 "Holo" 除外）

**检查清单**（每次新增页面/组件时必须确认）：
- [ ] 所有 `Text(...)` 内容为中文
- [ ] 所有 `TextField(placeholder:)` 占位符为中文
- [ ] 所有 `Button` 标签为中文
- [ ] 日期/时间使用 `Locale(identifier: "zh_CN")`
- [ ] 未使用 `Text(date, style:)` 或 `date.formatted(.dateTime)`（必须用 DateFormatter + zh_CN）
- [ ] 货币金额使用 `NumberFormatter.currency`（已配置为人民币）
- [ ] `NavigationTitle` / `alert` / `confirmationDialog` 均为中文

### 0.1️⃣ iOS 页面布局规范（含底部固定栏）

> 来源于财务页（FinanceView）布局优化，适用于「顶部 Header + 中间可滚动内容 + 底部固定 Tab/按钮」的页面。

**滚动条隐藏**
- 所有 `ScrollView` 必须设置 `showsIndicators: false`，不显示滚动指示器
- 适用于横向和纵向滚动视图

**顶部 Header**
- Header 与返回键应**贴近顶部安全区**，避免上方留白过大。
- 推荐：`.padding(.top, 12)`，不再使用 40pt 以上的大数值（如 56）。

**底部固定栏（Tab 栏 / 浮动按钮）**
- 使用 **`safeAreaInset(edge: .bottom, spacing: 0)`** 将整块底部栏固定在屏幕最底部（等效于 fixed bottom-0）。
- 主内容无需手动计算底部留白，系统会自动避开 inset 区域。
- Tab 栏需**适配 iOS 底部安全区**（Home Indicator）：用 `GeometryReader` 读取 `geo.safeAreaInsets.bottom`，在栏内底部使用 `16 + geo.safeAreaInsets.bottom` 等 padding；背景可用 `.ignoresSafeArea(edges: .bottom)` 延伸至屏幕最底。

**浮动主按钮（如记账加号）**
- 放在 `safeAreaInset` 的底部栏内，位于 Tab 栏**正上方**、**水平居中**（`HStack { Spacer(); Button; Spacer() }`），与对应 Tab 图标垂直对齐。

**中间内容区**
- 采用**上固定、下滚动**：`VStack { header; 固定卡片; ScrollView { 列表 }.frame(maxHeight: .infinity) }`。
- 中间列表区域用 `ScrollView` + `.frame(maxHeight: .infinity)` **占满剩余高度**并单独滚动，保证内容可自适应且不顶到固定栏。

### 0.2️⃣ 财务页 Tab 栏与底部栏约定（必读）

> 以下为财务模块（FinanceView）已确定的 UI 约定，后续开发与迭代须遵守。

**Tab 栏吸底**
- 底部导航使用 **fixed bottom-0 left-0 w-full**，不设圆角、不设额外 bottom 外边距，背景贴满屏幕最底部。
- 使用 `safeAreaInset(edge: .bottom, spacing: 0)` 实现固定贴底；栏内 **pb = env(safe-area-inset-bottom)**，可用 `max(geo.safeAreaInsets.bottom, 20)` 做 fallback；**pt-2**（约 8pt）作为栏内顶部留白。
- 背景使用 `.ignoresSafeArea(edges: .bottom)` 填满至物理底部，图标与文字保持在安全区上方。

**Tab 三等分与中间「账本/记一笔」合一**
- 底部三个 Tab 严格**三等分**：统计 | 中间 | 设置，每个 `.frame(maxWidth: .infinity)`。
- **中间一格**与加号合一：
  - **在记账页（账本）时**：中间显示**红色圆形 + 号**，文案「记一笔」，点击弹出记一笔 Sheet。
  - **在统计或设置页时**：中间显示**账本图标**，文案「账本」，点击切回账本页。
- 不再使用悬浮 FAB overlay，记一笔入口仅通过中间 Tab 进入。

**视觉层次**
- Tab 栏增加**顶部弱投影**：`shadow(color: .black.opacity(0.05), radius: 10, x: 0, y: -2)`，与上方列表区分边界。

**顶部 Header**
- Header 容器**无多余 pt/mt**，使用 `.padding(.top, 0)`，标题与返回键紧贴状态栏（灵动岛）下方，由系统安全区控制。

**交易记录列表行**
- 每行使用 **flex 等价布局**：分类名称与金额 **justify-between items-center**，金额在右侧严格对齐（`Spacer(minLength: 0)` + 金额 `fixedSize` + `frame(alignment: .trailing)`）。

## 🧩 iOS / Core Data 常见坑总结（写代码前必读）

> 本节来源于真实 debug 过程，属于**高频踩坑集合**。
> 每次写 iOS + Core Data 相关代码前，请先快速过一遍这一页。

### 1️⃣ Core Data 模型与代码定义不一致

**必须保证三处完全对齐**：
- **数据模型**：`NSManagedObjectModel` / `.xcdatamodeld` 中的字段名、类型、是否可空
- **实体类**：`@NSManaged` 属性定义
- **实际赋值代码**：Repository 中对属性的赋值逻辑

**典型错误 1：属性类型不匹配**
- 模型：`amount` 使用 `decimalAttributeType`
- 代码：`@NSManaged public var amount: Double` ❌
- 正确：`@NSManaged public var amount: NSDecimalNumber` ✅

**典型错误 2：关系未在模型中声明**
- 代码里使用 `transaction.category`
- 模型里只有 `categoryId`，没有 `category` 的 `NSRelationshipDescription`
- 结果：`unrecognized selector setCategory:` 崩溃

**典型错误 3：多余的必填字段不赋值**
- 模型里保留了必填的 `categoryId`，但代码只给 `category` 关系赋值
- 结果：保存时报 `Multiple validation errors occurred.`
- **规范**：要么删掉冗余字段，要么同时给 ID 和关系赋值

---

### 2️⃣ Core Data 并发与上下文使用

**禁止在错误线程访问 `NSManagedObject`**
- `EXC_BAD_ACCESS` 往往意味着在**非创建线程**访问了某个托管对象

**当前项目约定**：
- 所有 Repository 操作标记为 `@MainActor`
- 只用 **`viewContext`** 做增删改查

---

### 3️⃣ Core Data 报错信息快速排查

**`unrecognized selector setXxx:`**
- 检查模型里是否有 `xxx` 这个字段/关系
- 检查实体类是否用对了类型（尤其是 `NSDecimalNumber`）

**`Multiple validation errors occurred.`**
- 这是保存时的**字段校验失败汇总**
- 检查非可选属性/关系是否有值

### 4️⃣ SwiftUI 列表不刷新的典型问题

**症状**：保存成功后，列表看不到新数据，必须返回再进入一次。

**规范做法**：
```swift
// FinanceView 中
@State private var transactions: [Transaction] = []

.task { await loadTransactions() }

// Sheet 关闭时重新加载
.onChange(of: showAddTransaction) { _, isShowing in
    if !isShowing {
        Task { await loadTransactions() }
    }
}
```

**通用规范**：任何「弹出编辑/新增页面」的场景，**关闭时必须触发一次数据刷新**。

### 5️⃣ 分类图标（SF Symbols）

**图标来源**：
- 分类图标已从自定义 SVG 迁移为 Apple SF Symbols
- 旧图标映射表：`Category+CoreDataProperties.swift` → `legacyIconMapping`

**SF Symbol 名称验证（强制）**：
- **禁止**假设 SF Symbol 名称存在，**必须先验证再使用**
- 验证方法：`NSImage(systemSymbolName: "xxx", accessibilityDescription: nil) != nil`（macOS）或 `UIImage(systemSymbolName: "xxx") != nil`（iOS）
- Apple 的 SF Symbols 应用中不存在的名称，在运行时会渲染为空白
- 任何新增或修改 SF Symbol 名称时，**必须先跑验证脚本**：
```bash
swift -e 'import AppKit; print(NSImage(systemSymbolName: "你的图标名", accessibilityDescription: nil) != nil ? "✅" : "❌")'
```

**踩坑记录**（2026-04-11）：
- `apple.meditation`、`lipstick`、`couch.fill` 这三个名称**不存在于 SF Symbols 库**，导致水果/美妆/家具分类图标显示为空白
- 原因：迁移时只做了名称映射的数量覆盖度检查，没有验证名称是否真实存在
- 修复：`apple.meditation` → `carrot.fill`，`lipstick` → `sparkles`，`couch.fill` → `sofa.fill`

**多 target / 多 Assets**：
- 确认当前 **Run 的 target** 使用的是哪一个 `Assets.xcassets`
- 若有多个 xcassets，**每个都要有 CategoryIcons**

### 7️⃣ NSManagedObject 删除后访问崩溃

> **⚠️ 高频踩坑**：此问题反复出现多次。**删除 NSManagedObject 前必须先从 @State 数组中移除，否则 EXC_BREAKPOINT。**

**核心规则：先移除本地引用，再调用 context.delete + save。**

`context.save()` 后对象从 store 中删除，Core Data 无法再 fulfill fault。如果 `@State` 数组仍持有该对象引用，SwiftUI 下一次渲染时读取属性（`name`、`color` 等）会直接 EXC_BREAKPOINT。

#### 场景 A：列表内直接删除（swipe / 按钮触发）

```swift
// ✅ 正确：先从本地数组移除，再执行 Core Data 删除
private func confirmDelete(_ category: Category) {
    guard !category.isDeleted else { return }

    // 1️⃣ 先从本地数据移除 — 防止 SwiftUI 渲染已删除对象
    items.removeAll { $0.objectID == category.objectID }
    // 如果有子项映射，也要同步清理
    subItemsMap.removeValue(forKey: category.id)

    // 2️⃣ 再执行 Core Data 删除（内含 context.delete + save）
    Task {
        do {
            try await repository.deleteCategory(category)
        } catch {
            errorMessage = error.localizedDescription
        }
        await loadData()  // 3️⃣ 重新加载保证数据一致性
    }
}

// ❌ 错误：直接删除，不移除本地引用
Task {
    try await repository.delete(category)  // context.save() 后对象已删除
    // → 此时 items 数组仍持有已删除对象
    // → SwiftUI 渲染 ForEach 时读取 category.name → EXC_BREAKPOINT
    await loadData()
}
```

#### 场景 B：Sheet 关闭时删除（onDismiss 回调）

```swift
// 父视图（列表页）
@State private var habits: [Habit] = []  // 本地缓存
@State private var pendingDeleteId: UUID? = nil

.sheet(isPresented: $showDetailSheet, onDismiss: {
    if let id = pendingDeleteId {
        habits.removeAll { $0.id == id }  // 先移除
        DispatchQueue.main.asyncAfter(deadline: .now() + 0.1) {
            Task { @MainActor in
                try? repository.deleteById(id)
            }
        }
        pendingDeleteId = nil
    }
}) { ... }
```

#### 场景 C：其他视图可能持有已删除对象的引用

删除操作完成后，检查所有可能持有该对象引用的 `@State` 变量：

```swift
/// 清理已删除对象的引用（在 loadCategories / loadData 后调用）
private func cleanStaleRefs() {
    let validIDs = Set(items.map { $0.objectID })
    if let selected = selectedItem, !validIDs.contains(selected.objectID) || selected.isDeleted {
        selectedItem = nil
    }
    if let parent = drillDownParent, !validIDs.contains(parent.objectID) || parent.isDeleted {
        drillDownParent = nil
    }
}
```

## 9. AI 对话模块

### 9.1 Prompt 管理

| 规则 | 说明 |
|------|------|
| 使用 Swift 内嵌模板 | Prompt 模板必须内嵌在 `PromptManager.swift` 的字典中，**禁止使用 JSON 文件** |

| 规则 | 说明 |
|------|------|
| 必须用 KeychainService | API Key 存储必须使用 `KeychainService`（`kSecAttrAccessibleWhenUnlockedThisDeviceOnly`） |
| 禁止 UserDefaults | 禁止将 API Key、Token 等敏感信息存入 UserDefaults |
| 禁止硬编码 | 禁止在源代码中出现任何 API Key |

### 9.3 流式响应

| 规则 | 说明 |
|------|------|
| AsyncThrowingStream | 使用 `URLSession.bytes` + `AsyncThrowingStream` 实现 SSE 流式接收 |

| 渲染策略 | 流式中渲染纯文本 + 闪烁光标，完成后切换 Markdown 渲染 |

### 9.4 Provider 抽象

| 规则 | 说明 |
|------|------|
| AIProvider 协议 | 所有 LLM Provider 必须实现 `AIProvider` 协议 |

### 9.5 意图路由

| 规则 | 说明 |
|------|------|
| IntentRouter 单例 | 意图 → Repository 操作的映射集中在 `IntentRouter` |
| 置信度判断 | 高置信度（≥0.7）直接路由执行，低置信度进入对话模式 |
| 分类匹配 | 记账分类通过 `CategoryMatcherService` 匹配，不硬编码 |

## 10. 首次启动卡死 / Core Data 死锁排查规范

> 来源于 2026-05 真实踩坑：首次启动卡死修了 4 次都没好，根因不在 CoreDataStack 本身，而在整条数据管道的线程安全问题。

### 10.1 正确的 Core Data 异步加载架构

**三层协作模式**（缺一不可）：

```
HoloApp.init()
  └→ CoreDataStack.shared.prepareIfNeeded()    // 仅触发，不阻塞
       └→ persistentContainer lazy init
            └→ loadPersistentStores(async)     // 后台加载 store

HomeView.task {                               // .task 在首帧渲染后执行
  await CoreDataStack.shared.waitUntilReady()  // 挂起协程等 store 就绪
  repo.setup()                                 // 安全访问 Core Data
}

ChatMessageRepository.loadMessagesAsync()
  await CoreDataStack.shared.waitUntilReady()  // 同样先等 store
  Task.detached {                              // 后台线程查数据
    context.perform { ... }
  }
}
```

### 10.2 六个必须遵守的规则

| # | 规则 | 说明 | 违反后果 |
|---|------|------|----------|
| 1 | `shouldAddStoreAsynchronously = true` | Store 异步加载，不阻塞调用线程 | 设 `false` → 后台线程同步阻塞或主线程死锁 |
| 2 | 用 `CheckedContinuation` 等待就绪 | `waitUntilReady()` 挂起协程而非阻塞线程 | `DispatchSemaphore.wait()` 在 cooperative pool 中会导致线程饥饿 |
| 3 | 值类型全部标 `nonisolated` | `AIIntent`、`AIParseBatch`、`ChatMessageViewData` 等跨 actor 使用的类型必须 `nonisolated` | 缺少标注 → Swift 隐式插入 MainActor hop → 主线程等 Core Data → 死锁 |
| 4 | 后台加载用 `dictionaryResultType` | `NSFetchRequest<NSDictionary>` + `.dictionaryResultType`，不直接 fetch managed objects | 后台线程访问 `NSManagedObject` 是线程安全违规 |
| 5 | 值类型提供 `init(dictionary:)` | 从 `[String: Any]` 构造值类型，脱离 managed object | 用 `init(message:)` 读取 managed object 属性要求在 context 线程上 |
| 6 | Repository `init()` 零 I/O | lazy context + deferred `setup()` | init 中 fetch → `@StateObject` 在 body 求值时同步创建 → 阻塞主线程 |

### 10.3 卡死排查 Checklist

遇到「首次启动卡死 / 白屏 / 黑屏」时，按以下顺序检查：

- [ ] `CoreDataStack.buildContainer()` 是否 `shouldAddStoreAsynchronously = true`
- [ ] `waitUntilReady()` 是否用 `CheckedContinuation`（而非 semaphore / sleep / while 循环）
- [ ] 所有在后台线程使用的 struct/enum 是否标记 `nonisolated`
- [ ] 后台 fetch 是否使用 `dictionaryResultType`，而非直接 fetch managed objects
- [ ] 值类型是否有 `init(dictionary:)` 路径（不依赖 managed object）
- [ ] 所有 Repository 的 `init()` 是否零 I/O
- [ ] `HoloApp.init()` 是否只调用 `prepareIfNeeded()`（不等待加载完成）
- [ ] `HomeView.task` 是否先 `await waitUntilReady()` 再做数据操作

### 10.4 错误做法 vs 正确做法对比

```swift
// ❌ 错误：同步加载 + 信号量 + 非 Sendable 值类型

    await withCheckedContinuation { ... }
}
nonisolated struct ChatMessageViewData: Sendable { ... } // nonisolated
```

## 11. Swift Charts 坐标系与触摸交互（必读）

> 来源于 2026-05 真实踩坑：图表手势错位修了 3 轮未果，根因是搞混了 `ChartProxy` 的坐标空间。

### 11.1 核心规则：`proxy.position(forX:)` 返回 plot area 局部坐标

**绝对不要把 `proxy.position(forX:)` / `proxy.position(forY:)` 当作全局坐标使用。**

| API | 返回的坐标空间 | 说明 |
|-----|---------------|------|
| `proxy.position(forX:)` | **plot area 局部** | 以 plot area 左上角为原点 (0, 0) |
| `proxy.position(forY:)` | **plot area 局部** | 同上 |
| `proxy.value(atX:)` | 不推荐使用 | 对分类轴（String label）坐标映射不可靠，**禁止使用** |
| `DragGesture.value.location` | **GeometryReader local** | 以 GeometryReader 左上角为原点，覆盖整个图表含轴标签 |
| `proxy.plotFrame` → `geometry[$0]` | **GeometryReader local** | plot area 在 GeometryReader 中的位置 |

### 11.2 坐标转换公式

```
触摸 → plot area:   touchXInPlot = value.location.x - plotFrame.minX
plot area → overlay: overlayX    = plotFrame.minX + proxyX
```

**正确用法**（手势查找最近数据点）：
```swift
// ✅ 正确：先把触摸点转换到 plot area 坐标系，再和 proxy.position 比较
let touchXInPlot = value.location.x - plotFrame.minX
let pointPositions = dataPoints.compactMap { proxy.position(forX: $0.label) }
// 找最小距离的点...
```

**错误用法**（之前导致偏移的写法）：
```swift
// ❌ 错误：把触摸点当全局坐标，加 globalFrame.minX 反而偏了
let globalX = value.location.x + globalFrame.minX  // 多加了一层偏移！
proxy.value(atX: globalX)  // 这个 API 对分类轴也不可靠
```

### 11.3 Tooltip 定位同理

```swift
// proxy.position 返回 plot area 坐标，加 plotFrame.minX 回到 overlay 坐标
let localX = plotFrame.minX + xPos  // ✅
let localY = plotFrame.minY + yPos  // ✅
```

### 11.4 触摸命中工具

使用 `ChartTouchSelection.nearestPointIndex()`（已实现在 `FinanceAggregation.swift`），参数全部为 plot area 局部坐标：

```swift
ChartTouchSelection.nearestPointIndex(
    touchXInPlot: touchXInPlot,       // plot area 局部 X
    plotWidth: plotFrame.width,        // plot area 宽度
    pointXPositions: pointPositions    // proxy.position(forX:) 返回值数组
) -> Int?
```

### 11.5 排查 Checklist（图表触摸/定位异常时）

- [ ] 是否混用了全局坐标和 plot area 局部坐标？
- [ ] `proxy.value(atX:)` 是否用于分类轴？→ 改用 `proxy.position(forX:)` + 手动最近点

### 11.6 Canvas 自绘饼图/环形图规范（必读）

> 来源于 2026-05 统计分析类别饼图真实踩坑：灰色/红色覆盖、页面无法滚动、悬停金额丢失反复修复，根因分别是选中态 dim、SwiftUI 拖拽手势抢占 ScrollView、极小扇区固定间隔角反向成整圈。

**颜色策略**
- 类别统计页的饼图和图例必须共享同一组图表调色板，不能直接依赖 `category.swiftUIColor`。
- 导入数据后，多个新建/父子分类可能共享同一个默认色或父级色；如果饼图按分类色绘制，会出现整图近似单色。
- 鼠标悬停、手势滑动、选中扇区时，不要用全局 dim 覆盖其他扇区；如需突出选中项，优先使用扇区外移、描边或中心金额变化。

**扇区路径**
- 给扇区添加间隔角时，间隔角必须按扇区自身角度动态收缩。
- 禁止用固定 `1.5°` 直接扣所有扇区。0.3% 这类极小扇区只有约 `1.08°`，固定间隔角会让 `startAngle/endAngle` 反向，Canvas 可能把小扇区画成接近整圈，覆盖整张图。
- 推荐规则：`inset = min(preferredInset, spanAngle * 0.45)`，确保 `spanAngle - inset > 0`。

**手势与滚动**
- `ScrollView` 内的自绘图表禁止使用 `DragGesture(minimumDistance: 0)` 抢占整块图表区域。
- 需要同时支持「点击选中」「横向/悬停查看金额」「纵向页面滚动」时，优先使用 UIKit 透明触摸层：
  - `UITapGestureRecognizer` 负责点击选中，`cancelsTouchesInView = false`
  - `UIPanGestureRecognizer` 只在横向位移大于等于纵向位移时开始追踪高亮
  - 纵向拖动直接让外层 `ScrollView` 接管
  - `shouldRecognizeSimultaneouslyWith` 返回 `true`

**回归测试清单**
- [ ] 选中/悬停一个扇区后，其他扇区 opacity 仍为 `1.0`
- [ ] 类别统计页使用图表调色板，不直接使用分类色
- [ ] 横向移动可更新中心分类/金额，纵向拖动可滚动页面
- [ ] 极小扇区 `spanAngle - inset > 0`，不会反向成整圈

## 12. UIImagePickerController + fullScreenCover 异步数据处理（必读）

> 来源于 2026-05 真实踩坑：拍照添加附件卡死修了 6 轮未果，根因有三个，缺一不可。

### 12.1 三条铁律

| # | 规则 | 说明 | 违反后果 |
|---|------|------|----------|
| 1 | **立即转 Data** | UIImagePickerController delegate 中拿到 UIImage 后，必须立即 `jpegData(compressionQuality:)` 转为 `Data`，**不要传 UIImage 引用出 delegate** | UIImage 的 backing data 可能绑定 picker 内部 photo pipeline，picker dismiss 后后台线程渲染该 UIImage 会隐式等待主线程 → 死锁 |
| 2 | **用 onDismiss 延迟处理** | `fullScreenCover(isPresented:onDismiss:content:)` 中，重量级操作（Core Data save + loadActiveTasks + @Published cascade）放在 `onDismiss` 回调里，**不要放在 `onChange` 里** | onChange 在 dismiss 动画期间触发，Core Data 操作 + SwiftUI re-render 与 dismiss 动画冲突 → 卡死 |
| 3 | **统一走 Data 路径** | 所有图片保存统一用 `addAttachment(imageData:to:)`，让 `saveImageDataInBackground` 在后台解码+压缩+写文件 | `addAttachment(image:to:)` 把 UIImage 传给 `Task.detached`，如果 UIImage 的 backing data 有主线程依赖，后台渲染会卡住 |

### 12.2 正确模式

```swift
// ✅ 正确：delegate 中立即转 Data
class Coordinator: UIImagePickerControllerDelegate {
    let onCapture: (Data) -> Void  // 传 Data，不传 UIImage

    func imagePickerController(_, didFinishPickingMediaWithInfo info: _) {
        if let image = info[.originalImage] as? UIImage,
           let data = image.jpegData(compressionQuality: 0.85) {
            onCapture(data)   // picker 还活着，数据完全提取
        }
    }
}

// ✅ 正确：fullScreenCover onDismiss 中处理
.fullScreenCover(
    isPresented: $showCamera,
    onDismiss: {
        guard let data = pendingImageData else { return }
        pendingImageData = nil
        Task {
            try await repository.addAttachment(imageData: data, to: task, sourceType: "camera")
        }
    }
) {
    CameraView(onCapture: { data in
        pendingImageData = data          // 存 Data
        showCamera = false               // 触发 dismiss
    })
}
```

### 12.3 错误模式（导致卡死）

```swift
// ❌ 错误 1：传 UIImage 出 delegate
CameraView(onCapture: { image in   // UIImage 引用绑定 picker 生命周期
    pendingImage = image
    showCamera = false
})

// ❌ 错误 2：onChange 中处理（dismiss 动画期间执行）
.onChange(of: pendingImage) { _, newValue in
    Task {
        try await repository.addAttachment(image: image, to: task)  // context.save 与 dismiss 动画冲突
    }
}

// ❌ 错误 3：走 UIImage 路径而非 Data 路径
repository.addAttachment(image: image, ...)  // Task.detached 渲染 UIImage → 可能卡主线程
```

### 12.4 排查 Checklist（fullScreenCover + UIKit 组件卡死时）

- [ ] delegate 回调是否立即将 UIKit 对象转为值类型（UIImage → Data）？
- [ ] 重量级操作是否放在 `fullScreenCover` 的 `onDismiss` 而非 `onChange`？
- [ ] 是否统一走已验证可行的 Data 路径，而非混用 UIImage 和 Data？
- [ ] 后台 `Task.detached` 中是否使用了可能依赖主线程的 UIKit 对象？

## 13. Core Data 关系建模与数据刷新规范（必读）

> 来源于 2026-05 真实踩坑：删除分类导致整个交易模块卡死，记账保存失败，修复了 4 轮未果。
> 根因有三：① 缺少反向关系 ② denyDeleteRule 放错方向 ③ refreshAllObjects 破坏 SwiftUI 渲染。

### 13.1 反向关系是强制要求

**每条 NSRelationshipDescription 必须有对应的 inverseRelationship。** Core Data 没有反向关系时，save 操作的对象图一致性校验行为未定义，可能表现为：
- `context.save()` 卡死（无限循环校验）
- `context.save()` 抛出未知验证错误
- 删除/新增后 UI 不更新（通知链断裂）

```swift
// ❌ 错误：只有单向关系
let categoryRelation = NSRelationshipDescription()
categoryRelation.name = "category"
categoryRelation.destinationEntity = categoryEntity
// 没有 inverseRelationship

// ✅ 正确：双向关系
let categoryRelation = NSRelationshipDescription()
categoryRelation.name = "category"
categoryRelation.destinationEntity = categoryEntity

let categoryTransactionsRelation = NSRelationshipDescription()
categoryTransactionsRelation.name = "transactions"
categoryTransactionsRelation.destinationEntity = transactionEntity

categoryRelation.inverseRelationship = categoryTransactionsRelation
categoryTransactionsRelation.inverseRelationship = categoryRelation
```

实体类也必须有对应的 `@NSManaged` 属性：

```swift
// Transaction.swift（to-one 侧）
@NSManaged public var category: Category

// Category.swift（to-many 侧）
@NSManaged public var transactions: Set<Transaction>?
```

### 13.2 denyDeleteRule 的方向性

**`denyDeleteRule` 放在 to-many 侧（反向关系），to-one 侧用 `nullifyDeleteRule`。**

| 侧 | 关系 | deleteRule | 说明 |
|----|------|-----------|------|
| Transaction.category | to-one | `nullifyDeleteRule` | 交易删除时，分类不受影响 |
| Category.transactions | to-many | `denyDeleteRule` | 有交易引用时，禁止删除分类 |

```swift
// ✅ 正确
categoryRelation.deleteRule = .nullifyDeleteRule        // to-one 侧
categoryTransactionsRelation.deleteRule = .denyDeleteRule // to-many 侧

// ❌ 错误：deny 放在 to-one 侧
categoryRelation.deleteRule = .denyDeleteRule  // 阻止 Transaction 本身保存/删除！
```

**为什么 to-one 侧的 denyDeleteRule 会卡死**：
- `denyDeleteRule` 意味着「如果有对象引用目标，就拒绝删除目标」
- 放在 Transaction.category 上，Core Data 在 save Transaction 时需要校验 Category 是否被其他对象引用
- 没有反向关系时，Core Data 无法完成这个校验 → 卡死或抛错

### 13.3 禁止 `refreshAllObjects()`

```swift
// ❌ 禁止：让所有已注册对象重新 fault
context.refreshAllObjects()
// SwiftUI 正在渲染的 @Published 对象会丢失属性值
// 视图重新访问属性时触发 fault → 数据可能不一致 → 显示异常

// ✅ 正确：重新 fetch 需要的数据
func refreshData() async {
    summaryCache.removeAll()
    await loadMonthSummaries(for: currentMonth)
    await loadSelectedDayData()
    await loadPreviousPeriodComparison()
}
```

### 13.4 数据变更后的刷新模式

**增删改后，必须用 `await` 顺序刷新，禁止 fire-and-forget。**

```swift
// ✅ 正确：await 保证数据刷新完成后再继续
private func deleteTransaction(_ tx: Transaction) {
    Task {
        do {
            try await repository.deleteTransaction(tx)
            await calendarState.refreshData()  // await 等待刷新完成
        } catch { ... }
    }
}

// ❌ 错误：fire-and-forget，UI 可能在数据刷新前就渲染了旧数据
private func deleteTransaction(_ tx: Transaction) {
    Task {
        try await repository.deleteTransaction(tx)
        calendarState.refreshAfterDataChange()  // Task { await refreshData() } — 异步不等待
    }
}
```

### 13.5 排查 Checklist（数据变更后页面不更新 / 模块卡死时）

- [ ] 所有 NSRelationshipDescription 是否都有 `inverseRelationship`？
- [ ] `denyDeleteRule` 是否放在 to-many 侧（反向关系），to-one 侧是否用 `nullifyDeleteRule`？
- [ ] 是否使用了 `refreshAllObjects()`？→ 改为重新 fetch
- [ ] 数据变更后的刷新是否用 `await`（而非 fire-and-forget Task）？
- [ ] to-many 侧的实体类是否有 `@NSManaged public var transactions: Set<Transaction>?`？
- [ ] `isDeleted` 检查是否只是防御手段，而非根本解决方案？

## 14. SwiftUI ScrollView 滚动交互规范（必读）

> 来源于 2026-05 真实踩坑：HoloAI 对话页「上滑加载更多」试了 3 种方案都没生效。
> 根因：LazyVStack 中的 `onAppear` 和 `GeometryReader` 在视图被回收后不再上报。

### 14.1 ScrollView 顶部「下拉加载更多」— 用 `.refreshable`

**这是 iOS 原生的下拉刷新 API，适用于任何需要在 ScrollView 顶部触发加载的场景。**

```swift
// ✅ 正确：系统级下拉刷新，可靠、自带动画
ScrollView {
    LazyVStack { ... }
}
.refreshable {
    await loadMoreData()
}
```

**为什么其他方案不行：**

| 方案 | 问题 |
|------|------|
| `onAppear` 放在 LazyVStack 顶部 item 上 | 首次渲染就触发（用户还没滚动），滚动回来后 LazyVStack 可能不重新触发 `onAppear` |
| `GeometryReader` + `PreferenceKey` 放在 LazyVStack 内部 | LazyVStack 回收不可见视图后，GeometryReader 不再上报偏移量 |
| `GeometryReader` + `PreferenceKey` 放在 LazyVStack 外部 | 零高度视图的坐标空间上报不稳定，阈值难以确定 |

### 14.2 `onAppear` 在 LazyVStack 中的行为

**`onAppear` 在 LazyVStack 中不适合做「滚动到顶部」检测。**

- LazyVStack 预渲染可见区域附近的视图，首帧就可能触发顶部 item 的 `onAppear`
- 滚走再滚回来时，LazyVStack 可能复用已有视图实例而不重新触发 `onAppear`
- 防重入 guard（如 `isLoading`）可以阻止首次误触发，但无法让「滚动回来」正确触发

**`onAppear` 在 LazyVStack 中的正确用途：**
- 触发懒加载（如 `viewModel.loadMetadataIfNeeded(for: id)`）
- 上报可见性事件（如已读标记）

### 14.3 防止初始加载误触发

使用 `.refreshable` 时，加载方法内部应加门禁防止初始渲染期间误触发：

```swift
private func triggerLoadEarlier(proxy: ScrollViewProxy) async {
    guard didInitialScrollToBottom,      // 首次滚到底部后才允许触发
          viewModel.hasEarlierSessions,
          !viewModel.isLoadingEarlierSession else { return }

    if let anchorId = await viewModel.loadEarlierSession() {
        withAnimation(.easeOut(duration: 0.15)) {
            proxy.scrollTo(anchorId, anchor: .top)
        }
    }
}
```

### 14.4 排查 Checklist（ScrollView 顶部交互不生效时）

- [ ] 是否用了 `onAppear` 做滚动位置检测？→ 改用 `.refreshable`
- [ ] 是否用了 `GeometryReader` + `PreferenceKey` 放在 LazyVStack 内部？→ LazyVStack 会回收该视图
- [ ] `.refreshable` 的闭包是否为 `async`？加载方法签名是否匹配？
- [ ] 加载方法是否防重入（guard `isLoading` / `hasEarlierSessions`）？
- [ ] 是否需要 `didInitialScrollToBottom` 门禁防止首帧误触发？

## 15. ScrollView 内手势方向锁定规范（必读）

> 来源于 2026-05 真实踩坑：左滑操作按钮在上下滚动时误触，多轮修复均未解决。
> 根因：方向判定逻辑错误 — 垂直优先否决 + 锁定后继续监控垂直 = 互相打架导致卡顿和误判。
> 最终方案：`HorizontalGestureLock`（三态锁定 + 主方向优势比），已稳定运行。

### 15.1 核心问题

在 ScrollView 内添加水平手势（左滑操作、左右切换、图表交互等）时，水平手势与 ScrollView 的垂直滚动手势产生冲突。需要**准确且一次性**地判定用户真实意图。

### 15.2 错误方案（已踩坑，禁止复用）

| 方案 | 问题 |
|------|------|
| SwiftUI `DragGesture` | 与 ScrollView 手势冲突严重，无法同时识别 |
| 垂直优先否决（v > 阈值就取消水平） | 斜向滑动时水平可能先到阈值，垂直检查白跑 |
| 水平确认后继续监控垂直 | 锁定后回头检查 → 已设置的 offset 被重置 → 卡顿 |
| 减去 `confirmedTranslationX` 补偿 | 确认瞬间有视觉跳变 |
| `gesture.state = .cancelled` 强制取消 | `.cancelled` 分支仍会执行速度逻辑，可能误触发 |

**关键教训：方向判定必须是单向的 — 一旦确认，整个手势期间不再更改。**

### 15.3 正确方案：`HorizontalGestureLock`

文件位置：`Utils/HorizontalGestureLock.swift`

**核心原理：**

1. **三态状态机**：`undecided` → `horizontal` 或 `vertical`，锁死后不回头
2. **Touch slop 观望期**：位移 < 6pt 时保持 `undecided`，避免起步抖动误判
3. **主方向优势比**：确认方向时要求该方向比另一方向强 12%（`directionDominanceRatio = 1.12`），避免 45° 附近来回争抢
4. **锁定后直接用累计位移**：不做补偿减法，无缝衔接

**参数表：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `touchSlop` | 6 | 观望期最小距离，轻触和起步抖动不判定 |
| `horizontalConfirmDistance` | 12 | 水平确认距离，偏小以减少左滑卡顿感 |
| `verticalConfirmDistance` | 10 | 垂直确认距离，略低于日历/列表切换阈值 |
| `directionDominanceRatio` | 1.12 | 主方向需比副方向强 12% |

**核心代码：**

```swift
struct HorizontalGestureLock {
    private(set) var axis: GestureAxisLock = .undecided

    mutating func reset() { axis = .undecided }

    mutating func update(translation: CGSize) -> GestureAxisLock {
        guard axis == .undecided else { return axis }
        let h = abs(translation.width), v = abs(translation.height)
        guard max(h, v) >= tuning.touchSlop else { return .undecided }
        if h >= tuning.horizontalConfirmDistance,
           h >= v * tuning.directionDominanceRatio {
            axis = .horizontal; return .horizontal
        }
        if v >= tuning.verticalConfirmDistance,
           v >= h * tuning.directionDominanceRatio {
            axis = .vertical; return .vertical
        }
        return .undecided
    }
}
```

### 15.4 标准使用模式

```swift
// 1. 声明
private var gestureLock = HorizontalGestureLock()

// 2. .began 重置
case .began:
    gestureLock.reset()

// 3. .changed 判断方向（只调一次 update）
case .changed:
    switch gestureLock.update(translation: translation) {
    case .undecided: return   // 观望中
    case .vertical:   return  // 垂直滚动，不处理
    case .horizontal: break   // 锁定水平，继续
    }
    // 确认水平后才禁用 ScrollView
    disableScrollView(gesture)
    // 直接用 translation.x，不做补偿
    let offset = ...

// 4. .ended 检查锁定状态
case .ended, .cancelled:
    enableScrollView()
    if gestureLock.axis != .horizontal {
        // 非水平手势 → 回弹，不处理速度
        return
    }
    // 处理速度和 snap 动画
    gestureLock.reset()
```

### 15.5 强制规则

- **禁止**在 `.began` 阶段禁用 ScrollView，必须等方向确认后
- **禁止**锁定后继续监控另一方向的位移（锁定不回头）
- **禁止**对 `translation` 做补偿减法
- **禁止**在 `.ended` 中无条件处理速度（必须先检查 `axis != .horizontal` 就跳过）
- **禁止**用 `gesture.state = .cancelled` 做方向取消（改用 `return` 跳过）
- 新增水平手势场景时，**必须**复用 `HorizontalGestureLock`，不要自定义方向判断
