## 1. 一句话目标

把想法编辑器的 `#` 标签和 `@` 引用从「纯文本 + 保存时正则提取」升级为「带唯一 ID 的结构化行内 Token」——输入即触发候选面板，选中后插入不可拆分的节点，关系数据以结构化内容为准重建。

## 4. 功能范围

### ✅ 第一版做

**`#` 标签**：工具栏/键盘触发候选面板 → 搜索/创建/多级路径（`#工作/Holo/编辑器`）→ 行内 Token（原子化、整删）→ 点击筛选该标签下的想法 → 重命名/删除沿用现有全局管理能力。

**`@` 引用**：工具栏/键盘触发候选面板 → 最近编辑/关键词搜索想法 → 行内 Token（含首行快照）→ 点击跳转目标想法 → 反向引用展示 → 目标删除后保留快照。

### ⏸️ 第一版不做（与原方案一致，另加 3 项）

- AI 自动打行内标签 / AI 自动插入引用（AI 归类走现有 assignment 体系，与行内 Token 无关）
- 知识关系图、引用具体段落、编辑器内完整引用卡片、标签别名、拖拽调层级
- **埋点**（无基础设施，后续单独立项）
- **编辑器内实时自动保存**（保持现有显式保存语义，见 §8.3）
- **标签详情独立页面**（复用现有筛选/知识树入口，见 §10.2）

## 5. 数据模型演进

### 5.1 Thought：双字段，content 保留

```text
Thought（现有字段保留）
+ richContentJSON: String?   // 新增，可空。Token 结构化事实源
+ firstLine: String?         // 新增，可空。@ 候选列表标题，保存时派生
  content: String            // 保留！语义=派生平文本（含 Markdown 标记、#路径、@显示文字）
```

**关键决策：不废弃 `content`。** 原方案用 `richContentJSON + plainText` 双字段，本方案让现有 `content` 直接承担 plainText 角色：

- `richContentJSON` 是 Token 身份（tagId / noteId / 快照）的**唯一事实源**
- `content` 由 JSON **派生**（保存时重新生成），供 AI 分类、全文搜索、卡片摘要、知识树、记忆画廊等全部下游继续无感使用
- `richContentJSON == nil` → 传统纯文本想法，零迁移成本
- 下游消费方**零改动**（这是保留 content 的最大收益），只需回归验证

派生示例：

```text
richContentJSON: [text("今天重新思考了 "), tag(id, "工作/Holo"), text("，和 "), reference(noteId, "标签体系设计", snapshot), text(" 有关")]
content:         今天重新思考了 #工作/Holo，和 @标签体系设计 有关
```

### 5.2 ThoughtReference：加快照

```text
ThoughtReference（现有 id/createdAt/sourceThought/targetThought 保留）
+ displayText: String?    // 插入时目标首行（@ 后显示的文字）
+ snapshot: String?       // 插入时目标正文摘要（目标删除后展示）
```

顺手修复现有缺陷：目标想法删除后 `targetThought` 被 nullify，引用行悬空且无残留信息；有快照后可展示「原记录已删除 + 历史摘要」。

### 5.3 编辑器内容节点

```swift
enum HoloContentNode: Codable {
    case text(value: String)                          // 含原始 Markdown 标记
    case tag(id: UUID, displayPath: String)           // displayPath 为快照，渲染时按 tagId 取最新名
    case reference(noteId: UUID, displayText: String, snapshot: String)
}
```

## 6. 编辑器架构：双层模型（对原方案的核心修订）

### 6.1 问题：原方案与现有重渲染管线冲突

原方案以 `NSAttributedString` 自定义属性为 Token 载体。但现有 `MarkdownTextView` 的渲染管线会**从 Markdown 文本重建整个 attributedText**——重建即销毁仅存于富文本属性中的 Token。这是 Sprint 1 的真正难点，必须在架构层解决，不能靠打补丁。

### 6.2 决策：ContentNode[] 为编辑期事实源，NSAttributedString 为纯渲染产物

```text
ContentNode[]（编辑期内存模型，唯一事实源）
    ↓ 单向渲染
NSAttributedString = Markdown样式(text节点) + Token样式(tag/reference节点)
    ↑ 用户编辑
文本变化只回写 text 节点；Token 节点不可进入、不可局部修改
```

- 每次重渲染都从 ContentNode[] 重新生成富文本，Token 样式天然不会丢失
- 保存 = 序列化 ContentNode[] → richContentJSON + 派生 content
- Token 原子化（光标不可进入、退格整删）在 Coordinator 层拦截 `shouldChangeTextIn`，与原方案一致

### 6.3 模块划分（扩展现有，非新建）

```text
Views/Thoughts/Editor/（新目录）
├── HoloTextView.swift            // 由 MarkdownTextView 演进：渲染管线改为 ContentNode 驱动
├── EditorCoordinator.swift       // 抽出现有 Coordinator，新增触发检测/Token 拦截
├── TriggerDetector.swift         // 新增：光标前向扫描 #/@ 触发上下文
├── TokenManager.swift            // 新增：Token 插入/整删/点击命中/样式
├── HoloContentNode.swift         // 新增：节点模型
├── RichContentSerializer.swift   // 新增：ContentNode[] ↔ JSON ↔ 派生 content
├── SuggestionPanelView.swift     // 新增：键盘上方候选面板（SwiftUI）
└── SuggestionSearchService.swift // 新增：标签/想法候选查询 + 150ms 防抖
```

沿用原方案的 `EditorState` 状态机（idle/composing/searchingTag/searchingReference/tokenSelected）与中文输入法规则（`markedTextRange != nil` 时不渲染、不搜索、不动 Token）——现有 Coordinator 已有同样处理，合并而非两套。

## 7. 多级标签：路径即名称（核心决策）

### 7.1 为什么不建 parentId 实体层级

原方案的 `HoloTag`（name/fullPath/parentId）是真实体层级。但 Holo 现有标签体系**全链路按名称工作**：

- AI prompt 喂的是标签名（`existingTagExamples`），AI 输出的也是名称
- `ThoughtTagNormalizer.key` 按名称归一化去重
- 拒绝偏好（UserDefaults）按名称记录
- 全局重命名/合并按名称操作（`renameTagEverywhere`）
- `Topic.associatedTagNames` 是名称字符串缓存，喂给 AI 对话工具

建实体层级意味着这条链全部重写，且 AI 无法稳定输出层级 ID。**采用「路径即名称」**：`ThoughtTag.name` 直接存完整路径 `工作/Holo/编辑器`，层级由 `/` 切分在展示/搜索层派生。与 flomo/Bear 相同做法。

- 模型改动仅一处：`ThoughtTag + lastUsedAt: Date?`（候选面板「最近使用」排序需要；usageCount 已有）
- 父级 `工作` 不强制存在实体——浏览层按路径前缀聚合即可；用户显式打 `#工作` 时它本身也是独立标签
- 升级逃生门：若未来需要拖拽调层级/父级权限继承，再立项演进为实体层级，路径名称可平滑映射

### 7.2 归一化与重命名

- `ThoughtTagNormalizer` 扩展为**逐段归一化**：`工作/Holo` 的 key = `normalize(工作)/normalize(Holo)`，防 `工作 /Holo`、`工作/holo` 产生重复路径
- 重命名 `工作` → `项目`：现有 `renameTagEverywhere` 按名称操作，需扩展为**路径前缀重命名**——`工作` 改名影响 `工作`、`工作/Holo`、`工作/Holo/编辑器` 整棵子树（逐条改 name + 拒绝偏好联动沿用现有逻辑）
- 删除标签沿用 `deleteTagGlobally`：只删精确匹配的路径，不级联删子路径（`#工作` 删除不影响 `#工作/Holo`）；是否提供「含子级删除」选项留给实施期评审

### 7.3 InlineTagDetector 扩展

现有正则 `#[\p{L}][\p{L}\p{N}_]*` **不支持 `/`**，`#工作/Holo` 只会提取出 `工作`。扩展：

- 提取正则支持段间 `/`：`#[\p{L}][\p{L}\p{N}_]*(/[\p{L}\p{N}_]+)*`
- 触发条件按原方案收紧：`#` 位于文首，或前一字符为空格/换行/中英文标点（现正则会误匹配 `abc#产品`，借本次统一）
- **提取与触发共用同一份规则实现**（原方案要求「快捷按钮和手动输入同一套逻辑」，此处同理，禁止两套正则漂移）

### 7.4 与 AI 分类/知识树的关系（第一版并存，互不干扰）

- AI 归类继续走 `ThoughtTagAssignment`（source 五态不变），AI 产出的扁平标签名=根级路径，自然兼容
- AI prompt 的 `existingTagExamples` 现在会包含路径名（`工作/Holo`），AI 可直接复用路径标签——无需改 prompt，但实施期需验证 token 增量可接受（路径比单词长）
- Topic 知识树维持现状：Topic ↔ 标签关联不变，标签路径化后 `associatedTagNames` 缓存照常重算（复用 2026-07-17 的 `refreshAssociatedTagNamesCache`）
- 用户感知的「三套标签来源」统一策略：**行内 # Token（source=inline）与手动 chip（source=manual）继续并存**，UI 上编辑器标签区收敛为只读展示「本文标签」（行内+手动+AI 归并展示），新增标签统一引导到行内 #——避免 chip 和 Token 两个新增入口并存。手动 chip 数据链路保留（存量不动），仅入口收敛

## 8. 保存、重命名与同步

### 8.1 保存机制（沿用原方案核心原则）

保存时执行（与现有 `ThoughtRepository.create/update` 合并，不另起链路）：

```text
1. ContentNode[] → richContentJSON
2. 派生 content（平文本）与 firstLine
3. 从 content 提取 # 路径标签 → getOrCreateTag + source=inline assignment（复用现有 extract 链路）
4. 从 JSON 提取 reference 节点 → 重建 ThoughtReference（含 displayText/snapshot）
5. 以当前结构化内容为准全量重建两类关系（不只做增量）
6. 同一 context 事务提交
```

第 5 点是原方案最有价值的原则：撤销/整段删除/粘贴覆盖后，关系以编辑器最终内容为准重建，不会腐化。现有 inline 标签已是此语义（保存时重新 extract），引用关系对齐到同样语义。

### 8.2 标签重命名/删除的 Token 语义增强

现有全局管理（2026-07-17 已实施）在 Token 化后语义自动增强，无需新链路：

- **重命名**：渲染层按 `tagId` 查最新路径名 → 历史想法立即显示新名（原方案 §15.1 设计不变）；`content` 派生文本允许短暂滞后，在该想法下次保存时刷新（接受此滞后，不批量改写 N 条想法污染 CloudKit 同步与 updatedAt）
- **删除**：`deleteTagGlobally` 删实体后，Token 按 tagId 查不到 → 渲染时降级为快照文字的普通文本样式（`#工作/Holo` 保留文字、失去链接），与现有「删除不改写正文」边界一致，且不再有「编辑复活」问题（Token 已结构化，不依赖正文正则）

### 8.3 保存时机：保持显式保存

不引入原方案的 500~800ms 自动保存。现有编辑器是「保存/取消」表单语义，自动保存会让「取消」失去意义。Token 插入/删除仍走现有 `hasUnsavedChanges` + 保存按钮 + 退出自动保存路径。App 进后台保存沿用现有生命周期处理（若无则本次补上，范围仅限编辑器页）。

### 8.4 同步（CloudKit 现实）

- 新增字段（`richContentJSON`/`firstLine`/`lastUsedAt`/`displayText`/`snapshot`）全部是普通 Core Data 属性，走 lightweight migration + CloudKit 镜像，**无自建队列、无 syncStatus**
- 冲突语义：同一条想法多端同时编辑 = 记录级字段 LWW（CloudKit 默认），`richContentJSON` 与 `content` 在同次保存中同事务写入，不会出现 JSON 新、文本旧的撕裂
- 原方案 `ReferenceStatus.syncing` 状态删除；`unavailable`（目标尚未同步到本机）保留——按 tagId/noteId 查不到且非已删除时显示「暂时无法读取」

## 10. 阅读态与下游

### 10.1 阅读态渲染

`ThoughtDetailView` / `ThoughtCardView` 渲染 Token 的规则：

- 有 `richContentJSON` → 用同一套 ContentNode→NSAttributedString 渲染管线（只读模式，复用 `RichContentSerializer`，禁止另写一套 SwiftUI 拼装）
- 无 JSON → 走现有 Markdown 渲染，零变化
- 卡片摘要保持纯文本（读 `content`），不渲染 Token 样式，控制列表性能

### 10.2 标签浏览入口

- 阅读态点击 # Token → 现有想法列表按该标签路径筛选（`ThoughtsView` drawerSelection 体系已支持按标签筛选，扩展支持路径前缀匹配「含子级」）
- 知识树抽屉标签池展示路径标签时按 `/` 缩进分组（展示层派生，不动数据）

### 10.3 引用删除态

目标 `isSoftDeleted` 或实体不存在 → Token 灰色 + 「原记录已删除」，点击展示 snapshot 快照。「从回收站恢复」入口依赖回收站 UI——若无现成恢复入口，第一版只提供「查看历史摘要 / 删除当前引用」。

### 10.4 编辑器「引用区」收敛

行内 @ 上线后，`ThoughtEditorView` 底部的引用区 + `ReferenceSelectorView` sheet 入口**移除**（统一走行内 @，避免两套新增入口）；`references` 数据关系不变，详情页反向引用展示（`referencedBy`）保留并补充「被 N 条记录引用」列表 UI。

## 11. 存量数据迁移

**策略：零迁移，惰性升级。**

- 存量想法 `richContentJSON == nil`，`content` 原样，阅读和编辑走现有链路
- 编辑存量想法：加载时 content → 单 text 节点；正文中已有的 `#标签` 字符串**保持纯文本，不自动升级为 Token**（避免静默改变用户数据）；保存时若用户未触碰，仍按现有 extract 链路建关系，行为与今天完全一致
- 用户新打的 #/@ 才是 Token，保存后该想法转为 JSON 格式
- `firstLine` 为可空派生字段，@ 搜索对存量想法回退为取 content 首行（查询时计算，不批量回填）

## 12. 复制、粘贴、撤销（沿用原方案）

- 复制到外部：Token 复制为 `#工作/Holo` / `@标题` 平文本（不携带 ID）
- Holo 内部剪贴板：自定义 UTI `com.holo.rich-content` 携带节点 JSON
- 外部粘贴一律按纯文本进 text 节点，不自动 Token 化
- Undo：Token 插入/整删/搜索词替换为 Token 各为一次完整 undo 单元（`UITextView.undoManager` 分组）；注意重渲染管线参与后 undo 需以 ContentNode 模型快照为单位，不能只对 attributedText 做 diff——实施期 Sprint 1 验证

## 15. 决策点（评审用）

| # | 决策点 | 推荐 | 备选 |
|---|---|---|---|
| 1 | 多级标签实现 | 路径即名称（name 存全路径，展示层派生层级） | 实体 parentId 层级（重写名称链路，工作量大） |
| 2 | content 字段 | 保留为派生平文本，下游零改动 | 废弃 content 改 plainText（全部下游改字段名） |
| 3 | 编辑期事实源 | ContentNode[] 模型，富文本为渲染产物 | 富文本属性为源（与 Markdown 重渲染管线冲突） |
| 4 | 保存时机 | 保持显式保存/退出自动保存 | 500~800ms 防抖自动保存（改变取消语义） |
| 5 | 手动标签 chip 入口 | 收敛为只读展示，新增统一走行内 # | chip 与行内 # 双入口并存 |
| 6 | 存量正文 #标签 | 不自动升级 Token，惰性演进 | 全量迁移（静默改写用户数据） |
| 7 | 编辑器引用区 | 行内 @ 上线后移除 | 保留 sheet 为第二入口 |

## 16. 验收标准

- [ ] 输入 `#`/`@` 或点工具栏按钮，候选面板稳定触发；中文拼音组合期间不触发、不跳光标
- [ ] 多级标签 `#工作/Holo/编辑器` 可创建、搜索（按段名/路径前缀）、插入；Token 不可拆开编辑、退格整删
- [ ] Markdown 加粗/列表与 Token 在同一正文中样式共存，重渲染后 Token 不丢
- [ ] 标签重命名后，历史想法的行内 Token 即时显示新路径；删除后保留文字失去链接
- [ ] `@` 引用插入后点击跳转目标想法；目标删除后展示「原记录已删除 + 快照」不崩溃
- [ ] 反向引用：目标想法详情页可见「被 N 条记录引用」并可跳转来源
- [ ] 撤销一次 = 撤销整个 Token 插入，恢复搜索原文
- [ ] 存量想法（无 JSON）阅读/编辑/AI 分类/搜索零回归
- [ ] iCloud 多端：标签/引用关系随实体同步，无自建队列代码
- [ ] 核心逻辑（序列化器、提取正则、归一化、保存重建关系、快照）单测覆盖，`test_sim` 通过（注意 build_sim 不含测试 target）
