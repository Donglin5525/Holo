## 1. 背景与目标

### 1.1 背景

观点模块当前是「单条 AI 打标签」（`ThoughtOrganizationService.organizeThought`）。本方案把它升级为「笔记列表 + 左侧知识树抽屉 + 渐进主题归并」。完整设计见关联 spec。

### 1.2 三期交付目标

| 期 | 目标 | 后端 | 预估 |
|---|---|---|---|
| **P1** | 左侧抽屉骨架、AI 标签池、未归类、AI 整理预告入口跑通；Topic 区空态 | 零后端 | 2-3 天 |
| **P1.5** | Topic 本地闭环：Repository/Service、按 Topic 筛选、来源词主源、收纳判断、右边缘关闭、手动移入移出、幂等合并 | 零后端 | 4-6 天 |
| **P2** | 后端跨观点收敛：`thought_tag_convergence` purpose、归并确认 UI、建议级拒绝、iCloud 冲突 | 含后端 | 1-2 周 |

> P1 / P1.5 不改 Core Data schema。P2 的建议级拒绝实体涉及 schema migration，单独评估。

### 1.3 非目标

见 spec §3。关键：不做领域实体层级、不做全屏右滑手势、AI 不静默改用户标签、不做拖拽重排。

## 4. 数据与查询设计（P1/P1.5 不改 Core Data schema）

### 4.1 不新增实体/字段

P1、P1.5 完全复用现有 `Topic` / `ThoughtTag` / `ThoughtTagAssignment` / `Thought` 及其关系。`associatedTagNames`(string) 与 `associatedTags`(relationship) 共存——定 `associatedTags` 为查询主源，`associatedTagNames` 仅展示派生（spec 决策 16）。

### 4.2 AI 标签池聚合查询（P1 新增）

现有 Repository 无此能力，新增（TDD）：

```swift
// ThoughtRepository.swift 新增
struct AITagBucket: Identifiable {
    var id: String { tagName }
    let tagName: String
    let assignmentCount: Int          // 命中该标签的 .ai/.confirmedAI assignment 数
    let sourceBreakdown: [String: Int] // {"ai": n, "confirmedAI": m}
}

/// 聚合 AI 标签池：按 source ∈ [.ai, .confirmedAI] 且 rejectedAt == nil 的 assignment 聚合
/// - P1：不排除已收纳（无 Topic 收纳关系）
/// - P1.5：excludeAbsorbed = true 时排除三者交集命中的 assignment（见 4.3）
func fetchAITagBuckets(excludeAbsorbed: Bool = false) throws -> [AITagBucket]
```

### 4.3 收纳判断三者交集（P1.5）

判断某 assignment 是否「已被主题收纳」（spec §6.3 展示规则）：

```text
assignment.thought ∈ activeTopic.thoughts
AND assignment.tag ∈ activeTopic.associatedTags
AND assignment.source ∈ [.ai, .confirmedAI]
AND assignment.rejectedAt == nil
```

不是只看 `tag ∈ Topic.associatedTags`（会误伤同名标签）。

### 4.4 主主题展示层取（P1.5）

数据层 `Thought.topics` 多对多不动。展示时取主主题：

```swift
// TopicService 新增
/// 取一条观点的主展示主题：thoughts.count 最高的 active Topic；无则 nil
func primaryDisplayTopic(for thought: Thought) -> Topic?
```

不用 `updatedAt`（iCloud 跨设备时钟不可靠）、不用缓存 `thoughtCount`。

### 4.5 Topic 幂等（P1.5，不加 schema）

```swift
// TopicService 新增
/// 归一化标题作为运行时查重键（不持久化）
static func normalizedKey(title: String) -> String {
    title.trimmingCharacters(in: .whitespacesAndNewlines).lowercased()
}

/// 创建前先按 normalizedKey 查同名 Topic，命中复用，避免重复
func getOrCreateTopic(title: String, sourceTerms: [String]) throws -> Topic
```

CloudKit 同步后若出现重复 Topic，用现有 `mergedToTopic/mergedFromTopics` 关系合并（spec 决策 17）。

### 4.6 抽屉节点模型（DrawerNode）

抽屉每个可点节点统一用 `DrawerNode` 表达，作为「右侧列表筛选意图」的载体（Phase 1.4 起接入 `ThoughtListView`）：

```swift
/// 抽屉节点类型（右侧列表的筛选意图载体）
enum DrawerNode: Hashable {
    case allNotes          // 全部笔记
    case unclassified      // 未归类（未进入任何 Topic）
    case aiTag(String)     // AI 标签池某标签（tagName）
    case topic(UUID)       // 某主题（topicId，P1.5）
    case aiOrganize        // AI 整理入口（非筛选，触发预告）
}
```

### 4.7 抽屉 → 列表筛选映射

| DrawerNode | 筛选作用 | P1 落地方式 |
|---|---|---|
| `.allNotes` | 不加筛选 | 清空 `selectedTagName` / `currentFilters` / `drawerSelection` |
| `.unclassified` | 仅 `thought.topics.isEmpty` | P1 Topic 表空 → 等价全部；新增 `topics.isEmpty` 谓词为 P1.5 就位 |
| `.aiTag(name)` | 命中该 tag 的 `.ai/.confirmedAI` assignment 的观点 | **新增基于 assignment 的筛选查询**，**不复用** `selectedTagName`/`fetchByTag`（二者走 `Thought.tags`，无 source 维度，见 §3-#12） |
| `.topic(id)` | 该 Topic 下观点 | P1.5 `fetchThoughts(byTopic:)` |
| `.aiOrganize` | 非筛选 | 弹「待整理 N 条 / 功能开发中」（Phase 1.3） |

> ⚠️ 抽屉筛选与 chip 标签筛选**互斥**：选抽屉节点清空 chip 的 `selectedTagName`；反之点 chip 清空 `drawerSelection`。
> ⚠️ Phase 1.2 实施前确认：`.ai` assignment 是否同步写入 `Thought.tags`（仅影响点 AI 标签后筛选覆盖率验证；聚合查询本身统一走 assignment 不变）。

## 9. 风险与回滚

| # | 风险 | 缓解 |
|---|---|---|
| 1 | 抽屉 overlay 与 `swipeBackToDismiss` 手势竞争 | P1 关闭仅用点击（不用手势）；抽屉用 `.transition(.move)`，手势隔离；P1.5 右边缘手势仅抽屉打开时挂载 |
| 2 | AI 标签池聚合跨线程访问 NSManagedObject | 聚合在 Repository context 队列执行，返回值类型（`AITagBucket`）不持有 NSManagedObject |
| 3 | `getOrCreateTag` 提权影响面 | 仅 private → internal，无行为变化；P1.5 前确认调用方 |
| 4 | Topic 多对多 + iCloud 计数漂移 | `thoughtCount` 实时算不缓存（决策 14）；展示主主题用 `thoughts.count` |
| 5 | 同步后重复 Topic | Phase 1.5.7 `mergeDuplicateTopics` 兜底；强幂等留 P2 |
| 6 | 三者交集判断性能（大量 assignment × Topic） | 加索引/谓词优化；P1.5 数据量小，先实现后测性能 |
| 7 | P2 后端 prompt 未同步部署 | Phase 2.1 有明确部署步骤 + 验证 curl（CLAUDE.md 双端同步流程） |
| 8 | P2 schema migration（拒绝实体）风险 | 单独评估，参考 `docs/_common/plans/2026-05-17-Holo-iCloud-CloudKit同步实施方案-定稿.md` |

**回滚**：P1/P1.5 均为本地改动，无 feature flag 强需求；如抽屉引入回归，回退菜单按钮 + 抽屉文件即可，不影响现有观点列表逻辑（抽屉是叠加层）。
