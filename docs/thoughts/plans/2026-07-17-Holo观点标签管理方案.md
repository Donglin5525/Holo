## 1. 一句话目标

让用户能对观点标签做**全局编辑（重命名/合并）和全局删除**——特别是 AI 自动打出的碎标签，看到不顺眼的，一处操作、全部想法生效，且 AI 以后不再打。

---

## 2. 背景：现在缺什么

### 2.1 已有能力（单条想法粒度）

| 能力 | 入口 | 实现 |
|---|---|---|
| 手动标签加/删 | 编辑器 TagInputView | `ThoughtRepository.update(_:tags:)` 双写 |
| AI 标签逐条确认 ✓ | 详情页 AI 归类区 | `confirmTagAssignment`（source: ai → confirmedAI） |
| AI 标签逐条拒绝 ✗ | 详情页 AI 归类区 | `rejectTagAssignment`（source → rejectedAI）+ UserDefaults 拒绝偏好（`rejectedAITags`，50 条/90 天，喂 prompt 防再生） |

### 2.2 缺失能力（全局粒度）

| 缺失 | 现状 | 后果 |
|---|---|---|
| 全局删除标签 | `deleteTag(_ name:)` 只在「无任何想法使用」时才允许删（否则抛 `tagInUse`） | 等于废的。AI 打的碎标签只能逐条想法 ✗，几十条想法要点几十次 |
| 全局重命名标签 | 无接口。`mergeTags(oldName:newName:)` 存在但**无调用方且有 bug**（只迁移 `thoughts` 关系，assignments 被 cascade 删除，新标签在知识树里丢失归属） | AI 把「工作」打成「工作事业」，用户想统一改名，做不到 |
| 改颜色 | `ThoughtTag.color` 字段和 `ColorPickerPopover` 组件都存在，但**无入口**（死代码） | 标签颜色全部是默认色 |

### 2.3 数据模型（现状，无需迁移）

- `ThoughtTag`：name / color / usageCount；关系 `thoughts`（多对多 nullify）、`assignments`（一对多 cascade）、`associatedTopics`（多对多 nullify）
- `ThoughtTagAssignment`：source ∈ {manual, inline, confirmedAI, ai, rejectedAI}、rejectedAt
- `Topic.associatedTagNames`：逗号拼接的**字符串缓存**（从 associatedTags 派生，重命名/删除标签后会变脏）
- `usageCount`：只在 `getOrCreateTag` 时 +1，从不递减 → 已是失真值，仅用于排序

## 3. 范围

### ✅ 本次做

1. **全局删除标签**：一处删除，所有想法上摘掉；纯 AI 标签额外写入拒绝偏好防再生
2. **全局重命名标签**：改名；目标名已存在时自动合并（修复 mergeTags 的 assignments 迁移 bug）
3. **入口**：知识树抽屉（ThoughtKnowledgeDrawerView）标签池行长按菜单 —— 用户看到碎标签的地方就是管理的地方

### ⏸️ 本次不做（后续立项）

- 独立「标签管理」页（搜索、批量多选删除、批量合并）：等长按菜单验证使用频率后再立项
- 改颜色入口：ColorPickerPopover 已存在，接入成本低，但本次不扩散范围
- 正文 `#标签` 文本的同步清理：删除标签不改写想法正文（见 §6.3 边界说明）

## 4. 设计

### 4.1 全局删除：统一语义 + 拒绝偏好防再生

**核心决策（对抗评审修订）**：无论标签来源（manual / inline / confirmedAI / ai），全局删除**统一写 UserDefaults 拒绝偏好**。理由：

- 「删除」是用户的最新意志，效力高于历史来源。用户曾确认过又删掉，以删除为准
- 拒绝偏好只喂 AI prompt（「用户已拒绝标签：X」），**不拦截**用户手动添加和正文 `#标签` 提取——用户想再用这个名字随时可用，无副作用
- 按来源区分两套语义，删除确认文案也要分两套，产品和代码都更复杂，统一后一句话说清：「从 N 条想法移除，AI 以后不再推荐」

**删除动作**：删除该 tag 全部 assignments → 删除 ThoughtTag 实体 → 写拒绝偏好（复用现有 `rejectedAITags`，50 条/90 天容量）。

**为什么删除后不保留 rejectedAI assignment 而是直接删实体**：防再生的生效层是 UserDefaults 拒绝偏好（喂 prompt），不是 rejectedAI assignment（那是单条想法粒度的展示过滤）。tag 删除后其 assignments 被 cascade 清掉，数据更干净；拒绝偏好 90 天有效期内足够覆盖 AI 重打周期。

**新增 Repository 方法**：

```swift
/// 全局删除标签：删除所有 assignments + 删除 tag 实体
/// - Returns: 删除结果（摘掉的 assignment 数供 toast 反馈）
@discardableResult
func deleteTagGlobally(name: String) throws -> TagDeletionResult

struct TagDeletionResult {
    let removedAssignmentCount: Int // 摘掉的 assignment 数（供 toast 反馈）
    let affectedTopicIds: [UUID]    // 缓存已重算的 Topic（测试断言用）
}
```

配套清理（同一方法内完成）：
- `associatedTopics` 关系断开（nullify 自动）
- 受影响 Topic 的 `associatedTagNames` 字符串缓存重算（复用 TopicRepository 现有派生逻辑，见 §5 涉及文件）
- 发送 `.thoughtDataDidChange` 通知刷 UI

### 4.2 全局重命名：改名 or 合并

```swift
/// 全局重命名标签
/// - 目标名（归一化后）不存在 → 直接改 tag.name
/// - 目标名已存在 → 合并：assignments/thoughts/associatedTopics 全部迁移到目标 tag，删除源 tag
/// - Throws: ThoughtError.tagNameEmpty / .tagNotFound
func renameTag(from oldName: String, to newName: String) throws
```

**合并路径必须修复现有 mergeTags 的三个 bug**（合并路径直接重写，旧方法删除）：

| bug | 修复 |
|---|---|
| 只迁移 `thoughts` 关系，assignments 被 cascade 删除 | assignments 逐条改指 `assignment.tag = 目标tag`（保留 source/rejectedAt 历史），迁移后按（thought, 归一化名）去重——同一想法同一名称只留最高优先级来源的一条（manual/inline > confirmedAI > ai > rejectedAI），与 `createAssignmentInternal` 去重规则一致 |
| usageCount 失真扩大 | 合并后重算：`usageCount = 可见来源 assignments 的去重想法数`（与 fetchAITagBuckets 口径一致） |
| Topic 缓存不变 | 迁移 associatedTopics 关系 + 重算受影响 Topic 的 `associatedTagNames` 缓存 |

**重命名后的副作用链**（对抗评审修订，renameTag + Service 层协同处理）：
- `ThoughtTag.name` 变更 → 展示层全部走 `ThoughtTagNormalizer.displayName`，无需额外处理
- **旧名写入拒绝偏好**（Service 层）：防 AI 重打旧名造成标签再分裂。典型场景：AI 打了「工作事业」，用户改名「工作」——若「工作事业」不进拒绝偏好，AI 下次整理新想法又造出「工作事业」，标签池再次分裂
- **新名从拒绝偏好移除**（Service 层）：若新名恰好在拒绝偏好里（用户先拒绝后又决定启用），必须移除——否则新名同时在「用户已认可」池和「用户已拒绝」池，prompt 层信号矛盾，AI 行为不可预测
- `fetchUserRecognizedTagNames` / `fetchUnrecognizedAITagNames` / `fetchAITagBuckets` 均按归一化 key 聚合，改名后自动归位，无需改动

### 4.3 入口 UI：抽屉标签行长按菜单

在 `ThoughtKnowledgeDrawerView` 的 `aiTagRow` 上加 `.contextMenu`：

```
长按标签行「工作事业」(3)
├── 重命名        → 弹 alert 输入新名字（目标已存在时提示将合并）
├── 删除标签      → 弹确认 alert：「将从 N 条想法移除，AI 以后不再推荐」
└── 取消
```

**为什么用 contextMenu 而不是左滑**：抽屉行已是 Button（点击=筛选），左滑手势与抽屉自身的右边缘左滑关闭手势（P1.5 已实现）冲突风险高；contextMenu 零手势冲突，且 iOS 用户习惯成熟。技术上：抽屉是 ZStack overlay 层，alert/contextMenu 均为 window 级 presentation，挂在抽屉内部视图无层级问题。

**为什么不新增管理页**：标签池行就是用户看到碎标签的现场，操作路径最短。管理页（搜索/批量）等验证需求强度后单独立项。

**删除/重命名后的筛选状态重置（对抗评审补充）**：`drawerSelection` 是 ThoughtsView 的 @State，以标签名为标识（`.aiTag(String)`）。若用户删除/改名的正是当前筛选中的标签，selection 悬空 → 列表变空态。处理：抽屉持有 selection 的 @Binding，删除/改名成功后直接置 `selection = .allNotes`（重命名也可选置为 `.aiTag(新名)` 保持筛选连续性，实施时按体验取一，推荐重置为全部笔记，语义最简单）。

**操作后反馈**：toast 或临时状态条显示「已从 N 条想法移除标签 #X」/「已合并到 #Y」，复用现有 UI 反馈组件（实施时确认项目内现有 toast 机制）。

### 4.4 服务层编排

`ThoughtOrganizationService` 新增两个方法（保持 Repository 纯数据、Service 管偏好/通知的分层）：

```swift
/// 全局删除标签（统一写拒绝偏好防 AI 再生）
func deleteTagEverywhere(name: String)

/// 全局重命名标签（旧名写拒绝偏好防分裂，新名从拒绝偏好移除）
func renameTagEverywhere(from oldName: String, to newName: String) throws
```

- `deleteTagEverywhere`：调 `deleteTagGlobally` → 调现有 `addRejectedTag(name:)` 写拒绝偏好 → post `.thoughtDataDidChange`
- `renameTagEverywhere`：调 `renameTag` → `addRejectedTag(name: 旧名)` → `removeRejectedTag(name: 新名)` → post 通知

## 6. 边界与坑

### 6.1 混合来源标签的删除
同一归一化名下可能既有 manual 又有 ai assignment（用户和 AI 打了同一个词）。`deleteTagGlobally` 按整个 tag 删，全部摘掉，统一写拒绝偏好——以「删除」这一最新用户意志为准，不再按历史来源区分（§4.1）。

### 6.1.1 标签池盲区：已被 Topic 完全收纳的标签（对抗评审补充）
抽屉标签池走 `fetchAITagBuckets(excludeAbsorbed: true)`：一个标签的全部 assignment 对应想法都被 Topic 收纳后，该标签**不在标签池显示**，长按入口覆盖不到它。它仍存在于筛选器、标签推荐池等位置。本次接受此盲区（完全收纳的标签已有 Topic 归属，可从 Topic 视角管理）；若后续反馈强烈，管理入口随「独立标签管理页」一起立项。

### 6.2 合并时的 assignment 去重
A 标签（ai 来源，挂在想法 1、2）合并进 B 标签（manual 来源，挂在想法 2、3）→ 想法 2 会同时收到 A 迁来的 ai assignment 和已有 manual assignment。**必须去重**：保留 manual，丢弃 ai。去重规则与 `createAssignmentInternal` 同源（高优先级来源胜出），抽成共享私有方法，禁止两套逻辑漂移。

### 6.3 正文 #标签 复活（接受的边界）
inline 来源标签来自想法正文的 `#标签` 文本提取。全局删除标签**不改写正文**——用户下次编辑该想法并触发 inline 提取时，标签会复活。不处理的代价可接受（编辑是低频主动行为，复活后用户可见可再删）；要处理则需在保存编辑器时同步剔除正文 token，属于编辑器逻辑改动，本次不扩散。方案文档显式记录此边界，避免后续当 bug 返工。

### 6.4 Topic 缓存一致性
`Topic.associatedTagNames` 是逗号拼接的字符串缓存（HoloThoughtDataSource 读它喂 AI 对话工具）。删除/重命名/合并标签后必须重算受影响 Topic 的缓存，否则 AI 对话里主题关联标签会出现已删除的名字。重算逻辑以 `associatedTags` 关系为准派生，与 TopicRepository `setSourceTerms` 现有写法一致——抽出独立方法 `refreshAssociatedTagNamesCache(topic:)` 供三处复用，禁止再复制一份拼接逻辑。受影响 Topic 集合在删除/合并 tag **之前**先取出（删除后关系已 nullify，取不到）。

### 6.5 usageCount 口径（范围收敛，对抗评审修订）
`usageCount` 失真的主源是「只增不减」：`getOrCreateTag` 每次复用 +1，删除 assignment 从不减。完全修正需改动所有 assignment 删除路径（逐条拒绝、编辑器移除等），超出本次范围。**本次只在全局删除/合并路径重算受影响 tag**（`usageCount = 可见来源 assignments 的去重想法数`，与 fetchAITagBuckets 口径一致），日常 +1 维持现状——usageCount 仅用于排序（getAllTags / 推荐池），无持久化语义，失真可接受，不追求完美。

### 6.6 iCloud 同步
标签删除/改名是普通 Core Data 属性+关系变更，随现有同步链路走，无特殊处理。拒绝偏好（UserDefaults）本机有效——与现状一致（逐条拒绝也是本机），不扩散。

## 7. 实施步骤（TDD）

核心逻辑在 Repository 层，按项目规范走 TDD（RED → GREEN → REFACTOR）：

1. **测试先行**：`ThoughtRepositoryTagManagementTests.swift`

3. **Service 编排**：`deleteTagEverywhere` / `renameTagEverywhere` + `removeRejectedTag`

5. **验证**：`test_sim` 跑测试（build_sim 不含测试 target），模拟器手动验证抽屉交互

## 8. 决策点（评审用）

| # | 决策点 | 推荐 | 备选 |
|---|---|---|---|
| 1 | 入口形态 | 抽屉标签行长按菜单（路径最短） | 独立标签管理页（功能全但链路长，列为后续立项） |

| 3 | 重命名的旧名处理 | 旧名写拒绝偏好（防 AI 重打旧名造成再分裂），新名从拒绝偏好移除 | 不动拒绝偏好（AI 可能重打旧名） |
| 4 | 删用户标签是否改写正文 `#标签` | 不改写，接受编辑复活边界 | 改写正文（编辑器逻辑扩散，本次不做） |
| 5 | 改颜色入口 | 本次不做，仅文档记录 ColorPickerPopover 可待接入 | 顺带接入（范围扩散） |

## 9. 验收标准

- [ ] 长按抽屉标签 → 删除：所有想法上该标签消失；删除后触发 AI 整理不再产出该标签（拒绝偏好生效）
- [ ] 长按抽屉标签 → 重命名为新名：全部想法标签显示新名，知识树归位；AI 后续不再产出旧名
- [ ] 长按抽屉标签 → 重命名为已存在名：两标签合并，想法不重复显示，来源优先级正确，旧标签消失
- [ ] 删除/改名当前筛选中的标签：列表自动回到「全部笔记」，不出现空白筛选态
- [ ] 删除/合并后：AI 对话工具中主题关联标签无脏名字
- [ ] 全部新增/修改路径有单测覆盖且 `test_sim` 通过
