# Holo 观点标签统一识别方案

> 日期：2026-07-11
> 模块：观点（Thoughts）
> 状态：待评审

## 1. 一句话目标

让 AI 在给想法打标签时，**平等识别并优先复用全部「用户认可的标签」**——无论这个标签是用户手动建的，还是 AI 之前打过又被用户保留的。不再出现「用户手动建的标签 AI 看不见、不停造同义重复标签」的割裂。

---

## 2. 背景：现在为什么割裂

用户反馈：分类体验混沌，分不清哪些是 AI 分的、哪些是自己打的。代码层面根因有三层：

| 层 | 现状 | 问题 |
|---|---|---|
| 取数 | `organizeThought` 调 `getRecentTagNames(20)`，按 `usageCount` 降序取前 20 个标签名 | 用户手动标签用量低（1-2 次），被 AI 高频碎标签挤出前 20，**AI 根本看不到** |
| prompt | 后端 `thought_organization` 只有一句软引导：`参考已有标签风格：{{existingTagExamples}}` | 是「参考」不是「必须复用」，AI 可以无视已有标签随便造新的 |
| 来源 | `getRecentTagNames` 直接查 `ThoughtTag` 表，**该表无 source 字段** | 无法区分标签来源，没法把「用户认可」的标签单独提出来优先喂给 AI |

结论：**用户标签和 AI 标签在数据表里是一个池子（`ThoughtTag`），但 AI 打标签这条路径上，对用户标签是「半瞎」状态**——看不到全部、也不被要求复用。

---

## 3. 范围

### ✅ 本次做
1. 新增「用户认可标签」取数，让 AI 看见全部用户标签
2. prompt 从「参考风格」改为「优先复用」（双端同步）
3. 命中已有标签走复用，不造同义重复（梳理现有机制）

### ⏸️ 本次不做（后续立项）
- **7 个固定领域大类**：用户用法各异，硬塞固定大类不通用（已决策排除）
- **语义匹配**（认得「工作」=「工作事业」）：需 embedding 或额外 LLM 判别，单独立项
- **Topic 大类层级重构 / `topicCandidate` 接通**：涉及 Topic 模型大改，单独立项
- **现有碎标签数据迁移**：不迁移，靠新机制自然收敛演进

## 4. 设计

### 4.1 取数：新增「用户认可标签」+ AI 高频补充

**关键认知**：`ThoughtTag` 表没有 `source` 字段，来源信息只存在于 `ThoughtTagAssignment` 上。所以「用户认可的标签」必须通过 assignment 反查。

**「用户认可的标签」定义**：存在至少一条 `ThoughtTagAssignment`，其 `source ∈ {manual, inline, confirmedAI}`，且所属想法未删除。

- `manual` / `inline`：用户亲手打的，纯用户意志
- `confirmedAI`：AI 打过、用户点过 ✓ 保留的，等同用户认可
- `ai`（未确认）：不进「认可」池，仅作高频补充
- `rejectedAI`：排除

**新方法** `ThoughtRepository.fetchUserRecognizedTagNames(limit:) -> [String]`：
- 查 `ThoughtTagAssignment`，`source IN {manual, inline, confirmedAI}` 且 `thought.isSoftDeleted == NO`
- 按 `tag` 去重，取 `tag.name`
- 排序：用户认可标签内部按 `tag.usageCount` 降序（高频的排前，AI 更容易注意到）

**AI 高频补充**：保留 `getRecentTagNames` 思路，取 `source = ai` 的高频标签补充，但**排除已进入「认可」池的**，避免重复。

**组合喂给 prompt**：
- 用户认可标签（全量，典型 <30）排前
- AI 高频标签补充到总量上限（建议 60，控 token）
- 超过上限按 usageCount 截断，并在 prompt 里说明「（还有更多未列出）」

> **实现注意（对抗评审 #1）**：优先在 DB 层过滤而非内存去重。直接查 `ThoughtTag`，predicate 用 `SUBQUERY(assignments, $a, $a.source IN %@ AND $a.thought.isSoftDeleted == NO).@count > 0` 一次性拿到「有认可 assignment」的标签，避免把上万条 assignment 全 fetch 到内存。`usageCount` 排序仍走 `ThoughtTag` 自身字段。

### 4.2 prompt：参考 → 优先复用（双端同步，重点）

**后端 `defaultPrompts.json` → `thought_organization`** 新版要点：

```
## 标签规则
- 生成 1-3 个标签，每个标签 2-6 个字
- 标签应该是内容关键词，不是情感分类
- 避免过于宽泛的标签（如"生活""思考""日常""想法""记录"）

## 复用规则（重要）
以下标签已经存在，能准确描述本条想法的【必须复用】，不要生成同义重复的标签：
{{existingTagExamples}}

不要生成以下标签（用户已拒绝）：{{rejectedTags}}
```

核心变化：把「参考已有标签风格」升级为「**必须复用，不要生成同义重复**」——从软引导变硬约束。

**双端同步（按 CLAUDE.md / 记忆 `feedback_prompt_dual_sync`）**：
1. 改 iOS `PromptManager.swift` 内嵌模板（后备）+ 升级 `promptVersions` 版本号
2. 同步改后端 `HoloBackend/src/prompts/defaultPrompts.json`（**生效端**）
3. 重建并部署后端：`ssh root@123.56.104.9` → `cd /root/Holo/HoloBackend/deploy` → `docker compose build --no-cache && docker compose up -d`
4. 部署后验证：`curl http://localhost:8787/v1/prompts/thought_organization` 确认版本和内容
5. 真机杀掉 App 重开（清 `HoloBackendPromptService` 2 分钟 metaTTL 缓存）

**顺带清理（对抗评审 #7）**：现有 prompt 让 AI 输出 `topicCandidate` / `matchedTopicId` 字段，但客户端 `parseOrganizationResponse` 早已丢弃这俩（`Topic` 层级本次不动）。改 prompt 时顺手把输出格式简化为 `suggestedTags / confidence / reason`，去掉废弃字段——减小 AI 输出 token、聚焦标签任务、降低 AI 被无关字段干扰。

### 4.3 复用机制：撞名走已有标签

现有机制已部分到位，本次确认保留 + prompt 侧强化：

- `getOrCreateTag(name:)`：经 `ThoughtTagNormalizer.key` 归一化（去 `#`、lowercase、折叠全角/变音）后命中已有标签则复用，不新建 → **保留**
- `createAssignmentInternal`：同一想法同一标签不重复建 assignment（含 manual 覆盖 ai 的优先级逻辑）→ **保留**
- 本次增强点不在复用代码，而在 **prompt 引导 AI 尽量输出已有标签名**——AI 输出名命中已有 key，自然就走 `getOrCreateTag` 复用

> 局限（明确写入非目标）：字面不同但同义（「工作」vs「工作事业」）仍无法自动合并，靠 4.2 的 prompt 引导 + 用户手动收敛。语义匹配留后续。

## 7. token 成本估算

- 用户认可标签典型 <30 个，每个均 ~4 字 + 分隔符 ≈ 6 字符 → ~180 字符 ≈ **60 token**
- AI 高频补充到总量上限 60 → ~360 字符 ≈ **120 token**
- 相比单条想法正文（截 240 字 ≈ 80-150 token）和 system prompt，**增量可忽略**
- 比现有 `getRecentTagNames(20)` 多约 40-60 token，换「AI 看见全部用户标签」，性价比高
