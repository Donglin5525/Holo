# HoloAI 灵活查询时间与结果一致性设计

## 状态

Accepted — 用户于 2026-07-11 选择“最近一个月 = 包含今天的最近 30 个自然日”，并要求继续推进系统性修复。

## 背景与根因

用户询问“最近一个月吃了多少顿麦当劳，花了多少钱，平均一顿多少钱”时，卡片显示 `2026-06-16 ~ 2026-06-16`，点击“查看全部 6 顿”却出现 5 月记录。

CodeGraph 调用链确认了两个确定性根因：

1. `FlexibleQuerySummary.dateRange` 实际表示命中数据的最早/最晚日期，却被卡片和回答标成查询时间范围。
2. “查看全部”只把商户关键词传给 `FinanceSearchView`，没有传查询日期或命中集合，因此目标页重新搜索了全部历史记录。

同时发现三个会继续制造低级错误的边界缺陷：

- Planner 日期只做字符串大小比较；无效日期解析失败后，Executor 会静默忽略该时间条件并查全历史。
- Executor 硬编码 `fetchLimit = 200`；超过 200 个候选时，总数、总额和均价会静默算错。
- `summary.totalMatched` 基于完整命中集合，而持久化 evidence rows 受 plan limit 截断，不能把 rows 冒充“全部结果”。

## 明确口径

- `最近一个月`、`近30天`：包含今天的最近 30 个自然日。
- 以 2026-07-11、Asia/Shanghai 为例：`2026-06-12...2026-07-11`。
- `本月`：当月 1 日到今天。
- `上个月`：上一自然月首日至末日。
- 数据库查询统一使用半开区间 `[startOfDay, dayAfterEnd)`，UI 展示使用包含首尾的日期字符串。
- 用户输入中可确定解析的时间约束优先级高于 LLM Planner 输出。

## 方案对比

### 方案 A：只修卡片文案并给搜索页增加日期参数

改动最小，但搜索页仍会重新执行另一套筛选；类别、排除词、包含字段、limit 和未来新条件仍可能丢失，也无法保证编辑前后的结果与原回答一致。

### 方案 B：标准化查询范围 + 完整结果快照（采用）

Planner 之后统一规范时间；Executor 生成准确聚合、有限预览和完整命中 ID 快照；卡片、回答与详情共同消费这个结果。改动适中，但能直接建立一致性不变量，并兼容旧消息。

### 方案 C：重写整个财务查询引擎

长期最完整，但当前问题不需要替换 Core Data、全部筛选 UI 和所有查询操作，风险和验证范围过大。

## 架构决策（ADR）

### 决策 1：显式时间先本地规范化

新增无副作用的时间范围解析/编码组件。它从原始问题识别确定性时间表达，并在 deterministic resolver 与 LLM fallback 之后统一覆盖 plan 日期。Planner validator 严格验证 ISO 日期、首尾顺序和可支持范围；任何非法日期都失败或请求澄清，不得静默忽略。

后果：模型仍负责开放语义，但不能覆盖用户明确说出的时间。时间测试可以固定 `now`、Calendar 和 TimeZone。

### 决策 2：查询范围与命中数据范围分离

结果模型明确区分：

- `queryDateRange`：用户请求并实际用于 predicate 的范围。
- `matchedDataRange`：命中记录真实分布，仅用于需要时解释数据覆盖。

卡片和自然语言回答默认展示 `queryDateRange`，不再把 `matchedDataRange` 标成“时间范围”。

### 决策 3：结果详情使用完整 ID 快照

Executor 对所有匹配记录计算 summary/average，并保存全部匹配 transaction IDs；evidence rows 仍只保留有限预览。详情页按 ID 快照读取记录，绝不重新用关键词搜索历史。

旧消息没有完整快照时，只能展示其已持久化 evidence IDs，并将按钮文案改成实际可展示数量；不能退回全历史搜索。

### 决策 4：精确聚合不得受展示 limit 影响

移除候选查询的任意 200 条截断。`plan.limit` 只控制回答/卡片预览，不参与 total count、sum、average 的计算。

## 数据流

```text
原始问题 + intent extractedData
        ↓
确定性时间范围解析（显式约束优先）
        ↓
Merchant fast-path 或 LLM Planner
        ↓
Plan 规范化与严格校验
        ↓
Executor 使用同一半开日期区间查询完整集合
        ↓
准确 summary/calculation + preview evidence + allMatchedIDs
        ↓
持久化 FlexibleQueryResult
        ├─ 文字回答：queryDateRange + 总数/总额/均价
        ├─ 卡片：queryDateRange + 3 条预览
        └─ 全部明细：allMatchedIDs 精确加载
```

## 用户可见结果

对于 2026-07-11 的原始问题，回答格式应为：

> 麦当劳共 6 顿，合计 ¥285.00，平均每顿 ¥47.50。查询范围：2026-06-12～2026-07-11。

卡片副标题显示同一查询范围；点“查看全部 6 顿”后只能看到这次结果中的 6 条记录，任何 5 月记录都不能出现。

## 兼容与失败处理

- 新增持久化字段使用可选字段或自定义 decoding 默认值，旧聊天消息必须继续解码。
- 快照中的记录后来被删除时，详情页显示当前仍存在的记录数，不用其它历史记录补位。
- 快照记录被编辑后仍按 ID 展示最新内容，并提示这是原查询结果快照。
- 非法日期、不可解析日期或 start > end 必须产生明确错误，不得降级成无日期查询。
- 空结果仍展示实际查询范围，让用户知道系统查了哪里。

## 测试与验收

### 时间语义

- 2026-07-11 的“最近一个月”严格为 2026-06-12～2026-07-11。
- 覆盖跨月、跨年、闰年/月末、Asia/Shanghai 与非默认时区。
- `本月`、`上个月` 与 `最近一个月` 不混淆。
- LLM 返回单日或非法日期时，用户明确的最近 30 天约束仍胜出。

### 数据一致性

- 创建 5 月和近 30 天的麦当劳记录，聚合与详情只包含近 30 天。
- 查询范围、回答、卡片、预览和详情断言完全一致。
- 超过 200 条记录时 count/sum/average 仍准确。
- 超过 preview limit 时详情仍能展示全部快照 IDs。
- 旧结果 JSON 可解码且不会跳到全历史搜索。

## 风险与控制

- 完整 ID 快照增加少量消息 JSON 体积：个人账本规模可接受；evidence 内容仍受 limit 控制。
- 移除 200 条候选上限可能增加本地查询时间：先用日期/type/amount Core Data predicate 粗筛，再内存执行文本匹配；用大样本测试观察耗时。
- 主分支存在用户无关脏改：只对本任务路径 scoped stage/commit，任何现有修改都不覆盖。
