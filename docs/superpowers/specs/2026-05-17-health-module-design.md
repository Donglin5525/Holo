# HOLO 健康模块设计方案

> 日期：2026-05-17  
> 状态：设计稿待审阅  
> 范围：iOS SwiftUI 健康模块前端体验、HealthKit 状态处理、跨模块洞察入口  
> 原型：
> - `.superpowers/brainstorm/10215-1778950591/content/health-layout-v2-ac-insights.html`
> - `.superpowers/brainstorm/10215-1778950591/content/health-detail-and-states-v1.html`

## 1. 背景与目标

健康模块是 HOLO 个人数据资产体系的核心组成部分。它不只是展示步数、睡眠、站立时长，而是把用户的身体状态纳入生活数据闭环，与任务、习惯、记账、思考模块形成可解释的长期个人画像。

本阶段目标是完成健康模块的产品与前端体验设计，作为后续 SwiftUI 实现依据。设计方向已经确定为：

- 保留 Apple Watch Activity Rings 的三环隐喻。
- 采用 HOLO 自有的暖色纸感、卡片秩序和品牌色，不直接复制 Apple Health。
- 与 Apple Health 只读集成，并在页面进入、手动刷新、系统健康数据变化时尽量及时更新。
- 首页展示一条核心洞察，并提供轻量跨模块洞察列表。
- 兼容无 Apple Watch、未授权、部分无数据等真实使用状态。

## 2. 设计原则

### 2.1 视觉优先

首页以三环作为第一视觉锚点，用户进入页面后先理解“今天身体状态如何”，再看具体数字。数字是解释层，不是主入口。

### 2.2 即时反馈

步数、睡眠、站立进度随 HealthKit 数据刷新而更新。MVP 至少支持页面进入和手动刷新；后续可用 `HKObserverQuery` / `HKAnchoredObjectQuery` 提升接近实时的刷新体验。用户刚授权或刚刷新时，页面需要清楚表达同步状态，避免误以为数据丢失。

### 2.3 个性化与可扩展

MVP 默认目标为：

- 步数：10,000 步
- 睡眠：8 小时
- 站立：12 小时

后续支持用户自定义目标，并扩展心率、体重、活动分钟等指标。

### 2.4 HealthKit 信任感

授权页和数据源卡必须明确：

- HOLO 只读取健康数据。
- 不写入 Apple Health。
- 不上传原始健康记录。
- 使用健康数据是为了展示进度和生成生活洞察。

### 2.5 包容性设计

没有 Apple Watch 的用户不能看到残缺页面。站立环在无 Watch 场景下降级为“活动分钟 / 久坐风险”替代环，仍保持三环体验。

## 3. 信息架构

健康模块包含以下页面与状态：

1. 健康首页
2. 步数详情页
3. 睡眠详情页
4. 站立详情页
5. 首次授权页
6. 未授权 / 权限被关闭状态
7. 已授权但无数据状态
8. 无 Apple Watch 兼容状态

首页是健康模块主入口。详情页负责解释单项指标趋势和行动建议。状态页负责处理 HealthKit 真实边界。

## 4. 首页设计

### 4.1 页面结构

首页采用从状态到解释的纵向结构：

1. 顶部标题区：`健康`、同步状态、设置入口
2. 三环主视觉：身体状态分数 + 步数 / 睡眠 / 站立三环
3. 三项指标摘要：步数、睡眠、站立的当前值和目标进度
4. Apple Health 数据源卡：连接状态、只读同步、读取指标
5. 今日核心洞察：一条最重要的 HOLO 建议
6. 生活闭环洞察：习惯、财务、思考等跨模块关联
7. 7 天趋势入口：默认展示睡眠趋势，可按后续交互切换

### 4.2 三环主视觉

三环表示：

- 外环：步数，使用 `holoPrimary` 橙色
- 中环：睡眠，使用 `holoChart1` 蓝色
- 内环：站立，使用 `holoPurple` 紫色

中心展示身体状态分数，例如 `81`。分数是一个面向用户的综合提示，不作为医学评价。计算建议：

```text
身体状态分数 = 步数进度 * 0.3 + 睡眠进度 * 0.45 + 站立/活动进度 * 0.25
```

睡眠权重最高，因为它对任务效率、情绪和习惯完成有更强解释价值。分数上限为 100，低数据可信度时不显示精确分数，改为“数据不足”。

### 4.3 指标摘要

首页三项指标卡保持轻量：

- 步数：`8,432`，目标 `10,000`
- 睡眠：`7.2h`，目标 `8h`
- 站立：`8h`，目标 `12h`

点击指标摘要或三环区域中的对应环，进入对应详情页。

### 4.4 Apple Health 数据源卡

数据源卡显示在核心指标之后，用于建立信任：

```text
Apple Health 已连接
只读同步 · 步数 / 睡眠 / 站立
在线
```

如果部分指标无权限，需要显示“部分授权”，并在详情中说明哪个指标不可用。

### 4.5 洞察结构

首页洞察采用“一主三辅”结构。

主洞察示例：

```text
你睡眠超过 7.5h 时，第二天任务完成率提升 28%。今晚建议 22:30 前上床，优先保住睡眠环。
```

轻量跨模块洞察示例：

- 习惯：运动习惯正在拉动步数
- 财务：低睡眠日咖啡消费偏高
- 思考：压力记录集中在久坐日

MVP 可以先使用本地规则生成洞察文案，后续接入 HoloAI 分析上下文。

## 5. 详情页设计

### 5.1 通用详情页结构

三类详情页共享结构：

1. 导航标题
2. 大圆环进度
3. 当前值、目标值、差距说明
4. 7 天统计摘要
5. 7 天趋势图
6. 单项洞察或跨模块关联

### 5.2 步数详情页

步数详情页偏行动建议。

核心文案方向：

- “还差 1,568 步”
- “晚饭后 15 分钟散步，大概率能补齐今天的活动环”
- “本周低步数日通常也是久坐日”

统计摘要：

- 7 天平均步数
- 7 天最高步数
- 达标天数

### 5.3 睡眠详情页

睡眠详情页是健康模块最重要的洞察页，承担恢复状态和生活闭环解释。

核心文案方向：

- “恢复良好”
- “连续两晚接近目标，今天适合安排高专注任务”
- “高睡眠日，任务完成率更稳定”
- “低睡眠日，咖啡支出更高”

睡眠日期归属采用“起床日归属”：跨午夜睡眠记入醒来当天。

### 5.4 站立详情页

站立详情页偏久坐识别与提醒。

核心文案方向：

- “还差 4 小时”
- “久坐集中在 14:00-17:00”
- “建议开启下午站立提醒”

如果用户没有 Apple Watch，站立详情页降级为活动分钟详情页。

## 6. HealthKit 状态处理

### 6.1 首次授权

首次进入健康模块，如果未请求过权限，展示授权页。

授权页内容必须包含：

- 连接 Apple Health
- HOLO 只读取数据
- 不写入、不上传原始健康记录
- 读取步数、睡眠、站立的用途说明
- 主按钮：授权 Apple Health
- 次按钮：稍后再说

模拟数据只用于模拟器和 SwiftUI Preview，不作为真机用户功能入口。MVP 不提供手动录入健康数据。

### 6.2 未授权

用户拒绝授权或在系统设置关闭权限时，展示未授权状态：

- 告知无法访问健康数据
- 提供“打开系统设置”
- 保留模块说明，不展示随机数据

### 6.3 部分授权

HealthKit 可能出现部分指标有数据、部分指标无权限或无数据。页面不应整体失败。由于 HealthKit 对读取权限的可见性有限，产品状态需要结合授权请求结果、查询结果、设备支持情况和错误信息综合判断，而不是只依赖单个全局 `isAuthorized`。

处理规则：

- 有权限且有数据：正常展示
- 有权限但无数据：展示等待/空状态
- 无权限：展示锁定状态和设置入口
- 无 Apple Watch 导致无站立数据：进入替代环模式

### 6.4 已授权但无数据

展示“今天还没有可用健康数据”，并解释：

- 刚授权时可能需要等待
- Apple Health 本身无记录时不会生成误导性结论
- 可下拉或点击刷新

### 6.5 无 Apple Watch

无 Apple Watch 时，不隐藏健康模块。站立环替换为：

```text
活动分钟 / 久坐风险
```

替代指标可以由步数分布、活动分钟或 iPhone 可用运动数据组合生成。页面必须标明“无 Watch 模式”，避免用户误解为标准站立数据。

## 7. 数据与状态模型建议

### 7.1 指标类型

现有 `HealthMetricType` 可扩展为：

```swift
enum HealthMetricType {
    case steps
    case sleep
    case standHours
    case activeMinutes
}
```

`activeMinutes` 仅在无 Apple Watch 或无站立数据时作为替代指标显示。

### 7.2 授权状态

建议从单一 `isAuthorized` 改为按指标表达：

```swift
enum HealthMetricAvailability {
    case available
    case unauthorized
    case noData
    case unsupported
}
```

首页根据每个指标状态独立渲染。

实现时不要把 `requestAuthorization` 的 `success` 直接等同于所有读取权限都已授予。`success` 更适合表示授权请求流程完成；指标是否可读需要通过后续查询和错误状态确认。

### 7.3 数据来源状态

建议新增：

```swift
enum HealthDataSourceState {
    case notRequested
    case connected
    case partiallyConnected
    case denied
    case unavailable
}
```

该状态驱动 Apple Health 数据源卡和授权/设置入口。

## 8. 跨模块洞察策略

MVP 洞察先以规则为主：

- 睡眠达到目标且任务完成率高：提示睡眠与效率正相关
- 睡眠低于 6.5h 且咖啡支出上升：提示低睡眠日消费关联
- 站立/活动不足且压力类思考增多：提示久坐与压力记录关联
- 运动习惯连续打卡且步数提升：提示习惯对健康指标的拉动

后续接入 HoloAI 时，健康模块应进入 `AnalysisContext`，使聊天分析可以引用健康数据，但不暴露原始 HealthKit 明细。

## 9. 视觉规范

颜色沿用 HOLO 设计系统：

- 步数：`Color.holoPrimary`
- 睡眠：`Color.holoChart1`
- 站立：`Color.holoPurple`
- 成功状态：`Color.holoSuccess`
- 背景：`Color.holoBackground`
- 卡片：`Color.holoCardBackground`

卡片圆角遵循现有设计系统：

- 普通卡片：`HoloRadius.lg`
- 页面主视觉卡片：可使用 `HoloRadius.xl`
- 小指标卡：`HoloRadius.md`

页面应保持温和、清晰、克制，不做医疗应用式冷色临床风，也不做过重游戏化。

## 10. MVP 范围

本次实现建议包含：

- 新首页布局
- 三环主视觉
- 三个指标摘要
- Apple Health 数据源卡
- 今日核心洞察卡
- 生活闭环洞察列表
- 步数、睡眠、站立详情页
- 首次授权页
- 未授权/无数据状态
- 无 Apple Watch 替代环展示

暂不包含：

- 写入 Apple Health
- 医疗级健康建议
- 长周期月报/年报
- 自定义目标编辑页
- 手动录入健康数据
- 复杂心率、血氧、体重分析
- 服务端同步原始 HealthKit 数据

## 11. 验收标准

- 首次进入时，授权页明确说明只读和用途。
- 授权成功后，首页显示三环、三项指标和数据源卡。
- 有数据时，进度与 Apple Health 读取结果一致。
- 无数据时，不显示随机数据或误导性洞察。
- 无 Apple Watch 时，站立环降级为活动分钟/久坐风险替代环。
- 点击步数、睡眠、站立可进入对应详情页。
- 详情页展示 7 天趋势、平均值、最高值、达标天数。
- 跨模块洞察区能展示规则生成的至少 2 条关联。
- 模拟器可使用模拟数据开发，但真机逻辑不依赖模拟数据。
- 编译无错误，无 force unwrap，无调试 print。

## 12. 待用户审阅的问题

当前设计已默认以下决策：

- 首页采用 A+C 组合方案。
- 洞察采用“一条核心洞察 + 三条轻量跨模块洞察”。
- 无 Apple Watch 时用活动分钟/久坐风险替代站立环。
- 睡眠在详情页中拥有最高洞察优先级。
