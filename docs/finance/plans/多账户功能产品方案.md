# HOLO 多账户功能产品方案

## 背景

HOLO 记账模块目前已有基础的 `Account` 实体（现金/微信/支付宝/信用卡），但仅作为交易的标签使用，**没有余额追踪、没有账户管理 UI、没有转账功能、没有按账户统计**。多账户功能的目标是回答用户一个核心问题：**"我的钱在哪里？"**

### 用户决策
- **实施范围**：按建议分期（P0 先做余额 + 管理 UI + 净资产，P1 再做转账 + 分析筛选）
- **入口位置**：账户总览放在**财务设置页**内（NavigationLink push）
- **信用卡功能**：暂时不需要特殊处理，未来有需求再加

---

## 现有基础与关键问题

### 现有数据
- `Account` 实体：id, name, type (cash/digital/card/other), isDefault
- 4 个种子账户：现金(cash, default=true)、微信(digital)、支付宝(digital)、信用卡(card)
- `AccountPicker` 组件：存在但仅被 `AddTransactionView`（旧版页面）使用
- `Transaction` 与 `Account` 有必填的一对一关系

### 关键问题（当前代码现状）

**Q1：主记账入口不经过账户选择**
- `AddTransactionSheet`（主入口）保存时直接调用 `getDefaultAccount()`，不展示 AccountPicker
- `AddTransactionView`（旧版）有 AccountPicker 但不在主流程使用
- 结果：用户创建的所有交易都挂在默认账户上

**Q2：默认账户规则缺失**
- 没有切换默认账户的 UI 或方法
- 没有删除默认账户的兜底逻辑
- `getDefaultAccount()` 返回 nil 时 save 仅 print 警告，静默失败

**Q3：AccountType.card 语义模糊**
- 当前 `card` = "银行卡"，但种子数据里的"信用卡"也用 card
- 无法区分储蓄卡（资产）和信用卡（负债）

## 第一期（P0）：账户基础

### 功能 1：账户余额追踪
- 向 `Account` 添加 `initialBalance` 字段（初始余额）
- **实时计算**当前余额：`initialBalance + 收入 - 支出`
- 不存储余额（个人用户交易量小，实时计算足够快）

### 功能 2：主记账闭环 — 账户选择
- **在 `AddTransactionSheet` 中集成账户选择**，补上主流程缺失的一环
- 添加 `@State private var selectedAccount: Account?` 状态
- 在类别选择器下方（或金额区域旁）添加账户选择入口
- 保存时使用 `selectedAccount ?? defaultAccount`（有用户选择优先，否则回退默认）
- `canSave` 校验：`selectedAccount != nil || defaultAccount != nil`
- 记住上次选择的账户（`@AppStorage("lastSelectedAccountId")`），下次打开预选
- **编辑交易时**：加载交易已有的账户，允许修改

### 功能 3：默认账户规则

| 规则 | 约定 |
|------|------|
| 唯一性 | 系统始终有且仅有一个默认账户 |
| 初始默认 | 种子数据的"现金"账户为初始默认 |
| 切换默认 | 用户可在账户管理中将任意非归档账户设为默认 |
| 删除默认 | 禁止删除默认账户。UI 提示"请先将其他账户设为默认" |
| 归档默认 | 禁止归档默认账户。UI 提示同上 |
| 兜底 | 若因数据异常导致无默认账户，自动将 `isArchived == false` 的第一个账户（按 sortOrder）设为默认 |

### 功能 4：账户管理 UI
- **入口**：`FinanceSettingsView` 中新增"账户管理"入口（NavigationLink push）
- **账户总览页** (`AccountListView`)：
  - 顶部：总资产、总负债、净资产卡片
  - 按类型分组的账户卡片（图标 + 名称 + 余额 + 最近交易时间）
  - "+" 按钮添加新账户
- **账户详情页** (`AccountDetailView`)：
  - 账户信息头部（图标、名称、余额、类型标签）
  - 本月收支统计（收入/支出/净变动）
  - 该账户的交易历史列表（按日期分组）
  - 底部操作：编辑、调整余额
- **添加/编辑账户页** (`AddAccountSheet`)：
  - 名称、类型选择器、图标、颜色、初始余额、备注
  - 编辑模式：可修改名称、图标、颜色、备注；初始余额仅创建时可设
  - 删除规则：有交易的账户不可删除（UI 提示"该账户有 N 笔交易，无法删除"）
- **余额调整** (`AdjustBalanceSheet`)：
  - 显示当前计算余额 vs 实际余额输入
  - 创建调整交易来弥合差异

### 功能 5：余额调整的记账语义

"调整交易"的定义：

| 属性 | 约定 |
|------|------|
| 交易类型 | 复用现有 `income`/`expense`（不新增系统类型） |
| 方向 | 余额调高 → income；余额调低 → expense |
| 分类 | 新增一个系统内置分类"余额调整"（不可删除、不可编辑） |
| 统计图表 | **参与**月度收支总金额，但在分析页类别筛选中可单独看到 |
| 导入导出 | 正常导出为 income/expense + "余额调整"分类 |
| 交易列表 | 显示"[余额调整]"标签，备注字段记录用户填的差异原因 |

为什么不新增 `adjustment` 类型：
- 避免 TransactionType 从 2 种变 3 种带来的全部分支逻辑改动
- 复用 income/expense 让所有现有的聚合查询自动兼容
- 用分类区分即可满足过滤需求

### 功能 6：资产/负债口径

| 账户类型 | 余额正负 | 归属 |
|----------|----------|------|
| cash（现金） | 正 = 有钱 | 资产 |
| digital（数字钱包） | 正 = 有钱 | 资产 |
| bank（储蓄卡） | 正 = 有钱 | 资产 |
| card（银行卡/信用卡） | 正 = 有余额 | 资产（先储值后消费的场景） |
| other（其他） | 视正负 | 正→资产，负→负债 |

**净资产计算**：
- 总资产 = 所有账户中 `balance >= 0` 的余额之和
- 总负债 = 所有账户中 `balance < 0` 的余额绝对值之和
- 净资产 = 总资产 - 总负债

**第一期简化**：不对 card 做信用额度等特殊处理，统一按余额正负判断资产/负债。

### 功能 7：AccountType 迁移策略

当前种子数据里的"信用卡"用的是 `card` 类型。新增 `bank` 后的迁移规则：

| 场景 | 处理方式 |
|------|----------|
| 已有 `card` 类型账户 | **保留不变**，语义仍为"银行卡/信用卡" |
| 新建储蓄卡账户 | 使用新类型 `bank` |
| 新建信用卡账户 | 仍使用 `card`（第一期不区分） |
| 种子数据 | "信用卡"仍为 card，新增的"储蓄卡"为 bank |
| 迁移代码 | **不需要**。Core Data 轻量迁移自动添加新 raw value，旧数据不受影响 |

**存量数据 backfill**：
- 旧用户升级后，所有已有账户补齐 `sortOrder`
- 补齐规则：按当前查询顺序写入连续值（默认账户优先，其余保持现有顺序）
- 若出现多个默认账户：仅保留排序最靠前的一个，其余设为 `isDefault = false`
- 若没有默认账户：执行 `ensureDefaultAccount()`，将第一个未归档账户设为默认
- 新增的"储蓄卡"种子账户**仅对新安装用户生效**；老用户不自动补建，避免污染用户账户列表

### 功能 8：净资产展示
- 在 `AccountListView` 顶部展示总资产/总负债/净资产
- 仅当用户设置了至少一个账户初始余额时显示详细数字
- 增强 `AccountPicker` 在每个账户旁显示当前余额

## 第二期（P1）：转账与分析

### 功能 9：账户转账
- 新增 `TransactionType.transfer`（转账类型）
- 转账交易包含双账户引用：`account`（转出）+ `transferAccount`（转入）
- 转账**不计入**总收入/总支出（避免虚增数据）
- 可选的转账手续费字段 `transferFee`
- `AddTransactionSheet` 新增转账模式（From/To 双账户选择器）

**转账基础约束**（P1 实现时需遵循）：

| 约束 | 规则 |
|------|------|
| 分类 | 转账不需要分类（category 字段可选） |
| 手续费 | 独立记为一笔 expense 交易，挂在转出账户上 |
| 导入导出 | CSV 新增 `transferTo` 列，旧 CSV 无此列按普通交易导入 |
| 分析排除 | 所有按 category 的聚合查询增加 `type != transfer` 过滤 |
| 显示逻辑 | `Transaction.formattedAmountWithSign` 新增 `.transfer` case，显示为 "A→B ¥500" |
| 编辑/删除 | 原子操作：删除转账必须同时撤销双方余额影响 |

**转账数据模型补充**：
- `Transaction.category` 需要从必填改为可选，或为转账引入专用系统分类；**推荐改为可选**，语义更干净
- `Transaction` 新增 `transferAccount: Account?`，表示转入账户
- `Transaction` 新增 `transferGroupId: UUID?`，用于将主转账记录和手续费记录关联为一组
- `Transaction` 新增 `transferFeeAmount: Decimal?` 仅用于 UI 回显；实际余额影响仍由独立 expense 交易承担
- 所有依赖 `category` 的展示和聚合逻辑都需先判断 `type == transfer`

### 功能 10：按账户筛选分析
- 分析标签页添加账户筛选器
- 所有图表和聚合按选定账户过滤
- 新增"账户"分析维度：各账户支出分布

### 功能 11：账户排序与归档
- 长按拖动重新排序
- 归档不再使用的账户（从选择器隐藏，保留历史数据）

### Account 实体新增字段

| 字段 | 类型 | 默认值 | 用途 |

| `createdAt` | Date | now | 创建时间 |
| `updatedAt` | Date | now | 更新时间 |

### AccountType 枚举扩展
- 新增 `.bank = "bank"`（储蓄卡），图标 `"building.columns"`
- 已有 `.card = "card"` 保留不变，显示名改为"银行卡/信用卡"

### Category 实体新增
- 新增系统内置分类"余额调整"（`isSystem: true`，不可删除/编辑）
- 用于余额调整交易
- `Category` 新增 `isSystem: Bool` 字段，默认 `false`
- `CategoryManagementView` 中对 `isSystem == true` 的分类隐藏编辑/删除入口

### FinanceRepository 新增方法

```swift
// 账户 CRUD
func addAccount(name:type:icon:color:initialBalance:notes:) -> Account
func updateAccount(_:name:icon:color:notes:)
func deleteAccount(_:)               // 有交易时抛错

func unarchiveAccount(_:)

// 默认账户

// 余额计算
func getAccountBalance(_:) -> Decimal

// 余额调整

// 账户详情数据

// 排序

// 筛选查询（增强现有方法）
func getTransactions(from:to:accountId:) -> [Transaction]
```

### Transaction 实体（P1 预留字段）

| 字段 | 类型 | 默认值 | 用途 |
|------|------|--------|------|
| `transferAccount` | Account? | nil | 转账转入账户 |
| `transferGroupId` | UUID? | nil | 关联转账主记录与手续费 |
| `transferFeeAmount` | Decimal? | nil | 转账表单回显用手续费金额 |

说明：
- P0 不实现转账 UI，但建议在文档层先固定字段，避免 P1 再次改方案
- 若决定将 `Transaction.category` 改为可选，该变更也应在 P1 一并进入数据模型变更清单

### 导入导出兼容策略

#### JSON 备份（权威格式）
- `TransactionDTO` 新增 `accountId`、`categoryId`
- 保留 `accountName`、`categoryName` 仅作展示与兼容，不再作为主关联键
- 导入 JSON 时优先按 `id` 建立账户/分类关联，找不到时才降级按名称匹配
- `AccountDTO` 新增：`icon`、`color`、`initialBalance`、`sortOrder`、`isArchived`、`notes`、`createdAt`、`updatedAt`
- `CategoryDTO` 新增：`isSystem`

#### CSV（兼容格式）
- CSV 继续以可读性为主，保留按名称导出
- P1 若支持转账，CSV 新增 `transferTo` 列；未识别该列时按普通交易处理
- 对于同名账户，CSV 导入无法 100% 无损恢复，文档中应明确推荐“完整备份请使用 JSON”

## 验证方式

### 正常流程

1. **余额计算**：添加账户设初始余额 1000 → 记一笔支出 200 → 验证余额显示 800
2. **多账户**：添加 3 个不同类型账户 → 验证总资产 = 各账户余额之和
3. **账户选择记账**：在 AddTransactionSheet 选择"微信" → 保存 → 验证交易挂在微信账户
4. **账户选择记忆**：选"微信"记一笔 → 关闭 → 重新打开 → 验证预选仍为"微信"
5. **默认账户切换**：将"支付宝"设为默认 → 新建交易不选账户 → 验证挂在支付宝
6. **余额调整**：实际余额 950 但应用显示 800 → 调整到 950 → 验证差额记录为 income + "余额调整"分类
7. **导入/导出**：导出包含新字段的 JSON → 重新导入 → 验证数据完整

### 边界场景

8. **无默认账户**：手动将所有账户归档（保留默认不可归档规则） → 或数据库异常导致无默认 → 验证 ensureDefault 自动兜底
9. **默认账户被归档**：尝试归档默认账户 → 验证 UI 拒绝并提示
10. **有交易账户删除**：尝试删除有历史交易的账户 → 验证 UI 拒绝并提示交易数量
11. **负余额账户**：信用卡账户消费超过初始余额 → 验证余额显示负数、归入负债
12. **多个同名账户**：添加两个名为"储蓄卡"的账户 → 验证不崩溃、显示正常
13. **同名账户 JSON 备份恢复**：两个同名账户分别记账 → 导出 JSON → 导入 → 验证交易仍关联原账户（按 id 恢复）
14. **迁移后旧数据**：升级后已有 card 类型账户 → 验证类型不变、交易关联正常
15. **存量账户 backfill**：升级前账户无 `sortOrder` → 升级后验证排序稳定、默认账户唯一
16. **调整交易与统计**：创建余额调整 → 验证月度收支总额包含调整金额、分析页可按"余额调整"分类看到
17. **系统分类保护**：尝试编辑/删除"余额调整"分类 → 验证 UI 拒绝
18. **归档账户在导入导出中**：归档一个账户 → 导出 → 导入 → 验证归档状态保留
19. **余额为 0 的账户**：从未设置初始余额 → 验证净资产卡片隐藏或显示合理提示
