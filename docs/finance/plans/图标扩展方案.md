# 记账模块图标库扩容与分组展示修订方案

> 最后更新：2026-04-19
> 状态：Draft
> 适用范围：`CategoryManagementView` / `IconPickerGrid`

---

## 一、背景

当前记账模块的图标选择器 `IconPickerGrid` 使用一组硬编码的 SF Symbol 列表，现状有两个明显问题：

1. 可选图标数量偏少，很多自定义分类找不到贴切图标。
2. 图标以扁平列表展示，浏览效率较低，用户需要反复滚动查找。

但这里要明确一个边界：

**本次需求只做“图标库扩容 + 分组展示优化”，不改财务模块的业务分类体系，不改 `Category` 数据模型，不改已有分类层级语义。**

也就是说：

- `expenseHierarchy` / `incomeHierarchy` 仍然是当前财务模块的业务分类来源。
- `Category.icon` 仍然只是一个 SF Symbol 字符串。
- `IconPickerGrid` 中的分组只是 UI 展示分组，不代表业务上的一级分类。

---

## 二、当前实现基线

### 2.1 当前代码现状

- 文件：`Holo/Holo APP/Holo/Holo/Components/IconPickerGrid.swift`
- 当前 `presetCategoryIcons` 数量：**88 个**
- 展示方式：单层 `LazyVGrid`，4 列，按数组顺序平铺
- 数据结构：`[String]`

### 2.2 当前业务边界

当前财务模块已有明确的分类体系与系统分类约束：

- 业务分类来自 `Category+CoreDataProperties.swift` 的 `expenseHierarchy` / `incomeHierarchy`
- 系统分类单独由 `systemCategories` 维护，目前核心是“余额调整”
- `CategoryPicker` 会过滤掉系统分类，不把它们暴露为普通可选分类

因此，本方案不能把“系统图标”“账户调整”“初始余额”之类内部概念混进普通图标池，否则会误导后续产品和实现。

## 三、目标与非目标

### 3.1 本期目标

1. 保留当前 88 个图标，避免已有分类在编辑时出现“当前图标不在候选列表中”的问题。
2. 新增一批**已验证可用**的 SF Symbol，让总量提升到约 **170 个**。
3. 将图标选择器从扁平列表改为**按主题分组展示**，降低查找成本。
4. 增加自动校验，确保方案中的图标在当前支持的系统版本上可解析。
5. 补齐兼容策略：如果用户已有历史图标不在新图标池中，编辑时仍能正确显示和保留。

### 3.2 非目标

本期明确不做以下内容：

- 不调整 `Category` / `Transaction` / `Account` 的数据结构
- 不修改 `expenseHierarchy` / `incomeHierarchy` 的业务分组定义
- 不新增或修改系统分类
- 不做图标搜索、智能推荐、最近使用排序
- 不做图标迁移脚本，因为当前预设图标已经是 SF Symbol

## 四、关键修订决策

### 4.1 决策一：分组名称改为“展示分组”，不再使用“一级分类”表述

原因：

- “一级分类”在财务模块中是有明确业务含义的；
- 如果文档继续写“13 个一级分类”，很容易被误解为要改现有分类模型；
- 本次实际上只是图标浏览分组，不应与业务分类混用。

结论：

- 文档、代码、评审口径统一使用：**展示分组 / 图标主题组**
- 不再出现“新增系统一级分类”“新增家庭一级分类”这类描述

### 4.2 决策二：系统概念不进入普通图标池

以下内容不作为普通图标分组展示：

- 初始余额
- 账户调整
- 自动记账
- 数据导入
- 重置 / 同步

原因：

- 这些不是用户自定义分类的稳定命名场景；
- 其中一部分是账户体系概念，不是分类概念；
- 会污染图标池语义边界。

结论：

- 普通 `IconPickerGrid` 仅包含用户可用于分类命名的图标
- 系统分类如需独立图标，继续在各自逻辑中硬编码或单独维护

### 4.3 决策三：新增图标必须先通过程序化校验

上一版方案中存在一批在当前 SDK 中不可用的符号名，例如：

- `bbq`
- `arcade.stick.fill`
- `gem`
- `shopping.bag.fill`
- `spa`
- `wand.stars`
- `star.bubble.left.fill`
- `tooth.fill`
- `human.anatomy.fill`
- `spraybottle.fill`
- `bell.and.vase.fill`
- `broom.fill`

结论：

- 文档中不再直接写未校验的候选名
- 所有新增图标均以 `UIImage(systemSymbolName:) != nil` 为准
- 测试不通过的符号必须替换，不允许“先写进方案，实施时再看”

### 4.4 决策四：当前图标缺失时，编辑页必须兼容显示

风险场景：

- 用户历史数据可能包含旧候选池中的图标；
- 未来也可能出现手工写入或版本切换导致的“图标不在目录中”情况。

结论：

- 编辑分类时，如果 `selectedIcon` 不在当前 catalog 中：
  - 在网格顶部插入一个“当前图标”小节；
  - 保留该 icon 可选；
  - 用户不改图标时可直接保存。

这样可以保证这次扩容不会影响已有数据编辑体验。

### 4.5 决策五：SF Symbol 可用性以 iOS 部署目标为准

当前 app 部署目标为 **iOS 26.2**（`IPHONEOS_DEPLOYMENT_TARGET`），因此本方案中所有 SF Symbol 在目标平台上均可解析。但以下规则仍须遵守：

- 新增图标时，**必须使用 `UIImage(systemSymbolName:) != nil` 校验**（iOS 运行时），而非 `NSImage`（macOS 运行时）。两者的可用符号集存在差异。
- 校验应在 **iOS Simulator 或真机** 上执行，不能仅在 macOS 命令行验证。
- 若未来降低部署目标，需重新运行 7.4 节的自动化校验测试。

## 五、展示分组设计

### 5.1 分组原则

本次分组只服务于“找图标”而不是“建模分类”，因此按用户搜索习惯组织，而不是强行映射财务一级分类。

分组原则如下：

1. 高辨识度优先，用户能快速猜到某个图标大致在哪组。
2. 尽量减少跨组重复；允许极少数图标因语义更贴合被放入单一最优组。
3. 收入相关图标单独成组，方便新增收入类分类时快速查找。
4. 系统概念不单独成组。

## 七、实现方案

### 7.1 数据结构

不再把图标目录硬编码在 `IconPickerGrid.swift` 内部，改为单独抽出 catalog 常量文件。

建议新增：

- `Components/CategoryIconCatalog.swift`

建议结构：

```swift
struct IconPickerSection: Identifiable {
    let id: String
    let title: String
    let icons: [String]
}

enum CategoryIconCatalog {
    /// 12 个展示分组，每个分组包含图标名称数组
    static let sections: [IconPickerSection] = [
        IconPickerSection(id: “food”, title: “餐饮”, icons: [...]),
        IconPickerSection(id: “transport”, title: “交通”, icons: [...]),
        // ... 共 12 个分组
    ]

    /// 所有图标的扁平列表，从 sections 派生，保持分组顺序
    static let allIcons: [String] = {
        sections.flatMap(\.icons)
    }()

    /// 原有 88 个预设图标的集合，用于测试兼容性校验
    static let legacyPresetIcons: Set<String> = [
        “fork.knife”, “sunrise.fill”, /* ... 全部 88 个 */
    ]
}
```

设计要点：

- **单一数据源**：`sections` 是唯一定义图标的位置，`allIcons` 由它派生，避免维护多个数组再合并去重。
- `legacyPresetIcons` 仅用于测试（确保旧图标不丢失），不参与运行时逻辑。
- `IconPickerGrid` 只负责渲染，不再同时承担”目录定义”职责；
- 后续若要加搜索、最近使用，也可以复用同一份 catalog。

### 7.2 UI 渲染

`IconPickerGrid` 从单层 `LazyVGrid` 改为：

- 外层 `ScrollView`
- 外层按 section 使用 `LazyVStack`（section 级别懒加载）
- 每个 section 内部使用**非 Lazy 的 4 列布局**（单 section 内图标数量有限，约 10-18 个，无需懒加载）
- section 顶部显示标题

> **注意**：SwiftUI 中嵌套 Lazy 容器（如 `LazyVStack` 内嵌 `LazyVGrid`）会导致内层丧失懒加载能力，可能一次性渲染全部单元格。正确做法是外层 Lazy + 内层非 Lazy。

交互保持不变：

- 单选模式
- 点击切换选中
- 当前选中高亮

新增兼容逻辑：

- 若 `selectedIcon` 不在 `CategoryIconCatalog.allIcons` 中，最顶部插入一个“当前图标”分组

### 7.3 兼容现有调用方

以下调用方无需改业务逻辑，只需适配新的 section 化图标数据：

- `AddCategorySheet`
- `EditCategorySheet`

其中：

- 默认值仍可使用 `presetCategoryIcons.first` 的思路，但应改为 `CategoryIconCatalog.allIcons.first`
- 编辑态不强制要求当前 icon 一定在 catalog 中

### 7.4 自动化校验

建议新增测试文件：

- `HoloTests/Components/CategoryIconCatalogTests.swift`

至少覆盖以下测试：

1. `testAllSymbolsResolvable`
   - 遍历 `CategoryIconCatalog.allIcons`
   - 校验 `UIImage(systemSymbolName:) != nil`（iOS 运行时，不是 macOS 的 `NSImage`）

2. `testLegacyPresetIconsStillIncluded`
   - 确保当前 88 个基线图标全部包含在新 catalog 中

3. `testNoDuplicateIconsAcrossCatalog`
   - 确保 `allIcons` 去重后数量与预期一致

4. `testSelectedIconFallbackSection`
   - 当传入不在目录中的 `selectedIcon` 时，UI 仍能渲染并保持可选

自动化校验要成为验收条件之一，不能只依赖 Preview 肉眼检查。

## 十、验收标准

满足以下条件方可算完成：

1. 所有现有 88 个图标仍在候选池中。
2. 新增图标全部通过程序化解析校验。

## 十一、风险与缓解

| 风险 | 级别 | 说明 | 缓解措施 |
|------|------|------|---------|
| 图标过多导致选择疲劳 | 中 | 数量上升后浏览成本仍可能偏高 | 控制在约 170，分组清晰，后续再评估搜索 |
| 部分图标在小尺寸下辨识度差 | 中 | 虽然可解析，但 22pt 下未必好看 | 优先 `.fill` 变体，人工预览后再微调 |
| 历史图标不在目录中 | 中 | 可能来自旧版本或手工写入 | 提供 fallback section，允许保留 |
| 目录和业务语义再度混淆 | 高 | 后续评审可能又把分组当成业务分类 | 文档、代码命名统一使用“展示分组” |

## 十三、结论

这版修订后的方案把边界收窄为一件明确的事：

**只升级图标选择体验，不碰财务分类建模。**

具体落地上，采用：

- 保留现有 88 个图标
- 新增一批已验证可用的 SF Symbol
- 用 12 个“展示分组”替代扁平列表
- 用自动化校验替代纯手工验证
- 用 fallback 机制保证历史数据编辑兼容

这版可以直接作为研发实施稿继续推进。
