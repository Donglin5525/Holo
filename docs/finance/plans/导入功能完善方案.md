# 财务导入功能完善方案

## Context

当前导入功能处于半成品状态：CSV 解析和字段映射可用，但分类匹配结果用户无法审核/修改，新科目静默创建，模糊匹配阈值过低（0.6）存在误匹配风险，导入后无法可靠撤回，重复导入也会产生全量重复交易。需要将导入能力完善到生产可用级别。

本方案的核心目标不是“让 CSV 能导进去”，而是让财务数据导入满足三个底线：

1. **导入前可审查**：字段映射、日期解析、分类匹配、疑似重复都要在预览阶段可见。
2. **导入中可追踪**：每条导入交易、自动创建的分类/账户、导入批次都有可追踪标识。
3. **导入后可恢复**：24 小时内支持按批次撤回，但必须处理用户导入后编辑过交易的冲突。

## 关键产品规则

### 分类匹配规则

分类匹配必须使用 `type + primaryCategory + subCategory` 三元信息，不能只按二级分类名匹配。

- 精确匹配：二级名称相同且一级分类也匹配，才可视为 `.exact`。
- 同义词匹配：一级分类和二级分类分别经过 `CategorySynonymMapping` 标准化后都匹配，才可视为 `.synonym`。
- 学习映射：**必须修复 key 碰撞问题**。当前 key 格式 `type|sub.lowercased()` 导致 `餐饮/其他` 和 `购物/其他` 串线。改为 `type|primary|sub.lowercased()`，`CategoryLearnedMapping.record` 和 `lookup` 都要加 `primaryCategory` 参数。这是 P0 阻塞项。**旧 key 迁移**：key 格式变更后旧数据无法可靠复用，启动时只删除旧格式 key（只有 1 个 `|`），保留已经是 `type|primary|sub` 的新格式 key；可用 `mappingSchemaVersion` 避免重复扫描。
- 模糊匹配：阈值提升到 0.75；即使 confidence 很高，也必须标记 `.fuzzy` 并要求用户确认。
- 一级分类不匹配时，精确/同义词降级为 `.fuzzy`（需确认但保留推荐候选），而非直接降为 `.unmatched`（用户看到空白选择框）。这样用户仍能看到合理的推荐，减少手动操作。

### 字段解析规则

导入前必须展示字段映射，通用 CSV 必须允许用户修改映射后重新解析。

- 金额列仍为唯一硬必填字段。
- 日期解析失败不能静默使用今天，必须生成 `ParseWarning`，默认阻止该行导入；用户在预览中明确确认后，才允许使用今天作为兜底日期。
- 类型缺失或无法识别时可以按金额正负回退，但必须在预览中显示警告计数。
- 当前 CSV parser 若仍采用逐行解析，文档和 UI 必须明确“不支持引号内换行”；如果要支持，需要改为真正的 CSV 状态机 parser，先解析完整文件再拆行。

### 重复导入规则

导入前生成稳定指纹，用于检测疑似重复交易：

```swift
date(day-level or second-level) + amount + type + accountName + primaryCategory + subCategory + normalizedNote
```

预览阶段展示疑似重复数量，并提供导入策略：

- `skipDuplicates`：默认策略，跳过疑似重复交易。
- `importAll`：用户确认后强制全部导入。
- `reviewDuplicates`：展示重复候选，逐条选择导入/跳过。

### 撤回规则

撤回不是简单删除 `importBatchId == batchId` 的交易。必须先检查冲突：

- **P1 简化方案**：撤回时只检查 `updatedAt == importOriginalUpdatedAt`（即导入后未被编辑）。存在已编辑交易时阻止整批撤回，UI 提示冲突条数。
- `importOriginalUpdatedAt` 字段仅在 P1 保留，P0 不需要。
- 自动创建的分类/账户只在没有任何剩余交易引用时删除。

撤回入口不只记录最近一次导入。需要持久化最近 24 小时内的 `ImportBatchRecord` 列表，支持用户从批次历史中选择撤回。

### 2.2 CategoryMatcherService.swift

**阈值提升**：`fuzzyMatchThreshold: 0.6` → `0.75`

**父分类映射**：

- `buildParentNameMap(categories:) -> [UUID: String]`
- `parentName(for category: Category, parentNameMap: [UUID: String]) -> String?`
- `primaryMatches(originalPrimary:parentName:type:) -> Bool`
- `primaryCompatibilityScore(originalPrimary:parentName:type:) -> Double`

**匹配顺序**：

1. 在同类型分类中查找“一级 + 二级”均匹配的精确项。
2. 查找一级/二级同义词均匹配的项。
3. 查找用户学习映射，学习映射必须包含 type、originalPrimary、originalSub。
4. 模糊匹配二级名，并按一级分类兼容度加权。
5. 无匹配时返回 unmatched。

**强约束**：

- 不允许只因为二级分类名相同就返回 `.exact`。
- 一级分类完全不匹配时，最高只能作为候选，不能直接通过。
- fuzzy 无论分数多高都必须确认。

### 3.4 撤回实现

```swift

```

流程：

1. 查询该批次所有交易。
2. 如果任一交易 `updatedAt != importOriginalUpdatedAt`，返回或抛出“存在已编辑交易”的业务错误，默认阻止整批撤回。
3. 删除该批次未编辑交易。
4. 删除该批次创建且没有任何交易引用的分类。
5. 删除该批次创建且没有任何交易引用的账户。
6. save。
7. 删除对应 `ImportBatchRecord`。
8. 发送 `.financeDataDidChange` 通知。

## 验证计划

### 单元测试（必须覆盖）

匹配引擎改造是最容易出回归的部分，以下场景必须有单元测试：

- 三元匹配：相同 sub 不同 primary 时不应作为 `.exact` / `.synonym` 自动通过（`餐饮/其他` ≠ `购物/其他`），最多降级为需确认的 `.fuzzy` 候选
- 一级分类加权：primary 匹配时 confidence 提升，不匹配时衰减
- `CategoryLearnedMapping` 新 key 格式：写入 `type|primary|sub` 后能正确读回，旧格式 key 不串线
- 指纹一致性（P1）：相同数据→相同指纹，微小差异→不同指纹
- 降级行为：一级不匹配时精确/同义词降级为 `.fuzzy` 而非 `.unmatched`

### 功能验证

1. **编译验证**：全量 `xcodebuild` 编译通过。
2. **Core Data 迁移验证**：在已有数据的设备上安装新版本，确认自动轻量迁移成功（添加可选字段无需手动迁移脚本，但需验证）。
3. **BatchImportResult 编译面验证**：确认 `batchImportTransactions`、`batchImportTransactionsWithMatchResults`、所有初始化点均补齐新字段。
4. **导出回归**：导出 CSV/JSON 不包含 `importBatchId`、`importFingerprint`、`importOriginalUpdatedAt`。
5. **字段映射验证**：通用 CSV 自动映射错误时，手动改映射后重新解析，预览数据随之更新。
6. **日期警告验证**：导入日期无法解析的行，预览显示 warning，不再悄悄当作今天。
7. **一级分类匹配验证**：存在 `餐饮/其他` 与 `购物/其他` 时，导入 `购物/其他` 不得匹配到 `餐饮/其他`。
8. **模糊匹配验证**：高 confidence fuzzy 仍必须确认后才能导入。
9. **学习映射验证**：手动把 `餐饮/早饭` 映射到 `餐饮/早餐` 后，下次相同 type+primary+sub 可自动匹配；不同 primary 不串线。
10. **重复导入验证**：同一 CSV 连续导入两次，默认跳过重复项；选择强制导入时才新增重复交易。
11. **撤回成功验证**：导入一批后立即撤回，交易删除，自动创建且未被引用的分类/账户删除。
12. **撤回冲突验证**：导入后编辑其中一条交易，再撤回，UI 阻止整批撤回并展示冲突条数。
13. **多批撤回验证**：连续导入两批，最近 24 小时批次列表中两批都可见，撤回其中一批不影响另一批。
14. **刷新验证**：导入、跳过重复、撤回后均发送 `.financeDataDidChange`，账本列表和统计页刷新。
