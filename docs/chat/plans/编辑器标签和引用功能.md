# Holo iOS 编辑器 `@` 引用与 `#` 标签功能方案

## 1. 方案概述

Holo 的 `@` 和 `#` 不应该只是输入后变色的普通文字，而应该是两种带唯一 ID 的结构化内容。

| 类型 | 功能定位         | 展示示例              | 实际关联对象 |
| ---- | ---------------- | --------------------- | ------------ |
| `#`  | 对笔记进行分类   | `#工作/Holo`          | 标签 ID      |
| `@`  | 引用一条历史记录 | `@关于标签体系的思考` | 笔记 ID      |

推荐技术方案：

```text
SwiftUI 页面
    ↓
UIViewRepresentable
    ↓
UITextView 编辑器
    ↓
NSAttributedString 富文本
    ↓
TagToken / ReferenceToken
    ↓
结构化 JSON + 纯文本 + 关系表
```

核心产品原则：

1. 编辑器底部、系统键盘上方固定展示 `#` 和 `@` 快捷按钮。
2. 用户也可以直接通过键盘输入 `#` 或 `@`。
3. 候选面板展示在系统键盘上方，打开期间不收起键盘。
4. 标签和引用选择完成后，作为不可从中间拆开的完整节点插入。
5. 编辑器中使用轻量行内样式，不展示大型引用卡片。
6. 查看笔记时，再展示引用预览和反向引用。

# 2. 第一版功能范围

## 2.1 `#` 标签功能

第一版支持：

- 输入 `#` 调出最近使用的标签。
- 输入关键词搜索标签。
- 选择已有标签。
- 创建不存在的标签。
- 支持多级标签，例如 `#工作/Holo/编辑器`。
- 点击标签查看该标签下的全部笔记。
- 标签重命名后，历史笔记同步展示新名称。
- 标签删除后保留原文字，但解除标签关系。

第一版暂不支持：

- AI 自动添加标签。
- AI 自动合并标签。
- 拖拽调整标签层级。
- 标签别名。
- 标签关系图。

## 2.2 `@` 笔记引用功能

第一版支持：

- 输入 `@` 展示最近编辑的笔记。
- 输入关键词搜索历史笔记。
- 搜索笔记正文、标签和日期。
- 引用另一条笔记。
- 点击引用进入目标笔记。
- 查看哪些笔记引用了当前笔记。
- 被引用笔记删除后保留引用快照。
- 当前笔记不能引用自己。

第一版暂不支持：

- AI 自动推荐相关笔记。
- 在编辑器中展开完整引用卡片。
- 引用正文中的具体段落。
- 跨用户引用。
- 双向同步修改引用文字。

# 3. iOS 编辑器界面结构

## 3.1 默认布局

```text
┌─────────────────────────────┐
│                             │
│         笔记正文编辑区        │
│                             │
│                             │
├─────────────────────────────┤
│  图片   语音   待办   #   @   │
├─────────────────────────────┤
│                             │
│         iOS 系统键盘          │
└─────────────────────────────┘
```

建议将 `#` 和 `@` 固定放在键盘上方的编辑工具栏中。

中文键盘输入 `#` 和 `@` 的操作成本较高，快捷按钮可以提升功能发现率和使用频率。

## 3.2 点击快捷按钮

用户点击 `#` 后：

1. 在当前光标位置插入 `#`。
2. 进入标签搜索状态。
3. 在键盘上方展开标签候选面板。
4. 光标继续停留在 `#` 后面。
5. 用户继续输入关键词。

用户点击 `@` 后：

1. 在当前光标位置插入 `@`。
2. 进入笔记搜索状态。
3. 展示最近编辑的笔记。
4. 用户继续输入关键词。
5. 选择笔记后替换当前的 `@关键词`。

快捷按钮和手动输入必须共用同一套触发逻辑。

# 4. `#` 标签详细交互

## 4.1 输入 `#`

用户输入：

```text
今天重新设计了编辑器 #
```

候选面板展示：

```text
最近使用

#工作/Holo
#产品设计
#灵感
#数据分析
#生活
```

默认排序：

```text
最近使用时间
> 使用次数
> 标签创建时间
```

## 4.2 输入标签关键词

用户继续输入：

```text
今天重新设计了编辑器 #产品
```

候选面板展示：

```text
#产品
#产品/Holo
#产品/Holo/编辑器
#产品/交互设计

＋ 创建“#产品”
```

候选排序权重：

```text
完整名称匹配
> 完整路径前缀匹配
> 子级名称匹配
> 最近使用
> 使用频率
> 关键词包含
```

只有不存在完全相同的标签时，才展示“创建标签”。

## 4.3 选择标签

用户选择：

```text
#产品/Holo/编辑器
```

编辑器内容变为：

```text
今天重新设计了编辑器 #产品/Holo/编辑器 
```

系统执行：

1. 用标签节点替换用户输入的 `#产品`。
2. 在标签后自动插入一个空格。
3. 光标移动到空格后。
4. 关闭候选面板。
5. 保持系统键盘打开。
6. 更新标签最近使用时间。
7. 标记当前笔记内容已发生变化。

## 4.4 标签视觉样式

建议样式：

- 正文同字号。
- Holo 品牌紫色文字。
- 浅紫色背景。
- 圆角 4～6pt。
- 左右增加少量内边距。
- 不使用尺寸过大的胶囊按钮。
- 深色模式下自动切换背景与文字颜色。

示例：

```text
#产品/Holo/编辑器
```

## 4.5 标签触发条件

以下场景允许触发：

```text
#产品

今天记录一下 #产品

今天记录一下，#产品

换行后输入
#产品
```

以下场景不触发：

```text
abc#123

https://example.com/#page

邮箱或链接中的#

#define
```

判断规则：

```text
# 位于正文开头
或者
# 前一个字符是空格、换行或标点
```

标签输入遇到以下字符后结束：

- 空格。
- 换行。
- 中文标点。
- 英文标点。
- 光标移动到其他位置。

`/` 不结束输入，因为需要支持多级标签。

## 4.6 标签点击行为

### 阅读状态

点击标签：

```text
进入标签详情页
→ 展示该标签下的所有记录
```

### 编辑状态

编辑过程中点击标签，不直接跳转。

建议先选中标签，并在标签附近展示操作菜单：

```text
查看标签
移除标签
```

选择“移除标签”后：

```text
#产品/Holo
```

转为：

```text
产品/Holo
```

即解除标签关系，但保留文字内容。

# 5. `@` 笔记引用详细交互

## 5.1 输入 `@`

用户输入：

```text
这个想法和之前的内容有关 @
```

候选面板默认展示最近编辑的笔记：

```text
最近记录

关于标签体系应该如何设计
标签既可以由用户创建，也可以由 AI……
#Holo/标签体系
今天 01:42

Holo 编辑器结构调整
目前 TextEditor 无法满足……
#Holo/编辑器
昨天 22:18
```

每个候选项建议展示：

- 笔记首行。
- 正文摘要。
- 最多两个相关标签。
- 创建或更新时间。
- 是否已被当前笔记引用。

## 5.2 搜索笔记

用户输入：

```text
@标签
```

搜索范围：

1. 笔记首行。
2. 笔记正文。
3. 标签名称。
4. 笔记日期。
5. 已经引用过的笔记。

排序权重：

```text
首行完全匹配
> 首行前缀匹配
> 标签匹配
> 正文匹配
> 最近编辑时间
```

当前正在编辑的笔记必须从搜索结果中排除。

## 5.3 选择笔记

用户选择笔记后，将：

```text
@标签
```

替换为：

```text
@关于标签体系应该如何设计
```

推荐展示样式：

```text
🔗 @关于标签体系应该如何设计…
```

视觉规则：

- 使用轻量链接图标。
- 使用品牌紫色文字。
- 使用浅紫色背景。
- 整个引用允许跟随正文换行。
- 不允许用户进入引用内部编辑。
- 最多显示约 20～24 个字符。
- 超出部分使用省略号。

内部实际存储：

```json
{
  "type": "reference",
  "noteId": "4A02E6F1-8DB8-4A42-BD10-9821B53D41F8",
  "displayText": "关于标签体系应该如何设计",
  "snapshot": "标签既可以由用户创建，也可以由 AI 创建……"
}
```

## 5.4 引用点击行为

### 阅读状态

点击引用：

```text
打开目标笔记详情页
```

建议使用 `NavigationStack` 正常入栈，而不是全屏弹窗。

### 编辑状态

点击引用后先选中引用节点，并展示：

```text
查看记录
取消引用
```

防止用户编辑正文时误触进入其他页面。

## 5.5 被引用笔记删除

目标笔记删除后，来源笔记中的引用不应直接消失。

引用显示为：

```text
@原记录已删除
```

点击后展示插入引用时保存的摘要快照：

```text
原记录已删除

标签既可以由用户创建，也可以由 AI 创建……
```

可提供以下操作：

- 查看历史摘要。
- 删除当前引用。
- 从回收站恢复原记录。

# 6. 候选面板设计

## 6.1 展示位置

候选面板放在 Holo 编辑工具栏上方：

```text
┌─────────────────────────────┐
│                             │
│         正文编辑区域          │
│                             │
├─────────────────────────────┤
│       标签或引用候选列表       │
│       高度 180～280pt         │
├─────────────────────────────┤
│ 图片   语音   待办   #   @     │
├─────────────────────────────┤
│         iOS 系统键盘          │
└─────────────────────────────┘
```

展示规则：

- 小屏幕设备最多展示约 3 条。
- 大屏幕设备最多展示约 5 条。
- 列表内部可以滚动。
- 候选面板不遮挡当前光标所在行。
- 键盘始终保持打开。
- 输入关键词时实时更新结果。

## 6.2 面板关闭规则

iOS 不需要考虑 ESC 键。

通过以下行为关闭面板：

1. 用户选择一个候选项。
2. 用户点击面板关闭按钮。
3. 用户删除触发字符 `#` 或 `@`。
4. 用户输入空格或换行，结束搜索。
5. 用户点击正文其他位置。
6. 光标离开当前触发范围。
7. 用户主动收起系统键盘。
8. 用户离开编辑页面。
9. 用户点击工具栏其他功能。

关闭面板时，不删除用户已经输入的普通文字。

例如用户输入：

```text
@标签体系
```

随后主动关闭候选面板，保留：

```text
@标签体系
```

但它只是普通文本，不建立笔记引用关系。

# 7. 编辑器技术架构

## 7.1 编辑器选型

Holo 页面继续使用 SwiftUI，但编辑器内部建议使用：

```text
UIViewRepresentable + UITextView
```

不建议直接使用 SwiftUI 原生 `TextEditor`。

原因：

- 需要准确获取光标位置。
- 需要监听选区变化。
- 需要设置局部富文本属性。
- 需要阻止用户进入 Token 内部编辑。
- 需要处理标签和引用的点击事件。
- 需要处理中文输入法组合状态。
- 需要控制复制、粘贴和删除。
- 需要支持 Undo 和 Redo。
- 需要处理富文本序列化。

## 7.2 模块划分

```text
HoloRichTextEditor
├── HoloTextView
├── EditorCoordinator
├── TriggerDetector
├── TokenManager
├── TagSuggestionService
├── ReferenceSuggestionService
├── RichContentSerializer
├── RelationIndexer
└── EditorSuggestionPanel
```

### `HoloRichTextEditor`

负责：

- SwiftUI 和 UIKit 的通信。
- 接收当前笔记内容。
- 向外输出编辑结果。
- 控制候选面板状态。
- 控制编辑器焦点。

### `HoloTextView`

基于 `UITextView` 封装。

负责：

- 富文本输入。
- 光标和选区。
- 点击位置识别。
- Token 原子化处理。
- 动态字体和深色模式。

### `EditorCoordinator`

实现：

```swift
UITextViewDelegate
```

负责：

- 监听正文变化。
- 监听光标变化。
- 判断中文输入法状态。
- 判断是否出现 `#` 或 `@`。
- 处理 Token 删除。
- 调用标签和笔记搜索服务。
- 通知 SwiftUI 更新状态。

### `TriggerDetector`

负责识别当前光标对应的触发状态：

```swift
enum TriggerContext {
    case none

    case tag(
        range: NSRange,
        query: String
    )

    case reference(
        range: NSRange,
        query: String
    )
}
```

### `TokenManager`

负责：

- 插入标签节点。
- 插入引用节点。
- 删除完整节点。
- 判断光标是否进入节点。
- 获取点击位置对应的节点。
- 更新节点样式。
- 将 Token 转为普通文本。

### `RichContentSerializer`

负责以下数据转换：

```text
NSAttributedString
↔
ContentNode 数组
↔
JSON
```

# 8. 编辑器状态机

建议定义以下状态：

```swift
enum EditorState {
    case idle

    case composing

    case searchingTag(
        triggerRange: NSRange,
        query: String
    )

    case searchingReference(
        triggerRange: NSRange,
        query: String
    )

    case tokenSelected(
        tokenId: UUID
    )
}
```

状态说明：

| 状态                 | 说明                               |
| -------------------- | ---------------------------------- |
| `idle`               | 普通编辑状态                       |
| `composing`          | 中文拼音、日文假名等输入法正在组合 |
| `searchingTag`       | 正在搜索标签                       |
| `searchingReference` | 正在搜索引用笔记                   |
| `tokenSelected`      | 用户选中了已有标签或引用           |

状态流转：

```text
idle
  ↓ 输入 #
searchingTag
  ↓ 选择标签
idle
idle
  ↓ 输入 @
searchingReference
  ↓ 选择笔记
idle
idle
  ↓ 点击已有 Token
tokenSelected
  ↓ 点击正文
idle
```

# 9. 中文输入法处理

中文输入法是 iOS 编辑器最容易出问题的部分。

用户使用拼音输入：

```text
chanpin
```

在最终确认成“产品”之前，中间内容属于输入法的组合文字。

因此监听文字变化时，必须判断：

```swift
func textViewDidChange(_ textView: UITextView) {
    guard textView.markedTextRange == nil else {
        editorState = .composing
        return
    }

    detectCurrentTrigger(textView)
}
```

规则：

```text
markedTextRange 不为空
→ 不重新设置富文本样式
→ 不搜索标签
→ 不搜索笔记
→ 不修改 Token
→ 不移动光标
```

等用户确认中文后，再执行触发检测。

否则可能出现：

- 拼音还没确认就不断刷新候选列表。
- 光标位置跳动。
- 中文候选文字被富文本样式破坏。
- 输入法候选栏异常。
- 拼音输入到一半被替换成 Token。

# 10. 触发检测逻辑

每次用户完成一次有效输入后：

1. 获取当前光标位置。
2. 从光标位置向前扫描当前输入片段。
3. 找到最近的 `#` 或 `@`。
4. 判断触发字符是否合法。
5. 获取触发字符到光标之间的关键词。
6. 更新编辑器状态。
7. 调用对应搜索服务。

伪代码：

```swift
func detectTrigger(
    text: NSString,
    cursor: Int
) -> TriggerContext {

    guard cursor > 0 else {
        return .none
    }

    let scanStart = findCurrentSegmentStart(
        text: text,
        cursor: cursor
    )

    let range = NSRange(
        location: scanStart,
        length: cursor - scanStart
    )

    let segment = text.substring(with: range)

    if segment.hasPrefix("#") {
        return .tag(
            range: range,
            query: String(segment.dropFirst())
        )
    }

    if segment.hasPrefix("@") {
        return .reference(
            range: range,
            query: String(segment.dropFirst())
        )
    }

    return .none
}
```

不要在每次输入时扫描整篇笔记。

只需要扫描：

- 当前光标所在段落。
- 当前光标前的一小段文本。
- 或当前空格、换行之后的内容。

# 11. Token 实现方案

## 11.1 使用富文本自定义属性

标签和引用都使用 `NSAttributedString` 自定义属性：

```swift
extension NSAttributedString.Key {
    static let holoTokenType =
        NSAttributedString.Key("holoTokenType")

    static let holoEntityId =
        NSAttributedString.Key("holoEntityId")

    static let holoDisplayText =
        NSAttributedString.Key("holoDisplayText")
}
```

标签节点属性：

```swift
[
    .holoTokenType: "tag",
    .holoEntityId: tag.id.uuidString,
    .holoDisplayText: tag.fullPath
]
```

引用节点属性：

```swift
[
    .holoTokenType: "reference",
    .holoEntityId: note.id.uuidString,
    .holoDisplayText: note.firstLine
]
```

## 11.2 不建议第一版使用 `NSTextAttachment`

第一版不建议把标签和引用做成图片附件。

富文本属性更适合 Holo：

- 可以正常换行。
- 支持动态字体。
- 支持深色模式。
- 复制时能够保留文字。
- VoiceOver 更容易读取。
- 字体基线更稳定。
- 开发复杂度更低。

## 11.3 Token 原子化编辑

已确认的 Token 不允许光标进入中间。

例如：

```text
#产品/Holo
```

不允许出现：

```text
#产品/|Holo
```

用户点击 Token 时，只允许：

- 将光标放在 Token 前面。
- 将光标放在 Token 后面。
- 选中整个 Token。

按退格键删除：

```text
正文 #产品/Holo|
```

一次退格删除整个标签，而不是只删除最后一个字。

可以在以下代理方法中拦截：

```swift
func textView(
    _ textView: UITextView,
    shouldChangeTextIn range: NSRange,
    replacementText text: String
) -> Bool
```

处理逻辑：

```text
如果删除范围碰到 Token
→ 扩大删除范围到完整 Token
→ 手动删除完整 Token
→ 返回 false
```

程序主动修改 `attributedText` 后，需要主动执行：

- 更新编辑器状态。
- 更新结构化内容。
- 标记笔记已修改。
- 触发自动保存。

```json
  {
```

| `plainText`       | 全文搜索、AI 分析、导出、分享 |
| `firstLine`       | 引用候选列表标题              |
| `syncStatus`      | 判断本地和服务端同步状态      |

```text

- name
```

| ------ | ---------------- | -------- |

```text

- noteId
```

唯一约束：


```text

- sourceNoteId
- targetNoteId
- displaySnapshot
- createdAt
- deletedAt
```

建议唯一约束：

```text
sourceNoteId + targetNoteId
```

一条笔记中可以多次显示同一引用，但反向引用列表只展示一条来源关系。

# 13. 保存机制

每次保存笔记时执行：

```text
1. 将 NSAttributedString 解析为 ContentNode
2. 序列化为 richContentJSON
3. 生成 plainText
4. 提取全部 tagId
5. 提取全部 targetNoteId
6. 对比原有标签关系
7. 对比原有引用关系
8. 在同一个数据库事务中更新
9. 将变更加入同步队列
```

不要只在用户选择标签或引用时写关系表。

因为用户可能执行：

- 撤销操作。
- 删除整段内容。
- 粘贴覆盖。
- 清空笔记。
- 恢复历史版本。
- 删除包含 Token 的选区。

最终应该以编辑器当前的结构化内容为准，重新校验关系。

## 13.1 自动保存建议

建议触发时机：

- 输入停止约 500～800 毫秒。
- 插入标签后立即保存。
- 插入引用后立即保存。
- 删除 Token 后立即保存。
- App 进入后台前保存。
- 用户离开编辑页面前保存。

本地保存和服务器同步分开处理：

```text
编辑器变更
→ 立即保存本地
→ 加入同步队列
→ 网络可用时上传服务端
```

# 14. 本地搜索方案

## 14.1 标签搜索

查询为空时展示：

```text

```

输入关键词后依次匹配：

1. 完整路径完全匹配。
2. 完整路径前缀匹配。
3. 当前节点名称匹配。
4. 子级名称匹配。
5. 包含关键词。

最多返回 20 条。

## 14.2 引用笔记搜索

查询为空时展示：

```text
最近编辑的笔记
最近引用过的笔记
```

输入关键词后搜索：

```text
firstLine
plainText
标签路径
创建日期
更新时间
```

第一版建议完全使用本地数据搜索，保证：

- 离线可用。
- 响应足够快。
- 不依赖网络请求。
- 尚未上传的笔记也能互相引用。

## 14.3 搜索防抖

建议用户停止输入约 150 毫秒后再查询：

```swift
Task {
    try await Task.sleep(
        for: .milliseconds(150)
    )

    guard !Task.isCancelled else {
        return
    }

    await searchSuggestions()
}
```

每次关键词变化时，取消上一次搜索任务，避免旧结果覆盖新结果。

# 15. 标签重命名与删除

## 15.1 标签重命名

标签节点内部保存：

```text
tagId + displayPath 快照
```

加载笔记时：

1. 根据 `tagId` 查询最新标签名称。
2. 查询成功时展示最新路径。
3. 查询失败时展示保存时的快照文字。

例如标签从：

```text
#工作/Holo
```

重命名为：

```text
#项目/Holo
```

历史笔记自动展示新名称。

## 15.2 标签删除

删除标签时，不直接删除历史笔记中的文字。

处理方式：

```text
结构化标签：#工作/Holo
        ↓
普通文本：#工作/Holo
```

系统执行：

- 删除标签 ID。
- 删除标签关系。
- 保留原始文字。
- 将该 Token 转成普通文本。

# 16. 引用更新与删除

## 16.1 目标笔记被修改

引用节点中的显示文字建议保留插入时的快照。

例如目标笔记原名称：

```text
标签体系应该如何设计
```

后来修改为：

```text
Holo 标签体系完整方案
```

来源笔记仍可以显示：

```text
@标签体系应该如何设计
```

用户点击后进入目标笔记的最新内容。

这样做的原因是，引用文字本身属于来源笔记当时的表达语境，不应该被目标笔记自动改写。

## 16.2 目标笔记被删除

引用状态：

```swift
enum ReferenceStatus {
    case available
    case deleted
    case syncing
    case unavailable
}
```

展示方式：

| 状态       | 展示                     |
| ---------- | ------------------------ |
| 正常       | 紫色引用                 |
| 已删除     | 灰色引用                 |
| 同步中     | 引用右侧展示轻量加载状态 |
| 本地不存在 | 展示“暂时无法读取”       |

# 17. 复制、粘贴和分享

## 17.1 复制到其他应用

标签复制为普通文字：

```text

```

引用复制为普通文字：

```text
@标签体系应该如何设计
```

外部应用不会获得 Holo 内部的标签 ID 或笔记 ID。

## 17.2 Holo 内部复制

Holo 可以定义自己的剪贴板数据格式：

```text
com.holo.rich-content
```

用户在 Holo 内部复制和粘贴时，可以保留：

- 标签 ID。
- 引用笔记 ID。
- Token 类型。
- 富文本样式。

## 17.3 从外部应用粘贴

外部粘贴内容先按普通文本处理。

例如粘贴：

```text
今天完成了设计 #产品
```

第一版可以保留为普通文字。

后续可以增加提示：

```text

```

外部粘贴的：

```text

```

不能自动转换成引用，因为无法确定它指向哪一条笔记。

# 18. 撤销与恢复

以下操作必须作为一次完整的 Undo 操作：

- 插入一个标签。
- 删除一个标签。
- 插入一条引用。
- 删除一条引用。
- 将搜索文字替换成 Token。

例如：

```text

```

用户选择笔记后变成：

```text
@标签体系应该如何设计
```

用户撤销一次后，应恢复成：

```text

```

而不是逐字删除引用文字。

# 24. 最终技术原则

第一版最重要的不是视觉动画，而是以下六件事：

```text
输入稳定
中文输入法不出错
Token 不会被破坏
标签和引用 ID 永远准确
离线状态正常工作
关系数据可以可靠恢复
```

标签和引用必须保存为结构化节点，不能只保存为普通字符串。

正确方式：

```json
{
  "type": "tag",
  "id": "TAG-001",
  "displayPath": "产品/Holo"
}
{
  "type": "reference",
  "noteId": "NOTE-1024",
  "displayText": "标签体系应该如何设计"
}
```

不建议只保存：

```text
今天研究了 #产品/Holo，并引用了 @标签体系应该如何设计
```

因为普通字符串无法稳定解决：

- 标签重命名。
- 标签删除。
- 引用跳转。
- 同名笔记区分。
- 反向引用。
- 离线同步。
- 多设备冲突。
- 数据恢复。

因此，Holo 的实现核心应该是：

```text
用户输入
    ↓
识别 # 或 @
    ↓
搜索并选择对象
    ↓
插入结构化 Token
    ↓
保存结构化正文
    ↓
更新标签与引用关系
    ↓
本地保存并异步同步
```
