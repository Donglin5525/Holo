# Holo 观点编辑器改造方案

日期：2026-04-15
方向：A 方案，纯文本优先 + 智能辅助

## 1. 背景

当前「观点」模块的编辑器不好用，核心不是功能少，而是实现方式和使用场景错位了。

这个模块的真实目标更像“快速记录想法 / 观点”，但当前实现已经变成了一套自定义 markdown 富文本系统：编辑时用富文本，保存时再序列化回 markdown，还叠加了颜色、下划线、自定义标签解析等能力。结果是：

- 输入不自然，像在和编辑器机制对抗
- markdown 语法暴露过多，用户负担重
- 编辑态和展示态耦合，维护复杂
- 低频格式能力过多，高频写作体验反而不好

## 2. 现状审查结论

### 2.1 编辑器实现过重

当前 `MarkdownTextView` 采用的是：

- 编辑时维护 `UITextView + NSAttributedString`
- 保存时整段转回 markdown 字符串
- 再通过自定义 parser / renderer 渲染回展示态

这个结构的问题是：

- 光标、选区、输入法、样式状态容易互相影响
- 一次普通输入，背后其实在走“富文本状态机”
- 一旦加上列表、标签、引用，就更容易出现边界问题
- 为了适配这个结构，代码复杂度远超当前需求

### 2.2 markdown 语法设计不合理

现在不仅有常规 markdown，还引入了自定义语法：

- 下划线：`++text++`
- 颜色：`{color:#hex}text{/color}`

这两个问题很明显：

- 不符合常见 markdown 心智
- 用户几乎不会主动手写
- 与“快速记录观点”不匹配
- 会污染内容存储和后续渲染逻辑

### 2.3 工具栏能力偏离真实场景

当前工具栏提供：

- 加粗
- 斜体
- 下划线
- 颜色
- 有序/无序列表
- 标签

问题不是“按钮多”，而是能力优先级错了：

- 颜色、下划线属于低频能力
- 标签和引用才是观点模块真正有区分度的能力
- 手机端编辑首先要解决“顺手输入”，而不是“排版丰富”

### 2.4 保存与数据同步逻辑有风险

现有逻辑里有几个明显风险点：

- 取消时会自动保存，不符合常见编辑器预期
- 编辑已有观点时，引用关系没有完整同步更新
- 标签 `usageCount` 只增不减，长期会失真
- 搜索 / 筛选逻辑分散在 Repository 和 View 两层，行为不容易一致

## 3. 改造目标

把「观点」编辑器改成一个更像 Bear / iA Writer 的轻量文本编辑器：

- 默认像普通文本输入一样自然
- markdown 尽量隐藏，不让用户感到“在写语法”
- 只保留高频、真正有价值的增强能力
- 编辑器代码从“富文本系统”降回“文本输入 + 智能规则”
- 历史数据继续可读，不做破坏性迁移

## 4. 方案选择

已确认采用 A 方案，并进一步确认：

- markdown 对用户尽量隐藏
- 不走完整所见即所得
- 不做彻底结构化表单
- 不引入大范围富文本模型改造

也就是说，最终方向是：

“看起来像普通文本编辑器，用起来像成熟写作 App，但内部仍可兼容轻量 markdown 存储。”

## 5. 核心方案

### 5.1 编辑器交互重构

编辑器默认表现为纯文本输入区，不再做“边输边富文本渲染”。

保留的高频能力只有 4 类：

- 列表
- 加粗
- 标签
- 引用

交互规则：

- 列表按钮：给当前行加 `- ` 或 `1. `
- 回车时自动续写列表项
- 空列表项回车时自动退出列表
- 加粗按钮：对选中文本做包裹，不引入复杂样式状态
- 标签按钮：插入 `#` 并触发联想
- 引用按钮：打开独立引用选择器，不把引用卡片塞进正文

### 5.2 markdown 语法收缩

后续可编辑语法只保留：

- 段落
- 无序列表 / 有序列表
- `#标签`
- `**加粗**`

移除新写入路径中的：

- `++下划线++`
- `{color:...}` 颜色语法
- 其他低频自定义格式

策略不是“马上删掉历史能力”，而是：

- 详情页继续兼容旧语法显示
- 新编辑器不再暴露这些能力
- 老内容重新编辑后，自动归一化到新语法子集

### 5.3 标签与引用重新收口

标签：

- 保留正文内 `#标签`
- 保留显式标签区
- 保存时统一提取、去重、归一化

引用：

- 继续作为独立关系存储
- 不放进正文当占位符
- 编辑已有观点时必须同步新增和删除的引用关系

### 5.4 取消与保存逻辑修正

取消不再自动保存。

新的行为建议：

- 无改动：直接关闭
- 有改动：弹出操作确认
- 选项为：
  - 继续编辑
  - 放弃修改
  - 保存

这会明显减少“我只是点了取消，结果被自动保存”的违和感。

### 5.5 搜索与数据逻辑统一

需要把搜索、筛选、标签统计这些逻辑往 Repository 收：

- View 不再自己再过滤一层
- 标签使用次数要能增减平衡
- 引用、标签、正文的保存动作走统一入口
- 让列表、详情、搜索结果的行为保持一致

## 6. 代码层建议

优先改动的核心文件：

- `Views/Thoughts/ThoughtEditorView.swift`
- `Views/Thoughts/MarkdownTextView.swift`
- `Views/Thoughts/RichTextToolbarView.swift`
- `Utils/MarkdownParser.swift`
- `Utils/MarkdownRenderer.swift`
- `Data/Repositories/ThoughtRepository.swift`

建议的重构方向：

### 6.1 `ThoughtEditorView`

负责：

- 页面结构
- 草稿状态
- 保存 / 取消流程
- 标签 / 心情 / 引用入口
- 调用统一保存接口

不再负责：

- 自己拼接复杂存储逻辑
- 自己决定引用如何更新

### 6.2 `MarkdownTextView`

重做为：

- 轻量文本编辑控件
- 处理输入规则
- 处理列表自动续写
- 处理选区包裹
- 处理标签触发

不再负责：

- 整段富文本序列化
- 自定义属性状态系统
- 颜色 / 下划线等复杂格式

### 6.3 `RichTextToolbarView`

收缩为轻量工具条，只保留：

- 加粗
- 无序列表
- 有序列表
- 标签
- 引用

移除：

- 颜色
- 下划线

### 6.4 `MarkdownParser` / `MarkdownRenderer`

改为“兼容层”而不是“主系统”：

- 优先支持标准 markdown 子集
- 保留 `#标签` 高亮
- 兼容旧语法读取
- 不再继续扩展自定义 AST 能力

### 6.5 `ThoughtRepository`

新增或重构为统一保存入口，例如：

- 保存正文
- 同步标签
- 同步引用
- 更新时间
- 修正标签使用次数

## 7. 测试重点

### 7.1 输入体验

- 普通输入不卡顿、不跳光标
- 中文输入法过程稳定
- 列表自动续写正确
- 空列表项退出列表正确
- 选中文本加粗结果正确

### 7.2 数据正确性

- 新建观点时正文、标签、引用、心情全部正确保存
- 编辑观点时引用关系能增删同步
- 标签统计值不会越用越脏
- 取消编辑不会偷偷落库

### 7.3 兼容性

- 旧内容还能正常展示
- 含旧语法的历史内容重新编辑后不会崩
- 旧语法可被安全降级

### 7.4 搜索与预览

- 卡片摘要不被旧语法污染
- 搜索内容、标签、筛选结果一致
- 列表、详情、引用视图数据一致

## 8. 默认边界

本轮不做的内容：

- 不引入第三方富文本编辑器
- 不做完整 Notion 式所见即所得
- 不重做 Core Data 模型
- 不把图片插入正文流作为本轮重点
- 不把观点改成结构化表单模型

## 9. 结论

这次改造最重要的不是“修一个编辑器 bug”，而是把观点模块从“过度设计的富文本系统”拉回“真正适合手机记录想法的输入器”。

方向上最合适的是：

- 保留轻量文本存储
- markdown 尽量隐藏
- 收缩语法能力
- 强化列表、标签、引用这些真实高频场景
- 降低实现复杂度和后续维护成本

如果后续继续推进实现，建议先从编辑器输入模型和保存逻辑下手，再处理渲染兼容层，最后收口搜索、标签统计和引用同步。
