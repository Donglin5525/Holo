# Holo Plus 订阅与 AI 使用限额方案审查稿

**Goal:** 将 Holo 早期商业化收敛为「免费版 + Holo Plus 订阅」两档，并通过每日次数限制与后端 token 保险丝控制 AI 成本。

**Recommendation:** 第一版不做 Pro，不做买断。Plus 同时包含核心增强功能和 AI 能力，定价先用 `¥12/月`、`¥98/年` 的早鸟策略。

**Review Target:** 给 GLM 做产品、成本、合规和工程可行性审查。

---

## 1. 产品结论

Holo 当前阶段不建议做 `Free / Plus / Pro` 三档，也不建议做买断。

更适合的第一版商业化结构是：

| 档位 | 定位 | 用户心智 |
|---|---|---|
| 免费版 | 完整记录生活 | 我可以放心把生活数据记在 Holo 里 |
| Holo Plus | 更好用，也更懂我 | Holo 帮我省事，并提供 AI 分析与整理 |

这样做的原因：

1. Holo 还在早期，过早拆 Pro 会显得商业化复杂度高于产品成熟度。
2. AI 成本可以通过每日次数、月度 token 预算、单次输出上限控制，不必单独拆 Pro。
3. 买断与持续 AI 成本天然冲突，即使有每日次数，也会留下长期履约压力。
4. 两档方案更容易在 App 内解释，也更容易通过付费弹窗讲清楚价值。

## 2. 定价方案

### 2.1 首发价格

| SKU | 价格 | 角色 |
|---|---:|---|
| Holo Plus 月付 | `¥12/月` | 低门槛入口 |
| Holo Plus 年付 | `¥98/年` | 首发主推 |

建议定价页主推年付：

```text
Holo Plus
¥12/月
¥98/年，限时早鸟
```

### 2.2 后续价格预案

如果真实 AI 成本、留存和转化验证良好，可继续保持 `¥98/年`。

如果重度用户占比高，或后续接入更贵模型，可以将标准年付切到：

| SKU | 标准价 |
|---|---:|
| Holo Plus 月付 | `¥15/月` |
| Holo Plus 年付 | `¥128/年` |

### 2.3 不做买断的原因

不建议做 Plus 买断，核心原因不是 token 当下贵，而是商业承诺不稳定：

| 风险 | 说明 |
|---|---|
| 长期成本不确定 | 模型价格、用户使用强度、上下文长度都会变化 |
| 权益解释变复杂 | 如果买断不含 AI，用户会困惑；如果买断含 AI，成本不可控 |
| 早期产品仍会调整 | 现在买断会限制后续权益重组空间 |
| 审核和客服成本增加 | 需要解释买断、订阅、AI 次数、未来服务边界 |

结论：第一版只上订阅，不上买断。

## 3. 功能权益分层

### 3.1 免费版权益

免费版必须完整保留基础记录能力，不制造“记到一半要付费”的挫败感。

| 模块 | 免费版 |
|---|---|
| 财务 | 收入、支出、转账、分类、备注、基础汇总 |
| 任务 | 创建任务、截止日期、完成状态、基础列表 |
| 想法 | 记录想法、手动标签、搜索 |
| 习惯 | 创建习惯、打卡、连续天数、基础统计 |
| 健康 | HealthKit 接入、基础数据查看 |
| 记忆 | 基础时间线、基础回看 |
| AI | 少量每日体验 |

### 3.2 Plus 权益

Plus 同时提供“核心增强”和“AI 能力”。

| 模块 | Plus 权益 |
|---|---|
| 桌面小组件 | 财务、待办、习惯、健康、记忆、生活星图等组件 |
| 财务增强 | 财务分期、周期账单、订阅扣费提醒、预算进度 |
| 统计增强 | 高级趋势、跨月对比、年度视图、自定义看板 |
| 提醒增强 | 周期提醒、临近提醒、异常提醒 |
| 个性化 | 自定义首页模块、分类展示、工作台布局 |
| AI 聊天 | 更高每日次数 |
| AI 深度分析 | 财务、健康、任务、习惯、想法的跨模块分析 |
| AI 自动整理 | 想法聚类、标签建议、主题沉淀 |
| AI 周报/月报 | 个性化回顾，不只是静态报表 |
| 生活观察 | 每日一次 AI 观察，识别近期变化 |

## 4. AI 使用限额

### 4.1 用户可见限额

用户只需要理解“每天可以用多少次”，不要暴露 token 概念。

| AI 能力 | 免费版 | Plus |
|---|---:|---:|
| AI 聊天 / 问答 | `5 次/天` | `30 次/天` |
| AI 深度分析 | `1 次/周` | `1 次/天` |
| AI 自动整理想法 | `3 次/周` | `3 次/天` |
| AI 周报 | `1 次/月体验` | `1 次/周` |
| AI 月报 | 预览版 | `1 次/月` |
| 生活观察 | 不开放或本地规则版 | `1 次/天` |
| 重型 Agent 任务 | 不开放 | `5 次/月` |

### 4.2 内部隐藏限额

每日次数解决用户理解问题，内部还需要 token 保险丝解决成本问题。

| 限制 | 用户是否可见 | 目的 |
|---|---:|---|
| 每日次数限制 | 可见 | 明确权益边界 |
| 每分钟请求限制 | 不主动展示 | 防连点、防刷 |
| 单次 maxTokens | 不展示 | 防输出失控 |
| 月度 token 软上限 | 不展示 | 防极端长上下文 |
| 月度 token 硬上限 | 特殊场景提示 | 防成本穿透 |
| 失败重试上限 | 不展示 | 防异常重试放大成本 |

建议 Plus 内部预算：

```text
AI 聊天：30 次/天
深度分析：1 次/天
生活观察：1 次/天
重型 Agent：5 次/月
月度 token 软上限：2,000,000 tokens
月度 token 硬上限：3,000,000 tokens
```

软上限触发后：

1. 缩短上下文窗口。
2. 降低深度分析证据包大小。
3. 暂停或降级重型 Agent。
4. 优先复用缓存输入。

硬上限触发后：

1. 基础记录功能不受影响。
2. 普通 AI 聊天可降级为短上下文。
3. 深度分析、重型 Agent 延后到下月恢复。
4. 用温和文案告知用户。

## 5. Token 成本测算

### 5.1 价格依据

DeepSeek 官方 API 文档当前列出的 `deepseek-v4-flash` 价格为：

| 计费项 | 美元价格 | 约合人民币 |
|---|---:|---:|
| 输入 tokens，cache miss | `$0.14 / 1M tokens` | `约 ¥1 / 1M tokens` |
| 输入 tokens，cache hit | `$0.0028 / 1M tokens` | `约 ¥0.02 / 1M tokens` |
| 输出 tokens | `$0.28 / 1M tokens` | `约 ¥2 / 1M tokens` |

折算假设：`1 USD ≈ ¥7.2`。

来源：DeepSeek API Docs - Models & Pricing  
https://api-docs.deepseek.com/quick_start/pricing

> 注意：DeepSeek 官方文档提示产品价格可能调整，因此后端必须保留模型切换、限额调整和预算开关。

### 5.2 单次功能成本估算

| AI 功能 | 单次 token 估算 | 单次成本估算 |
|---|---:|---:|
| 普通 AI 聊天 | `1,500 input + 500 output` | `约 ¥0.0025` |
| 深度分析 | `6,800 input + 2,100 output` | `约 ¥0.011` |
| 生活观察 | `2,000 input + 500 output` | `约 ¥0.003` |
| AI 自动整理想法 | `1,500 input + 500 output` | `约 ¥0.0025` |
| 重型 Agent | `25,000 input + 8,000 output` | `约 ¥0.04` |

### 5.3 Plus 用户极限使用成本

如果 Plus 用户每天用满推荐额度：

| 能力 | 用量 | 月成本估算 |
|---|---:|---:|
| AI 聊天 | `30 次/天` | `约 ¥2.25/月` |
| 深度分析 | `1 次/天` | `约 ¥0.33/月` |
| 生活观察 | `1 次/天` | `约 ¥0.09/月` |
| AI 自动整理想法 | `3 次/天` | `约 ¥0.23/月` |
| 重型 Agent | `5 次/月` | `约 ¥0.20/月` |
| 合计 | - | `约 ¥3.10/月` |

加入失败重试、上下文变长、价格波动和非缓存输入，按 `2-3 倍安全系数`：

```text
Plus 重度用户月成本：约 ¥6-10/月
```

如果年付 `¥98/年`，Apple 抽成后实际收入可能约为 `¥68-83/年`，折合 `约 ¥5.7-6.9/月`。因此：

1. `¥98/年` 适合作为早鸟价。
2. 长期标准价建议预留到 `¥128/年`。
3. 必须依赖每日次数和月度 token 保险丝。

## 6. 付费触发与弹窗策略

### 6.1 推荐链路

```text
用户点击 Plus 功能
→ 展示 Holo 自己的价值说明弹窗
→ 用户选择月付或年付
→ 调起 Apple StoreKit 支付确认
→ 支付成功后回到原功能并立即解锁
```

不要直接从功能入口拉起系统支付窗口。用户应先理解“为什么值得付费”。

### 6.2 弹窗文案方向

Plus 弹窗不讲“你不能用”，而讲“升级后 Holo 更懂你”。

示例：

```text
让 Holo 更好用，也更懂你

升级 Holo Plus 后，你可以使用桌面小组件、财务分期、高级统计，
并解锁更多 AI 问答、深度分析、自动整理和生活观察能力。
```

按钮：

```text
开通 Plus 年付 ¥98
月付 ¥12
暂时不用
```

### 6.3 超限文案

免费用户：

```text
今天的免费 AI 次数已用完
升级 Holo Plus 后，每天可以使用更多 AI 问答、深度分析和自动整理能力。
```

Plus 用户：

```text
今天的 AI 深度分析次数已用完
你仍然可以继续记录和查看数据。Plus 的深度分析次数将在明天恢复。
```

月度硬上限特殊情况：

```text
本月高级 AI 使用量较高
为了保证服务稳定，部分深度分析能力将在下月恢复。基础记录和普通查看不受影响。
```

## 7. 支付与 App Store 审核注意事项

Holo Plus 属于 App 内数字功能解锁，应使用 Apple In-App Purchase / StoreKit。

Apple App Review Guidelines 3.1.1 说明，如果要在 App 内解锁功能或内容，例如订阅、高级内容、完整版解锁等，通常必须使用 in-app purchase。

来源：Apple App Review Guidelines  
https://developer.apple.com/app-store/review/guidelines/

审核准备建议：

1. App Store Connect 中配置 `Holo Plus Monthly` 和 `Holo Plus Yearly`。
2. App 内提供“恢复购买”入口。
3. 审核说明中写清 Plus 解锁内容、AI 次数限制、免费版可用范围。
4. 付费页不要暗示 Apple 之外的付款方式。
5. 价格、权益、次数限制必须与 App Store 元数据一致。

## 8. 工程落地建议

### 8.1 客户端需要的概念

```text
SubscriptionTier:
- free
- plus

AIQuotaType:
- chat
- deepAnalysis
- thoughtOrganization
- weeklyReport
- monthlyReport
- dailyObserver
- agentTask
```

客户端只负责：

1. 展示当前会员权益。
2. 在入口处判断是否需要展示 Plus 弹窗。
3. 展示剩余次数。
4. 处理 StoreKit 购买和恢复购买。
5. 将订阅状态传给后端或由后端校验 receipt 后同步。

### 8.2 后端需要的概念

```text
usage_limits:
- user_id
- period_type: daily / weekly / monthly
- quota_type
- used_count
- input_tokens
- output_tokens
- reset_at

entitlements:
- user_id
- tier
- subscription_source
- expires_at
- last_verified_at
```

后端必须做最终限制，不能只依赖客户端。

后端限制顺序：

```text
校验订阅状态
→ 校验 quota_type 次数
→ 校验 per-minute 限流
→ 校验月度 token 软/硬上限
→ 执行模型调用
→ 记录 input/output tokens
→ 返回剩余次数或下次恢复时间
```

### 8.3 当前 Holo 后端现状参考

当前 `HoloBackend/src/config.js` 已有部分基础限制和 route 输出预算：

| 当前配置 | 默认值 |
|---|---:|
| `HOLO_CHAT_REQUESTS_PER_DAY` | `50` |
| `HOLO_CHAT_MAX_TOKENS` | `1024` |
| `HOLO_INTENT_MAX_TOKENS` | `4096` |
| `HOLO_INSIGHT_MAX_TOKENS` | `2048` |
| `HOLO_HEALTH_INSIGHT_MAX_TOKENS` | `1600` |
| `HOLO_MEMORY_OBSERVER_MAX_TOKENS` | `2048` |
| `HOLO_AGENT_LOOP_MAX_TOKENS` | `8192` |

后续商业化落地时，需要把“全局 chatRequestsPerDay”升级为“按用户、按会员、按能力类型”的 quota。

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 价格过低导致重度用户亏损 | 年付收入无法覆盖 AI 成本 | 早鸟价限时；保留 `¥128/年` 标准价；启用 token 硬上限 |
| 用户不理解为什么 AI 有次数 | 付费体验受损 | 明确写成“每日 AI 次数”，不要讲 token |
| 频繁弹窗打扰 | 降低信任感 | 只在用户触发 Plus 功能或超限时弹，不做无关打断 |
| AI 失败重试放大成本 | 后端账单异常 | 限制自动重试次数，失败后降级为短上下文 |
| 订阅状态被客户端伪造 | 权益绕过 | 后端校验 receipt 或服务端维护 entitlement |
| App Review 认为 IAP 不清晰 | 审核延迟 | 审核说明写清 Plus 权益、次数限制、免费版可用范围 |
| 用户达到硬上限后不满 | 体验受损 | 硬上限设置足够高，只作为极端保险丝；基础功能不受影响 |

## 10. 上线阶段

### Phase 1: 最小商业化闭环

上线：

1. Plus 月付、年付。
2. StoreKit 购买、恢复购买。
3. Plus 付费弹窗。
4. 桌面小组件、财务分期、高级统计权益控制。
5. AI 聊天每日次数限制。

暂不上：

1. 买断。
2. Pro。
3. 复杂积分包。
4. 用户可购买额外 AI 次数。

### Phase 2: AI 能力扩展

上线：

1. AI 深度分析每日 1 次。
2. AI 自动整理想法每日 3 次。
3. AI 周报/月报。
4. 生活观察每日 1 次。
5. usage dashboard 内部监控。

### Phase 3: 成本校准

观察 2-4 周：

1. Plus 转化率。
2. 年付比例。
3. 人均 daily AI 次数。
4. P90 / P99 token 消耗。
5. 失败率和重试率。
6. 单 Plus 用户月均成本。

再决定：

1. 是否从 `¥98/年` 切到 `¥128/年`。
2. 是否调整每日次数。
3. 是否开放更多 Agent 能力。

## 11. GLM 审查问题

请重点审查以下问题：

1. `免费 + Plus` 两档是否足够支撑 Holo 第一版商业化？
2. 不做买断的理由是否充分？
3. `¥12/月`、`¥98/年` 是否与 AI 成本、Apple 抽成和早期转化目标匹配？
4. Plus 的 AI 每日限额是否太宽或太紧？
5. 月度 token 软上限 `2,000,000`、硬上限 `3,000,000` 是否合理？
6. 是否存在 Apple 审核风险或权益表达不清的问题？
7. 后端按用户、按 quota type 计数是否还有遗漏场景？
8. 免费用户的 AI 体验次数是否足够引导转化？
9. 重型 Agent 是否应第一版暂不上？
10. 这套方案是否会让用户觉得“Plus 还是不够用”？

## 12. 当前建议定稿

第一版建议定为：

```text
免费版：
完整记录生活。

Holo Plus：
¥12/月，¥98/年。
包含桌面小组件、财务分期、高级统计、AI 聊天、AI 深度分析、AI 自动整理、AI 周/月报和生活观察。

限制：
AI 聊天 30 次/天。
深度分析 1 次/天。
生活观察 1 次/天。
重型 Agent 5 次/月。

不做：
买断。
Pro。
积分包。
```

这套方案的核心取舍是：牺牲一部分高阶商业化精细度，换取更清晰的用户心智和更低的第一版实现复杂度。
