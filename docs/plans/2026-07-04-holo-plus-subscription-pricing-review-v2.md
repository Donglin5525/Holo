## 1. 产品结论

**两档结构（免费版 + Holo Plus）正确，第一版不做 Pro、不做买断。**

第一性原理上，真正的理由不是 v1 说的"商业化复杂度高于产品成熟度"，而是：

> **Holo 当前 0 真实付费用户、未上架，任何档位/定价/限额假设都无法被证伪。因此应上最少档位、最简限额，把工程精力放在"能收集真实成本与转化数据"的最小闭环上，而不是构建精细的多档计费体系。**

这条理由比 v1 的理由更根本，也直接约束了后续所有数字的性质——**v2 里所有定价、次数、token 上限都是"待 Phase 3 真实数据校准的临时值"，不是定稿**。

| 档位 | 定位 | 用户心智 |
|---|---|---|
| 免费版 | 完整记录生活 | 我可以放心把生活数据记在 Holo 里 |
| Holo Plus | 更好用，也更懂我 | Holo 帮我省事，并提供 AI 分析与整理 |

## 2. 定价方案

### 2.1 首发价格

| SKU | 价格 | 角色 |
|---|---:|---|
| Holo Plus 月付 | `¥12/月` | 低门槛入口 |
| Holo Plus 年付 | `¥98/年`（限时/限量早鸟） | 首发主推，仅用于冷启动 |
| Holo Plus 年付（标准） | `¥128/年` | 标准价，早鸟结束后默认展示 |

早鸟建议写死边界，避免把"后续切价"当作止损机制：

```text
¥98/年早鸟价：首发 14 天或前 500 名，以先到者为准。
早鸟结束后恢复 ¥128/年标准价。
```

### 2.2 毛利分析（基于 §5 全链路成本）

第三轮修订：毛利表拆为 Phase 1 和 Phase 2。因为重型 Agent 已从 Phase 1 移到 Phase 2，不能把 Phase 2 成本提前压到首发 Plus 毛利上。

#### Phase 1：不含重型 Agent

| 项目 | 数值 |
|---|---:|
| 年付早鸟 ¥98 × 70%（Apple 抽 30%） | `¥68.6/年 ≈ ¥5.7/月` |
| 重度 Plus 用户月成本（1x 实际） | `约 ¥5.3/月`（见 §5.3，剔除重型 Agent） |
| 重度 Plus 用户月成本（2x 压力测试） | `约 ¥10.6/月` |
| **重度用户月毛利（1x）** | **约 +¥0.4** |
| **重度用户月毛利（2x）** | **约 -¥4.9** |
| 轻度 Plus 用户月成本（1x 实际） | `约 ¥2-3/月` |
| **轻度用户月毛利** | **约 +¥3/月** |

**结论**：Phase 1 下 `¥98/年` 不是立即必亏，但毛利很薄；它适合作为限时/限量早鸟，不适合作为长期标准价。

#### Phase 2：含重型 Agent

| 项目 | 数值 |
|---|---:|
| 年付早鸟 ¥98 × 70%（Apple 抽 30%） | `¥68.6/年 ≈ ¥5.7/月` |
| 重度 Plus 用户月成本（1x 实际，含 Agent） | `约 ¥5.8/月`（见 §5.3） |
| 重度 Plus 用户月成本（2x 压力测试） | `约 ¥11.6/月` |
| **重度用户月毛利（1x）** | **约 -¥0.1** |
| **重度用户月毛利（2x）** | **约 -¥5.9** |

**结论**：Phase 2 若开放重型 Agent，`¥98/年` 基本不适合作为长期价。标准价应回到 `¥128/年`，并保留 Agent 月度次数限制。

### 2.3 价格切换与早鸟结束条件（修正 R6 / T4）

| 信号 | 动作 |
|---|---|
| 首发 14 天结束或早鸟名额用完 | 新用户默认展示 `¥128/年` |
| Phase 3 观察 2-4 周，Plus 用户中**重度占比 >30%** | 不延长早鸟；评估是否下调每日 AI 次数 |
| Plus 用户中重度占比 <15%，且转化率不达标 | 可短期延长早鸟，但必须限时 |
| 真实单 Plus 用户月成本 > ¥8（1x，非安全系数） | 立即停止早鸟，检查 Agent / ASR / 隐式调用限额 |

> 第三轮修正：后续切 `¥128` 只能影响新用户，无法修复已售年付 cohort 的亏损。因此 `¥98` 必须一开始就定义为限时/限量早鸟，而不是等成本异常后再切。

### 2.4 不做买断（保留 v1）

| 风险 | 说明 |
|---|---|
| 长期成本不确定 | 模型价格、ASR 价格、用户使用强度都会变化 |
| 权益解释变复杂 | 买断不含 AI 用户困惑；含 AI 成本不可控 |
| 早期产品仍会调整 | 买断限制后续权益重组空间 |
| 审核和客服成本增加 | 需解释买断/订阅/AI 次数/未来服务边界 |

## 4. AI 使用限额（重大修订 R2 / R3）

### 4.1 用户可见限额

| AI 能力 | 免费版 | Plus |
|---|---:|---:|
| AI 聊天 / 问答 | `3 次/天`（R2：v1 为 5） | `30 次/天` |
| 自然语言/语音记账解析 | `20 次/天`（R3） | `50 次/天`（L1：100→50，撞墙成本可控） |
| 自然语言/语音任务解析 | `10 次/天`（R3） | `30 次/天`（L1：50→30） |
| 单次 ASR 时长上限 | `60 秒/次`（L2，硬截断） | `60 秒/次` |
| AI 深度分析 | `1 次/周` | `1 次/天` |
| AI 自动整理想法 | `3 次/周` | `3 次/天` |
| AI 周报 | `1 次/月体验` | `1 次/周` |
| AI 月报 | 预览版 | `1 次/月` |
| 生活观察 | 不开放或本地规则版 | `1 次/天` |

> **R2 理由**：免费额度要让大多数活跃用户在 1-2 周内撞墙，才能触发付费。5 次/天 × 30 = 150 次/月，工具型 App 普通用户日均 1-3 次足够，多数人永远撞不到墙。

### 4.2 隐式 LLM 调用限额（R3 新增，关键）

**问题**：v1 把"AI 次数"等同于"用户可见的 AI 功能次数"。但 Holo 的核心交互（自然语言记账、语音记想法）本身就依赖 LLM 解析（`finance_action_parser` / `task_action_parser`）。免费用户即使从不用 AI 聊天，只要用语音/自然语言记账，每次都触发 LLM + 可能的 ASR 调用——**且这部分成本完全不在 v1 的"5 次/天"限额覆盖范围内**。

**对策**：
1. 免费用户的 `action_parser` 类调用单独设每日上限（见 §4.1）。
2. 超限后，免费用户仍可记账，但**降级为表单手动输入**（不调 LLM），文案：「免费自然语言记账已达今日上限，可手动选择分类，或升级 Plus」。
3. Plus 用户给高额度，不写"不限"。否则 Plus 会和"每日次数控制成本"的商业化原则冲突。
4. 后端必须对 `finance_action_parser` / `task_action_parser` 这类 purpose 也走 `usageStore.consume`，不能只限 `chat`。
5. **intent 调用不单独消耗用户可见 quota**（L3）：intent 作为内部分流调用，其成本已计入主调用（聊天/分析）的单次成本，避免一次对话扣两次配额，导致 30 次/天实际只够 15 次对话。
6. **单次 ASR 时长硬截断 60 秒**（L2）：防止长音频按秒计费穿透成本（10 分钟音频 = `¥0.186/次`）；总时长暂不独立限额，由"语音聊天次数 + 语音记账次数"间接约束，Phase 3 视数据决定是否加。

### 4.3 内部隐藏限额（简化 R5）

| 限制 | 用户是否可见 | Phase |
|---|---:|---|
| 每日次数限制 | 可见 | Phase 1 |
| 每分钟请求限制 | 不展示 | Phase 1 |
| 单次 maxTokens | 不展示 | Phase 1（已有，强化） |
| 失败重试上限 | 不展示 | Phase 1 |
| 月度 token 软/硬上限 + 4 级降级 | 特殊场景提示 | **Phase 3**（R5：从 Phase 1 推迟） |

> **R5 理由**：每日次数已能锁住约 90% 成本（30 次/天 × 单次成本 = 单用户月成本硬顶）。软上限的 4 级降级（缩上下文/降证据包/停 Agent）会让"同一功能不同时间表现不一样"，客服难解释——这是为"数万付费用户 + 专人盯 P99"阶段设计的系统，不是 0 用户阶段。Phase 3 真实数据出来后再决定是否需要。

## 5. 全链路成本测算（R1 全新）

### 5.1 价格依据（2026 已核验）

**DeepSeek V4 Flash**（官方价，R9 已核验）：

| 计费项 | 美元价 | 约合人民币 |
|---|---:|---:|
| 输入 tokens，cache miss | `$0.14 / 1M` | `¥1.008 / 1M` |
| 输入 tokens，cache hit | `$0.0028 / 1M` | `¥0.020 / 1M` |
| 输出 tokens | `$0.28 / 1M` | `¥2.016 / 1M` |

来源：https://api-docs.deepseek.com/quick_start/pricing

**DashScope ASR**（qwen3-asr-flash-realtime，阿里云国内）：

| 计费项 | 价格 |
|---|---:|
| 实时语音识别 | `约 ¥1.12/小时 ≈ ¥0.00031/秒` |

来源：阿里云百炼模型价格 https://help.aliyun.com/zh/model-studio/model-pricing

**等效单价假设**：
- system prompt 复用，估算 input 的 30% 命中 cache。等效 input 单价 = `0.7×¥1 + 0.3×¥0.02 ≈ ¥0.7/1M`。
- 取整用于估算：input `¥0.7/1M`、output `¥2/1M`、ASR `¥0.00031/秒`。

保守测算口径：

| 口径 | input 单价 | 适用场景 |
|---|---:|---|
| 乐观 / 默认 | `¥0.7/1M` | 假设 system prompt 有 30% cache hit |
| 保守 | `¥1.0/1M` | 假设 cache hit 不稳定或上下文变化大 |

后续 Phase 3 必须用真实 `usage.input_tokens`、`usage.output_tokens` 和 provider cache 命中数据回填本节。

### 5.2 单次功能全链路成本

> 关键修订（R1）：v1 按"单次 LLM 调用"估，v2 按"用户完成一次表达的全链路"估。Holo 一次交互背后通常 = ASR（语音时）+ intent（分流）+ 主调用，可能 + action_parser（记账时）。

| 功能 | 调用链 | input tok | output tok | ASR | **单次成本** |
|---|---|---:|---:|---:|---:|
| 文本聊天 | intent + chat | 2,300 | 700 | - | `¥0.003` |
| 语音聊天（10s） | ASR + intent + chat | 2,300 | 700 | ¥0.003 | `¥0.006` |
| 深度分析 | intent + insight | 7,600 | 2,300 | - | `¥0.010` |
| 生活观察 | memory_observer | 2,000 | 500 | - | `¥0.002` |
| 自动整理想法 | thought_org + convergence | 3,000 | 1,000 | - | `¥0.004` |
| 重型 Agent（单步） | agent_loop ×1 | 25,000 | 8,000 | - | `¥0.034` |
| 重型 Agent（3 步循环） | agent_loop ×3 | 75,000 | 24,000 | - | `¥0.103` |
| 文本记账（隐式） | finance_action_parser | 300 | 100 | - | `¥0.0004` |
| 语音记账 10s（隐式） | ASR + finance_action_parser | 300 | 100 | ¥0.003 | `¥0.003` |

> 注：intent 是否每次对话必走取决于客户端编排（`HoloBackendAIProvider`）。本表按"每次必走"保守估算；若 intent 仅复杂查询触发，聊天/分析成本再降 15-20%。落地时以客户端埋点实测为准。

### 5.3 Plus 重度用户月成本（全链路）

假设 Plus 重度用户：聊天 50% 语音、记账 50% 语音、Agent 平均 3 步。

| 能力 | 月用量 | 月成本 |
|---|---:|---:|
| AI 聊天（50% 语音） | `30 次/天 × 30 = 900 次` | `¥4.05` |
| AI 深度分析 | `1 次/天 × 30` | `¥0.30` |
| AI 生活观察 | `1 次/天 × 30` | `¥0.06` |
| AI 自动整理 | `3 次/天 × 30 = 90 次` | `¥0.36` |
| 重型 Agent（3 步） | `5 次/月` | `¥0.52` |
| 隐式记账解析（50% 语音） | `10 次/天 × 30 = 300 次` | `¥0.51` |
| **Phase 1 合计（不含重型 Agent，1x 实际）** | - | **`约 ¥5.3/月`** |
| **Phase 1 合计（不含重型 Agent，2x 压力测试）** | - | **`约 ¥10.6/月`** |
| **Phase 2 合计（含重型 Agent，1x 实际）** | - | **`约 ¥5.8/月`** |
| **Phase 2 合计（含重型 Agent，2x 压力测试）** | - | **`约 ¥11.6/月`** |

**2x 安全系数吸收项**（R4 明确）：maxTokens 顶满（chat `1024` vs 估 500、agent_loop `8192` vs 估 8000 均接近上限）、失败重试、长上下文爬升、价格波动。

> 对比 v1 的 `¥6-10/月`：v1 偏低约 40%，主要漏了 ASR（语音聊天 + 语音记账）和 intent 调用。

**撞墙成本上限**（L5）：上表按"预期用量"估。若用户撞满所有限额（L1 额度修订后），Phase 2 1x 成本约 `¥9.4/月`、2x 约 `¥18.7/月`——是预期用量的 1.6 倍。其中 ASR 是最大变量：上表假设平均 10 秒/次语音，若用户大量用满 60 秒上限（L2），ASR 部分可 ×6，撞墙成本可达 `¥30+/月`。这是 `¥98/年` 只能做限时早鸟、且 ASR 必须有单次时长上限的根本原因。

### 5.4 免费用户月成本（关键盲区）

| 能力 | 月用量 | 月成本 |
|---|---:|---:|
| AI 聊天（3 次/天，50% 语音） | `90 次` | `¥0.41` |
| 隐式记账解析（8 次/天，50% 语音） | `240 次` | `¥0.41` |
| **合计（1x 实际）** | - | **`约 ¥0.8-1.1/月`** |

**关键洞察**：免费用户隐式记账解析成本（¥0.41）占总成本约 **37%**，而 v1 限额完全不覆盖。1000 免费用户 = 月成本 `¥800-1100`，且 **0 收入**。若被脚本滥用批量语音记账，成本穿透。

> 这正是 R3 的依据：**限额必须覆盖 action_parser 这类隐式调用，否则免费侧成本失控**。

**撞墙成本上限**（L5）：上表按"预期用量 8 次/天"估。若免费用户撞满 `20 次/天` 记账限额（50% 语音），仅记账月成本就 `约 ¥0.9`；叠加聊天撞墙 `¥0.41`，免费用户撞墙上限约 `¥1.3/月`。1000 免费用户撞墙 = 月成本 `¥1300`，且 0 收入。这是免费版必须给自然语言记账设独立硬限额、且 ASR 加单次时长上限的依据。

## 7. Apple 审核准备（补 R8）

Holo Plus 属于 App 内数字功能解锁，必须用 StoreKit IAP（App Review Guidelines 3.1.1）。

## 8. 工程落地（补 Phase 0，R7）

### 8.1 当前后端现状（关键落差）

当前 `HoloBackend/src/app.js` 默认使用 `createSqliteUsageStore(database.db)`，不是纯内存限流；`inMemoryUsageStore` 主要是可替换实现 / 测试用。第三轮修正后的关键落差是：已有 SQLite 次数限流，但尚未升级为商业化会员计费系统。

| 维度 | 现状 | 商业化需要 |
|---|---|---|
| 计费主体 | `deviceId`（匿名） | `user_id` 或 receipt 绑定 |
| 存储 | 默认 `SQLite rate_limits`，按 key/count/expires_at 计数 | 可查询、可审计、可按订阅周期聚合的 quota ledger |
| 计量维度 | 仅 `used_count`（次数） | + input/output token 累计 |
| purpose 覆盖 | `/v1/ai/chat/completions` 下所有 purpose 会走 `usageStore.consume`；`asr` 也单独限流 | 需要把 purpose 映射成商业 quota type，例如 chat / naturalLanguageInput / deepAnalysis |
| 会员权益 | 无 entitlement | 需要 StoreKit receipt 校验与 free / plus 配置 |
| 返回信息 | 只返回 429，不返回业务化剩余次数 | 客户端需要 remaining / resetAt 以展示友好超限提示 |
| 每日重置时区 | `toISOStringDay()` 按 UTC 切日，中国用户早 8 点才恢复次数（L4） | 按 `Asia/Shanghai`（UTC+8）切日 |
| 软上限降级 | 0 行 | Phase 3 再上（R5） |

> R7 第三轮修正：不是"持久化从 0 待建"，而是"现有 SQLite 次数限流需要升级为会员 entitlement + quota ledger"。这比 v2 原表述更准确，也会降低 Phase 0 的实际工程量。

### 8.2 客户端概念（保留 v1）

```text
SubscriptionTier: free | plus

AIQuotaType:
- chat
- naturalLanguageInput    // R3 新增：覆盖 action_parser
- deepAnalysis
- thoughtOrganization
- weeklyReport
- monthlyReport
- dailyObserver
```

客户端只负责：展示权益、入口判断 Plus 弹窗、展示剩余次数、处理 StoreKit 购买与恢复、把订阅状态同步后端。

### 8.3 后端概念（修订：purpose 覆盖扩大）

```text
usage_limits:
- user_id (或 receipt-bound device_id)
- period_type: daily / weekly / monthly
- quota_type          // 含 naturalLanguageInput（R3）
- used_count
- input_tokens        // Phase 3 启用
- output_tokens       // Phase 3 启用
- reset_at

entitlements:
- user_id
- tier
- subscription_source
- expires_at
- last_verified_at
```

后端限制顺序（Phase 1 简化版，无 token 上限）：

```text
校验订阅状态（entitlements）
→ 校验 quota_type 次数（含 naturalLanguageInput）
→ 校验 per-minute 限流
→ 执行模型调用
→ 记录 used_count
→ 返回剩余次数或下次恢复时间
```

**关键约定**：
- intent 调用走 `usageStore.consume` 内部计数，但**不消耗用户可见 quota**（L3），避免一次对话扣两次配额。
- ASR 单次时长硬截断 60 秒（L2），在 `/v1/asr/transcriptions` 入口校验，超限返回 413。
- daily reset 按 `Asia/Shanghai` 时区切片（L4），不用 UTC。

## 10. 上线阶段（补 Phase 0）

### Phase 0: 计费后端前置工程（R7 新增）

上线：

1. 基于现有 `createSqliteUsageStore` 扩展 quota ledger，而不是新建一套平行计数系统。
2. 建立 `purpose -> quota_type` 映射：`chat`、`naturalLanguageInput`、`deepAnalysis`、`thoughtOrganization`、`dailyObserver`。
3. 将 `finance_action_parser` / `task_action_parser` 归入 `naturalLanguageInput`（R3）。
4. receipt → device_id / user_id 绑定机制（或匿名 device_id 作为 Phase 0 过渡）。
5. 客户端订阅状态 → 后端 entitlement 同步。
6. 限额响应返回 `remaining`、`resetAt`、`quotaType`，供客户端展示友好超限文案。
7. intent 调用走内部计数，不消耗用户可见 quota（L3）。
8. `/v1/asr/transcriptions` 加单次时长硬截断 60 秒（L2）。
9. daily reset 切日从 UTC 改为 `Asia/Shanghai`（L4）。

完成标志：重启后端后，免费用户的限额计数不丢失；`action_parser` 调用受 `naturalLanguageInput` 限额拦截；客户端能显示剩余次数和恢复时间；intent 不双重扣额；ASR 单次超 60 秒被拒；免费用户在北京时间 0 点（非早 8 点）恢复次数。

### Phase 1: 最小商业化闭环

上线：

1. Plus 月付、年付（StoreKit）。
2. 恢复购买。
3. Plus 付费弹窗（含 AI 局限披露）。
4. 桌面小组件、财务分期、高级统计权益控制。
5. AI 聊天每日次数（免费 3、Plus 30）。
6. 自然语言记账每日次数（免费 20、Plus 50，R3 / T2 / L1）。
7. 自然语言任务解析每日次数（免费 10、Plus 30，R3 / T2 / L1）。
8. 单次 ASR 时长硬截断 60 秒（L2）。
9. Family Sharing 关闭（商业配置项，不作为审核硬要求）。

暂不上：买断、Pro、积分包、额外 AI 次数包、**重型 Agent**（R4）。

### Phase 2: AI 能力扩展

上线：

1. AI 深度分析每日 1 次。
2. AI 自动整理想法每日 3 次。
3. AI 周报/月报。
4. 生活观察每日 1 次。
5. **重型 Agent**（R4：从 Phase 1 移来，含 5 次/月限额）。
6. usage dashboard 内部监控。

### Phase 3: 成本校准

观察 2-4 周：

1. Plus 转化率。
2. 年付比例。
3. 人均 daily AI 次数 + **隐式调用次数**（R3 新增埋点）。
4. P90 / P99 token 消耗。
5. 失败率和重试率。
6. 单 Plus 用户月均成本（实测对照 §5.3）。
7. **重度用户占比**（R6 切价依据）。

再决定：

1. 是否结束或延长 `¥98/年` 早鸟（按 §2.3 的限时/限量规则）。
2. 是否调整每日次数。
3. 是否启用月度 token 软/硬上限（R5）。
4. 是否开放更多 Agent 能力。
