## 📝 核心开发原则

### 0️⃣ 全局中文化（必读）

**原则**：Holo 是一款面向中文用户的 APP，**所有用户可见的文本内容必须使用简体中文**。

**强制要求**：
- ✅ **界面文本**：所有按钮、标签、标题、占位符、提示语、错误信息必须为简体中文
- ✅ **日期格式**：使用 `zh_CN` locale，例如「2026年3月7日 星期六」
- ✅ **货币格式**：默认人民币（¥），`NumberFormatter` 使用 `currencyCode: "CNY"` + `locale: zh_CN`
- ✅ **代码注释**：使用中文注释，符合项目注释规范
- ❌ **禁止英文 UI 文本**：不得在界面中出现 "Expense"、"Income"、"Today's Ledger"、"View all" 等英文文案
- ❌ **禁止混合语言**：同一界面中不得中英混用（品牌名 "Holo" 除外）

### 0.1️⃣ iOS 页面布局规范（含底部固定栏）

> 来源于财务页（FinanceView）布局优化，适用于「顶部 Header + 中间可滚动内容 + 底部固定 Tab/按钮」的页面。

| 区域 | 规范 |
|------|------|
| **顶部 Header** | 贴近安全区，`.padding(.top, 12)`，避免 40pt+ 大留白 |
| **底部固定栏** | 用 `safeAreaInset(edge: .bottom)` 固定贴底；Tab 栏内用 `GeometryReader` 取 `safeAreaInsets.bottom` 做底部 padding，背景可 `.ignoresSafeArea(edges: .bottom)` |
| **浮动主按钮** | 放在底部栏内、Tab 栏正上方，`HStack { Spacer(); Button; Spacer() }` 水平居中 |
| **中间内容区** | `VStack { header; 固定区; ScrollView { 列表 }.frame(maxHeight: .infinity) }`，列表占满剩余高度并单独滚动 |

### 0.2️⃣ 财务页 Tab 栏与底部栏约定（必读）

| 项目 | 约定 |
|------|------|
| **Tab 吸底** | fixed bottom-0 left-0 w-full，无圆角；`safeAreaInset(edge: .bottom)`；pb = `max(safeAreaInsets.bottom, 20)`，pt-2；背景 `.ignoresSafeArea(edges: .bottom)` |
| **三等分** | 统计 \| 中间 \| 设置，每格 `.frame(maxWidth: .infinity)` |
| **中间与加号合一** | 在账本页：中间显示红色 +、「记一笔」，点击弹出记一笔 Sheet；在统计/设置页：中间显示账本图标、「账本」，点击切回账本。不再使用悬浮 FAB |
| **顶部投影** | Tab 栏顶部弱投影 `shadow(..., y: -2)` |
| **Header** | `.padding(.top, 0)`，紧贴状态栏下方 |
| **交易行** | 分类与金额 justify-between，金额右侧严格对齐 |

### 0.3️⃣ 记账模块业务规则（必读）

> 记账是财务核心功能，以下规则为**基本业务原则**，所有金额相关的 UI 和计算逻辑必须遵守。

#### 金额显示规则

| 规则 | 说明 | 示例 |
|------|------|------|
| **不显示正负号** | 无论是收入还是支出，前端显示的金额**不带正负号**。类型通过颜色、图标或标签区分 | ✅ `¥100.00` ❌ `-¥100.00` `+¥100.00` |
| **保留两位小数** | 全局金额统一保留到「分」，使用 `NumberFormatter.currency` 格式化 | ✅ `¥10.50` ✅ `¥100.00` ❌ `¥10.5` ❌ `¥100` |
| **数据库存正数** | 数据库中金额字段始终存储**正数**，交易类型通过 `type` 字段区分收支 | `amount: 100.00, type: "expense"` |

```swift
// ✅ 正确：使用 formattedAmount（不带符号）
Text(transaction.formattedAmount)

// ❌ 错误：使用 formattedAmountWithSign（带符号）
Text(transaction.formattedAmountWithSign)

// NumberFormatter 配置（已定义在 NumberFormatter+Currency.swift）
formatter.minimumFractionDigits = 2
formatter.maximumFractionDigits = 2
```

#### 计算器键盘规则

| 规则 | 说明 |
|------|------|
| **支持四则运算** | 键盘必须支持 `+` `-` `×` `÷` 四则运算，满足日常记账需求（如分摊账单） |
| **浮点数计算** | 除法必须使用**浮点数运算**，禁止整数除法。`23÷32=0.72`，不是 `0` |
| **运算优先级** | 遵循数学运算优先级：先乘除后加减。`10+5×2=20`，不是 `30` |
| **键盘切换** | 备注输入时隐藏数字键盘（系统键盘弹出）；点击金额区域显示数字键盘 |

```
键盘布局（5行4列）：
÷   ×   -   +
7   8   9   ⌫
4   5   6   AC
1   2   3   ↩︎（预留）
.   0   00  ✓
```

#### 金额输入规则

| 规则 | 说明 |
|------|------|
| **小数位数限制** | 每个数字段最多 2 位小数 |
| **运算符替换** | 连续输入运算符时，后者替换前者 |
| **表达式计算** | 点击 `=` 或 `✓` 时计算表达式结果 |
| **结果格式化** | 计算结果保留 2 位小数，去除尾部多余的 0 |

### 1️⃣ 小步快跑，频繁提交

**原则**：每次完成一个功能点都要主动 commit 并推送到 GitHub

**为什么重要**：
- ✅ **代码安全**：避免意外丢失工作成果
- ✅ **版本可追溯**：清晰记录每个功能的开发历史
- ✅ **易于回滚**：出现问题可快速恢复到稳定版本
- ✅ **团队协作**：及时同步进展，便于代码审查
- ✅ **心理优势**：每次提交都是正向反馈

## 🔄 标准开发流程

```
开始功能开发
    ↓
编写代码 + 本地测试
    ↓
✅ 验证功能正常
    ↓
git add .                    # 暂存更改
    ↓
git commit -m "feat: xxx"    # 提交并写清楚提交信息
    ↓
git push                     # 推送到远程仓库
    ↓
完成一个功能点 ✓
```

## 💡 提交时机检查清单

在以下情况**必须提交**：

- [ ] 完成一个新功能模块
- [ ] 修复一个 bug
- [ ] 重构一段代码（不改变功能）
- [ ] 添加/更新文档
- [ ] 准备切换分支做其他任务
- [ ] 每天下班前（即使功能未完成）
- [ ] 运行测试通过后

**提交频率建议**：
- ✅ 至少每天 1 次提交
- ✅ 每个功能点 1-3 次提交
- ❌ 避免累积超过 1 天的工作才提交

## 📋 Git 提交信息规范

### 提交格式

```
<type>: <简短描述>

<详细描述（可选）>

- 改动点 1
- 改动点 2

Closes #123  # 关联 Issue（可选）
```

### 提交类型（type）

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 实现首页 UI` |
| `fix` | Bug 修复 | `fix: 修复日程提醒位置错位` |
| `refactor` | 代码重构（不影响功能） | `refactor: 优化设计系统结构` |
| `docs` | 文档更新 | `docs: 添加开发规范` |
| `style` | 代码格式（不影响功能） | `style: 格式化代码` |
| `test` | 添加/更新测试 | `test: 添加首页组件测试` |
| `chore` | 构建/工具/配置变更 | `chore: 更新 Xcode 配置` |
| `perf` | 性能优化 | `perf: 优化启动速度` |

### 好的提交信息示例

```bash
✅ feat: 实现首页 UI - 基于 Figma 设计稿

新增功能:
- 首页视图 (HomeView) - 完整实现 Figma 设计稿布局
- 语音助手按钮 (VoiceAssistantButton) - 渐变背景 + 发光效果
- 功能入口按钮 (FeatureButton) - 任务/财务/健康/观点
- 底部导航栏 (BottomNavBar) - 个人/AI/记忆长廊

技术栈:
- SwiftUI + iOS 17.0
- 所有组件支持 #Preview

已验证:
- 模拟器运行正常
- 无编译错误和 Linter 警告
```

```bash
✅ fix: 修复日程提醒位置错位

问题：日程提醒显示在屏幕顶部，位置过于靠上
解决：调整 offset 值从 -60 改为 -20，并简化视图结构
影响：日程提醒现在位于 Header 下方、语音按钮上方的正确位置
```

### 不好的提交信息示例

```bash
❌ commit -m "更新代码"
❌ commit -m "fix bug"
❌ commit -m "asdfasdf"
❌ commit -m ""  # 空提交信息
```

## 🚀 Git 常用命令速查

### 日常开发

```bash
# 查看当前状态
git status

# 查看变更内容
git diff

# 暂存所有更改
git add .

# 暂存特定文件
git add <文件名>

# 提交并写信息
git commit -m "feat: 描述"

# 推送到远程
git push

# 拉取最新代码
git pull
```

### 查看历史

```bash
# 查看提交历史（简洁）
git log --oneline

# 查看提交历史（详细）
git log

# 查看某次提交的详情
git show <commit-hash>

# 查看文件历史
git log -- <文件名>
```

### 分支管理

```bash
# 查看分支
git branch

# 创建新分支
git checkout -b <分支名>

# 切换分支
git checkout <分支名>

# 合并分支
git merge <分支名>

# 删除分支
git branch -d <分支名>
```

### 撤销操作

```bash
# 撤销工作区修改
git checkout -- <文件名>

# 撤销暂存区修改
git reset HEAD <文件名>

# 撤销上一次提交（保留更改）
git reset --soft HEAD~1

# 撤销上一次提交（不保留更改）
git reset --hard HEAD~1
```

## 🔒 安全注意事项

### 禁止提交的内容

- ❌ `.env` 文件（包含环境变量和密钥）
- ❌ `credentials.json`、`config.json`（包含密码）
- ❌ API Key、Secret Key
- ❌ 数据库连接字符串
- ❌ 个人身份信息
- ❌ `.DS_Store`（macOS 系统文件）
- ❌ `build/`、`DerivedData/`（编译产物）

### 使用 .gitignore

确保项目根目录有 `.gitignore` 文件，包含：

```gitignore
# macOS
.DS_Store

# Xcode
DerivedData/
*.xcuserstate
*.xcscmblueprint

# 构建产物
build/
*.app

# 密钥和配置
.env
*.pem
*.key
credentials.json
```

## 🎯 最佳实践

### ✅ 推荐做法

1. **功能分支开发**
   ```bash
   git checkout -b feature/home-page
   # 开发...
   git commit -m "feat: 实现首页布局"
   git push origin feature/home-page
   ```

2. **小提交优于大提交**
   - 一个提交只做一件事
   - 提交越小，越容易审查和回滚

3. **提交前自检**
   ```bash
   git status          # 确认要提交的文件
   git diff --staged   # 预览提交内容
   ```

4. **使用图形化工具**
   - GitKraken、SourceTree、Fork
   - Xcode 内置的 Source Control

### ❌ 避免做法

1. **累积多天的工作才提交**
2. **提交信息写不清楚**
3. **本地开发很久不推送**
4. **提交包含敏感信息**
5. **提交编译产物和大文件**
