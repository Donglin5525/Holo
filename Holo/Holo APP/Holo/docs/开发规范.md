## 📝 核心开发原则

### 0️⃣ 全局中文化（必读）

**原则**：Holo 是一款面向中文用户的 APP，**所有用户可见的文本内容必须使用简体中文**。

**强制要求**：
- ✅ **界面文本**：所有按钮、标签、标题、占位符、提示语、错误信息必须为简体中文
- ✅ **日期格式**：使用 `zh_CN` locale，例如「2026年3月7日 星期六」
- ✅ **货币格式**：默认人民币（¥），`NumberFormatter` 使用 `currencyCode: "CNY"` + `locale: zh_CN`
- ✅ **代码注释**：使用中文注释，符合项目注释规范
- ❌ **禁止英文 UI 文本**：不得在界面中出现 "Expense"、"Income"、"Today's Ledger"、"View all" 等英文文案
- ❌ **禁止混合语言**：同一界面中不得中英混用（品牌名 "Holo" 除外）

**检查清单**（每次新增页面/组件时必须确认）：
- [ ] 所有 `Text(...)` 内容为中文
- [ ] 所有 `TextField(placeholder:)` 占位符为中文
- [ ] 所有 `Button` 标签为中文
- [ ] 日期/时间使用 `Locale(identifier: "zh_CN")`
- [ ] 货币金额使用 `NumberFormatter.currency`（已配置为人民币）
- [ ] `NavigationTitle` / `alert` / `confirmationDialog` 均为中文

### 0.1️⃣ iOS 页面布局规范（含底部固定栏）

> 来源于财务页（FinanceView）布局优化，适用于「顶部 Header + 中间可滚动内容 + 底部固定 Tab/按钮」的页面。

**顶部 Header**
- Header 与返回键应**贴近顶部安全区**，避免上方留白过大。
- 推荐：`.padding(.top, 12)`，不再使用 40pt 以上的大数值（如 56）。

**底部固定栏（Tab 栏 / 浮动按钮）**
- 使用 **`safeAreaInset(edge: .bottom, spacing: 0)`** 将整块底部栏固定在屏幕最底部（等效于 fixed bottom-0）。
- 主内容无需手动计算底部留白，系统会自动避开 inset 区域。
- Tab 栏需**适配 iOS 底部安全区**（Home Indicator）：用 `GeometryReader` 读取 `geo.safeAreaInsets.bottom`，在栏内底部使用 `16 + geo.safeAreaInsets.bottom` 等 padding；背景可用 `.ignoresSafeArea(edges: .bottom)` 延伸至屏幕最底。

**浮动主按钮（如记账加号）**
- 放在 `safeAreaInset` 的底部栏内，位于 Tab 栏**正上方**、**水平居中**（`HStack { Spacer(); Button; Spacer() }`），与对应 Tab 图标垂直对齐。

**中间内容区**
- 采用**上固定、下滚动**：`VStack { header; 固定卡片; ScrollView { 列表 }.frame(maxHeight: .infinity) }`。
- 中间列表区域用 `ScrollView` + `.frame(maxHeight: .infinity)` **占满剩余高度**并单独滚动，保证内容可自适应且不顶到固定栏。

### 0.2️⃣ 财务页 Tab 栏与底部栏约定（必读）

> 以下为财务模块（FinanceView）已确定的 UI 约定，后续开发与迭代须遵守。

**Tab 栏吸底**
- 底部导航使用 **fixed bottom-0 left-0 w-full**，不设圆角、不设额外 bottom 外边距，背景贴满屏幕最底部。
- 使用 `safeAreaInset(edge: .bottom, spacing: 0)` 实现固定贴底；栏内 **pb = env(safe-area-inset-bottom)**，可用 `max(geo.safeAreaInsets.bottom, 20)` 做 fallback；**pt-2**（约 8pt）作为栏内顶部留白。
- 背景使用 `.ignoresSafeArea(edges: .bottom)` 填满至物理底部，图标与文字保持在安全区上方。

**Tab 三等分与中间「账本/记一笔」合一**
- 底部三个 Tab 严格**三等分**：统计 | 中间 | 设置，每个 `.frame(maxWidth: .infinity)`。
- **中间一格**与加号合一：
  - **在记账页（账本）时**：中间显示**红色圆形 + 号**，文案「记一笔」，点击弹出记一笔 Sheet。
  - **在统计或设置页时**：中间显示**账本图标**，文案「账本」，点击切回账本页。
- 不再使用悬浮 FAB overlay，记一笔入口仅通过中间 Tab 进入。

**视觉层次**
- Tab 栏增加**顶部弱投影**：`shadow(color: .black.opacity(0.05), radius: 10, x: 0, y: -2)`，与上方列表区分边界。

**顶部 Header**
- Header 容器**无多余 pt/mt**，使用 `.padding(.top, 0)`，标题与返回键紧贴状态栏（灵动岛）下方，由系统安全区控制。

**交易记录列表行**
- 每行使用 **flex 等价布局**：分类名称与金额 **justify-between items-center**，金额在右侧严格对齐（`Spacer(minLength: 0)` + 金额 `fixedSize` + `frame(alignment: .trailing)`）。

### 1️⃣ 小步快跑，频繁提交

**原则**：每次完成一个功能点都要主动 commit 并推送到 GitHub

**为什么重要**：
- ✅ **代码安全**：避免意外丢失工作成果
- ✅ **版本可追溯**：清晰记录每个功能的开发历史
- ✅ **易于回滚**：出现问题可快速恢复到稳定版本
- ✅ **团队协作**：及时同步进展，便于代码审查
- ✅ **心理优势**：每次提交都是正向反馈

```

<详细描述（可选）>
```

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 实现首页 UI` |
| `fix` | Bug 修复 | `fix: 修复日程提醒位置错位` |

## 🧩 iOS / Core Data 常见坑总结（写代码前必读）

> 本节来源于首个记账功能的真实 debug 过程，属于**高频踩坑集合**。  
> 每次写 iOS + Core Data 相关代码前，请先快速过一遍这一页。

### 1️⃣ Core Data 模型与代码定义不一致

- **必须保证三处完全对齐**：  
  - **数据模型**：`NSManagedObjectModel` / `.xcdatamodeld` 中的字段名、类型、是否可空  
  - **实体类**：`Transaction.swift` / `Category.swift` / `Account.swift` 里的 `@NSManaged` 属性  
  - **实际赋值代码**：`FinanceRepository` 中对属性的赋值逻辑  
- **典型错误 1：属性类型不匹配**
  - 模型：`amount` 使用 `decimalAttributeType`  
  - 代码：`@NSManaged public var amount: Decimal` / `Double`  
  - 结果：运行时出现 `unrecognized selector setAmount:` / 崩溃  
  - **规范**：  
    - 只要是 Core Data 的 `decimalAttributeType`，**实体类必须用 `NSDecimalNumber`**，所有计算再转为 `Decimal`。
- **典型错误 2：关系未在模型中声明**
  - 代码里使用 `transaction.category` / `transaction.account`  
  - 模型里只有 `categoryId` / `accountId` 这样的 UUID 字段，没有 `category` / `account` 的 `NSRelationshipDescription`  
  - 结果：运行时 `-[Transaction setCategory:]: unrecognized selector sent to instance`  
  - **规范**：  
    - 只要实体类里有 `@NSManaged public var xxx: SomeEntity`，模型中就必须有同名 relationship（`name` 一定要一致）。
- **典型错误 3：多余的必填字段不赋值**
  - 模型里保留了必填的 `categoryId` / `accountId` 字段，但代码只给 `category` / `account` 关系赋值，没有赋 ID 字段  
  - 结果：保存时报 `Multiple validation errors occurred.`  
  - **规范**：  
    - 要么删掉这类冗余字段，只保留关系；  
    - 要么在保存前**同时**给 ID 字段和关系字段赋值，并保证非可选字段一定有值。

### 2️⃣ Core Data 并发与上下文使用

- **禁止在错误线程访问 `NSManagedObject`**
  - `EXC_BAD_ACCESS (code=1, address=0xbeadde81d9e0)` 往往意味着在**非创建线程**访问了某个托管对象  
  - 典型场景：在 `backgroundContext.perform {}` 中创建/获取的对象，传到主线程的 SwiftUI 视图里直接用  
- **记账 MVP 的约定**：
  - 所有仓库操作（`FinanceRepository`）标记为 `@MainActor`  
  - 只用 **`viewContext`** 做增删改查，避免初期业务中过早引入复杂并发  
  - 如需后台写入，必须遵守：  
    - 后台上下文只负责持久化；  
    - UI 层只使用主线程 `viewContext` 重新通过 `objectID` fetch 出来的对象。

### 3️⃣ Core Data 报错信息快速排查指南

- **`unrecognized selector setXxx:`**
  - 优先检查：  
    - 模型里是否有 `xxx` 这个字段 / 关系？  
    - 实体类里是否用对了类型（尤其是 `NSDecimalNumber`）？  
- **`Multiple validation errors occurred.`**
  - 这是保存时的**字段校验失败汇总**，常见原因：  
    - 非可选字段未赋值（`isOptional = false` 但代码里没设值）；  
    - 关系最小数量 `minCount = 1` 却为 nil；  
  - 排查步骤：  
    - 在 `save()` 前打断点，检查所有非可选属性/关系是否有值；  
    - 对照 `createDataModel()` / `.xcdatamodeld` 和实体类，找出多余或遗漏的字段。

### 4️⃣ SwiftUI 列表不刷新的典型问题

- **症状**：保存成功后，列表/「今天」页面看不到新数据，必须返回再进入一次。  
- **根因**：数据源只在 `.task` 首次执行时加载一次，**保存后没有重新触发加载**。  
- **当前项目中的标准做法**（记账列表 `FinanceView`）：  
  - `FinanceView` 内部维护 `@State private var transactions: [Transaction]`  
  - 首次进入时，在 `.task` 中 `await loadTransactions()`  
  - 从 `AddTransactionView` 保存返回时，通过  
    ```swift
    .onChange(of: showAddTransaction) { _, isShowing in
        if !isShowing {
            Task { await loadTransactions() }
        }
    }
    ```  
    在 sheet 关闭的那一刻重新加载数据。  
- **通用规范**：  
  - 任何「弹出编辑/新增页面」的场景，**关闭时必须触发一次数据刷新**（网络请求或本地 fetch），不能只依赖初次加载。

### 5️⃣ iOS Asset Catalog 与 SVG 分类图标（为何多次才成功）

> 本节来源于「记一笔 · 选择分类」71 个 Figma 图标接入的完整踩坑过程。  
> 接入自定义 SVG 图标到 Xcode Asset Catalog 时，请按本节的**最终成功做法**执行，可少走弯路。

#### 问题回顾：为什么搞了几次才成功？

| 现象 | 可能原因 | 最终解决方式 |
|------|----------|--------------|
| **图标完全不显示**（只有底色和文字） | 1）Asset 未打进包：项目里存在**两个** `Assets.xcassets`（根目录 Holo/ 与 Holo APP/Holo/Holo/），运行 target 可能用的是没有 CategoryIcons 的那一个<br>2）资源名不对：用了 `Image("CategoryIcons/icon_xxx")` 但 Catalog 未开命名空间，或反过来 | 1）把 CategoryIcons 也复制到**根目录**的 `Holo/Assets.xcassets/`，保证当前运行的 target 用的那份里有图标<br>2）代码里**先试** `UIImage(named: "CategoryIcons/\(name)")` **再试** `UIImage(named: name)`，任一击中即用该名加载；找不到再回退到 SF Symbol |
| **Xcode 里看不到 CategoryIcons 文件夹** | 在左侧**项目导航**里找「CategoryIcons」→ 它不在顶层，而在 **Assets.xcassets 内部** | 单击 **Assets.xcassets**，在**资源库左侧列表**里找 CategoryIcons（与 AppIcon、AccentColor 同级） |
| **图标在 Xcode 预览里是灰块，真机也不显示** | SVG 里用了 `fill="var(--fill-0, #HEX)"`，iOS 在**模板图（Template）**渲染下不识别该 CSS 变量，导致不绘制 | 把所有 CategoryIcons 下 SVG 的 `fill="var(--fill-0, #HEX)"` **统一替换为** `fill="#000000"`，再用 `.renderingMode(.template)` + `.foregroundColor(...)` 着色 |
| **Type '()' cannot conform to 'View'** | 在 `@ViewBuilder` 里写了 `print(...)` 等返回 `Void` 的语句，Swift 要求该闭包内每句都产出 `View` | 不在 ViewBuilder 里写 `print`；调试用断点或单独写非 View 的辅助方法 |
| **图标太小** | 初始尺寸 22×22 在 48 圆里偏小 | 按设计需求放大（如 2 倍：44×44 图标 + 64×64 圆） |

#### 最终成功做法（必读）

1. **SVG 内容**
   - 用于**模板着色**的图标：SVG 内 path 的 fill 必须为**纯色**，推荐 `fill="#000000"`。  
   - **不要**在 SVG 里写 `fill="var(--fill-0, #HEX)"`，iOS Asset Catalog 的模板渲染不支持该变量，会导致不显示。

2. **Asset Catalog 结构**
   - 子文件夹（如 CategoryIcons）的 `Contents.json` 可**不设** `provides-namespace`，用短名 `icon_xxx` 更稳。  
   - 若设了命名空间，代码里需同时尝试 `"CategoryIcons/icon_xxx"` 与 `"icon_xxx"`（见下）。

3. **代码加载方式**
   - 用 `UIImage(named:)` 做**运行时检测**，兼容两种命名：
     - 先 `UIImage(named: "CategoryIcons/\(category.icon)")`
     - 再 `UIImage(named: category.icon)`
   - 若都不为 nil，用 `Image(uiImage: img).renderingMode(.template)` 并设置 `foregroundColor`。  
   - 若为 nil，回退到 `Image(systemName: "tag.fill")` 等 SF Symbol，避免空白。

4. **多 target / 多 Assets 时**
   - 确认当前 **Run 的 target** 使用的是哪一个 `Assets.xcassets`（Build Phases → Copy Bundle Resources）。  
   - 若项目中有多个 xcassets（例如根目录一份、子工程一份），**每个会被打进的 xcassets 里都要有 CategoryIcons**，或统一只用一份并确保 target 引用正确。

5. **图标尺寸建议（记一笔选择分类）**
   - 图标：**44×44 pt**（在 2 倍屏上清晰可辨）。  
   - 圆形背景：**64×64 pt**，与图标比例协调。

### 6️⃣ SwiftUI + Core Data 视图卡死/白屏问题（必读）

> 本节来源于习惯功能（HabitsView）开发中的反复 debug，属于**高频致命坑**。

#### 问题 6.1：在 body 中执行 Core Data 查询

- **现象**：打开页面**白屏/卡死**，无响应
- **原因**：在 `var body: some View` 计算属性中直接调用 Core Data 查询方法
- **典型错误代码**：
  ```swift
  var body: some View {
      VStack {
          // ❌ 在 body 中直接调用 Core Data 查询
          let streak = repository.calculateStreak(for: habit)
          Text("连续 \(streak) 天")
      }
  }
  ```
- **规范做法**：
  ```swift
  @State private var streak: Int = 0
  
  var body: some View {
      VStack {
          Text("连续 \(streak) 天")  // ✅ 只读取 @State 值
      }
      .onAppear { loadData() }
      .onReceive(NotificationCenter.default.publisher(for: .dataDidChange)) { _ in
          loadData()
      }
  }
  
  private func loadData() {
      Task { @MainActor in
          streak = HabitRepository.shared.calculateStreak(for: habit)
      }
  }
  ```

#### 问题 6.2：`@StateObject` 包装 `@MainActor` 单例

- **现象**：白屏、线程断言、`EXC_BREAKPOINT`
- **原因**：多个视图用 `@StateObject private var repo = SomeMainActorSingleton.shared` 包装同一个单例，`@StateObject` 会尝试管理对象生命周期，与单例模式冲突
- **典型错误代码**：
  ```swift
  // ❌ 多个视图都这样写，会冲突
  @StateObject private var repository = HabitRepository.shared
  ```
- **规范做法**：
  ```swift
  // ✅ 方案1：需要监听 @Published 变化时，用 @ObservedObject
  @ObservedObject private var repository = HabitRepository.shared
  
  // ✅ 方案2：不需要监听时，直接调用，不包装
  private func loadData() {
      Task { @MainActor in
          let data = HabitRepository.shared.activeHabits
      }
  }
  ```

#### 问题 6.3：直接访问 `@MainActor` 单例（未用 Task 包装）

- **现象**：`EXC_BREAKPOINT`、线程隔离错误
- **原因**：Swift Concurrency 的 `@MainActor` 要求所有访问必须在主线程隔离域内
- **规范做法**：
  ```swift
  // ✅ 所有对 @MainActor 单例的访问都用 Task 包装
  private func loadData() {
      Task { @MainActor in
          let repo = HabitRepository.shared
          habits = repo.activeHabits
          progress = repo.getTodayProgress()
      }
  }
  ```

### 7️⃣ NSManagedObject 删除后访问崩溃（必读）

> 本节来源于习惯删除功能的反复闪退 debug，是 Core Data + SwiftUI 结合时的**经典陷阱**。

#### 问题 7.1：视图持有已删除的 NSManagedObject

- **现象**：删除后闪退，`EXC_BREAKPOINT`，堆栈指向 `UUID._unconditionallyBridgeFromObjectiveC`
- **原因**：`NSManagedObject` 被 Core Data 删除后，SwiftUI 视图仍持有该引用并尝试访问其属性
- **典型崩溃堆栈**：
  ```
  HabitCardView.loadStatus()
  -> repository.isTodayCompleted(for: habit)
  -> habit.id 访问
  -> EXC_BREAKPOINT
  ```

#### 问题 7.2：`.sheet(item:)` 绑定 NSManagedObject

- **现象**：删除时崩溃
- **原因**：`.sheet(item: $selectedHabit)` 直接绑定 `Habit`（NSManagedObject），删除后 SwiftUI 仍尝试访问
- **典型错误代码**：
  ```swift
  // ❌ 危险：selectedHabit 是 NSManagedObject
  @State private var selectedHabit: Habit? = nil
  .sheet(item: $selectedHabit) { habit in
      HabitDetailView(habit: habit)
  }
  ```

#### 问题 7.3：NotificationCenter 触发时访问已删除对象

- **现象**：删除成功但 app 崩溃
- **原因**：`notifyDataChange()` 发送通知 → 所有视图的 `onReceive` 触发 → 被删除对象的视图仍尝试访问

#### 解决方案汇总（NSManagedObject 删除的标准流程）

```swift
// ========== 父视图（列表页）==========

// ✅ 1. 使用 ID 而非对象引用
@State private var selectedHabitId: UUID? = nil
@State private var showDetailSheet: Bool = false
@State private var pendingDeleteId: UUID? = nil  // 待删除的 ID
@State private var habits: [Habit] = []          // 本地缓存

// ✅ 2. sheet 用 isPresented 而非 item
.sheet(isPresented: $showDetailSheet, onDismiss: {
    // ✅ 3. onDismiss 中执行删除
    if let id = pendingDeleteId {
        // ✅ 4. 先从本地数组移除，避免 ForEach 访问已删除对象
        habits.removeAll { $0.id == id }
        
        // ✅ 5. 延迟执行 Core Data 删除，确保 UI 更新完成
        DispatchQueue.main.asyncAfter(deadline: .now() + 0.1) {
            Task { @MainActor in
                try? HabitRepository.shared.deleteHabitById(id)
            }
        }
        pendingDeleteId = nil
    }
}) {
    // ✅ 6. 从本地数组查找对象，而非直接绑定
    if let id = selectedHabitId,
       let habit = habits.first(where: { $0.id == id }) {
        HabitDetailView(habit: habit, onWillDelete: { deleteId in
            pendingDeleteId = deleteId
            showDetailSheet = false
        })
    }
}

// ========== 子视图（详情页）==========

// ✅ 7. 访问前检查对象有效性
private func loadData() {
    guard !habit.isDeleted, habit.managedObjectContext != nil else { return }
    
    Task { @MainActor in
        guard !habit.isDeleted, habit.managedObjectContext != nil else { return }
        // 执行数据加载...
    }
}

// ✅ 8. 使用 isDeleted/managedObjectContext 时必须 import CoreData
import CoreData
```

#### 删除流程时序图

```
用户点击删除
    ↓
DetailView: onWillDelete(habitId)  // 通知父视图
    ↓
ListView: pendingDeleteId = id, showDetailSheet = false  // 设置待删除 ID，关闭 sheet
    ↓
sheet onDismiss 触发
    ↓
ListView: habits.removeAll { $0.id == id }  // 先从本地数组移除
    ↓
DispatchQueue.main.asyncAfter(0.1s)  // 延迟确保 UI 更新
    ↓
Repository.deleteHabitById(id)  // 执行 Core Data 删除
    ↓
notifyDataChange()  // 发送通知
    ↓
各视图 onReceive 触发，但 guard 检查跳过已删除对象
    ↓
✅ 安全完成
```

### 8️⃣ Git / 仓库结构相关坑

- **禁止嵌套多个 Git 仓库**  
  - 之前 iOS 工程目录 `Holo/Holo APP/Holo` 内部也有一个 `.git`，导致：  
    - 根仓库把它当作子模块/gitlink；  
    - 外层提交看不到内层实际文件变更，管理混乱。  
  - 当前规范：  
    - **根目录只允许一个 `.git`**；  
    - iOS 项目、前端、文档都作为普通子目录纳入统一仓库管理。
